# Figma 2 PNG Doc UI Case Generate

按以下顺序执行：

**Step 1：盘点输入 -> Step 2：先分析需求文档并让用户确认 -> Step 3：识别大白块并导出 PNG 后确认 UI 场景 -> Step 4：整合资料、补齐交叉场景与矩阵并统一确认测试设计 -> Step 5：强制生成测试用例初稿 -> Step 6：reviewer 复核并输出终稿**

目标不是复述页面，而是先拆清 `场景`、`数值`、`展示`、`报错`、`状态`、`交叉组合`，先形成可 review 的初稿，再由 reviewer 复核终稿。

## Step 1：盘点输入

围绕用户提供的文档、Figma 链接和已有截图做轻量盘点：

- 展示会参与分析的 `.docx`、`.pdf`、`.md` 文件。
- 标出 Figma URL、`fileKey`、`node-id`。
- 标出用户是否已经提供部分 PNG；若有，记录图片与场景名的对应关系。
- 记录用户额外约束：
  - 是否限制 Figma MCP 读取次数。
  - 是否要求先按 UI 场景拆解。
  - 是否要求 TestRail 风格输出。
  - 是否要求由你先判断该选哪个 layer 导出。
- 明确哪些是原始需求文档，哪些只是补充说明。

这一阶段只盘点，不急着深读 Figma，也不急着直接写 testcase。

## Step 2：先分析需求文档并让用户确认

### 2.1 读取文档

- `.md` 直接读取。
- `.docx` / `.pdf` 使用当前环境可行的本地读取方式。
- 如果是 SmartX 文档，优先读取并遵循 `../smartx-doc-2-product-and-keywords/SKILL.md`，先提炼产品、功能摘要和关键词。

### 2.2 抽取测试设计前置信息

进入 Figma 前，先完成这轮结构化分析：

1. 功能目标
2. 涉及页面或入口
3. 关键对象
4. 关键数值口径
5. 关键展示字段
6. 关键校验或限制
7. 已知边界和异常
8. 当前不确定点
9. 依据来源

特别优先提炼这些信息：

- 哪些 UI 入口会受影响。
- 哪些数字需要显示、计算、比较、扣减、归还、聚合或重算。
- 哪些文案属于提示、说明、错误、状态标签、tooltip、footnote。
- 哪些状态会切换，如正常、不足、过期、禁用、隐藏、半透明。
- 哪些规则是对象维度，哪些规则是集群维度，哪些规则是全局汇总维度。

### 2.3 先做交叉场景识别

在读文档阶段就显式枚举交叉维度，不要等看到界面后才补想：

1. 入口交叉
   - 同一规则是否在多个入口触发。
2. 对象交叉
   - 单对象、同集群多对象、多集群多对象。
3. 许可或状态交叉
   - 方式、类型、额度、版本、过期时间、启用态、禁用态、已消耗、未消耗。
4. 操作链路交叉
   - 创建后扩容、扩容后删除、首次默认后更新、切换后再次操作。
5. 边界交叉
   - `X>Y`、`X=Y`、`X<Y`、`新额度=当前已使用量`、`新额度<当前已使用量`。

如果文档里出现“切换”“转换”“首次默认”“不额外消耗”“最后一个才归还”“全局汇总”等信号，默认这是高优先级交叉场景，后续必须进矩阵。

### 2.4 需要澄清时的规则

- 如果用户提供了多个文档且看起来不是同一 feature，停止推进并指出冲突来源。
- 如果无法唯一确定测试主体页面、弹窗、卡片或表单，先列出 2-4 个候选入口让用户确认，再继续读 Figma。
- 如果某条规则会直接影响 testcase 正确性，但文档无法唯一判断，显式记为 blocker。

### 2.5 向用户展示并确认分析结果

继续前，显式展示：

1. 功能摘要
2. 主要入口
3. 关键对象
4. 关键数值口径
5. 关键展示 / 报错类型
6. 交叉维度清单
7. 当前不确定点

这一轮必须等待用户确认或修正。

## Step 3：识别大白块并导出 PNG 后确认 UI 场景

### 3.1 默认策略

优先少读、分块读，不要一上来围绕整页白板或整张长画布做整体分析。

默认顺序：

1. 先用一次 `get_metadata` 看页面结构、section 和关键 frame。
2. 从根节点或目标 section 下识别候选“大白块”。
3. 按候选块逐个导出 PNG。
4. 逐张加载 PNG 做 UI 场景分析。
5. 只有在 PNG 看不清或关键文案无法确认时，再补最少量的 Figma 精读。

### 3.2 “大白块”识别规则

优先把下面这些对象当作导出单元：

- 独立白底大 frame，内部包含完整页面、弹窗、表单或卡片。
- 独立业务场景块，内部已经包含标题、主体区、操作按钮或关键信息。
- 长白板中的多个并列白块，每个白块单独导出，而不是整板一起导出。

不要把下面这些对象当作独立导出单元：

- 纯文本标题标签。
- 便利贴、批注气泡、连线说明。
- 单个按钮、单个输入框、单个 icon。
- 只用于说明选中态的辅助元素。

### 3.3 选 layer 与导出规则

导出时优先选择该场景最外层的 `Frame` 或 `Section`，不要选内部零散图层。

具体规则：

1. 优先选中最外层白底容器。
2. 一个大容器里并排放了多个独立白块时，改选每个子 `Frame`。
3. 用户只有查看权限时，先判断当前链接能否通过选 layer 导出；若不能导出，明确说明这是权限限制。
4. 用户已手动导出 PNG 时，直接把这些 PNG 当作 Step 3 输入，不重复要求导出。

### 3.4 Figma MCP 调用预算

如果用户限制读取次数，把预算当作硬约束。

推荐预算分配：

1. `1 次`：根节点或目标 section 的 `get_metadata`
2. `N 次`：每个大白块一次 `get_screenshot`
3. `0-2 次`：仅在 PNG 无法辨认关键文案、数值或错误提示时补充 `get_design_context`

预算不足时优先导出：

- 会改变测试结论的块
- 主流程块
- 错误态块
- 状态差异明显的块
- 文案差异块

### 3.5 基于 PNG 的 UI 分析目标

围绕文档结论，对每张 PNG 完成以下分析：

1. UI 场景名称
2. 场景入口
3. 关键控件或信息块
4. 数值展示位置
5. 数值比较规则
6. 正常态展示
7. 异常态展示
8. 状态态展示
9. 报错位置与提示方式
10. 与通用组件相比，哪些内容变了，哪些必须保持一致

优先显式记录：

- 卡片 / 弹窗 / 表单 / 表格 / tooltip / footnote / footer error
- 提交前校验 vs 提交后反馈
- 单对象报错 vs 多对象联合报错
- 正常显示 vs 置灰 / 半透明 / 隐藏 / 禁用 / 过期
- 已用 / 可用 / 总额度 / 单位 / 上限 / 汇总

### 3.6 向用户展示并确认 UI 场景结果

继续前，显式展示：

1. 已导出的场景块清单
2. 每张 PNG 对应的 UI 场景
3. 每个场景的关键对象
4. 每个场景的关键数值
5. 每个场景的关键展示
6. 每个场景的关键报错 / 状态
7. 哪些结论来自 PNG 直接确认
8. 哪些结论只基于文档或 metadata 推断
9. 剩余未澄清点

这一轮必须等待用户确认或修正。

## Step 4：整合资料、补齐交叉场景与矩阵并统一确认测试设计

### 4.1 构建测试对象表

在正式生成 testcase 前，先把“需要被验证的独立对象”列出来。

推荐格式：

| 测试对象 | 来源 | 关键变量 | 关键结果 |
|----------|------|----------|----------|
| 对象名 | 文档 / PNG | 数值 / 状态 / 入口 | 展示 / 报错 / 生效 |

至少覆盖：

- 文档里定义的业务对象
- PNG 中实际出现的卡片、弹窗、区域、报错组件
- 会影响用例拆分的状态对象
- 会跨入口复用同一规则的对象

### 4.2 强制构建矩阵

生成 testcase 前，必须显式输出矩阵，而不是只写一串 case 标题。

最少要构建这些矩阵：

1. `entry_matrix`
2. `state_matrix`
3. `number_matrix`
4. `copy_matrix`
5. `display_matrix`
6. `transition_matrix`
7. `switch_matrix`

### 4.3 交叉场景判定规则

遇到以下任一信号，必须补交叉场景，不得只测单步：

1. 文档出现“首次默认”“后续更新”“支持互相转换”
2. 文档出现“不额外消耗”“最后一个才归还”
3. 文档出现“按集群”“按关联集群”“所有对象汇总”
4. 文档出现“扩容”“缩容”“删除”“导入”
5. 文档同时给出“默认值”“门槛值”“不允许小于”
6. 文档既有类型变化，又有方式变化或状态变化

对每条规则至少判断一次：

- 是否有单对象 vs 多对象差异
- 是否有同集群 vs 多集群差异
- 是否有首次操作 vs 二次操作差异
- 是否有切换前 vs 切换后差异
- 是否有展示层和校验层不一致风险

### 4.4 向用户展示并确认测试设计

进入初稿生成前，同时展示：

1. 测试对象表
2. 建议测试主题分组
3. `entry_matrix`
4. `state_matrix`
5. `number_matrix`
6. `copy_matrix`
7. `display_matrix`
8. `transition_matrix`
9. `switch_matrix`
10. 必须显式展开的边界点
11. 可以抽样覆盖的维度
12. 当前仍存在的疑点

这一轮必须等待用户确认后再生成初稿。

## Step 5：强制生成测试用例初稿

### 5.1 强制先出初稿，不直接出终稿

先生成 `testcase_draft.md`，不要直接把第一版当最终答案。

要求：

- 初稿必须显式吸收 Step 4 已确认的矩阵结论。
- 初稿必须覆盖主流程、异常、边界、状态变化和交叉组合。
- 初稿允许还不够完美，但不能跳过。

### 5.2 初稿结构要求

优先使用 TestRail 风格结构，参考 `references/template.md`：

- `## 模块`
- `### 子模块`
- `描述：`
- `#### 用例`
- `描述：`
- `【Step1】`
- `【Step2】`
- `【Step3】`

一条 case 只验证一个核心结果。

## Step 6：reviewer 复核并输出终稿

### 6.1 reviewer 复核输入

在向用户输出前，必须把 `testcase_draft.md` 作为 review 对象，重新对照：

- 原始需求文档
- 已确认的文档分析结果
- 已确认的 PNG 场景分析结果
- 已确认的测试对象表和矩阵
- 如有必要补读的关键 Figma 节点
- `references/template.md`

### 6.2 reviewer 复核重点

1. 是否遗漏文档里明确存在的入口、对象、条件、状态。
2. 是否遗漏 PNG 里明确变化的数值、文案、展示位置、错误方式。
3. 是否把数值规则写成了泛泛展示检查。
4. 是否把多个核心结果混在同一条 case。
5. 是否遗漏 `X>Y`、`X=Y`、`X<Y` 这类关键边界。
6. 是否遗漏单对象与多对象联合报错。
7. 是否遗漏同集群、多集群、汇总口径差异。
8. 是否遗漏首次默认、切换后、更新后、串联操作后的场景。
9. 是否遗漏双向切换、等值门槛、最后一个对象才归还等高风险组合。
10. 是否与 `references/template.md` 的层级、描述和 Step 结构对齐。
11. 是否把通用组件未变化部分误写成大量无效 testcase。

如果 reviewer 发现缺项、错项、粒度问题或格式问题，先回修初稿，再生成 `testcase_final.md`。

### 6.3 最终输出要求

- 以 `【测试用例处理完成】` 开头。
- 直接展示 review 后的 `testcase_final.md` 正文。
- 除非用户额外要求，否则不要附加长篇过程解释。

## 质量检查

完成前确认：

- 已先读文档，再处理 Figma 场景块，而不是只看整页截图。
- 已优先把长白板拆成多个大白块 PNG，而不是拿整张长图直接做用例设计。
- 已控制 Figma MCP 读取次数，并说明预算使用情况。
- 已在 testcase 生成前显式输出过“文档分析确认”“PNG 场景分析确认”“测试设计确认”。
- 已把场景、数值、展示三层拆开。
- 已显式做交叉场景识别，而不是只覆盖单步操作。
- 已显式构建并检查矩阵，而不是凭印象补 case。
- 已强制生成初稿，再做 reviewer 复核。
- 最终 case 重点覆盖页面场景、数值规则、展示结果、报错方式、状态变化、交叉组合和一致性回归。
- 最终 case 粒度清晰，单条 case 不臃肿。
