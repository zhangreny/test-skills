## 示例 1：用户要学习旧用例规律

用户意图：
“不要把几万个用例全看完，帮我总结旧用例的写法规律。”

合适输出：
1. 先缩到一个项目或一个 suite。
2. 先看结构，再看标题。
3. 最后输出命名、粒度、覆盖维度、反模式总结。

不合适输出：
1. 直接全文阅读。
2. 按 suite 流水账复述。

## 示例 2：用户要导出历史用例

用户意图：
“把 TestRail 的结果导出来，按组放好，后面我还要学习。”

合适输出：
1. 先看项目里有没有 `export_testrail_cases.py`。
2. 优先复用现有脚本。
3. 导出到 `testcase_tutorail_of_groups/`。
4. 一个 suite 一个 `md` 文件。

不合适输出：
1. 默认只生成一个超大文件。
2. 不保留项目和 suite 层级。

## 示例 3：好总结

```markdown
## 命名规律
1. 标题包含对象、条件、结果。
2. 常见句式是 `[对象] - [条件] - [结果]`。

## 粒度规律
1. 一条 case 只验证一个核心结果。
2. 跨 UI、日志、配置、功能结果时应拆条。

## 覆盖维度
1. 主流程。
2. 失败路径。
3. 对象差异。
4. 状态变化。
5. 故障恢复。
```

## 示例 4：坏总结

```markdown
这个 suite 第一条是 A，第二条是 B，第三条是 C。
这些用例都在测功能，整体看起来差不多。
```
