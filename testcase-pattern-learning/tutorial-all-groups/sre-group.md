# SRE 组历史测试用例补充规则

## 适用范围

本文件只用于 SRE / 平台运维类 feature 的补充规则，适用于以下主题：

1. 可观测平台、告警、静默、通知策略、仪表盘、报表
2. 巡检中心、巡检服务升级、巡检任务、报告导出
3. SFS / 文件存储集群、平台适配、任务 / 事件 / VIP 迁移 / DRS 联动
4. Neutree 这类控制面 + 监控组件一体的运维平台
5. 系统服务部署、升级、资源约束、任务 / 事件 / UI / 后端一致性

使用方式：

1. 先读 `all-groups-common.md`。
2. 如果当前 feature 属于 observability、inspection、system-service、alerting、SFS、平台运维、服务升级、任务 / 事件治理类，再读本文件。

5. 平台适配、资源配额、管理 VIP 迁移、DRS 联动、故障与恢复

## SRE 组特有结论

1. SRE 组 testcase 往往不是单点功能验证，而是“控制面配置 -> 任务执行 -> 后端服务状态 -> UI / 告警 / 报表回显”的完整闭环。
2. SRE 组默认要把“静态校验”“提交任务”“任务中 / 异常中”“任务后状态”分层写清楚。
3. 只要 feature 涉及服务升级或部署，默认补“文件 / 镜像来源差异”“版本 / 架构差异”“异常终止与清理”“升级后功能回归”。
4. 只要 feature 涉及任务或事件，默认补“任务详情”“错误 message”“重试 / 重建 / 恢复后状态收敛”。

## SRE 组 feature 类型与默认关注点

### 1. 告警 / 静默 / 通知策略类

典型对象：
告警规则、静默策略、通知策略、静默对象、版本门槛、Token Field 选择器。

默认关注维度：

1. 对象为空 / 非空
2. 版本符合 / 不符合
3. 集群 / Kubernetes 服务 / 系统服务类型差异
4. 全选 / 取消全选 / 搜索 / 勾选交互
5. 启用 / 禁用 / 结束时间
6. 任务全部成功 / 部分失败

重点验证：
UI 交互、文案提示、任务结果、静默对象同步结果和列表展示。

### 2. 服务升级 / 安装包管理类

典型对象：
升级文件、本地上传、URL 上传、CPU 架构、版本、解压、校验、升级任务。

默认关注维度：

1. 本地文件 / URL
2. 文件存在 / 不存在 / 格式错误
3. 架构一致 / 不一致
4. 目标版本高于 / 低于当前版本
5. 上传中 / 解压中 / 校验中 / 升级中
6. 手动停止 / 自动失败 / 清理成功 / 清理失败

重点验证：
按钮禁用态、预览信息、任务状态、错误信息、后端临时文件清理和升级后状态。

### 3. 平台部署 / 外部依赖类

典型对象：
控制面、监控组件、镜像仓库、K8s / VM 部署、外部 Grafana、远程 metrics。

默认关注维度：

1. 在线 / 离线 / Mirror Registry
2. VM / K8s
3. x86 / aarch64
4. Rocky / Ubuntu / openEuler 等 OS 差异
5. 内置监控 / 外部可观测服务
6. adminPassword / jwtSecret / imagePullSecrets 等配置

重点验证：
部署成功、组件 Pod 状态、访问入口、外部依赖联动和配置保留。

### 4. 任务 / 事件 / 资源约束类

典型对象：
任务列表、事件列表、operator 配额、CPU / 内存配额、长任务状态。

默认关注维度：

1. 任务开始 / 结束 / 顺序
2. 超限 / OOM / 强制注入异常
3. 默认阈值 / 超阈值
4. 升级前 / 升级后
5. 前端展示 / 后端状态

重点验证：
任务和事件是否一致、资源限制是否生效、异常后是否自动恢复或重建。

### 5. HA / VIP / DRS / 故障恢复类

典型对象：
管理 VIP、leader、ETCD、网卡、主机、DRS 黑名单、特例规则。

默认关注维度：

1. leader 所在节点 / 非 leader 节点
2. 单点故障 / 多点故障
3. 恢复前 / 恢复后
4. 升级 / 扩容 / 卸载 / 删除
5. Tower / AOC / 其它平台联动

重点验证：
VIP 迁移、服务可访问性、特例规则自动维护、删除或扩容后的黑名单状态。

## SRE 组推荐 section

1. 配置与静态校验
2. 任务与事件链路
3. 服务部署 / 升级
4. 平台联动与版本门槛
5. 资源 / 配额 / 限制
6. 告警 / 报表 / 列表 / 页面展示
7. 故障恢复与 HA

## SRE 组特有补充规律

1. 标题尽量写清“对象 / 前提状态 / 操作 / 结果”，例如“巡检服务升级时上传架构不一致的升级包，升级按钮保持禁用并提示架构不匹配”。
2. 默认把“UI 校验”“任务提交”“任务中状态”“任务完成后状态”拆成不同 testcase，而不是堆在一条里。
3. 涉及服务升级、安装包、镜像或 URL 时，默认补“输入非法 / 无法获取 / 版本低 / 架构错 / 手动停止 / 清理残留”。
4. 涉及告警、报表、静默、任务、事件时，默认补“列表展示”“详情页”“错误 message”“部分成功 / 部分失败”。

每写一个新的 SRE 组 feature，建议最少补以下 10 类：

1. 主流程
2. 表单和静态校验
3. 版本 / 架构 / 平台差异
4. 任务提交与任务详情
5. 异常 message 与停止 / 重试
6. 资源约束 / 配额 / 超限
7. 列表 / widget / 报表 / dashboard 展示
8. 升级 / 部署后回归
9. HA / VIP / leader / DRS / 特例规则
10. 收尾闭环：清理临时文件、禁用态恢复、异常恢复后的状态收敛

## SRE 组最简执行摘要

1. 先拆配置 / 任务 / 状态 / 回显四层。
2. 升级与部署类默认补来源、版本、架构、停止、清理。
3. 告警与静默类默认补对象选择、版本门槛、部分失败。
4. 平台运维类 feature 必须补任务 / 事件 / UI / 后端一致性。
5. HA / 故障类默认补恢复后的状态收敛，不只写故障本身。
