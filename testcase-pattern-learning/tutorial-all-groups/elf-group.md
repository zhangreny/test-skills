# ELF 虚拟化组历史测试用例补充规则

## 适用范围

本文件只用于 ELF 虚拟化组相关 feature 的补充规则，适用于以下主题：

1. 虚拟机生命周期 / 属性 / 资源变更
2. 虚拟机迁移 / 跨集群迁移 / v2v
3. VMTools / Guest 网络配置 / cloud-init 联动
4. ELF 与 ZBS 的块存储、双活、vhost、回收站联动
5. Tower 纳管、插件、UI 展示、告警与回归

使用方式：

1. 先读 `all-groups-common.md`。
2. 如果当前 feature 属于 ELF 虚拟化、虚拟机迁移、Guest 工具、虚拟化与块存储联动、虚拟化管理面联动类，再读本文件。

## ELF 虚拟化组特有结论

1. ELF 组 testcase 通常不是先按页面拆，而是先按“操作链路”拆：安装 / 配置 / 执行 / 迁移后 / 故障后 / 恢复后。
2. ELF 组比很多业务组更依赖源端与目标端差异，尤其是平台、版本、架构、vhost 状态、虚拟机电源状态、boot 模式、磁盘总线、磁盘数量、Guest 工具状态。
3. ELF 组不能只验证任务是否成功，还要验证目标端虚拟机形态、Guest 内部行为、管理面展示、日志脱敏、残留介质清理、迁移后回归。
4. 如果历史样本里已经稳定出现“支持路径 / 不支持路径”“设置静态 IP / 不设置静态 IP”“自动处理成功 / 自动处理失败后手动兜底”“升级前 / 升级后”的成对结构，生成新用例时默认继承这种成对写法。

## ELF 虚拟化组 feature 类型与默认关注点

### 1. 虚拟机生命周期 / 属性类

典型对象：
OVF 导入导出、虚拟机编辑、磁盘属性、磁盘队列、TRIM/UNMAP、HA 触发报警、多队列能力。

默认关注维度：

1. 新部署 / 升级 / 转换 / hotfix
2. vhost / 非 vhost
3. SMTXOS / SMTXELF / ACOS
4. x86 Intel / Hygon / Arm Kunpeng / Arm Feiteng
5. Windows / Linux / 不同 Guest OS 版本
6. BIOS / UEFI
7. virtio / SCSI / IDE
8. 单盘 / 多盘 / 快照存在
9. 默认配置 / 修改后配置
10. 功能成功 / 行为不支持 / 明确失败

重点验证：
配置值是否正确落地、Guest 是否正常可用、磁盘或队列行为是否符合预期、升级或转换后旧能力是否保持一致。

### 2. 迁移 / v2v / 路径兼容类

典型对象：
v2v 工具安装、站点关联、迁移任务、集群内迁移、跨集群热迁移、分段迁移、vmware/ELF/ACOS 之间路径兼容。

默认关注维度：

1. 源端 / 目标端平台组合
2. 源端 / 目标端版本组合
3. 架构组合：intel / hygon / arm
4. vhost / 非 vhost
5. 开机 / 关机 / 暂停 / 未知状态
6. boot 模式：BIOS / UEFI
7. 磁盘数量、总线类型、快照、cdrom / usb 挂载
8. 设置静态 IP / 不设置静态 IP
9. 支持路径 / 不支持路径 / 未来计划禁用路径
10. 单台迁移 / 批量迁移 / Tower 纳管联动

重点验证：
任务结果是否符合预期、失败提示是否准确、目标端虚拟机结构是否正确、资源预估是否正确、迁移后基础回归是否完整、敏感信息是否脱敏。

### 3. VMTools / Guest 网络配置类

典型对象：
VMTools 安装与升级、编辑网卡、静态 IPv4/IPv6、cloud-init、guestinfo / qga 采集。

默认关注维度：

1. 已安装 / 未安装 VMTools
2. 高版本 / 低版本 VMTools
3. 服务运行 / 服务停止 / 工具异常
4. IPv4 / IPv6 / 双栈 / 默认网关
5. 单网卡 / 多网卡 / VPC 网卡
6. netplan / NetworkManager / systemd-networkd / networking / ifup-ifdown
7. cloud-init 创建 / cloud-init 已配置静态 IP
8. 自动安装成功 / 自动安装失败 / 手动挂载兜底
9. 自动移除成功 / 自动移除失败 / 保留环境定位
10. 任务成功 / 资源锁定失败 / Guest 内服务异常失败

重点验证：
Guest 内配置是否真正生效、网络采集是否正常、失败是否有可定位信息、自动化处理失败后是否有可执行的兜底路径。

### 4. 管理面 / ZBS 联动类

典型对象：
块存储开关、双活、license、维护模式、告警、回收站 widget、回收站入口、表格操作、一键清空、Tower 插件与纳管。

默认关注维度：

1. 仅 ELF / 仅 ZBS / ELF+ZBS / ELF+ZBS+VMware ESXi
2. 版本阈值：如 570、630、Tower 480
3. 开启块存储 / 未开启块存储
4. 非双活 / 双活 / 可用域差异
5. 单个操作 / 批量操作 / 一键清空
6. 阈值内 / 超阈值
7. 资源存在 / 已被删除但未及时同步
8. 权限允许 / 权限限制
9. 升级前 / 升级中 / 升级后
10. 正常展示 / 不展示 / 跳转筛选 / 文案变化

重点验证：
入口和展示是否正确、跨产品联动是否一致、任务与资源状态是否同步、阈值和权限限制是否准确、升级与维护模式下业务是否连续。

## ELF 虚拟化组特有补充规律

1. 先按操作链路拆 testcase，再按源端、目标端、Guest、管理面四层变量补矩阵。
2. 标题尽量把源端、目标端、关键条件和预期结果写全，让 case 本身就能表达“什么路径在什么条件下应该成功或失败”。
3. 不只验证控制面任务结果，还要验证数据面和 Guest 面结果，例如磁盘形态、网卡配置、qga / guestinfo 采集、迁移后回归。
4. 只要出现自动化处理链路，就默认补“自动成功”“自动失败后手动兜底”“失败后残留清理”三类 case。

每写一个新的 ELF 虚拟化组 feature，建议最少补以下 10 类：

1. 主流程：创建、编辑、删除、开关机、迁移、恢复
2. 路径组合：源端 / 目标端平台、版本、架构、vhost 状态
3. Guest 差异：Windows / Linux、BIOS / UEFI、单网卡 / 多网卡、磁盘总线 / 数量
4. 正反结果：支持路径成功，不支持路径失败，失败提示准确
5. 生命周期：新部署、升级、转换、hotfix、开关能力前后
6. 工具链：VMTools、cloud-init、qga、guestinfo、自动安装与自动移除
7. 安全面：日志 / API 明文密码、漏洞扫描、端口扫描、文案脱敏
8. 联动面：Tower 纳管、插件、UI 文案、告警、回收站、任务同步
9. ZBS 边界：块存储开关、双活、license、维护模式、回收站恢复与一键清空
10. 收尾闭环：迁移后回归、残留 ISO 清理、已删除未同步资源、批量任务后继续操作

## ELF 虚拟化组最简执行摘要

如果只保留 ELF 虚拟化组最重要的补充规则，优先保留下面 8 条：

1. 先拆操作链路，再拆源端 / 目标端 / Guest / 管理面变量。
2. 迁移类默认显式展开平台、版本、架构、vhost、开关机状态、boot 模式、磁盘和网卡差异。
3. 一条 case 只验证一个核心结果，但结果要覆盖控制面、Guest 面或管理面中的一个明确观察点。
4. 标题必须尽量写清源端、目标端、条件、结果。
5. 成功路径和失败路径默认成对，自动处理和手动兜底默认成对。
6. 涉及 Guest 配置时，必须验证 VMTools / cloud-init / 网络服务差异，不只看任务成功。
7. 涉及 ZBS 联动时，必须验证版本阈值、资源组合、入口展示、批量操作和异常收尾。
8. 迁移、升级、回收类 feature 必须写迁移后 / 升级后 / 恢复后的回归闭环。
