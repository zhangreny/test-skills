# 备份组历史测试用例补充规则

## 适用范围

本文件只用于备份组相关 feature 的补充规则，适用于以下主题：

1. 备份存储库、备份计划、备份 job、恢复点
2. VM / 卷复制、复制计划、复制服务、跨站点联动
3. 故障转移、永久故障转移、故障恢复、恢复点删除
4. VPC 网络映射、复制网络设置、源端 / 目标端网络差异
5. 邮件通知、报表投递、任务 / UI 联动

使用方式：

1. 先读 `all-groups-common.md`。
2. 如果当前 feature 属于备份、复制、恢复点、故障转移、备份存储库、复制服务或跨站保护类，再读本文件。

## 来源与范围

本补充规则主要基于：

1. `former_cases/6_Backup/271_Master.md`
   主要提供备份 / 复制 / 恢复点 / 故障转移 / VPC 适配 / 邮件报表类 testcase 的结构和矩阵。

这份样本已经覆盖了备份组最常见的 testcase 设计模式，因此本次先不额外扩到 live grouped exports。

主要覆盖以下能力：

1. 备份存储库创建、编辑、删除、升级
2. 备份计划创建、编辑、增删保护对象、恢复点清理
3. 全量 / 增量备份、停止 job、备份文件删除
4. VM / 卷复制、手动 / 自动复制、服务部署与升级
5. 单个 / 批量故障转移、永久故障转移、故障恢复
6. 跨站站点管理、恢复点管理、删除计划与服务卸载
7. VPC 网络映射、IP / MAC / 浮动 IP 差异、UI / tooltip / 空状态
8. 邮件通知、报表发送、安全模式差异

## 备份组特有结论

1. 备份组 testcase 通常先按“保护链路”拆：存储库 / 计划 / 执行 / 恢复点 / 恢复或故障转移，再按对象和站点差异做矩阵。
2. 备份组默认要把“自动触发 / 手动触发”“全量 / 增量”“单个 / 批量”“VM / 卷”成对展开。
3. 复制或恢复场景不能只验任务成功，还要验恢复点、目标端对象状态、网络设置、UI 展示和清理闭环。
4. 只要涉及故障转移，就默认补“故障转移”“永久故障转移”“故障恢复”“恢复后继续复制 / 继续备份”。

## 备份组 feature 类型与默认关注点

### 1. 备份存储库 / 备份计划类

典型对象：
备份存储库、备份计划、被保护虚拟机、恢复点保留策略。

默认关注维度：

1. 新建 / 编辑 / 删除
2. 无 job 执行 / 有 job 执行
3. 增加保护对象 / 删除保护对象
4. 删除对象时是否同时删除恢复点
5. NFS4 等后端类型差异
6. 升级前 / 升级后

重点验证：
配置是否生效、计划编辑后对象与恢复点是否同步、任务与 UI 状态是否一致。

### 2. 备份 / 复制执行类

典型对象：
全量备份、增量备份、全量复制、增量复制、手动执行、自动触发。

默认关注维度：

1. 全量 / 增量
2. 自动 / 手动
3. VM / 卷
4. 执行成功 / 中途终止 / 异常失败
5. 批量 / 单个
6. 数据有变更 / 无变更

重点验证：
任务状态、数据传输行为、恢复点生成、停止任务后的清理和文件删除。

### 3. 故障转移 / 恢复类

典型对象：
故障转移、永久故障转移、故障恢复、站内 / 跨站复制。

默认关注维度：

1. 站内 / 跨站
2. 单个 / 批量
3. 普通故障转移 / 永久故障转移 / 故障恢复
4. 选择已有恢复点 / 立即复制
5. 目标端资源存在 / 不存在
6. 恢复后再次执行计划

重点验证：
目标端对象状态、网络设置、恢复点选取、故障后恢复路径是否闭环。

### 4. VPC 网络映射与 UI 交互类

典型对象：
VPC、子网、MAC 地址、浮动 IP、网络设置页面、tooltip、空态提示。

默认关注维度：

1. 源端 / 目标端网络类型
2. 相同 VPC / 不同 VPC / 相同子网 / 不同子网
3. 原机存在 / 原机被删除
4. 浮动 IP 存在 / 不存在 / 被占用
5. 单个 / 批量故障转移
6. 创建计划 / 编辑计划 / 执行故障转移

重点验证：
映射默认值是否正确、提示文案是否准确、异常数据时 UI 是否可理解。

### 5. 通知与报表类

典型对象：
SMTP、报表发送、安全模式、任务通知。

默认关注维度：

1. SSL / STARTTLS / 无安全模式
2. 发送成功 / 配置错误
3. 报表生成 / 发送链路

重点验证：
任务状态、发送结果、错误提示和最终报表可用性。

## 备份组推荐 section

1. 存储库与计划管理
2. 执行链路
3. 恢复点与清理
4. 故障转移与恢复
5. 网络映射与 UI
6. 服务升级与站点管理
7. 报表与通知

## 备份组特有补充规律

1. 默认把“配置对象”“执行对象”“恢复对象”分开写，不要把计划管理和执行链路压成一条 case。
2. 标题尽量同时写清对象、执行方式和结果，例如“手动执行增量复制时停止任务，复制任务失败且恢复点不保留”。
3. 只要是批量场景，默认补“全部成功”“部分异常”“依赖数据缺失”的分支。
4. 复制 / 恢复类 feature 必须补网络映射、MAC / IP、浮动 IP 或目标端资源存在性这类执行后验证点。

每写一个新的备份组 feature，建议最少补以下 10 类：

1. 主流程：创建、编辑、删除、执行
2. 自动 / 手动
3. 全量 / 增量
4. VM / 卷
5. 单个 / 批量
6. 恢复点管理
7. 故障转移 / 永久故障转移 / 故障恢复
8. UI / tooltip / 空态提示
9. 服务升级 / 站点管理
10. 收尾闭环：终止任务、删除文件、删除恢复点、删除计划后的残留清理

## 备份组最简执行摘要

1. 先拆保护链路，再拆对象和站点变量。
2. 全量 / 增量、自动 / 手动、单个 / 批量默认成对。
3. 故障转移类必须写恢复，不只写转移。
4. 复制 / 恢复场景必须补网络映射和目标端结果验证。
5. 任务成功不是终点，恢复点、文件、UI、清理也要验。
