# 历史测试用例通用提炼规则

## 文档目标

本文件沉淀一份对所有组都可复用的测试用例设计规则，目标不是复述旧 suite，而是帮助后续新 feature 稳定地产出结构清晰、粒度合理的 testcase。

重点回答：

1. 旧用例通常怎么组织。
2. 标题通常怎么命名。
3. 测试点通常怎么拆。
4. 一条 case 的理想粒度是什么。
5. 哪些覆盖维度默认要补。
6. 哪些写法应视为反模式。

## 使用方式

1. 先读本文件，获取通用的拆点、命名、粒度和复核规则。
2. 再按 feature 所属组别，按需读取 `tutorial-all-groups` 下对应组的补充文档。
3. 如果当前没有对应组的补充文档，只使用本文件即可推进。

## 核心结论

1. 高质量用例不是按页面堆功能，而是按能力和变量拆测试点。
2. 一条 case 应尽量只验证一个核心结果，不要把多个独立结果塞进同一条。
3. 标题必须同时体现对象、条件、结果，避免只有动作没有预期。
4. 复杂 feature 的关键不是单纯 happy path，而是不同路径、对象、故障条件下行为是否一致。
5. 好的 suite 通常会同时覆盖功能正确性、配置正确性、展示正确性、日志或计数、故障恢复、资源约束、升级兼容。
6. 当结果不直接展示在 UI 时，高质量用例必须说清楚“通过什么工具执行、通过什么信号观察”，而不是只写“结果正确”。
7. 如果输出仍像“测试角度大纲”而不是“对象 + 条件 + 动作 + 单一结果”的原子 case，就说明还没有拆到位，必须继续细化。

## 建议学习方式

不要从头扫完整库，建议按以下顺序学习：

1. 先定范围：先看一个项目、一个 suite、一个子领域。
2. 先看结构：先看 section 和 subsection 的组织方式。
3. 再看标题：看命名句式、场景拆分方式、成功失败是否成对。
4. 最后提炼规律：命名、粒度、覆盖维度、反模式。

如果规律已经稳定，就不需要继续扩大样本。停止继续阅读的信号通常是：

1. 标题句式已经重复出现。
2. 覆盖维度已经稳定。
3. 粒度规律已经明显。
4. 新读到的 case 只是重复旧规律。

## feature 类型与默认关注点

### 1. 页面展示 / 可视化类

典型对象：
概览页、详情页、拓扑、表格、筛选、排序、导出、告警、tooltip、卡片、字段展示。

默认关注维度：

1. 页面入口：概览 / 详情 / 弹窗 / 卡片 / 列表 / 报表
2. 数据视图：图 / 表 / 拓扑 / 详情面板
3. 筛选条件：单条件 / 多条件 / 混合条件
4. 条件变化：新增 / 删除 / 修改对象后的联动
5. 排序 / 导出 / 预览 / 告警
6. 跨页切换后状态保留
7. 空数据 / 异常数据 / 权限不足

重点验证：
展示是否正确、筛选结果是否正确、联动是否正确、导出或告警是否正确。

### 2. 运维生命周期类

典型对象：
部署、关联、取关、启用、停用、卸载、重新启用、迁移、主备切换、回切。

默认关注维度：

1. 部署
2. 关联 / 取消关联
3. 启用 / 停用
4. 编辑配置
5. 主备切换 / 回切
6. 卸载 / 删除
7. 清理残留
8. 反复操作
9. 顺序依赖
10. 操作失败时回滚

重点验证：
动作是否可执行、状态是否正确变化、资源是否清理、回滚是否正确、重复操作是否幂等。

### 3. 故障 / 恢复 / 稳定性类

典型对象：
服务异常、节点重启、链路抖动、维护模式、短时中断、长时中断。

默认关注维度：

1. 单点故障
2. 多点故障
3. 短时抖动
4. 长时间中断
5. 人工恢复
6. 自动恢复
7. 恢复后配置保持
8. 恢复后业务恢复
9. 恢复后日志 / 告警 / 状态正确

重点验证：
故障时业务表现、切换行为、恢复时长、恢复后状态、是否存在残留异常。

### 4. 许可 / 资源约束类

典型对象：
socket / vCPU 许可、试用 / 订阅 / 永久、额度显示、扩容限制、归还策略。

默认关注维度：

1. 许可类型
2. 许可维度
3. 已使用 / 可用额度
4. 正好满足
5. 不足
6. 完全用尽
7. 旧集群 / 新集群
8. 被关联 / 未关联
9. 资源归还
10. 报错文案

重点验证：
是否允许操作、额度显示是否正确、报错是否准确、释放后额度是否归还。

### 5. 版本 / 升级 / 兼容类

典型对象：
发版测试、升级路径、旧版本兼容、架构兼容、OS 兼容。

默认关注维度：

1. 升级前版本
2. 升级后版本
3. 升级前存量配置
4. 升级后新增配置
5. 升级后默认值
6. 组件组合版本
7. 平台或架构差异
8. 升级后功能回归
9. 升级后数据或配置保持
10. 升级失败 / 不支持路径

重点验证：
升级后功能是否仍正确、默认值是否正确、存量配置是否保留、兼容矩阵是否符合预期。

### 6. 接口 / 字段类

典型对象：
OAPI、字段展示、请求类型、接口返回、过滤参数、统计字段。

默认关注维度：

1. GET / POST / UPDATE / DELETE
2. 必填字段
3. 可选字段
4. 默认值
5. 只读字段
6. 字段联动
7. 返回计数 / 过滤结果
8. 非法输入
9. 资源存在 / 不存在
10. 兼容旧字段

重点验证：
接口行为是否正确、字段语义是否正确、返回结果是否正确、更新后字段是否同步。

### 7. 工具 / 诊断类

典型对象：
诊断工具、trace 工具、性能监控工具、命令行工具。

默认关注维度：

1. 参数正确
2. 参数错误
3. 权限要求
4. 输出正确
5. 过滤条件
6. 文件输出
7. 长时间运行
8. 并发运行
9. 不同平台
10. 异常退出

重点验证：
命令是否可执行、输出是否正确、错误提示是否准确、长时间运行是否稳定。

## 测试点提炼方法

拿到新 feature 后，先按以下顺序拆点，再生成 case：

### Step 1：明确主能力

1. 这个 feature 的核心价值是什么？
2. 它改变的是配置、展示、运维、许可、接口，还是工具能力？
3. 用户最直接感知的结果是什么？

### Step 2：明确主要对象

1. 涉及哪些资源对象？
2. 这些对象是否有多种类型？
3. 是否有主/备、客户端/服务端、单资源/多资源差异？

### Step 3：明确主要变量

1. 哪些条件变化会导致结果不同？
2. 是否存在版本、许可、平台、状态、对象差异？
3. 是否存在开启/关闭、关联/未关联、存在/不存在差异？

### Step 4：明确验证层

1. 验证结果体现在哪？
2. 是配置结果、UI 展示、日志、计数、接口返回、资源状态，还是报错文案？

### Step 5：明确生命周期动作

1. 是否存在创建、编辑、删除、启停、关联、取关、升级、卸载？
2. 是否需要验证动作前后状态变化？

### Step 6：明确执行工具与观测手段

1. 用户和测试通常通过 UI、API、CLI、脚本、诊断工具中的哪种入口执行？
2. 结果体现在哪：UI、日志、告警、任务状态、计数、报表、抓包、trace、性能指标？
3. 哪些场景如果不写明工具或观测路径，就会让 case 失去可执行性或可判定性？

### Step 7：明确异常和边界

1. 哪些失败场景是业务必须拦截的？
2. 哪些异常会导致不生效、部分生效、报错、回滚、禁用？

## 测试点默认类别

不要只生成主流程，默认至少从下面几类里选择：

1. 主流程
2. 参数或对象差异
3. 边界情况
4. 异常输入或错误配置
5. 状态切换
6. 故障恢复
7. 展示与观测
8. 执行工具与观测手段
9. 资源约束
10. 升级兼容
11. 回归历史问题

默认建议：

1. 小 feature 至少覆盖 4 类。
2. 中等 feature 至少覆盖 6 类。
3. 核心 feature 至少覆盖 8 类。

## 测试点优先级

### P0：必须覆盖

1. 主流程
2. 关键对象差异
3. 核心失败路径
4. 生命周期中的关键动作
5. 结果验证层

### P1：强烈建议覆盖

1. 边界情况
2. 状态切换
3. 展示或日志联动
4. 故障恢复
5. 升级兼容

### P2：按相关性补充

1. 许可约束
2. 多平台
3. 长时间运行
4. 并发
5. 历史问题回归

## 粒度规则

### 一条 case 的理想粒度

一条 case 只回答一个问题：

在某个明确条件下，
对某个明确对象，
执行某个明确动作后，
某个核心结果是否符合预期。

推荐结构：

`条件 + 动作 + 单一结果`

### 需要拆成多条 case 的信号

出现以下情况应拆条：

1. 一个标题里有多个并列结果。
2. 同时验证 UI、日志、配置多个层面。
3. 同时覆盖多个对象类型。
4. 同时覆盖多个生命周期动作。
5. 同时包含成功和失败。
6. 同时包含多个独立变量变化。

### 可以合并为一条 case 的条件

仅当以下条件同时满足时可以合并：

1. 属于同一个动作。
2. 验证的是同一个结果层。
3. 多个检查点只是同一结果的组成部分。

### 矩阵场景的粒度控制

当天然是矩阵型 feature 时，不要无脑全量笛卡尔积，遵循以下规则：

1. 核心路径必须全量。
2. 次要维度可以代表性抽样。
3. 若两个维度叠加后不会改变结果，可不全量组合。
4. 会显著改变结果的对象差异、状态差异、版本差异通常不能省。
5. 仅作为展示层差异的维度，可选代表场景。

## 标题命名规则

### 标题必须包含的 3 个元素

1. 测试对象
2. 关键条件
3. 预期结果

缺少任一元素，标题质量不足。

### 推荐标题模板

1. 路径型：`[路径/对象] - [条件] - [结果]`
2. 生命周期型：`[动作对象] - [动作] - [结果]`
3. 展示型：`[页面/入口] - [能力点] - [结果]`
4. 资源约束型：`[资源条件] - [动作] - [结果]`
5. 故障恢复型：`[故障动作] - [观察对象] - [恢复结果]`
6. 接口型：`[资源] - [请求类型/字段场景] - [结果]`

### 推荐词汇

推荐：

1. 生效
2. 不生效
3. 展示正确
4. 数据正确
5. 筛选结果正确
6. 日志生成正确
7. 配置保持
8. 配置恢复
9. 切换成功
10. 回切成功
11. 报错
12. 提示正确
13. 禁用
14. 自动移除
15. 自动恢复
16. 归还额度
17. 不归还额度

避免：

1. 功能测试
2. 验证正常
3. 场景测试
4. 其他测试
5. 综合测试
6. 各种情况
7. 没问题
8. 功能可用
9. 界面没问题

## 前置条件规则

会影响结果的前置条件，尽量写进标题或规则描述：

1. 已开启 / 已关闭
2. 已关联 / 未关联
3. 存在 / 不存在
4. 单对象 / 多对象
5. 主 / 备
6. 版本差异
7. 许可足够 / 不足
8. 故障中 / 恢复后

不必强行进标题的内容：

1. 资源具体名称
2. 安装包路径
3. 工具具体执行命令
4. 非关键环境准备步骤

## 验证层规则

生成 case 前，先决定验证点落在哪一层：

1. 结果层：功能生效、不生效、切换后仍正确
2. 配置层：配置生成正确、编辑后保持、升级后默认值正确、删除后清理正确
3. 展示层：列表展示正确、字段展示正确、过滤结果正确、tooltip 文案正确
4. 可观测层：日志生成、命中计数增加、告警触发、报表导出成功
5. 资源约束层：配额扣减、额度显示、报错阻止、额度归还
6. 恢复层：故障后切换、恢复后能力恢复、恢复后配置保持、恢复后无残留告警

默认规则：

1. 一条 case 只主打一层。
2. 如果需要跨层验证，按层拆条。

## suite 组织规则

### 推荐骨架

新 feature suite 默认按以下顺序组织：

1. 基础功能
2. 对象或参数差异
3. 编辑与状态切换
4. 异常与限制
5. 展示与日志
6. 故障与恢复
7. 资源与许可
8. 升级与兼容
9. 回归与收尾

核心 feature 通常至少要覆盖前 6 部分。

### 不同类型 feature 的推荐 section

展示类：

1. 页面入口
2. 数据正确性
3. 筛选与排序
4. 条件联动
5. 导出与告警
6. 异常与空态

运维类：

1. 部署
2. 关联
3. 编辑
4. 启停
5. 取关
6. 卸载
7. 残留清理

许可类：

1. 展示
2. 正好满足
3. 不足
4. 完全用尽
5. 释放归还
6. 报错文案

## 默认覆盖清单

如果没有特殊限制，新 feature 默认至少检查：

1. 主流程是否可执行
2. 关键对象差异是否覆盖
3. 关键路径差异是否覆盖
4. 成功与失败是否成对
5. 创建、编辑、删除或启停是否闭环
6. 是否至少覆盖一个非结果层验证点
7. 关键 case 是否说明了如何执行和如何观察结果
8. 是否考虑故障恢复或状态切换
9. 是否考虑升级兼容或存量配置
10. 是否考虑资源约束或禁用态
11. 是否避免一个 case 同时验证过多点

## 反模式清单

1. 标题只有动作，没有结果。
2. 标题过于笼统，如“验证功能正常”。
3. 一个标题包含多个独立结果。
4. 把备注、讨论、推测写进标题。
5. 把环境搭建细节塞进标题。
6. 只测 UI，不测真实生效。
7. 只测成功，不测失败路径。
8. 只写 happy path，不写故障恢复。
9. 隐藏结果类 case 没有写明执行工具或观测手段。
10. 变量过多但不拆。
11. 同一 suite 命名风格混乱。
12. 用“基础功能验证”“异常场景验证”“权限验证”这类主题标题直接充当 testcase，而没有进一步拆成原子结果。

## 最简执行摘要

如果只保留最重要的规则，优先保留下面 12 条：

1. 先判定 feature 类型，再选维度。
2. 先拆能力，再拆变量，再写标题。
3. 一条 case 只验证一个核心结果。
4. 标题必须有对象、条件、结果。
5. 正向和反向场景默认成对。
6. 展示类优先拆页面、数据、筛选、联动、导出。
7. 运维类优先拆部署、关联、启停、卸载、清理。
8. 故障类必须写恢复，不只写故障。
9. 许可类必须写额度、限制、报错、归还。
10. 升级类必须写存量、默认值、路径、回归。
11. 接口类必须拆请求类型、字段语义、错误输入。
12. 避免一个 case 同时覆盖多层结果。

## 结论

这份文档可作为后续设计各组新 feature 测试用例的上位规则。

它关注的不是某个旧 suite 的具体内容，而是：

1. 测试点如何抽取。
2. 测试维度如何展开。
3. 测试粒度如何控制。
4. 标题如何统一命名。
5. 哪些反模式必须避免。
