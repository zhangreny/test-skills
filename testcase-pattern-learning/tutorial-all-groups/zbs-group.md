# ZBS 组历史测试用例补充规则

## 适用范围

本文件只用于 ZBS 组相关 feature 的补充规则，适用于以下主题：

1. ZBS CSI、storageClass、PV / PVC、快照、克隆、恢复
2. 副本数、精简 / 厚置备、常驻缓存、QoS、加密
3. 单活 / 双活、multiCluster、topo、nvmf、chap 等参数行为
4. ZBS 升级检查、在线 / 离线升级、升级后回归
5. ZBS 故障、主机 / 服务 / 网卡 / 接入 IP 变更

使用方式：

1. 先读 `all-groups-common.md`。
2. 如果当前 feature 属于 ZBS、ZBS CSI、块存储、卷参数、存储策略、快照 / 克隆 / 恢复、双活或升级检查类，再读本文件。

## 来源与范围

本补充规则主要基于：

1. `former_cases/18_ZBS CSI/1312_Master.md`
   主要提供 ZBS CSI 的参数矩阵、升级、故障和回归写法。
2. `former_cases/1_SmartX/300_SMTX ZBS.md`
   主要提供 ZBS 与其他产品联动、版本阈值、管理面与组合场景的写法。

主要覆盖以下能力：

1. 副本数在单活 / 双活集群下的默认和显式配置
2. 卷加密、快照 / 克隆 / 恢复、扩容与删除
3. 常驻缓存、精简 / 厚置备、QoS 组合
4. 升级检查、升级后存量资源回归、sidecar / controller 行为
5. 故障测试：服务 down、网卡 down、主机关机、主机增减、接入 IP 变更

## ZBS 组特有结论

1. ZBS 组 testcase 通常先按“卷生命周期或参数生命周期”拆，再按集群形态、参数值和资源类型展开矩阵。
2. ZBS 组默认要把“参数为空 / 缺省 / 支持值 / 不支持值 / 旧值到新值变化”写全。
3. 只要涉及升级，不能只验升级成功，还要验升级前创建的 SC / PVC / PV / 快照 / 克隆对象在升级后是否继续可用。
4. 故障类默认要验前台结果、后端资源状态、pod 绑定或 mount 清理，不只看告警。

## ZBS 组 feature 类型与默认关注点

### 1. 存储参数与默认值类

典型对象：
副本数、storagePolicy、defaultStoragePolicy、multiCluster、topo、nvmf、chap。

默认关注维度：

1. 单活 / 双活
2. 参数为空 / 缺省 / 显式值 / 非法值
3. 支持值 / 不支持值
4. SC 设置 / PVC 设置 / 继承默认值

重点验证：
自动生成的 PV 参数是否符合预期、报错是否准确、默认值是否按集群类型变化。

### 2. 卷加密 / 缓存 / 置备模式类

典型对象：
加密算法、常驻缓存、精简 / 厚置备、QoS 组合。

默认关注维度：

1. 开启 / 关闭
2. 支持算法 / 非法算法
3. Filesystem / Block
4. 创建 / 克隆 / 恢复 / 扩容 / 删除
5. 参数组合场景

重点验证：
参数是否真正生效、组合配置是否相容、变更后的卷行为是否符合预期。

### 3. 快照 / 克隆 / 恢复类

典型对象：
PVC、PV、快照、恢复后的 PV、克隆卷。

默认关注维度：

1. 源卷存在 / 不存在
2. 访问模式一致 / 不一致
3. 创建 / 恢复 / 克隆 / 删除
4. 升级前对象 / 升级后对象

重点验证：
任务结果、恢复卷属性、克隆卷属性、异常场景报错和数据面可用性。

### 4. 升级与兼容类

典型对象：
在线升级、离线升级、sidecar、controller、副本分布。

默认关注维度：

1. 升级检查通过 / 不通过
2. 存量特殊参数存在 / 不存在
3. 升级前创建资源 / 升级后新建资源
4. 在线 / 离线 / 镜像仓库差异

重点验证：
升级前后资源可用性、检查项准确性、挂载清理、controller 分布和 sidecar 状态。

### 5. 故障与恢复类

典型对象：
chunk 服务、接入网卡、bonding、主机上下线、接入 IP。

默认关注维度：

1. 单服务故障 / 多组件故障
2. 有 bonding / 无 bonding
3. 主机减少 / 增加
4. 网络变更 / IP 变更
5. 故障前 / 故障中 / 恢复后

重点验证：
卷是否持续可用、状态是否收敛、告警和恢复是否正确。

## ZBS 组特有补充规律

1. 标题尽量写清“集群类型 + 参数条件 + 资源类型 + 结果”，例如“使用 ZBS 双活集群且 SC 不设置副本数时，创建 PVC 自动生成 3 副本 PV”。
2. 只要 feature 涉及参数，就默认补“参数不填”“参数非法”“参数支持但与集群形态冲突”的 case。
3. 升级 feature 默认补“升级前资源继续可用”“升级后新建资源正常”“旧资源删除 / 快照 / 克隆 / 恢复仍可用”。
4. 故障类默认补“故障发生”“恢复后状态”“残留清理”三层。

每写一个新的 ZBS 组 feature，建议最少补以下 10 类：

1. 主流程
2. 单活 / 双活差异
3. 参数为空 / 缺省 / 合法 / 非法
4. Filesystem / Block
5. 快照 / 克隆 / 恢复
6. 加密 / 缓存 / QoS 组合
7. 升级前 / 升级后
8. 故障 / 恢复
9. Pod 绑定 / mount 清理 / sidecar / controller 行为
10. 管理面联动与版本阈值

## ZBS 组最简执行摘要

1. 先拆卷和参数生命周期，再拆集群形态和参数矩阵。
2. 参数类默认补空值、缺省、合法、非法。
3. 升级类必须覆盖存量对象回归。
4. 快照 / 克隆 / 恢复不能只验任务成功，要验卷属性和后续可用性。
5. 故障类默认写恢复和清理，不只写异常本身。
