# SDN 历史测试用例提炼规则

## 文档目标

本文件基于 `15_SDN` 历史用例总结而来，但目标不是复述旧 suite，而是沉淀一份可复用的 SDN 测试设计规则。

这份文档用于指导后续新 feature 的测试用例设计，重点回答：

1. 旧用例通常怎么组织。
2. 标题通常怎么命名。
3. 测试点通常怎么拆。
4. 一条 case 的理想粒度是什么。
5. 哪些覆盖维度默认要补。
6. 哪些写法应视为反模式。

## 来源与范围

本规则基于 `15_SDN` 下多类 suite 的共性提炼，主要覆盖以下能力：

1. 分布式防火墙 DFW
2. VPC / 子网 / 网关 / FIP / 对等连接
3. 负载均衡 LB / LBI / LBG / VIP / LIP
4. 流量可视化 / 可观测性 / 日志 / 报表 / 筛选
5. VDS / bond / network-tool / L2PING
6. 故障恢复 / 稳定性 / 维护模式 / 切换
7. 许可 / 配额 / 版本适配 / 升级路径
8. OAPI / API / 字段覆盖 / 请求类型
9. 测量工具 / 诊断工具

## 核心结论

1. 高质量 SDN 用例不是按页面堆功能，而是按能力和变量拆测试点。
2. 一条 case 应尽量只验证一个核心结果，不要把多个独立结果塞进同一条。
3. 标题必须同时体现对象、条件、结果，避免只有动作没有预期。
4. 网络类 feature 的关键不是单纯 happy path，而是不同路径、对象、故障条件下行为是否一致。
5. 好的 suite 通常会同时覆盖功能正确性、配置正确性、展示正确性、日志或计数、故障恢复、资源约束、升级兼容。

## 建议学习方式

不要从头扫完整库，建议按以下顺序学习：

1. 先定范围：先看一个项目、一个 suite、一个子领域。
2. 先看结构：先看 section 和 subsection 的组织方式。
3. 再看标题：看命名句式、场景拆分方式、成功失败是否成对。
4. 最后提炼规律：命名、粒度、覆盖维度、反模式。

如果规律已经稳定，就不需要继续扩大样本。停止继续阅读的信号通常是：

1. 标题句式已经重复出现。
2. 覆盖维度已经稳定。
3. 粒度规律已经明显。
4. 新读到的 case 只是重复旧规律。

## feature 类型与默认关注点

### 1. 网络路径 / 转发类

典型对象：
VPC 通信、网关转发、FIP、对等连接、L2/L3 连通性、策略生效路径、LB 流量路径。

默认关注维度：

1. 同节点 / 跨节点
2. 同子网 / 不同子网
3. 同 VPC / 跨 VPC
4. VLAN / VPC / 混合网络
5. 路由网关 / NAT 网关 / L2 网关 / 对等连接 / FIP
6. IPv4 / IPv6
7. TCP / UDP / ICMP / 特定协议
8. 单 IP / 多 IP
9. 正常连通 / 因策略失败 / 因未关联失败 / 因配置错误失败
10. 修改前 / 修改后

重点验证：
路径是否可达、转发是否正确、策略是否生效、失败是否符合预期。

### 2. 策略 / 对象匹配类

典型对象：
DFW、安全组、标签、IP 组、Pod 安全组、隔离、白名单、日志开关。

默认关注维度：

1. 对象类型：标签 / 安全组 / IP 组 / Pod 安全组 / IP 地址
2. 动作类型：创建 / 编辑 / 删除
3. 策略类型：放行 / 拒绝 / 全局白名单 / 诊断隔离 / 完全隔离
4. 匹配方向：源 / 目标 / 任意端
5. 成员变化：增 / 删 / 改
6. 命中行为：生效 / 不生效 / 计数变化 / 日志生成
7. 规则状态：启用 / 停用
8. 升级前存量 / 升级后新增
9. 关联对象存在 / 被删除 / 被修改

重点验证：
对象匹配是否正确、策略结果是否正确、编辑后是否即时生效、日志和计数是否一致。

### 3. 页面展示 / 可视化类

典型对象：
概览页、详情页、拓扑、表格、筛选、排序、导出、告警、tooltip、卡片、字段展示。

默认关注维度：

1. 页面入口：概览 / 详情 / 弹窗 / 卡片 / 列表 / 报表
2. 数据视图：图 / 表 / 拓扑 / 详情面板
3. 数据范围：虚拟机网络 / 系统网络 / 容器网络
4. 筛选条件：单条件 / 多条件 / 混合条件
5. 条件来源：安全组 / IP 组 / Pod / 集群 / 服务 / 时间范围
6. 条件变化：新增 / 删除 / 修改对象后的联动
7. 排序 / 导出 / 预览 / 告警
8. 跨页切换后状态保留
9. 空数据 / 异常数据 / 权限不足

重点验证：
展示是否正确、筛选结果是否正确、联动是否正确、导出或告警是否正确。

### 4. 运维生命周期类

典型对象：
部署、关联、取关、启用、停用、卸载、重新启用、迁移、主备切换、回切。

默认关注维度：

1. 部署
2. 关联 / 取消关联
3. 启用 / 停用
4. 编辑配置
5. 主备切换 / 回切
6. 卸载 / 删除
7. 清理残留
8. 反复操作
9. 顺序依赖
10. 操作失败时回滚

重点验证：
动作是否可执行、状态是否正确变化、资源是否清理、回滚是否正确、重复操作是否幂等。

### 5. 故障 / 恢复 / 稳定性类

典型对象：
网卡 down、交换机故障、节点重启、服务异常、维护模式、链路抖动、LACP 异常、ARP 老化。

默认关注维度：

1. 上游故障
2. 下游故障
3. 单点故障
4. 多点故障
5. 短时抖动
6. 长时间中断
7. 人工恢复
8. 自动恢复
9. 恢复后配置保持
10. 恢复后业务恢复
11. 恢复后日志 / 告警 / 状态正确

重点验证：
故障时业务表现、切换行为、恢复时长、恢复后状态、是否存在残留异常。

### 6. 许可 / 资源约束类

典型对象：
socket / vCPU 许可、试用 / 订阅 / 永久、额度显示、扩容限制、归还策略。

默认关注维度：

1. 许可类型
2. 许可维度
3. 已使用 / 可用额度
4. 正好满足
5. 不足
6. 完全用尽
7. 旧集群 / 新集群
8. 被关联 / 未关联
9. 资源归还
10. 报错文案

重点验证：
是否允许操作、额度显示是否正确、报错是否准确、释放后额度是否归还。

### 7. 版本 / 升级 / 兼容类

典型对象：
发版测试、升级路径、旧版本兼容、架构兼容、OS 兼容。

默认关注维度：

1. 升级前版本
2. 升级后版本
3. 升级前存量配置
4. 升级后新增配置
5. 升级后默认值
6. 组件组合版本
7. x86 / arm / 特定 OS
8. 升级后功能回归
9. 升级后数据或配置保持
10. 升级失败 / 不支持路径

重点验证：
升级后功能是否仍正确、默认值是否正确、存量配置是否保留、兼容矩阵是否符合预期。

### 8. 接口 / 字段类

典型对象：
OAPI、字段展示、请求类型、接口返回、过滤参数、统计字段。

默认关注维度：

1. GET / POST / UPDATE / DELETE
2. 必填字段
3. 可选字段
4. 默认值
5. 只读字段
6. 字段联动
7. 返回计数 / 过滤结果
8. 非法输入
9. 资源存在 / 不存在
10. 兼容旧字段

重点验证：
接口行为是否正确、字段语义是否正确、返回结果是否正确、更新后字段是否同步。

### 9. 工具 / 诊断类

典型对象：
network-tool、测量工具、trace 工具、性能监控工具。

默认关注维度：

1. 参数正确
2. 参数错误
3. 权限要求
4. 输出正确
5. 过滤条件
6. 文件输出
7. 长时间运行
8. 并发运行
9. 不同平台
10. 异常退出

重点验证：
命令是否可执行、输出是否正确、错误提示是否准确、长时间运行是否稳定。

## 测试点提炼方法

拿到新 feature 后，先按以下顺序拆点，再生成 case：

### Step 1：明确主能力

1. 这个 feature 的核心价值是什么？
2. 它改变的是转发、配置、展示、策略、运维、许可，还是工具能力？
3. 用户最直接感知的结果是什么？

### Step 2：明确主要对象

1. 涉及哪些资源对象？
2. 这些对象是否有多种类型？
3. 是否有源/目标、客户端/服务端、主/备、虚拟机/Pod、单 IP / 多 IP 差异？

### Step 3：明确主要变量

1. 哪些条件变化会导致结果不同？
2. 是否存在节点、子网、VPC、协议、版本、许可、平台差异？
3. 是否存在开启/关闭、关联/未关联、存在/不存在差异？

### Step 4：明确验证层

1. 验证结果体现在哪？
2. 是转发结果、UI 展示、日志、计数、接口返回、资源状态，还是报错文案？

### Step 5：明确生命周期动作

1. 是否存在创建、编辑、删除、启停、关联、取关、升级、卸载？
2. 是否需要验证动作前后状态变化？

### Step 6：明确异常和边界

1. 哪些失败场景是业务必须拦截的？
2. 哪些异常会导致不生效、部分生效、报错、回滚、禁用？

## 测试点默认类别

不要只生成主流程，默认至少从下面几类里选择：

1. 主流程
2. 参数或对象差异
3. 边界情况
4. 异常输入或错误配置
5. 状态切换
6. 故障恢复
7. 展示与观测
8. 资源约束
9. 升级兼容
10. 回归历史问题

默认建议：

1. 小 feature 至少覆盖 4 类。
2. 中等 feature 至少覆盖 6 类。
3. 核心网络 feature 至少覆盖 8 类。

## 测试点优先级

### P0：必须覆盖

1. 主流程
2. 关键对象差异
3. 核心失败路径
4. 生命周期中的关键动作
5. 结果验证层

### P1：强烈建议覆盖

1. 边界情况
2. 状态切换
3. 展示或日志联动
4. 故障恢复
5. 升级兼容

### P2：按相关性补充

1. 许可约束
2. 多平台
3. 长时间运行
4. 并发
5. 历史问题回归

## 粒度规则

### 一条 case 的理想粒度

一条 case 只回答一个问题：

在某个明确条件下，
对某个明确对象，
执行某个明确动作后，
某个核心结果是否符合预期。

推荐结构：

`条件 + 动作 + 单一结果`

### 需要拆成多条 case 的信号

出现以下情况应拆条：

1. 一个标题里有多个并列结果。
2. 同时验证 UI、日志、转发、配置多个层面。
3. 同时覆盖多个对象类型。
4. 同时覆盖多个协议。
5. 同时覆盖多个生命周期动作。
6. 同时包含成功和失败。
7. 同时包含多个独立变量变化。

示例：

不建议：
`创建网关后验证配置正确、转发正常、日志正确、页面展示正确`

应拆为：

1. 创建网关后配置正确
2. 创建网关后转发正常
3. 创建网关后日志正确
4. 创建网关后页面展示正确

### 可以合并为一条 case 的条件

仅当以下条件同时满足时可以合并：

1. 属于同一个动作。
2. 验证的是同一个结果层。
3. 多个检查点只是同一结果的组成部分。

示例：

可合并：
`概览页-按集群筛选-列表和统计数据正确`

不建议合并：
`概览页-按集群筛选-列表正确且导出报表正确且告警正确`

### 矩阵场景的粒度控制

当天然是矩阵型 feature 时，不要无脑全量笛卡尔积，遵循以下规则：

1. 核心路径必须全量。
2. 次要维度可以代表性抽样。
3. 若两个维度叠加后不会改变结果，可不全量组合。
4. 协议差异、对象差异、主备差异通常不能省。
5. 仅作为展示层差异的维度，可选代表场景。

## 标题命名规则

### 标题必须包含的 3 个元素

1. 测试对象
2. 关键条件
3. 预期结果

缺少任一元素，标题质量不足。

### 推荐标题模板

1. 路径型：`[路径/对象] - [条件] - [结果]`
2. 生命周期型：`[动作对象] - [动作] - [结果]`
3. 展示型：`[页面/入口] - [能力点] - [结果]`
4. 资源约束型：`[资源条件] - [动作] - [结果]`
5. 故障恢复型：`[故障动作] - [观察对象] - [恢复结果]`
6. 接口型：`[资源] - [请求类型/字段场景] - [结果]`

### 推荐词汇

推荐：

1. 连通正常
2. 无法连通
3. 生效
4. 不生效
5. 展示正确
6. 数据正确
7. 筛选结果正确
8. 日志生成正确
9. 计数增加
10. 配置保持
11. 配置恢复
12. 切换成功
13. 回切成功
14. 报错
15. 提示正确
16. 禁用
17. 自动移除
18. 自动恢复
19. 归还额度
20. 不归还额度

避免：

1. 功能测试
2. 验证正常
3. 场景测试
4. 其他测试
5. 综合测试
6. 各种情况
7. 没问题
8. 功能可用
9. 界面没问题

## 前置条件规则

会影响结果的前置条件，尽量写进标题或规则描述：

1. 同节点 / 跨节点
2. 同子网 / 不同子网
3. 同 VPC / 跨 VPC
4. 已关联 / 未关联
5. 已绑定 / 未绑定
6. 开启 / 关闭
7. 存在 / 不存在
8. 单 IP / 多 IP
9. 主 / 备
10. 版本差异
11. 许可足够 / 不足
12. 故障中 / 恢复后

不必强行进标题的内容：

1. 集群具体 IP
2. 虚拟机名称
3. 安装包路径
4. 工具具体执行命令
5. 非关键环境准备步骤

## 验证层规则

生成 case 前，先决定验证点落在哪一层：

1. 数据平面 / 转发层：连通、不通、精准转发、不泛洪、路径切换后仍可达
2. 控制平面 / 配置层：配置生成正确、编辑后保持、升级后默认值正确、删除后清理正确
3. 展示层 / UI 层：列表展示正确、字段展示正确、过滤结果正确、tooltip 文案正确
4. 可观测层：日志生成、命中计数增加、告警触发、报表导出成功
5. 资源约束层：配额扣减、额度显示、报错阻止、额度归还
6. 恢复层：故障后切换、恢复后通信恢复、恢复后配置保持、恢复后无残留告警

默认规则：

1. 一条 case 只主打一层。
2. 如果需要跨层验证，按层拆条。

## suite 组织规则

### 推荐骨架

新 feature suite 默认按以下顺序组织：

1. 基础功能
2. 对象或参数差异
3. 编辑与状态切换
4. 异常与限制
5. 展示与日志
6. 故障与恢复
7. 资源与许可
8. 升级与兼容
9. 回归与收尾

核心 feature 通常至少要覆盖前 6 部分。

### 不同类型 feature 的推荐 section

网络路径类：

1. 基础路径
2. 变量路径
3. 对象差异
4. 失败路径
5. 编辑后验证
6. 故障恢复

展示类：

1. 页面入口
2. 数据正确性
3. 筛选与排序
4. 条件联动
5. 导出与告警
6. 异常与空态

运维类：

1. 部署
2. 关联
3. 编辑
4. 启停
5. 取关
6. 卸载
7. 残留清理

许可类：

1. 展示
2. 正好满足
3. 不足
4. 完全用尽
5. 释放归还
6. 报错文案

## 默认覆盖清单

如果没有特殊限制，新 feature 默认至少检查：

1. 主流程是否可执行
2. 关键对象差异是否覆盖
3. 关键路径差异是否覆盖
4. 成功与失败是否成对
5. 创建、编辑、删除或启停是否闭环
6. 是否至少覆盖一个非数据平面验证层
7. 是否考虑故障恢复或状态切换
8. 是否考虑升级兼容或存量配置
9. 是否考虑资源约束或禁用态
10. 是否避免一个 case 同时验证过多点

## SDN 特有补充规律

除了上面的通用规则，SDN 领域还有几个更强的习惯：

1. 先按功能树或网络路径把问题空间拆开。
2. 再按关键变量做矩阵，不漏同/跨节点、同/跨子网、协议、对象、主备、版本。
3. 标题尽量把前提和结果写明，让 case 自解释。
4. 不只测功能成功，还要测展示、日志、故障恢复、许可限制和升级兼容。

每写一个新的 SDN feature，建议最少补以下 10 类：

1. 主流程：创建、编辑、删除、启用、停用
2. 关键组合：节点位置、子网关系、VPC/VLAN、协议类型
3. 对象差异：标签、安全组、IP 组、Pod、网关、VDS
4. 正反结果：成功与失败成对覆盖
5. 生命周期：升级前后、切换前后、关联前后、日志开关前后
6. 故障恢复：节点、网口、服务、交换机、ARP/转发表
7. 展示观测：UI、日志、报表、筛选、告警
8. 资源限制：许可、配额、版本限制、禁用态
9. 兼容性：平台、架构、OS、旧版本升级路径
10. 收尾闭环：删除后归还、取消关联后状态、卸载后资源清理

## 反模式清单

1. 标题只有动作，没有结果。
2. 标题过于笼统，如“验证功能正常”。
3. 一个标题包含多个独立结果。
4. 把备注、讨论、推测写进标题。
5. 把环境搭建细节塞进标题。
6. 只测 UI，不测真实生效。
7. 只测成功，不测失败路径。
8. 只写 happy path，不写故障恢复。
9. 变量过多但不拆。
10. 同一 suite 命名风格混乱。

## 最简执行摘要

如果只保留最重要的规则，优先保留下面 12 条：

1. 先判定 feature 类型，再选维度。
2. 先拆能力，再拆变量，再写标题。
3. 一条 case 只验证一个核心结果。
4. 标题必须有对象、条件、结果。
5. 正向和反向场景默认成对。
6. 网络类优先拆路径、拓扑、对象、协议。
7. 展示类优先拆页面、数据、筛选、联动、导出。
8. 运维类优先拆部署、关联、启停、卸载、清理。
9. 故障类必须写恢复，不只写故障。
10. 许可类必须写额度、限制、报错、归还。
11. 升级类必须写存量、默认值、路径、回归。
12. 避免一个 case 同时覆盖多层结果。
