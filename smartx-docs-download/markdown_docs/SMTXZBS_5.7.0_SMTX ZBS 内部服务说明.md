# SMTXZBS/5.7.0/SMTX ZBS 内部服务说明
## 关于本文档

# 关于本文档

本文档介绍了 SMTX ZBS 的内部服务、内部服务占用端口和服务失效的影响。内部服务是指保证 SMTX ZBS（读作 SmartX ZBS）的存储和运维功能正常运行所需要的服务，包括块存储内部服务和文件存储内部服务。

---

## 文档更新信息

# 文档更新信息

- **2026-02-06：文档随 SMTX 文件存储 1.3.0 正式发布**

  - 删除了 kube-vip 服务。
  - 删除了 sfs-ganesha 服务占用的 2051 端口。
- **2025-11-24：配合 SMTX ZBS 5.7.0 正式发布**

  相较于 SMTX ZBS 5.6.1，本版本更新内容如下：

  - **块存储内部服务**：

    - 新增 `netguard` 服务，`zbs-chunkd` 服务新增相关端口
    - 删除 `amphi` 服务
  - **文件存储内部服务**：

    - 新增 `etcd-operator` 服务
    - 删除 `autoheal` 服务

# 块存储内部服务

块存储内部服务包含 SmartX 独立开发的服务和所依赖的第三方服务。

## 随机端口说明

在实际场景中，NFS 服务和客户端本地也可能使用一些随机端口，具体说明如下：

- NFS 服务除了使用 `2049` 和 `111` 两个固定端口外，还会启动其他进程（如 status、nlockmgr 等），这些进程的端口由 RPC 服务进行随机分配，每次启动 NFS 服务时使用的端口均不相同。

  若需查看 NFS 当前使用的所有端口，或查看某一未知的端口对应的进程，可采用如下方法：

  - 查看 NFS 当前使用的随机端口：在主机执行 `rpcinfo -p` 命令查看。
  - 查看某一端口对应的服务进程：在主机执行 `netstat -nlp` 命令查看。
- 客户端本地也可能启用一些随机端口对接块存储集群服务。若扫描到此类未知的本地端口，可在主机执行 `netstat -nlp` 命令，查看该端口对应的源端口或目的端口，并根据块存储内部服务使用的固定端口，确定本地端口对应的块存储集群服务。

## 块存储内部服务 > 服务失效影响

# 服务失效影响

## 块存储相关服务

| **服务名** | **在单节点失效影响** | **整体失效影响** |
| --- | --- | --- |
| zbs-metad | - 在 Leader 节点失效：存储服务短暂性不可访问（约 40s） - 在 Follower 节点失效：无影响 | 失效条件：少于半数 meta 节点存活  - 块存储集群存储不可用 - 所有虚拟机 IO 超时 - SMTX 虚拟机服务和存储服务相关操作不可用 |
| zbs-chunkd | - 失效节点的数据无法访问 - 触发数据恢复 - 虚拟机 IO 短暂性卡顿 - 本地 Hypervisor 上的磁盘存储服务将转由远端提供，可能出现短暂卡顿 | 条件：全部失效   - 存储不可用，IO 中断 |
| zbs-taskd | 失效节点无法执行长时运行任务 | 条件：全部失效 块存储集群失去长时任务调度及运行能力 |
| vipservice | 失效节点无法承载 VIP | 条件：全部失效 VIP 服务不可用 |
| zbs-iscsi-redirectord | - 失效节点无法建立新的 iSCSI 连接 - 失效节点已经建立的 iSCSI 连接不受影响 | 条件：全部失效   - 块存储集群无法建立新的 iSCSI 连接 - 块存储集群已经建立的 iSCSI 连接不受影响 |
| zbs-aurora-monitord | vhost 模式的虚拟机无法启动 | 条件：全部失效  - vhost 模式的虚拟机无法启动。 - 已经启动的虚拟机无法读写。 |
| zbs-inspector | 失效节点无法进行数据巡检 | 条件：全部失效 集群失去数据巡检能力。 |
| zbs-aurorad | 失效节点 IO 重定向失败。 | 条件：所有节点的 zbs-chunkd 服务和 zbs-aurorad 服务失效。 集群不可用，IO 中断。 |
| timemachine | 只有 Meta Leader 所在节点的 timemachine 可以对外提供服务，该节点 timemachine 失效后集群无法提供定时快照的服务，其余节点失效无影响。 | 条件：全部失效   - 定时快照功能失效 - 跨站点备份/回滚/克隆服务失效 |
| zbs-watchdogd | 若 Zookeeper leader 节点上的 zbs-watchdogd 服务异常，则当节点的操作系统异常时，Zookeeper 服务可能无法快速重新选举。 | - |

## 文件存储内部服务

# 文件存储内部服务

文件存储内部服务包含了 SmartX 独立开发的服务和所依赖的第三方服务。

SMTX ZBS 文件存储集群由三个以上的文件控制器组成，对外提供文件存储服务。文件控制器运行在 SMTX ZBS 块存储集群的物理节点上，根据运行的服务类型，文件控制器分为两种节点：主节点、存储节点。

- **存储节点**运行文件存储集群的存储服务。
- **主节点**除了存储服务外，还运行着文件存储集群的管控服务。

> **注意**：
>
> 文件存储集群的主节点和存储节点与块存储集群的主节点和存储节点并非一一对应。例如，块存储集群的主节点不一定是文件存储集群的主节点所在的物理节点。

## 文件存储内部服务 > 服务失效影响

# 服务失效影响

## 文件存储相关服务

| 服务名 | 在单个文件控制器失效影响 | 在所有文件控制器失效影响 |
| --- | --- | --- |
| sfs-agent | docker 尝试自动拉起，若失败，则该文件控制器将在 20s 后被重启，I/O 短暂中断。 | docker 尝试自动拉起，若失败，则所有文件控制器将被重启，I/O 短暂中断。 |
| sfs-meta | docker 尝试自动拉起，若失败  - shard follower：无影响 - shard leader：如果文件系统的 meta shard 的leader 挂载在此文件控制器上，则该文件系统的元数据无法访问 | 所有文件系统无法访问。 |
| sfs-data | docker 尝试自动拉起，若失败  - shard follower：无影响 - shard leader：如果文件系统的某个或者某些 data shard 的 leader 挂载在此文件控制器上，则该文件系统分布在这些 data shard 上的数据无法访问 | 所有文件系统的数据无法访问。 |
| sfs-ganesha | docker 尝试自动拉起，若失败，则文件接入 IP 迁移到其他文件控制器，通过该文件接入 IP 挂载的文件系统的 I/O 短暂中断。 | 所有文件系统无法访问。 |
| sfs-manager | docker 尝试自动拉起，若失败  - follower：无影响 - leader：触发 leader 切换，短暂地无法通过 CloudTower 操作文件系统 | 文件存储集群异常，无法进行文件系统生命周期管理。 |
| sfs-cloud-provider | docker 尝试自动拉起，若失败  - follower：无影响 - leader：触发 leader 切换，短暂地无法通过 CloudTower 操作文件系统 | 文件存储集群异常，无法进行文件系统生命周期管理。 |
| etcd-operator | docker 尝试自动拉起，若失败  - kube-etcd 正常：无影响 - kube-etcd 异常：无法重启该文件控制器上的 kube-etcd 服务 | docker 尝试自动拉起，若失败  - kube-etcd 正常：无影响 - kube-etcd 异常：无法重启所有文件控制器上的 kube-etcd 服务 |
