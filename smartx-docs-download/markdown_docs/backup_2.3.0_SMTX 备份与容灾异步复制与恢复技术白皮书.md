## 关于本文档

# 关于本文档

SMTX 备份与容灾是由北京志凌海纳科技股份有限公司（以下简称 “SmartX”）自主研发的数据保护产品，可为运行在 SMTX OS（ELF）集群的虚拟机和 SMTX ZBS 块存储集群的存储卷提供全面的数据保护和灾难恢复解决方案。

异步复制与恢复主要提供以下功能：

- **异步复制**

  将运行在源集群中的保护对象异步复制到目标集群中，并在目标集群创建对应的副本对象。在异步复制保护对象的过程中，每次复制操作都会生成一个恢复点。通过复制操作和周期性的异步复制计划，可以实现数据同步，确保数据的安全性和可用性。
- **故障转移**

  当保护对象发生故障时，可使用副本对象接替保护对象继续提供服务。
- **故障测试转移**

  在测试环境中模拟故障转移场景，通过创建测试虚拟机复现真实故障切换流程，在不影响实际保护对象的前提下，验证故障转移方案的可用性与可靠性。
- **故障恢复**

  保护对象在发生故障转移后，当其恢复正常时，将故障转移期间副本对象产生的数据增量传输回保护对象，并恢复至由保护对象提供服务。故障恢复可以确保故障转移期间产生的数据变化不会丢失，并确保保护对象在恢复后业务能够无缝切换回原始的生产环境，保障业务的连续性。
- **永久故障转移**

  在保护对象无法恢复的情况下，将保护对象永久切换到副本对象，并使用副本对象持续提供服务。

## 概述 > 应用场景

# 应用场景

异步复制与恢复可应用于以下场景：

- **灾难恢复**

  防止因自然或人为灾害、勒索软件攻击、恐怖事件等导致的大规模数据损坏和丢失。一旦灾难发生，可快速将业务切换至灾备集群，并迅速恢复业务运行。
- **灾备演练**

  通过故障转移测试进行灾备演练，模拟灾难发生后的快速恢复过程，确保灾备方案的有效性和可靠性，验证恢复点的可恢复性。
- **数据中心迁移**

  结合异步复制和故障转移功能，将业务系统和数据迁移到灾备中心，实现零恢复点目标（RPO）的业务切换。完成切换后，即可对源数据中心硬件实施物理搬迁，确保业务系统与底层硬件设施解耦。
- **数据复用**

  利用集群原生的快照重建功能实现数据复用。通过从副本对象的快照重建资源对象进行开发、测试、故障排除等活动，避免对生产环境造成任何影响。数据复用功能能够在不影响主要业务的前提下，为开发人员和测试团队提供高效且隔离的环境，加快开发和测试过程，同时确保生产环境的稳定性和安全性。

## 相关概念

# 相关概念

本章解释了手册中使用的专业术语。在阅读过程中，如果遇到不熟悉的概念，可在此快速查找其定义和详细解释。

**SMTX 备份与容灾**

SMTX 备份与容灾是面向 SmartX IT 基础设施的数据保护产品，可为运行在 SMTX 基础架构的工作负载提供备份、复制和灾难恢复解决方案。

**复制服务**

复制服务负责管理异步复制计划，并执行虚拟机和存储卷的异步复制、故障转移、故障转移测试、故障恢复和永久故障转移等任务调度。

根据关联的集群不同，复制服务分为目标端复制服务和源端复制服务。

- **目标端复制服务**

  与目标集群关联的复制服务，主要负责执行异步复制计划、故障转移、故障转移测试、管理复制资源，并在故障恢复时从目标集群读取数据传输给源端复制服务。目标端复制服务负责确保复制数据的同步性，保障副本虚拟机和原虚拟机，以及副本卷和原卷的数据一致性。
- **源端复制服务（Peer）**

  与源集群关联的复制服务，主要负责从源集群读取数据并传输到目标端复制服务，并在故障恢复时接收目标端复制服务传输的数据并将数据写入源集群。

**站点**

站点是指由一个 CloudTower 管理的一个故障域。一个站点可以包含一个或多个集群和数据中心。每个站点在地理位置、硬件设备、管理平台和软件服务等方面都完全隔离，且不共享任何资源。

站点可以分为以下两种类型：

- **源站点**：源站点通常是实际运行业务的站点。复制服务将源站点的虚拟机和存储卷数据复制到目标站点的副本对象，实现数据的同步。
- **目标站点**：目标站点通常是用于容灾的站点，用于存储从源站点复制过来的副本对象。

**站内复制**

源集群和目标集群位于同一站点。当原虚拟机或原卷不可用时，可切换使用站点内其他集群中的副本对象，实现集群级别的容灾。若管理站点的 CloudTower 发生故障，无法发起故障转移。

**跨站点复制**

源集群和目标集群位于不同站点。当源站点内的原虚拟机或原卷不可用时，无论管理源站点的 CloudTower 是否发生故障，均可切换至目标站点的副本对象，实现站点级别的容灾。

跨站点复制包括以下两种类型：

- **入站复制**：目标站点为当前站点，源站点为其他关联站点。
- **出站复制**：源站点为当前站点，目标站点为其他关联站点。

**源集群**

通常为生产集群，承载实际运行的业务应用和数据，是原虚拟机或原卷所属的集群，位于源站点。

**目标集群**

通常为用于容灾的集群，用于存储从源集群复制过来的副本对象，位于目标站点。

**保护对象**

作为复制数据源的原始对象，可以是 SMTX OS 集群上的虚拟机或 SMTX ZBS 块存储集群上的存储卷。

保护对象通常是运行的生产环境中的虚拟机或者存储卷，负责承载实际的业务应用和数据。

**副本对象**

通过异步复制计划创建的保护对象的完整副本，包含和保护对象相同的数据和配置。副本对象通常用于灾难恢复和容灾目的，在保护对象发生故障时，它可以快速接替保护对象，确保业务的连续性和可用性。副本对象通常存储在用于容灾的目标集群，并通过周期性的异步复制计划保持和保护对象的同步。

**异步复制计划**

异步复制计划用于规划和安排数据的复制操作。通过异步复制计划，用户可以定义复制对象、复制类型，复制站点、目标集群、复制服务、复制周期、复制窗口、保留策略、副本对象等信息。

**复制对象**

一个保护对象和其对应的副本对象称为一组复制对象。进行虚拟机复制时，一个原虚拟机及其副本虚拟机称为一组复制对象；进行卷复制时，一个原卷及其副本卷称为一组复制对象。

**复制周期**

异步复制计划自动执行的频率和时间，用于指定数据复制的时间间隔。较短的复制周期可以实现更小的 RPO（Recovery Point Objective，故障恢复点目标）。例如，如果复制周期设置为 15 分钟，那么在发生灾难性事件时，最多可能丢失 15 分钟的数据。

**复制窗口**

允许异步复制计划自动执行的时间范围。

**全量异步复制**

全量异步复制是一种数据复制方式，旨在复制保护对象的所有数据，包括操作系统、应用程序、文件和配置等。在全量异步复制过程中，保护对象的所有数据都会被复制到副本对象，从而确保副本对象与保护对象完全一致，提供可靠的灾难恢复能力。全量异步复制通常发生在初次复制时，用于建立保护对象的初始副本，并为后续的增量异步复制提供基准。

**增量异步复制**

增量异步复制是复制自上次全量或增量异步复制后发生变化的数据。在增量异步复制过程中，只有保护对象中自上次复制以来发生变化的数据会被传输到副本对象。这样可以大大减少数据传输的量和时间，提高复制的效率。增量异步复制是全量复制的补充，使副本对象能够持续地跟踪和同步保护对象的变化，保障数据的准确性和一致性。

**指纹**

在异步复制过程中，复制服务将对保护对象数据分块，并对每一数据块计算指纹。在进行增量异步复制时，会对数据块的指纹进行比对，若同一数据块的指纹对比上次复制时相同，复制服务判断该数据块没有发生变化，不再传输该数据块。

**恢复点**

恢复点是在每次为复制对象执行复异步制、故障转移（立即生成恢复点模式）、故障转移测试（立即生成恢复点模式）或故障恢复时生成的数据状态，以快照形式保存在复制对象上。恢复点记录了复制对象的数据和配置，反映了复制对象在执行上述操作时的状态。

恢复点可以使复制对象在需要时快速回滚到特定的数据状态，例如在灾难恢复场景中将副本对象恢复到某个先前的时间点，以恢复副本对象的正常运行。通过定期生成恢复点，可以确保副本对象的数据是最新的，保障业务的连续性和可用性。

**故障处理**

故障发生后可进行的恢复操作，包括故障转移、故障转移测试、故障恢复和永久故障转移。

- **故障转移**

  故障转移是一种应对保护对象不可用的操作。当保护对象不可用时，可以将副本对象回滚到与指定恢复点一致的状态，并切换至使用副本对象。通过故障转移，系统可以在短时间内将服务快速恢复到故障发生前的状态，保障业务的连续性和稳定性。
- **故障转移测试**

  故障转移测试是一种模拟容灾演练的操作，在不中断源端生产业务的前提下，通过生成测试虚拟机，在测试虚拟机验证恢复机制的有效性。演练过程中的增量数据不会同步回被保护的虚拟机。
- **故障恢复**

  故障恢复是指当发生故障的保护对象恢复正常使用后，副本对象将故障转移期间变化的数据回传，然后切换回使用保护对象。故障恢复旨在将复制对象之间的数据进行同步，确保保护对象可以在故障解决后继续提供服务。
- **永久故障转移**

  永久故障转移是当保护对象无法从故障中恢复时采取的措施。在保护对象发生永久性故障时，副本对象将正式接替保护对象的服务角色，以确保业务的持续运行和可用性。

**RPO** (Recovery Point Objective)

数据恢复点目标，主要指在故障转移场景中允许的数据损失的时间范围。RPO 以时间为度量单位，标示了从最近执行的复制时间点到发生故障时的时间跨度。较小的 RPO 值表示系统要求在发生故障时允许的数据损失较少。较大的 RPO 值意味着在故障发生时系统可以容忍较长时间范围内的数据丢失。

**RTO** (Recovery Time Objective)

数据恢复时间目标。主要指业务系统从崩溃到恢复运行所能容忍的时间。较短的 RTO 值表示业务系统要求在发生故障时能够迅速恢复，较长的 RTO 值则意味着业务系统可以容忍更长的恢复时间。

**VPC**（Virtual Private Cloud）

虚拟专有云，是一套为云资源构建的逻辑隔离的网络环境，用户可以在 VPC 中自主设置 VPC 子网、IP 地址、路由表、浮动 IP、网关服务和安全服务等，以实现网络隔离和安全控制。副本虚拟机和原虚拟机可以部署在相同或不同的 VPC 中，并支持灵活的网络映射。测试虚拟机需部署在和原虚拟机不同的 VPC 中。

**VPC 子网**

VPC 子网是组成 VPC 的基础网络模块，用于划分 VPC 的 IP 地址空间。位于同一 VPC 子网的虚拟机默认可以直接通信，跨 VPC 子网虚拟机在在路由可达的情况下也可以互相通信。

**浮动 IP**

浮动 IP 是一种网络地址转换（NAT）机制，通过一对一映射的方式，将外部子网中的 IP 地址动态绑定到 VPC 内部的虚拟机上。这使得外部网络可以主动访问 VPC 内的虚拟机。浮动 IP 为 VPC 内的虚拟机提供了灵活的外部访问能力，同时也保证了网络安全性。

**网络映射**

旨在解决原虚拟机和副本虚拟机/测试虚拟机之间所处的网络环境差异问题的一项配置规则。

由于副本虚拟机/测试虚拟机和原虚拟机往往处于不同的网络环境，若副本虚拟机/测试虚拟机完全复制原虚拟机的网络配置，可能无法在目标集群环境正常配置或连通网络。通过配置网络映射规则，可将原虚拟机所属的虚拟机网络映射到目标集群中对应的虚拟机网络，从而保证副本虚拟机/测试虚拟机能够正常与网络相连。

**IP 映射**

旨在解决原虚拟机和副本虚拟机/测试虚拟机在不同网络环境下的 IP 地址不匹配的问题的一项配置规则。

由于副本虚拟机/测试虚拟机和原虚拟机往往处于不同的网络环境，若副本虚拟机/测试虚拟机完全复制原虚拟机的 IP 地址，可能无法在目标集群正常联通网络。通过配置 IP 映射规则，在故障转移和故障转移测试时，复制服务可以自动为副本虚拟机/测试虚拟机设置正确的 IP 地址，确保副本虚拟机/测试虚拟机可以按照预期的网络配置提供服务，无需手动去更新虚拟机 IP。

**ELF**

ELF 平台即指 SMTX OS 原生虚拟化计算平台，是当 SMTX OS 以超融合形态直接部署在 x86\_64/AArch64 架构的服务器上时，该超融合软件中基于 KVM 的虚拟化引擎组件。ELF 除提供虚拟机生命周期管理、电源操作、高可用、冷热迁移等基本功能外，通过与存储组件 ZBS 相结合，还能提供秒级快照、模板和克隆等高级的虚拟机服务。

ELF 简单易用，对 SMTX OS 超融合集群的所有操作均可通过 CloudTower 完成。

**CloudTower**

SmartX 多集群资源集中管理平台，旨在管理站点内数据中心及跨数据中心的多个 SmartX 集群和系统服务，并提供完整和标准的 RESTful API 和多种语言的 SDK。

## 设计方案 > 方案概要

# 方案概要

SMTX 备份与容灾的设计方案采用了 Kubernetes Operator 模式，遵循声明式编程（declarative programming）范式，只需对复制相关资源设定期望的状态，而无需描述其实现流程，系统即能自行确定需要执行的操作以达到用户期望的状态。

Kubernetes Operator 模式包含了资源（Resource）和控制器（Controller）两个基本概念：

- **资源**

  Kubernetes 将被管理的对象抽象为“资源”。资源包含两种状态：期望状态和实际状态。期望状态通常由用户进行定义，实际状态通常由控制器负责更新和维护。

  Kubernetes 系统通过控制器监视资源对象，并执行业务逻辑，让资源的实际状态最终达到期望状态，从而达成用户的预期结果。

  Kubernetes 提供了多种默认的资源类型，也支持用户创建自定义资源（Custom Resource，简称 CR）。

  异步复制与恢复包含以下自定义资源：

  | 英文名称 | 中文名称 | 含义 |
  | --- | --- | --- |
  | ReplicationGroup | 复制组 | 一个保护对象对应一个 ReplicationGroup，一个 ReplicationGroup 包含一个 ReplicationSlice。 |
  | ReplicationSlice | 复制恢复点链 | 代表一个保护对象的复制恢复点链。一个复制恢复点链由一个全量恢复点和多个增量恢复点组成。 |
  | ReplicationRestorePoint | 复制恢复点 | 一个可以用于故障转移的基本对象，记录了保护对象的数据和配置，反映了保护对象在复制时的状态。 |
  | MultiStorageConsistentSnapshot | 数据一致快照组 | 由多个在存储上数据一致的快照组成的组。 |
  | ReplicationPlan | 异步复制计划 | 包含保护对象、复制周期、保留策略和副本对象设置。 |
  | ReplicationExecution | 异步复制执行 | 手动或自动触发的一次复制计划的执行操作，包含多个 ReplicationJob。 |
  | ReplicationJob | 异步复制任务 | 代表一个保护对象的复制任务。 |
  | ReplicaGroup | 复制对象副本 | 代表一个故障转移/故障转移测试中的副本对象。 |
  | FailoverExecution | 故障转移/故障转移测试执行 | 手动触发的一次故障转移/故障转移测试执行操作，包含多个 ReplicationFailoverJob、BootJob 和 ReIPJob。 |
  | ReplicationFailoverJob | 复制组异步复制故障转移/故障转移测试任务 | 代表一个保护对象的异步复制/故障转移/故障转移测试任务。 |
  | BootJob | 开机任务 | 代表一个或多个虚拟机的开机任务。 |
  | ReIPJob | IP 映射任务 | 代表一个或多个虚拟机的 IP 映射任务。 |
  | FailbackExecution | 故障恢复执行 | 手动触发的一次故障恢复执行操作，包含多个 FailbackJob，保护对象为虚拟机时还包含 BootJob 和 ReIPJob。 |
  | FailbackJob | 故障恢复任务 | 代表一个副本对象的故障恢复任务。 |
  | FailbackReplicationJob | 恢复回传任务 | 代表一个副本对象的恢复回传任务，是故障恢复任务的一部分。 |
  | PermanentFailoverJob | 永久故障转移任务 | 代表一个副本对象的永久故障转移任务。 |
  | Peer | 对等复制服务 | 即源端复制服务，用于为目标端复制服务提供代理。 |
- **控制器**

  针对每一种资源，复制服务均有与之对应的控制器。控制器会监听资源的创建、编辑、删除等事件，并且根据资源的期望状态和实际状态，触发对应的执行逻辑，直至资源达到期望状态。如果对应资源的相关操作失败，且在有限的重试后无法达到期望状态，则对应的资源对象最终会处于失败或异常状态，且控制器不会再重试操作。

  例如，用户通过界面设置异步复制计划的执行周期后，系统会更新 ReplicationPlan 资源的期望状态（即执行异步复制计划的时间点）。负责 ReplicationPlan 资源的控制器，将根据 ReplicationPlan 资源的期望状态，自动在设置的复制时间点执行异步复制操作，从而使复制任务按照设定的周期执行。

## 设计方案 > 方案特点

# 方案特点

异步复制的设计方案具有如下特点：

- **无代理**

  无需在保护对象中安装代理，减少管理和运维负担，且异步复制过程不影响保护对象性能。
- **复制数据量优化**

  通过增量异步复制、跳过未分配的数据和指纹去重的方式，减少异步复制所需数据量，节省带宽和存储空间。
- **一致性保证**

  可借助虚拟机工具创建文件系统一致性的虚拟机快照，避免出现缓存数据和磁盘数据不一致，进而导致虚拟机故障转移后部分文件可能损毁的情况。
- **高可用**

  依赖于 ELF 的高可用功能，当复制服务所在节点发生故障时，可自动在其他节点继续运行并提供服务。

## 架构与组件 > CloudTower

# CloudTower

SMTX 备份与容灾使用 CloudTower 作为管理平台帮助用户完成复制和故障处理的相关操作。CloudTower 是 SmartX 的多集群管理平台，可以帮助用户高效统一地管理多个数据中心、多个集群内的虚拟机和存储卷，实现分布式云架构的构建。

CloudTower 为 SMTX 备份与容灾提供操作界面，是用户进行相关操作的请求入口。用户可以在操作界面进行管理备份与容灾安装包、管理备份与容灾软件许可、安装、升级复制服务、管理复制计划、管理关联站点等操作。

- **设置**

  用户在操作界面执行复制相关操作时，CloudTower UI 会将操作请求先发送至 CloudTower Server，然后由 CloudTower Server 将请求发送至目标端复制服务 ，由目标端复制服务执行具体的操作。在执行复制或故障处理操作时，目标端复制服务也会向 CloudTower Server 请求 GraphQL API 以查询要复制或故障处理的保护对象配置信息和创建快照等。CloudTower Server 和目标端复制服务两者可相互调用。
- **许可管理**

  CloudTower Server 上安装了 License Server，用于管理备份与容灾软件许可。License Server 会统计 CloudTower 中的服务已使用的许可单元，并对操作请求进行校验，当许可过期或者相关操作会超出许可限额时，会对请求进行限制。
- **运维管理**

  用户在操作界面执行上传复制服务安装包、安装复制服务、升级复制服务或是卸载复制服务等运维操作时，CloudTower UI 会将操作请求先发送至 CloudTower Server，然后由 CloudTower Server 中的 CloudTower Application 管理组件创建、部署或删除复制服务虚拟机。
- **站点管理**

  用户在操作界面上可以关联其他站点，实现与外部服务的互联互通。关联站点时，CloudTower Server 会进行鉴权，鉴权通过后会生成其他站点的 API Key，并将站点 API Key 注册在复制服务中。复制服务使用 API Key 与其他站点进行通信。

## 架构与组件 > 复制服务

# 复制服务

复制服务是异步复制的核心组件，以虚拟机的形式运行在集群中。

关联源集群和目标集群的复制服务在整个异步复制和故障处理过程中分别发挥着不同的作用，所使用的组件也不相同，具体说明如下。

---

## 架构与组件 > 复制服务 > 目标端复制服务

# 目标端复制服务

目标端复制服务负责执行异步复制和故障处理任务，管理关联的异步复制计划和复制对象等资源，并且与 CloudTower、ZBS 和源端复制服务进行数据的查询、读取和写入等交互操作。

在目标端复制服务虚拟机中，以容器形式运行了三个核心的服务模块：**kube-apiserver**、**ETCD** 和 **controller-manager**。

## kube-apiserver

kube-apiserver 是目标端复制服务的数据总线和数据中心，是目标端复制服务和其他组件进行数据交互和通信的枢纽。kube-apiserver 提供了管理元数据的 HTTP REST API 接口。

在创建异步复制计划、执行异步复制和故障处理任务等操作时，CloudTower Server 和 controller-manager 会通过 kube-apiserver 查询、创建、修改或删除资源对象。kube-apiserver 会将获取的目标端复制服务的所有元数据信息持久化存储至 ETCD。

## ETCD

ETCD 是一个分布式键值存储软件，也是 Kubernetes 默认的数据存储引擎。ETCD 将 kube-apiserver 获取的目标端复制服务的所有元数据信息，如复制计划、复制任务、恢复点等，进行持久化存储。

## controller-manager

controller-manager 是目标端复制服务负责异步复制计划调度、执行异步复制和故障处理逻辑的模块。

controller-manager 包含了多个控制器，每个控制器负责对应的一类资源的调度、管理和任务执行工作。具体包括：

- 依照设定的复制周期和复制窗口，定时触发异步复制计划的执行；
- 管理源端复制服务，定时探测与源端复制服务之间的网络联通性；
- 依照设定的恢复点保留策略，定时回收过期的恢复点；
- 执行异步复制任务，在任务成功或失败时清理临时资源；
- 执行故障转移任务，在任务失败时将副本对象回滚到转移前的状态；
- 执行故障转移测试任务，在任务失败时清理已经生成的测试虚拟机；
- 执行故障恢复任务，在任务成功或失败时清理临时资源；
- 执行永久故障转移任务，在任务成功时清理资源。

在执行复制和故障处理任务时，controller-manager 会与其他组件进行交互，包括：

- 监听 kube-apiserver 中资源对象的创建、编辑、删除事件，在执行逻辑后更新 CR 对象的状态等；
- 访问 CloudTower，查询并获取复制对象元数据信息和快照信息，创建和删除复制对象快照等；
- 访问 ZBS，通过 RPC 协议读取或写入卷数据；
- 访问源端复制服务，通过 RPC 协议发送 ZBS 代理请求。

---

## 架构与组件 > 复制服务 > 源端复制服务

# 源端复制服务

源端复制服务会在复制过程中通过复制网络与目标端复制服务建立连接，形成对等复制服务的关系。在复制或故障处理时，如果目标复制服务与源端复制服务源端集群不连通，源端复制服务可以成为目标端复制服务和源端集群之间的中介，用于代理 ZBS 请求。

---

## 架构与组件 > ZBS

# ZBS

ZBS 是 SMTX OS 集群和 SMTX ZBS 块存储集群的分布式块存储组件，为复制对象提供存储服务。复制对象在数据层面对应 ZBS 中一个或若干个卷。复制服务对保护对象的异步复制，即是对卷做异步复制。

复制服务对保护对象进行异步复制时，目标端复制服务上的 controller-manager 会通过 CloudTower 创建快照，然后利用源端复制服务代理查询 ZBS 的存储卷元数据信息以及读取保护对象快照数据。最后通过 RPC 协议将数据写入到目标集群 ZBS 中，完成数据的复制。

当需要对保护对象进行故障恢复时，目标端复制服务的 controller-manager 会通过 CloudTower 创建副本对象快照，然后通过 RPC 协议从 ZBS 中读取副本对象快照数据。最后利用源端复制服务 Agent 将数据写入到源集群 ZBS 中，完成数据的回传。

---

## 数据路径 > 复制数据路径

# 复制数据路径

将保护对象异步复制到副本对象时，涉及数据的复制和配置同步，此过程一般涉及以下步骤：

1. **构建数据链路**

   源集群和目标集群之间通过源端复制服务与目标端复制服务之间的复制网络构建数据链路，用于保护对象向副本对象传输数据。

   ![](https://cdn.smartx.com/internal-docs/assets/90f7479a/replication_whitepaper_03.png)

   所有的复制服务都会向 CloudTower 注册，由 CloudTower 维护各个服务的连接地址。用户可以在 CloudTower UI 界面上将复制服务与集群进行关联，当用户对源集群进行复制时，CloudTower 会将该集群所关联的复制服务作为对端复制服务传递给目标端复制服务。目标端复制服务会通过定时探测的方式检查与源端复制服务之间的网络联通性，在复制时，目标端服务将会和源端复制服务构建一条可用的链路。若源集群关联了多个复制服务，则目标端复制服务会随机选择一个可连通的源端复制服务作为对等复制服务，并建立数据链路，目标复制服务可以通过数据链路访问源集群的 ZBS。
2. **获取保护对象的当前状态**

   目标端复制服务向 CloudTower 发送请求，对保护对象进行快照，并获得保护对象快照的元数据。元数据包含了保护对象的配置信息以及 ZBS 卷快照，记录了保护对象在当前时刻的状态。
3. **将保护对象的配置同步给副本对象**

   在开始复制保护对象数据之前，目标端复制服务会先向 CloudTower 发出请求，将保护对象的配置信息复制给副本对象，确保副本对象与保护对象配置相同。

   - 若为首次复制，目标端服务会先在目标端集群创建副本对象，副本对象的配置和保护对象相同。
   - 若为增量复制，目标端服务会直接将保护对象的配置信息同步给副本对象。
4. **将保护对象的数据复制给副本对象**

   完成复制对象配置信息的同步后，保护对象的快照数据将同步给副本对象。

   - 首次复制将使用全量异步复制，将保护对象的所有快照数据传输给副本对象。
   - 后续复制将使用增量异步复制，并结合数据块指纹比对的方式，识别出上次复制以来实际发生过更新的数据，并将这些数据传输给副本对象。
5. **在副本对象创建恢复点**

   在完成保护对象数据的异步复制后，将会对副本对象创建快照作为异步复制的恢复点。后续当保护对象发生故障后，可执行故障转移任务，将副本对象从指定恢复点状态启动以接替保护对象继续提供服务。

---

## 数据路径 > 故障恢复数据路径

# 故障恢复数据路径

当保护对象从故障中恢复后，故障恢复功能可将业务运行从副本对象切换回保护对象，在此过程中，副本对象需将保护对象故障期间发生变更的配置信息和数据回传给保护对象，完成数据的同步。

1. **构建数据链路**

   源集群和目标集群之间通过源端复制服务与目标端复制服务之间的复制网络构建数据链路，用于副本对象向保护对象传输数据。

   ![](https://cdn.smartx.com/internal-docs/assets/90f7479a/replication_whitepaper_03.png)

   所有的复制服务都会向 CloudTower 注册，由 CloudTower 维护各个服务的连接地址。用户可以在 CloudTower UI 界面上将复制服务与集群进行关联，当进行故障恢复时，CloudTower 会将源集群所关联的复制服务作为对端复制服务传递给目标端复制服务。目标端复制服务会通过定时探测的方式检查与源端复制服务之间的网络联通性，在故障恢复时，目标端服务将会和源端复制服务构建一条可用的链路。若源集群关联了多个复制服务，则目标端复制服务会随机选择一个可连通的源端复制服务作为对等复制服务，并建立数据链路，目标复制服务可以通过数据链路访问源集群的 ZBS。
2. **获取副本对象的当前状态**

   目标端复制服务向 CloudTower 发送请求，对副本对象进行快照，并获得保护对象快照的元数据。元数据包含了副本对象的配置信息以及 ZBS 卷快照，记录了副本对象在当前时刻的状态。
3. **将副本对象的配置同步给保护对象**

   由于在故障转移期间，副本对象在接替保护对象提供服务的过程中可能涉及到配置信息的更新，因此，为确保在故障恢复后保护对象能够正常提供服务，目标端复制服务会向 CloudTower 发出请求，将副本对象的配置同步给保护对象。
4. **将副本对象的数据复制给保护对象**

   完成保护对象配置信息的同步后，副本对象的快照数据将传输给保护对象。目标端复制服务将通过指纹对比的方式，识别出副本对象相对于保护对象增量数据，并将数据传输给保护对象。
5. **在副本对象创建恢复点**

   在将副本对象的数据传输给保护对象后，目标端复制服务将会对副本对象创建快照作为复制的恢复点，并作为后续保护对象数据的复制基准。若保护对象再次发生故障，依然可执行故障转移任务，将副本对象从指定恢复点状态启动以接替保护对象继续提供服务。

## 工作执行流程 > 基本执行逻辑

# 基本执行逻辑

复制服务的 controller-manager 模块中包含了一组控制器，用于控制每种资源的状态。当用户对某项资源执行操作后，可视为调整了该资源的期望状态。控制器在观察到用户的操作后，会根据资源当前的实际状态和期望状态，不断执行调谐（Reconcile）逻辑，使得资源的实际状态最终达到期望状态。

复制服务的基本执行逻辑如下：

1. 用户在 CloudTower UI 上进行某项操作后，CloudTower UI 将操作请求发送至 CloudTower Server；
2. CloudTower Server 将操作请求发送至 kube-apiserver，由 kube-apiserver 对相应的资源对象进行创建、编辑和删除操作；
3. controller-manager 中相应的控制器观察到 kube-apiserver 中增加、修改、删除事件的发生，尝试查询最新的资源对象，进行相应的逻辑处理，并更新资源对象的实际状态；
4. CloudTower Server 观察最新的资源对象状态，并在 CloudTower UI 界面上展示。

![](https://cdn.smartx.com/internal-docs/assets/90f7479a/replication_whitepaper_04.png)

下面以异步复制计划的创建过程为例说明复制服务基本的执行逻辑，编辑和删除操作的逻辑也基本相同。

1. 用户在 CloudTower UI 执行创建异步复制计划操作后，CloudTower UI 将操作请求发送至 CloudTower Server；
2. CloudTower Server 将该操作请求发送至 Replication Service 的 kube-apiserver 模块，kube-apiserver 将创建一个 ReplicationPlan 资源对象；
3. controller-manager 模块的 ReplicationPlan 控制器观察到 kube-apiserver 创建了新的 ReplicationPlan 后，将根据用户的设置，为每个待复制的保护对象创建一个 ReplicationGroup 资源对象，同时计算异步复制任务的下一次执行时间，并在复制触发时间点创建一个 ReplicationExecution 资源对象，触发 ReplicationExecution 控制器执行相关复制操作；
4. CloudTower Server 观察到 ReplicationGroup 的创建后，会在复制计划详情页面展示复制对象，并且根据 ReplicationExecution 的状态，展示异步复制任务的执行结果。

## 工作执行流程 > 异步复制任务的执行流程

# 异步复制任务的执行流程

创建异步复制计划后，系统会根据设定的复制周期自动执行复制任务，用户也可选择手动执行，两种执行方式的内部处理流程基本相同。

在跨站点容灾场景中，手动执行异步复制任务可以在源站点或目标站点发起。执行任务时，如果复制服务需要操作保护对象或副本对象，会向对象所属的站点 CloudTower 发送请求。

![](https://cdn.smartx.com/internal-docs/assets/90f7479a/replication_whitepaper_05.png)

如下步骤以手动执行虚拟机的异步复制计划为例：

1. 用户在 CloudTower UI 界面上手动执行一个异步复制计划，前端将向 CloudTower Server 发送创建 ReplicationExecution 资源对象的请求；
2. CloudTower Server 向目标端复制服务下发创建 ReplicationExecution 的请求；
3. controller-manager 中的 ReplicationExecution 控制器在观察到 ReplicationExecution 的创建后，将会创建一个或多个 ReplicationJob 资源对象。ReplicationJob 控制器观察到 ReplicationJob 创建后，会通过 CloudTower 的 API 接口，为待复制的虚拟机创建虚拟机快照，该虚拟机快照会作为本次异步复制的数据源和下一次增量异步复制的对比源；
4. ReplicationJob 控制器读取虚拟机快照数据中的虚拟机配置信息，通过 CloudTower 的 API 接口，创建或者同步副本虚拟机使其配置与虚拟机快照相同，并根据 IP 映射规则，对 VPC 网卡执行 IP 映射操作；
5. ReplicationJob 控制器读取虚拟机快照数据，获取该虚拟机每块虚拟盘对应的 ZBS Snapshot 位置，并通过源端复制服务 Agent 与对应 ZBS Server 建立连接；
6. ReplicationJob 通过源端复制服务开始读取源端 ZBS Snapshot 的数据；
7. 目标端复制服务不断将读取的 ZBS Snapshot 流式写入到目标端 ZBS Server 中，直到完成所有虚拟盘的复制；
8. 根据异步复制任务的类型和结果，ReplicationJob 控制器会通过 CloudTower 的 API 接口删除虚拟机快照：

   - 如果当前异步复制成功，上次成功的异步复制任务在原虚拟机上保留的虚拟机快照将会过期，后续将会自动清理。而本次复制任务第 3 步中创建的虚拟机快照会在下一次异步复制任务成功后过期
   - 如果当前复制失败，则会删除本次复制任务第 3 步中在原虚拟机上创建的虚拟机快照。

存储卷与虚拟机的复制流程类似。

## 工作执行流程 > 故障转移的执行流程

# 故障转移的执行流程

用户可以通过选择已有恢复点或者立即生成恢复点对复制对象进行故障转移，两种模式的内部处理流程基本相同。

在跨站点容灾场景中，故障转移任务只能在目标站点发起。执行任务时，如果复制服务需要操作保护对象或副本对象，会向对象所属的站点 CloudTower 发送请求。

![](https://cdn.smartx.com/internal-docs/assets/90f7479a/replication_whitepaper_06.png)

如下为虚拟机故障转移的流程：

1. 用户在 CloudTower UI 界面创建故障转移任务后，前端将向 CloudTower Server 发送创建 FailoverExecution 资源对象的请求；
2. CloudTower Server 向目标端复制服务下发创建 FailoverExecution 的请求；
3. controller-manager 中的 FailoverExecution 控制器在观察到 FailoverExecution 的创建后，将会创建一个或多个 FailoverJob 资源对象。FailoverJob 控制器观察到 FailoverJob 创建后，如果以“立即生成恢复点”模式进行故障转移，将会创建 ReplicationJob 进行异步复制；如果以“选择已有恢复点”模式进行故障转移，会通过 CloudTower 的 API 接口，将副本虚拟机回滚到指定恢复点状态；
4. FailoverJob 将根据当前异步复制计划中网络/IP 映射规则对副本虚拟机进行网络映射和 VPC 网卡的 IP 映射，并根据用户选项进行 VPC 浮动 IP 设置；
5. FailoverExecution 控制器观察到 FailoverJob 完成后，将会创建一个或多个 BootJob 资源对象。BootJob 控制器观察到 BootJob 创建后，在等待指定的启动延迟后，会通过 CloudTower 的 API 接口，将副本虚拟机开机；
6. 若用户在创建故障转移任务时，在异步复制计划中设置了 IP 映射规则，且指定恢复点中虚拟机工具为运行中状态，对应的副本虚拟机才会创建 ReIPJob。ReIPJob 控制器观察到 ReIPJob 创建后，将会监控副本虚拟机的开机。在开机成功后，会通过 CloudTower 的 API 接口，将根据 IP 映射规则，将副本虚拟机 VLAN 网卡的 IP 地址更改为映射后的 IP。

存储卷故障转移与虚拟机的故障转移执行流程的 1～ 3 步骤类似。

## 工作执行流程 > 故障恢复的执行流程

# 故障恢复的执行流程

用户可以对故障转移中的复制对象进行故障恢复。

在跨站点容灾场景中，故障转移任务只能在目标站点发起。执行任务时，如果复制服务需要操作保护对象或副本对象，会向对象所属的站点 CloudTower 发送请求。

![](https://cdn.smartx.com/internal-docs/assets/90f7479a/replication_whitepaper_07.png)

故障转移虚拟机的流程如下，类似于异步复制与故障转移虚拟机流程的结合：

1. 用户在 CloudTower UI 界面上创建故障恢复任务后，前端将向 CloudTower Server 发送创建 FailbackExecution 资源对象的请求；
2. CloudTower Server 向目标端复制服务下发创建 FailbackExecution 的请求；
3. controller-manager 中的 FailbackExecution 控制器在观察到 FailbackExecution 的创建后，将会创建一个或多个 FailbackJob 资源对象。FailbackJob 控制器观察到 FailbackJob 创建后，会创建 FailbackReplicationJob 进行数据回传。FailbackReplicationJob 控制器观察到 FailbackReplicationJob 创建后，通过 CloudTower 的 API 接口，为故障恢复的副本虚拟机创建虚拟机快照；
4. FailbackReplicationJob 控制器读取虚拟机快照中的虚拟机配置信息，通过 CloudTower 的 API 接口，同步原虚拟机使其配置与虚拟机快照相同；
5. FailbackReplicationJob 控制器解析虚拟机快照，获取该虚拟机每块虚拟盘对应的 ZBS Snapshot 位置，并和对应 ZBS 建立连接，开始读取目标端 ZBS Snapshot 的数据；
6. FailbackReplicationJob 通过源端复制服务 Agent 与对应的源端 ZBS 建立连接；
7. 复制服务不断将读取的 ZBS Snapshot 流式写入到源端 ZBS 中，直到完成所有虚拟盘的数据回传；
8. FailbackJob 控制器将根据恢复点中的记录，将原虚拟机网卡配置恢复，并根据用户选项进行 VPC 浮动 IP 设置；
9. FailbackExecution 控制器观察到 FailbackJob 完成后，将会创建一个或多个 BootJob 资源对象。BootJob 控制器观察到 BootJob 创建后，在等待指定的启动延迟后，会通过 CloudTower 的 API 接口，将原虚拟机开机；
10. 若用户在创建故障转移任务时，指定恢复点中虚拟机工具为运行中状态，对应的原虚拟机才会创建 ReIPJob。ReIPJob 控制器观察到 ReIPJob 创建后，将会监控原虚拟机的开机。在开机成功后，会通过 CloudTower 的 API 接口，对原虚拟机进行 VLAN IP 配置。

存储卷与虚拟机的故障恢复执行流程的 1～ 7 步骤类似。

## 高级特性 > 复制数据量优化

# 复制数据量优化

复制服务采用如下策略组合来减少异步复制的数据量：

- **增量异步复制**

  源集群和目标集群通常运行在两个距离较远的站点之间，它们之间的带宽通常较小。为了尽可能充分地利用带宽，减少传输时间，如果源集群和目标集群曾经完成过一次异步复制任务，后续任务执行时，都会采用增量异步复制的方式来降低传输的数据量。

  在数据传输期间，目标端复制服务会对复制数据分块计算指纹，指纹将作为元数据存储在目标端复制服务中。进行增量异步复制时，目标端复制服务将会查询之前复制链中所有的指纹数据，并将上一次复制时数据块的指纹与数据块当前计算出的指纹进行比较。若发现数据块的指纹相同，说明该数据块实际没有发生更新，复制时将跳过这些数据块，这一机制确保在执行复制任务时，只传输和保存实际发生变化的增量数据，从而提高复制效率。
- **跳过未分配的数据块**

  当存储卷使用精简置备的存储策略时，复制服务会通过查询 ZBS 的存储卷元数据信息确认是否存在未分配的数据块，并在复制时跳过所有未实际分配的数据块，以减小复制文件的大小。

## 高级特性 > 一致性保证

# 一致性保证

若待复制的保护对象为存储卷或运行中且未安装虚拟机工具的虚拟机，SMTX 备份与容灾的复制服务将为保护对象创建崩溃一致性快照。此类快照仅记录磁盘数据，不会强制将缓存数据刷写到磁盘。因此，若保护对象存在缓存数据尚未全部写入磁盘的情况，创建的快照可能会出现缓存数据和磁盘数据不一致，进而可能导致保护对象恢复后出现部分文件损毁的情况。

若待复制的保护对象为处于运行中状态且安装了虚拟机工具的虚拟机，在执行复制时，复制服务将借助虚拟机工具创建文件系统一致性的虚拟机快照，保证缓存数据和磁盘数据的一致性，从而确保故障转移后的虚拟机可以正常工作。因此，建议用户为待异步复制的虚拟机安装虚拟机工具。

在不同操作系统中，通过虚拟机工具创建文件系统一致性快照的原理如下：

- **Linux 虚拟机**

  Linux 使用 FIFREEZE 来冻结文件系统，首先阻塞文件系统的写操作，然后 sync 文件系统的 inodes 信息和脏页。待文件系统冻结后，调用 ZBS 接口创建卷快照，创建成功后再解冻文件系统。
- **Windows 虚拟机**

  Windows 操作系统借助卷影复制服务（VSS）创建文件系统一致性快照，具体实现原理可参考 Windows [官方文档](https://docs.microsoft.com/zh-cn/windows-server/storage/file-server/volume-shadow-copy-service)。

## 高级特性 > 高可用

# 高可用

复制服务以虚拟机的形式运行在 SMTX OS（ELF）集群或 STMTX ZBS 块存储集群中，因此，复制服务具有 ELF 虚拟机的高可用特性。当复制服务虚拟机所在的主机宕机时，ELF 会在集群内另一台健康的主机上重新创建一台复制服务虚拟机，以保证复制服务可以继续运行并提供服务。

复制服务发生 HA 时如果存在正在执行的复制、故障转移、故障恢复和永久故障转移任务，将在虚拟机 HA 后重新执行。
