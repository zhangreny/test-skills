# SMTXOS/6.3.0/SMTX OS 故障场景说明
## 关于本文档

# 关于本文档

本文档介绍了 SMTX OS（读作 SmartX OS）集群在实际运行过程中可能遇到的故障场景和故障影响，以及 SMTX OS 在发生故障时的处理机制。

阅读本文档需了解 SMTX OS 超融合软件，了解虚拟化、软件定义网络/存储、分布式存储等相关技术背景知识，并具备丰富的数据中心操作经验。

## 物理盘故障

# 物理盘故障

本章节描述了物理盘损坏或者不可访问后的影响以及 SMTX OS 的处理机制。

---

## 物理盘故障 > 物理盘分区类型

# 物理盘分区类型

SMTX OS 集群的物理盘包含如下类型的分区：

- **操作系统分区**

  操作系统分区用于部署 SMTX OS 系统。它包含了操作系统所需的文件和组件，确保系统能够正常运行。
- **元数据分区**

  元数据分区承载着集群中关键的 ZooKeeper 和 Meta 数据。这些数据存储了有关集群配置、状态和元信息，用于保障集群的正常运行。
- **Journal 分区**

  Journal 分区是一个日志分区，用于记录物理盘上的写操作日志。当对存储进行写操作时，操作日志会被写入 Journal 分区，这样可以在发生异常情况时进行数据恢复和校验。
- **Cache 分区**

  Cache 分区是一个缓存分区，用于提高系统的性能。在不同介质的存储下，当有空间可用或者读取操作命中缓存时，I/O 操作会优先使用 Cache 分区，这样可以加快数据访问速度，提高系统的响应性能。在单一存储介质下没有读缓存。在分层模式中，Cache 分区还负责持久化性能层数据。
- **数据分区**

  数据分区是一个用于持久化存储的分区。在分层模式中，相对于 Cache 分区，数据分区的速率较低但容量较大，数据分区负责持久化容量层数据。在不分层模式中，数据分区性能较好，无须额外的 Cache 分区来提升读性能。

## 物理盘故障 > 物理盘故障影响 > 存储分层模式（全闪配置）

# 存储分层模式（全闪配置）

全闪配置下，所有物理盘均为高速的 SSD 盘。

由于 SMTX OS 系统可以安装在两块物理盘中并配置软 RAID 1，也可以独立安装在两块物理盘上组成硬 RAID 1。两种部署方式下，仅系统盘包含的分区不同，其他盘的分区相同。

单节点上物理盘具体说明如下：

- **SMTX OS 安装在硬 RAID 1 中**

  SMTX OS 系统安装在两块独立的 SSD 上，两块 SSD 与存储控制器组成硬 RAID 1 并作为系统盘，节点内其他物理盘作为数据盘。

  ![](https://cdn.smartx.com/internal-docs/assets/5c3f00ee/failure_scenario_4.png)
- **SMTX OS 安装在软 RAID 1 中**

  SMTX OS 系统安装在两块 SSD 上并配置软 RAID 1，剩余存储空间用于存放缓存和数据。节点内其他物理盘作为数据盘。

  ![](https://cdn.smartx.com/internal-docs/assets/5c3f00ee/failure_scenario_1.png)

## 物理盘故障 > 物理盘故障影响 > 存储分层模式（全闪配置） > 单节点物理盘故障

# 单节点物理盘故障

## 系统盘或含元数据分区的数据盘故障

- **系统盘故障**

  若 SMTX OS 系统安装在硬 RAID 1，当两块系统盘中任意一块发生故障时，操作系统将自动切换到节点另一块系统盘进行读写，整个过程中集群业务不受影响。

  当两块系统盘同时发生故障时，操作系统分区和元数据分区将彻底无法使用，进而导致该节点不可用。若故障节点上的虚拟机配置了高可用（HA），虚拟机可迁移至其他节点正常运行。由于节点导致部分数据块可用副本数量减少，少于预期副本数，将触发数据恢复。
- **含元数据分区的数据盘故障**

  若 SMTX OS 操作系统安装在软 RAID 1，当节点两块含元数据分区的数据盘中任意一块发生故障时，故障盘的操作系统分区和元数据分区将不可用，但由于两块含元数据分区的数据盘的操作系统分区组成 RAID 1，两个元数据分区也组成 RAID 1，因此操作系统将自动切换到节点上另一块盘的分区进行读写，在整个过程中 SMTX OS 集群业务不受影响。同时，由于故障盘上的 Journal 分区，Cache 分区不可用，节点的读写性能将产生下降。

  当两块含元数据分区的数据盘同时发生故障时，操作系统分区和元数据分区将彻底无法使用，进而导致该节点不可用。若故障节点上的虚拟机配置了高可用（HA），虚拟机可迁移至其他节点正常运行。

  由于物理盘故障导致部分数据块可用副本数量或编码块减少，少于预期副本数或者编码块数量，将触发数据恢复。

## 数据盘故障

数据盘主要包含 Journal 分区、Cache 分区和数据分区，分别用于记录物理盘写操作、数据缓存和数据存储。当数据盘发生故障时，故障盘的 Journal 分区，Cache 分区和数据分区都将不可用，影响分别如下：

- **Journal 分区不可用**

  系统可以继续使用其他物理盘的 Journal 分区，故障盘所在节点的写入性能将出现下降，但读取性能不受影响。
- **Cache 分区不可用**

  系统可以继续使用其他物理盘的 Cache 分区。故障后如果节点总的 Cache 分区可用空间足够，将不会影响读写性能。如果 Cache 分区可用空间不足，将触发更加积极的数据下沉和 I/O 限速，导致节点的读写性能下降。Cache 分区中持久化存储的性能层数据因为物理盘故障而损失的副本也会触发数据恢复。系统优先在其他节点上完成数据重建并恢复至期望的副本数。如果集群中仅剩余故障节点存在可用空间，也可以在当前节点的其他健康数据分区上重建数据。
- **数据分区不可用**

  数据分区中采用多副本或纠删码技术，保证数据块冗余在多个节点的数据分区上。在单节点上，若单块数据盘或多块数据盘发生故障，因为其他节点上仍有足够的数据副本或纠删码编码块，系统可以继续使用其他物理盘的数据分区，因此不会对集群业务的正常运行产生影响。同时，数据盘故障将触发数据恢复，利用健康数据盘上的数据，故障盘上的数据将自动恢复至其他节点的可用空间，从而保证副本数量或纠删码编码块数量符合预期。如果集群中仅剩余故障节点存在可用空间，数据也可以在当前节点上的其他健康数据分区上重建。

  此外，只要数据盘上的 Cache 分区可用空间足够，数据分区不可用的故障不会影响节点存储性能。

## 物理盘故障 > 物理盘故障影响 > 存储分层模式（全闪配置） > 多节点物理盘故障

# 多节点物理盘故障

当多节点的物理盘发生故障时，主要考虑故障对数据可用性的影响。数据采用不同冗余策略时，故障会有不同的影响。

- 若数据的冗余策略为**副本**策略，当多个节点上的物理盘同时发生故障时，只要有一个数据副本存活，即可保持数据的正常读写，对集群业务不会产生影响。但如果多块物理盘故障导致所有副本均不可用且无法恢复，则会导致数据完全损毁，将对集群业务的正常运行产生影响。
- 若数据的冗余策略为**纠删码**，正常情况下会以 K 个数据块和 M 个校验块的形式分别存储在不同节点上。当多个节点上的物理盘同时发生故障时，只要有不少于 K 个编码块（数据块或校验块）存活，即可保证数据的正常读写，对集群业务不会产生影响。如果少于 K 个编码块存活，则会导致数据不可访问，将对集群业务的正常运行产生影响。

## 物理盘故障 > 物理盘故障影响 > 存储分层模式（混闪配置）

# 存储分层模式（混闪配置）

混闪配置下，缓存盘和数据盘独立部署。高速的 SSD 盘部署为缓存盘，低速的 HDD 盘部署为数据盘。

由于 SMTX OS 系统可以安装在两块物理盘中并配置软 RAID 1，也可以独立安装在两块物理盘上组成硬 RAID 1。两种部署方式下，仅系统盘包含的分区不同，其他盘的分区相同。

单节点上物理盘具体说明如下：

- **SMTX OS 安装在硬 RAID 1 中**

  SMTX OS 系统安装在两块独立的 SSD 上，两块 SSD 与存储控制器组成硬 RAID 1 并作为系统盘，节点内其他物理盘作为缓存盘和数据盘。

  ![](https://cdn.smartx.com/internal-docs/assets/5c3f00ee/failure_scenario_6.png)
- **SMTX OS 安装在软 RAID 1 中**

  SMTX OS 系统安装在两块 SSD 上并配置软 RAID 1，剩余存储空间用于存放缓存和数据。节点内其他物理盘作为缓存盘和数据盘。

  ![](https://cdn.smartx.com/internal-docs/assets/5c3f00ee/failure_scenario_2.png)

## 物理盘故障 > 物理盘故障影响 > 存储分层模式（混闪配置） > 单节点物理盘故障

# **单节点物理盘故障**

## 系统盘或含元数据分区的缓存盘故障

- **系统盘故障**

  若 SMTX OS 系统安装在硬 RAID 1，当两块系统盘中任意一块发生故障时，操作系统将自动切换到节点另一块系统盘进行读写，整个过程中集群业务不受影响。

- **含元数据分区的缓存盘故障**

  当两块含元数据分区的缓存盘同时发生故障时，操作系统分区和元数据分区将彻底无法使用，进而导致该节点不可用。若故障节点上的虚拟机配置了高可用（HA），虚拟机可迁移至其他节点正常运行。由于节点导致性能层部分数据块可用副本数量减少，少于预期副本数，将触发数据恢复。

## 其他缓存盘故障

其他缓存盘主要包含 Journal 分区和 Cache 分区，主要用于记录物理盘写操作和数据缓存。发生故障时，故障盘的 Journal 分区，Cache 分区和数据分区都将不可用，影响分别如下：

- **Journal 分区不可用**

  系统可以继续使用其他物理盘的 Journal 分区，单个节点的写入性能将出现下降，但读取性能不受影响。
- **Cache 分区不可用**

  系统可以继续使用其他物理盘的 Cache 分区。故障后如果节点总的 Cache 分区可用空间足够，将不会影响读写性能。如果 Cache 分区可用空间不足，将触发更加积极的数据下沉和 I/O 限速，导致节点的读写性能下降。Cache 分区中持久化存储的性能层数据因为物理盘故障而损失的副本也会触发数据恢复。系统优先在其他节点上完成数据重建恢复至期望的副本数。如果集群中仅剩余故障节点存在可用空间，也可以在当前节点上的其他健康数据分区上重建数据。

> **说明**：
>
> 任意一块缓存盘发生故障时，不会导致其他缓存盘或数据盘掉线，且不会引起可用存储容量减少。

## 数据盘故障

数据分区采用多副本或纠删码技术，保证数据块冗余在多个节点的数据分区上。在单节点上，若单块数据盘或多块数据盘发生故障，因为其他节点上仍有足够的数据副本或纠删码编码块，系统可以继续使用其他物理盘的数据分区，因此不会对集群业务的正常运行产生影响。同时，数据盘故障将触发数据恢复，利用健康数据盘上的数据，故障盘上的数据将自动恢复至其他节点的可用空间，从而保证副本数量或纠删码编码块数量符合预期。如果集群中仅剩余故障节点存在可用空间，数据也可以在当前节点上的其他健康数据分区上重建。

## 物理盘故障 > 物理盘故障影响 > 存储不分层模式

# 存储不分层模式

不分层部署模式下，所有物理盘均为高速 SSD 盘。相较于分层模式，此部署模式下数据分区的性能和 Cache 分区相同，无需额外的读缓存，因此每个物理盘上都不包含 Cache 分区。

由于 SMTX OS 系统可以安装在两块物理盘中并配置软 RAID 1，也可以独立安装在两块物理盘上组成硬 RAID 1。两种部署方式下，仅系统盘包含的分区不同，其他盘的分区相同。

单节点上物理盘具体说明如下：

- **SMTX OS 安装在硬 RAID 1 中**

  SMTX OS 系统安装在两块独立的 SSD 上，两块 SSD 与存储控制器组成硬 RAID 1 并作为系统盘，节点内其他物理盘作为数据盘。

  ![](https://cdn.smartx.com/internal-docs/assets/5c3f00ee/failure_scenario_5.png)
- **SMTX OS 安装在软 RAID 1 中**

  SMTX OS 系统安装在两块 SSD 上并配置软 RAID 1，剩余存储空间用于存放数据。节点内其他物理盘作为数据盘。

  ![](https://cdn.smartx.com/internal-docs/assets/5c3f00ee/failure_scenario_3.png)

存储不分层模式下，数据的冗余策略只能为多副本策略。

  当两块含元数据分区的数据盘同时发生故障时，操作系统分区和元数据分区将彻底无法使用，进而导致该节点不可用。由于节点导致部分数据块可用副本数量减少，少于预期副本数，将触发数据恢复。

  数据盘故障将触发数据恢复，使用将健康物理盘上的数据自动恢复至其他节点的可用空间中，从而保证副本数量符合预期。如果集群中仅剩余故障节点存在可用空间，数据也可以在当前节点上的其他健康数据分区上重建。

## 物理盘故障 > 物理盘故障影响 > 存储不分层模式 > 多节点物理盘故障

# 多节点物理盘故障

当多节点的物理盘发生故障时，主要考虑故障对数据可用性的影响。当多个节点上的物理盘同时发生故障时，只要有一个数据副本存活，即可保持数据的正常读写，对集群业务不会产生影响。但如果多块物理盘故障导致所有副本均不可用且无法恢复，则会导致数据完全损毁，将对集群业务的正常运行产生影响。

---

## 节点故障

# 节点故障

节点故障是指节点发生如下问题导致节点不可用。

- 主机断电、关机、重启、断网；
- 发生内存或主板损坏等硬件故障。

---

## 节点故障 > 节点类型

# 节点类型

SMTX OS 集群中的节点分为主节点和存储节点两种类型。

---

## 节点故障 > 节点类型 > 主节点

# 主节点

主节点是指运行了 ZooKeeper 和 Meta 服务的节点。ZooKeeper 提供了分布式一致性协议，用于在分布式系统中协作多个任务；Meta 提供了元数据管理服务，主要负责管理存储对象、管理数据副本、控制访问权限和保障数据一致性等。主节点上也同时运行了 Chunk 等数据存储服务，也可以为集群提供数据存储服务。

主节点同时担任了一种 ZooKeeper 角色和一种 Meta 角色，具体可以分为以下四类：

- **ZooKeeper Leader / ZooKeeper Follower**

  SMTX OS 集群所有主节点上均运行了 ZooKeeper 服务，节点之间通过相互投票的方式，选举出 ZooKeeper Leader。ZooKeeper Leader 节点是 SMTX OS 集群的决策节点，负责处理数据的写入、修改和删除请求、控制节点之间的数据同步。未竞选成为 ZooKeeper Leader 的节点将作为 ZooKeeper Follower 节点，负责同步 ZooKeeper 元数据。

  在同一时刻， SMTX OS 集群中有且只有一个 ZooKeeper Leader 节点。若 ZooKeeper Leader 节点发生故障，其他 ZooKeeper Follower 节点将再次通过投票选举出新的 ZooKeeper Leader 节点。
- **Meta Leader / Meta Follower**

  运行了 Meta 服务的主节点可以分为 Meta Leader 和 Meta Follower。

  Meta 以集群的形式提供服务，并借助 ZooKeeper 服务确认一个主节点作为 Meta Leader。同一时刻， SMTX OS 集群中有且仅有一个 Meta Leader 对外提供 Meta 服务，Meta Follower 节点只负责同步元数据。当 Meta Leader 节点发生故障时，Meta Follower 节点将选举出新的 Meta Leader 节点。

---

## 节点故障 > 节点类型 > 存储节点

# 存储节点

存储节点是运行 Chunk 数据存储服务的节点，用于实际存储数据。与主节点不同的是，存储节点不会运行 ZooKeeper、Meta 等元数据等关键性服务，因此存储节点故障的影响范围更小，一般不会对整个 SMTX OS 集群整体的可用性产生影响。

---

## 节点故障 > 故障影响

# 故障影响

根据发生故障的节点类型，节点故障影响可以分为主节点故障影响和存储节点故障影响。

---

## 节点故障 > 故障影响 > 主节点故障

# 主节点故障

由于主节点上运行了 ZooKeeper 和 Meta 等关键服务，因此主节点故障对 SMTX OS 集群的可用性将产生影响。

## 单个主节点故障

单个主节点故障时，一般不会对 SMTX OS 集群的可用性产生影响，具体说明如下：

- **ZooKeeper Leader 和 Meta Leader 故障**

  ZooKeeper Leader 或 Meta Leader 故障将触发 ZooKeeper Leader 或 Meta Leader 重新选举。在此过程中，虚拟机可能会出现轻微 I/O 异常。
- **ZooKeeper Follower 和 Meta Follower 故障**

  ZooKeeper Follower 和 Meta Follower 节点运行时未实际承担元数据服务，发生故障时不会触发 ZooKeeper 或者 Meta 的重新选举，因此集群虚拟机不会出现 I/O 中断的情况。

此外，由于主节点同时运行了 ZBS Chunk 服务，因此主节点发生故障也会对 SMTX OS 集群存储产生影响，此部分故障影响可参考[存储节点故障](/smtxos/6.3.0/failure_scenario_guide/failure_scenario_guide_22)。

## 多个主节点故障

多个主节点发生故障将对 SMTX OS 集群的可用性产生影响。若 SMTX OS 集群超过一半的主节点不可用，则元数据服务不可用，集群无法正常工作，虚拟机停止 I/O。

以 3 节点和 5 节点 SMTX OS 集群为例，故障主节点数量对集群的影响如下表所示：

| **主节点数量** | **故障主节点数量** | **故障影响** |
| --- | --- | --- |
| 3 | 1 | 集群可正常提供服务 |
| ≥ 2 | 集群整体不可用 |
| 5 | ≤ 2 | 集群可正常提供服务 |
| ≥ 3 | 集群整体不可用 |

---

## 节点故障 > 故障影响 > 存储节点故障

# 存储节点故障

存储节点发生故障时，将对 SMTX OS 集群可用的数据副本或编码块产生影响。

## 单个存储节点故障

SMTX OS 采用多副本或纠删码技术，保证数据块冗余在多个节点上，提供单节点容错能力。

若数据的冗余策略采用多副本策略，当数据写入时，将同时被复制 2 份或 3 份存放在不同的节点上。因此，单节点故障发生时，由于其他节点仍然保存有完整的数据副本，不会引发数据的完全丢失，可保障业务的连续性和可靠性，即使只存在最后一个有效副本也是如此，数据会自动恢复到其他可用的节点上。

若数据的冗余策略采用纠删码，在数据写入时，纠删码技术将一段数据编码为多个编码块（包含原始数据块和校验块），每个编码块分别存放在不同的节点上。当单节点故障发生时，可以通过纠删码算法和剩余的编码块恢复出缺失的原始数据块，进而保证业务的连续性和可靠性。

存储节点发生故障的影响可分为如下几类：

- **引发主机异常告警**

  当节点发生故障时，界面将观察到主机异常的告警。
- **集群性能下降**

  由于 SMTX OS 集群能力是线性扩展的，因此节点故障通常会导致集群的存储性能下降，下降比例等同于故障节点提供的性能占集群的比例。
- **触发数据恢复**

  节点发生故障后，该节点上存放的数据副本或编码块将失效。系统在检测到副本或编码块失效后将触发数据恢复，并在其他正常节点的可用存储空间将数据进行恢复。在数据恢复期间，部分节点上的 I/O 受到恢复数据流的影响有额外的下降，但在数据恢复结束后将保持稳定。

## 多个存储节点故障

> **注意**：
>
> 大部分情况下，ZBS 存储服务中的数据遵循局部化分布策略。在集群节点数量较多时，1 ～ 3 个节点发生故障不会造成集群中所有逻辑数据都有数据损失，仅会影响少量数据。

当使用多副本策略时，多节点故障对数据的影响如下表所示。

| **副本数量** | **故障节点数量** | **故障影响** |
| --- | --- | --- |
| 2 | 1 | 数据安全 |
| ≥ 2 | 可能造成部分数据损毁 |
| 3 | ≤ 2 | 数据安全 |
| ≥ 3 | 可能造成部分数据损毁 |

当使用纠删码策略时，多节点故障对数据的影响如下表所示：

| 原始数据块数量 K | 校验块数量 M | 故障节点数量 | 故障影响 |
| --- | --- | --- | --- |
| ≥ 2 | = 1 | 1 | 数据安全 |
| ≥ 2 | 可能造成部分数据损毁 |
| = 2 | ≤ 2 | 数据安全 |
| ≥ 3 | 可能造成部分数据损毁 |
| ≥ 3 | = 3 | ≤ 3 | 数据安全 |
| ≥ 4 | 可能造成部分数据损毁 |
| ≥ 4 | = 4 | ≤ 3 | 数据安全 |
| = 4 | 极小概率出现数据损毁 |
| ≥ 5 | 可能造成部分数据损毁 |

---

## SCVM 故障

# SCVM 故障

SCVM（SmartX Controller Virtual Machine）故障指 SCVM 发生了关机、重启，以及虚拟机网络异常等故障。

---

## SCVM 故障 > SCVM 简介

# SCVM 简介

SCVM 是运行在 ESXi 主机上的一种特殊的虚拟机。当使用 VMware ESXi 平台部署 SMTX OS 时，每个节点都需要创建一个 SCVM 虚拟机用于安装 SMTX OS。

SCVM 上运行了 ZBS 服务，可以统一管理 ESXi 主机上的硬盘设备（SSD/HDD），集群中多个 SCVM 通过存储网络组成分布式存储集群，为 VMware ESXi 主机中的虚拟机提供存储服务。

正常情况下，ESXi 主机上的业务虚拟机的 I/O 请求会先发送到 ESXi 的 vmkernel，由 vmkernel 通过查询本地路由表，将 I/O 请求发送至本地的 SCVM，SCVM 中的 ZBS Chunk 服务将处理接收到的 I/O 请求。

---

## SCVM 故障 > 单个 SCVM 故障

# 单个 SCVM 故障

- **开启了 I/O 重路由**

  I/O 重路由能够检测 SCVM 的健康状况。当本地 SCVM 发生故障时，ESXi 路由表会将本地业务虚拟机的 I/O 请求目的地址由本地 SCVM 修改为远程 SCVM，本地业务虚拟机的 I/O 请求将重新路由到至其他 ESXi 主机上的 SCVM，进而完成故障链路的切换。但当本地 SCVM 恢复至正常状态且超过 3 分钟，I/O 请求会重新路由至本地 SCVM，并清除临时路由路径，此时的切换过程为毫秒级完成，通常不会引发可观测到的 I/O 中断。

  此外，若发生故障的 SCVM 运行了 ZooKeeper 和 Meta 服务，故障将触发 ZooKeeper Leader 和 Meta Leader 的重新选举，期间或出现短暂的 I/O 中断。
- **未开启 I/O 重路由**

  在关闭 I/O 重路由功能的情况下，若虚拟机配置了 HA，则 SCVM 故障将触发 SCVM 所在主机上的虚拟机 HA，否则虚拟机将无法进行任何读写操作。

---

## SCVM 故障 > 多个 SCVM 故障

# 多个 SCVM 故障

在单 SCVM 故障的基础上，若多 SCVM 故障导致了虚拟机的全部数据副本都不可用，则对应的虚拟机也不可用。如果磁盘物理上是完好的，则 SCVM 恢复后数据副本也可以恢复正常，受影响的虚拟机将恢复正常。

如果集群超过一半运行了 ZooKeeper 和 Meta 服务的 SCVM 发生故障，则元数据服务不可用，集群将无法正常工作。

---

## 机架机箱故障

# 机架机箱故障

机架机箱的故障主要包括如下：

- 机架故障：包括机架自身断网、断电，也包括机架中单个或多个主机故障。
- 机箱故障：包括机箱自身断网、断电，也包括机箱中单个或多个主机故障。
- 主机故障：包括主机本身关机、重启、网络异常，也包括因主机上的单个或多个数据盘故障导致主机不可用。

> **注意**：
>
> 以下描述的可容忍故障类型仅针对数据本身的安全性，数据的可用性还需要一并考虑故障是否会引起集群中超过半数的主节点故障。

---

## 机架机箱故障 > 多副本模式

# 多副本模式

SMTX OS 支持机架拓扑感知技术，利用拓扑信息将数据副本尽可能均匀地分配到不同的机架、机箱和主机上。因此，即使发生机架断电或损坏等极端状况，系统还能在其他机架的主机上找到存活的副本，从而提高集群整体的容错能力。

除了分配副本时主动选择最佳的位置存储副本，在集群的运行过程中，系统也会根据副本分布情况和机架拓扑信息，自动在主机之间迁移数据副本，使数据副本尽可能优先分布在不同的机架中，其次分布在相同机架的不同的机箱中，最次分布在相同机箱的不同主机中，以达到最佳的数据副本拓扑安全性。

通常，集群中机架、机箱数量越多，能提供的容错能力越强。以常见的 2 副本和 3 副本集群为例，副本数、机架数、机箱数与可容忍的故障类型的关系如下表所示：

| **副本数** | **机箱数** | **机架数** | **可容忍的故障类型** |
| --- | --- | --- | --- |
| 2 副本 | 1 | 1 | 任意 1 个主机故障 |
| ≥ 2 | 1 | 任意 1 个机箱故障 |
| ≥ 2 | ≥ 2 | 任意 1 个机架故障 |
| 3 副本 | 1 | 1 | 任意 2 个主机故障 |
| 2 | 1 | 任意 1 个机箱故障 |
| ≥ 3 | 1 | 任意 2 个机箱故障 |
| ≥ 2 | 2 | 任意 1 个机架故障 |
| ≥ 3 | ≥ 3 | 任意 2 个机架故障 |

对于发生故障的机架和机箱而言：

- 若虚拟机启用了 HA 功能，故障持续 3 分钟后将触发 HA，虚拟机将自动迁移到其他可用节点上；对于未配置 HA 的虚拟机，需要手动迁移到可用主机上开机才可以继续使用。
- 在管理界面将出现相关的报警说明。

---

## 机架机箱故障 > 纠删码模式

# 纠删码模式

与多副本模式类似，在纠删码模式下，SMTX OS 在进行数据块和校验块（统称为编码块）分配时，会根据集群拓扑信息，尽量将其均匀分配到不同的机架、机箱和主机上，以提高集群整体的容错能力。在运行过程中，也会根据编码块的分布和拓扑信息，自动进行迁移，以保证集群运行在高容错能力下。

通常，集群中机架、机箱数量越多，提供的容错能力越强。只要集群中故障节点数不超过校验块数量 M，剩余节点即可将数据恢复至期望状态，从而保证业务不受影响。根据机架和机箱的数量，以及节点在机架机箱中的分布情况，下表以推荐的纠删码配置（2+1、4+1、2+2、4+2、8+2）为例，展示了数据块数量 K、校验块数量 M、机箱数 B、机架数 R 与可容忍的故障类型的关系。

| 数据块数量 K | 校验块数量 M | 机箱数 B | 机架数 R | 可容忍的故障类型 |
| --- | --- | --- | --- | --- |
| 2 | 1 | ≤ 2 | ≤ 2 | 任意 1 个主机故障 |
| ≥ 3 | ≤ 2 | 任意一个机箱故障 |
| ≥ R | ≥ 3 | 任意一个机架故障 |
| 4 | 1 | ≤ 4 | ≤ 4 | 任意一个主机故障 |
| ≥ 5 | ≤ 4 | 任意一个机箱故障 |
| ≥ R | ≥ 5 | 任意一个机架故障 |
| 2 | 2 | 1 | 1 | 任意 2 个主机故障 |
| 2 ≤ B ≤ 3 | 1 | 任意一个机箱故障 |
| ≥ 4 | 1 | 任意 2 个机箱故障 |
| ≥ R | 2 ≤ R ≤ 3 | 任意 1 个机架故障 |
| ≥ R | ≥ 4 | 任意 2 个机架故障 |
| 4 | 2 | 1 ≤ B ≤ 2 | ≤ 2 | 任意 2 个主机故障 |
| 3 ≤ B ≤ 5 | ≤ 2 | 任意 1 个机箱故障 |
| ≥ 6 | ≤ 2 | 任意 2 个机箱故障 |
| ≥ R | 3 ≤ B ≤ 5 | 任意 1 个机架故障 |
| ≥ R | ≥ 6 | 任意 2 个机架故障 |
| 8 | 2 | 1 ≤ B ≤ 4 | ≤ 4 | 任意 2 个主机故障 |
| 5 ≤ B ≤ 9 | ≤ 4 | 任意 1 个机箱故障 |
| ≥ 10 | ≤ 4 | 任意 2 个机箱故障 |
| ≥ R | 5 ≤ R ≤ 9 | 任意 1 个机架故障 |
| ≥ R | ≥ 10 | 任意 2 个机架故障 |

---

## 网络故障

# 网络故障

网络故障是指 SMTX OS 集群中出现的与网络连接相关的问题。

网络故障通常由网卡硬件或驱动故障、网线模块故障、上游链路故障或拥塞等造成。这些故障会导致节点间的网络延迟或数据丢包，甚至整个网络断开连接。

当单个网卡发生异常时，系统能够快速检测并且自动尝试恢复。当节点的存储网口或接入网口进行了网口绑定时，个别网卡有较高的延迟或者丢包率，如果重置（ifconfig down/ ifconfig up）网卡后仍然无法恢复，可以将该网口从网口绑定中临时移除，待该网口恢复正常后再将其加回网口绑定。

---

## 网络故障 > 网口故障影响

# 网口故障影响

如果节点的一张网卡上的两个网口做了网口绑定，则支持故障链路自动切换。即某一网口发生故障时，其他网口可自动接替该故障网口转发数据流量。

采用 `active-backup` 绑定模式时，故障链路会在 4s 内完成切换，集群虚拟机会感知到一个短暂的延迟，I/O 性能出现略微下降，但完成切换后即可恢复正常；在采用非 `active-backup` 绑定模式时，单网口异常通常将导致节点最大写带宽减半。

若所有绑定的网口同时故障，则节点将无法提供对应网络平面（管理/存储/迁移）的连接能力。

---

## 网络故障 > 管理网络故障影响

# 管理网络故障影响

管理网络是主机自身管理服务使用的系统网络，用于用户访问和管理 SMTX OS 集群，以及对接 CloudTower。一个 SMTX OS 集群节点只允许有一个管理网络。

管理网络发生故障时， SMTX OS 集群仍然可以正常工作，但使用故障节点的管理 IP 地址将无法访问集群的管理界面。在 CloudTower 管理界面也将出现对应的故障报警说明。

---

## 网络故障 > 存储网络故障影响

# 存储网络故障影响

存储网络是主机上块存储服务所使用的网络，用于 SMTX OS 集群内部各节点之间进行数据交换。一个 SMTX OS 集群节点只允许有一个存储网络。

影响存储网络性能的主要因素有**网络丢包**、**网络延迟**和**网络中断**，它们的故障影响说明如下。

## 网络丢包

单个节点因为网卡等异常导致出现频繁丢包时，异常节点上的存储服务延迟会显著升高。通常丢包率达到或超过 10% 之后所有的连接都会难以正常维持，可以认为网络基本不可用，节点很有可能整体进入异常状态。同时， SMTX OS 集群的配置变更请求（包括存储对象的创建、删除、更新等）将出现显著的迟缓现象。

## 网络延迟

当存储网络延为 5 毫秒 ~ 500 毫秒时，不会对集群的可用性产生影响，但网络延迟越高，集群的性能表现就越低。在大于 500 毫秒之后，有可能会出现服务不稳定的现象。

当网络延迟超过 1 秒（ZooKeeper 的选举超时阈值）后，ZooKeeper Leader 无法选举成功， SMTX OS 集群将进入不可用的状态。

## 网络中断

单节点存储网络中断后，将触发节点状态异常与节点无法连通的告警。

单节点和多节点存储网络中断的故障影响等同于节点故障，具体可参考[节点故障](/smtxos/6.3.0/failure_scenario_guide/failure_scenario_guide_08)。

---

## 网络故障 > 迁移网络故障影响

# 迁移网络故障影响

迁移网络是为虚拟机迁移单独创建的专属网络，集群内热迁移的数据将通过迁移网络进行传输，避免占用存储网络的带宽。

未创建迁移网络，或者迁移网络发生故障的情况下：

- 集群内迁移或跨集群热迁移使用存储网络进行；若存储网络不可达，则使用管理网络迁移。
- 跨集群冷迁移和分段迁移直接使用管理网络进行迁移。

---

## 网络故障 > 虚拟机网络故障影响

# 虚拟机网络故障影响

虚拟机网络是虚拟机运行使用的网络。根据业务需求，集群中可以存在多个虚拟机网络，一台虚拟机可以通过不同的虚拟网卡连接不同的虚拟机网络。当虚拟机在集群内不同主机间进行迁移时，虚拟机网络可保证虚拟机提供的业务在迁移过程中不会中断，也无需重新配置网络数据库。

当虚拟机网络发生故障时，故障网络上关联的虚拟机将无法正常进行网络通信。

## 双活集群故障

# 双活集群故障

本章节描述了双活集群在发生节点和网络故障时的容灾处理方式及对业务的影响。对于物理盘和网卡故障，其影响及处理方式与未启用双活特性的集群一致，请参考本手册的对应章节。

本文以一个由 9 个节点组成的典型同城双活集群为例，其中：

- 优先可用域 A、次级可用域 B 以及仲裁节点 C 分别位于三个不同物理区域的数据中心 IDC A、IDC B 和 IDC C；
- 虚拟化平台为 ELF 平台或 VMware ESXi 平台，已启用 HA 并配置放置组策略。

![](https://cdn.smartx.com/internal-docs/assets/5c3f00ee/failure_scenario_7.png)

在双活集群中，集群粒度的优先可用域（IDC A）与次级可用域（IDC B）均可运行业务。每个虚拟机会选择自身长期运行的可用域作为优先可用域。两个可用域内节点发生故障时的系统行为基本一致。本文示例中假设虚拟机预期长期稳定运行在 IDC A 上，即 IDC A 为虚拟机的优先可用域。

集群已启用 HA 并配置放置组策略的情况下：

- 放置组策略默认行为：优先在虚拟机当前可用域内所有节点执行 HA；若条件允许，可在另一可用域内执行 HA。
- 若实际部署中放置组策略有特殊限制（例如仅允许在优先可用域内启动），则 HA 行为需遵守该限制。
- 数据层面的故障恢复行为不直接受放置组策略影响。

> **注意**：
>
> 无论发生何种故障，单个可用域内最多仅保存数据的 2 个副本，不会恢复至 3 副本。

## 双活集群故障 > 节点故障

# 节点故障

| **故障场景** | **双活集群处理方式及影响** |
| --- | --- |
| IDC A 中的单个存储节点（例如节点 3）故障 | - **故障节点上的虚拟机：**   - 若 IDC A 中其他节点有足够的计算资源（CPU 和内存），则故障节点上 3 上的虚拟机会触发 HA，并迁移到 IDC A 的其他节点。   - 若 IDC A 中节点空闲计算资源不足，而 IDC B 中有足够的计算资源，则虚拟机触发 HA 后会迁移至 IDC B 中。   - 若所有节点的空间计算资源不足，则虚拟机将无法正常 HA。 - **其他节点上的虚拟机：**   - 若虚拟机正在访问故障节点 3 上的 Extent 数据，将出现短暂的 I/O 延迟增高（通常为毫秒级别，最大不超过 7 s），然后恢复正常。   - 若虚拟机未访问故障节点 3 的数据，则无可观测到的影响。 - **数据恢复：**   - 若故障节点 3 上保存的副本对应的数据在异常期间发生写入，则将在更新发生后 1 分钟之内触发数据恢复。   - 若故障节点 3 上保存的副本对应的数据未发生过更新，将在 10 分钟之后触发数据恢复。   - 若 IDC A 内健康节点有足够空间，将从 IDC A 内的健康副本读取数据后恢复至 IDC A 内的其他健康节点。   - 若 IDC A 内尚未持有数据副本的健康节点没有可用空间，将从 IDC B 内的健康副本读取数据后恢复至 IDC B 内的其他健康节点。 |
| IDC A 中的多个存储节点（例如节点 3 和 4 ）同时故障 | - **故障节点上的虚拟机：**   - 若 IDC A 中其他节点有足够的计算资源（CPU 和内存），则故障节点上 3、4 上的虚拟机会触发 HA，并迁移到 IDC A 上的其他节点。若部分虚拟机数据的 2 个副本在 IDC A 中位于节点 3 和 4 上，HA 之后，业务可以通过访问 IDC B 上的副本而保持正常运行状态，但是存储性能将因所有请求都需要跨域访问而出现比较明显的下降（具体下降幅度取决于域间延迟和虚拟机的 I/O 模型）。   - 若 IDC A 中节点空闲计算资源不足，而 IDC B 中的计算资源足够，则虚拟机触发 HA 后会迁移到 IDC B 中。   - 若所有节点的空间计算资源不足，则虚拟机将无法正常 HA。 - **其他节点上的虚拟机：**   - 若虚拟机正在访问故障节点 3、4 的 Extent 数据，将出现短暂的 I/O 延迟增高（通常为毫秒级别，最大不超过 7 s），然后恢复正常。   - 若虚拟机未访问故障节点 3、4 的数据，则无可观测到的影响。 - **数据恢复：**   - 若故障节点 3、4 保存的副本对应的数据在异常期间发生写入，将在更新发生后 1 分钟之内触发数据恢复。   - 节若故障点 3、4 的保存的副本对应的数据未发生过更新，将在 10 分钟之后触发数据恢复。   - 若 IDC A 内健康节点有足够空间，将从 IDC A 内的健康副本读取数据后恢复至 IDC A 内的其他健康节点。   - 若 IDC A 内数据的 2 个副本分别位于故障节点 3、4 节点上，故障导致仅在 IDC B 内保留有 1 份健康副本。此时：     - 若 IDC A 内其他节点有足够的空间，将首先从 IDC B 内的健康副本读取数据后恢复至 IDC A 内的其他健康节点，恢复至 2 副本（IDC A 和 IDC B 各一份）状态。     - 而后，若 IDC A 内还有未持有副本的节点且有足够空间，则将再从 IDC A 内的健康副本读取数据后恢复至 IDC A 内的其他健康节点，最终恢复至 IDC A 2 副本，IDC B 1 副本状态。     - 若 IDC A 内不存在既具备足够存储空间又尚未持有副本的节点，但 IDC B 内尚未持有副本的节点空间足够，则将从 IDC B 内的健康副本读取数据后恢复至 IDC B 内的其他健康节点，最终恢复至 IDC A 1 副本，IDC B 2 副本状态。     - 若 IDC A 内的节点没有足够的空间，将从 IDC B 内的健康副本读取数据后恢复至 IDC B 内的其他健康节点，恢复至 2 副本（IDC B 2 份）状态。   - 若因为空间不足无法恢复至 3 副本，集群中将会观察到空间利用率高的告警和始终存在无法完成的数据恢复的现象。 |
| IDC A 中故障节点同时承担了 Meta Leader 或 ZooKeeper Leader 角色 | - 若异常主节点包含 ZooKeeper Leader 或 Meta Leader，所有正在 I/O 的虚拟机将观察到一次存储高延迟（≤ 9 s）现象，但不会有 I/O 错误。 - 虚拟机 HA 与数据恢复判定与存储节点故障现象一致。 - 若 IDC A 中所有主节点都故障，则集群虚拟机、虚拟卷的创建和编辑等操作会出现轻微延迟（通常为毫秒级别）。 |
| IDC A 内所有节点故障 | - 若 IDC B 上节点的可用资源足够，则所有虚拟机将 HA 至 IDC B 中。若 IDC B 中资源不足，则根据虚拟机配置的高可用优先级确定优先恢复的虚拟机。同一优先级别的虚拟机恢复顺序不固定。 - 所有数据会在 IDC B 内部尽力恢复至 2 副本状态。若 IDC B 内的预留空间不足以承载全部数据，则每个节点上的容量层空间有可能会被恢复至 100%。性能层空间则仅允许恢复至最大 90% 左右，尽量保证业务 I/O 写入正常。 - 单一可用域内容量层空间不足以承载所有数据时，性能层的数据无法下沉至容量层。可用域整体故障维持时间较久且在没有主动干预删除部分超出的数据的情况下，未采用厚置备模式的虚拟机可能因为没有空间可供写入而发生异常。 - 通常推荐单个 IDC 按照可以承载集群所有数据 2 个副本的容量进行规划。 |
| IDC B 内节点故障 | - 虚拟机继续稳定运行在 IDC A 的节点上，不会发生 HA。 - 若虚拟机当前写入的数据有一个副本位于异常节点上，则可能观察到一次 I/O 延迟（通常为毫秒级别，最大不超过 7 s）。 - 由于 IDC B 仅通常仅持有一份数据副本，对于被判定为异常的数据（故障发生后 1 分钟内仍有更新的数据，或 10 分钟内未发生更新的数据），系统将从 IDC A 中的健康副本读取数据，并同步恢复至 IDC B 内的其他健康节点，恢复至 IDC A 2 副本，IDC B 1 副本状态。 |
| 仲裁节点 C 故障 | - 若集群在仲裁节点故障发生前为稳定运行的健康状态（集群会保证仲裁节点不承担 Zookeeper Leader 角色），则业务不受影响。 - 若故障发生前集群经历了其他故障且刚刚恢复，仲裁节点此时存在小概率承担 Zookeeper Leader 角色，此时可能会观察到一次 I/O 延迟（通常为毫秒级，最大不超过 6 s）。 |
| 跨可用域多节点故障，但关键节点（主节点 + 仲裁节点）占比不超过 50% | - 虚拟机的 HA 行为由其配置的优先可用域及放置组规则决定，处理方式与其他普通节点发生故障时保持一致。 - 存在小概率出现某些数据的全部副本均位于异常节点的情况。此时关联虚拟机因无法访问数据而无法正常启动，且由于不存在健康副本，数据恢复无法触发。需等待故障修复，使至少一个数据副本恢复为健康状态后，方可正常启动虚拟机并触发数据恢复流程。 |

## 双活集群故障 > 网络故障

# 网络故障

## 网络连接中断

| **故障场景** | **双活集群处理方式及影响** |
| --- | --- |
| IDC A 存储网络中断（无法连接 IDC B 与仲裁节点 C），IDC B 与仲裁节点 C 连接正常 | 该场景的处理逻辑与**节点故障**章节中的“IDC A 内所有节点故障”场景的处理逻辑一致。 |
| IDC B 存储网络中断（无法连接 IDC A 与仲裁节点 C），IDC A 与仲裁节点 C 连接正常 | 该场景的处理逻辑与**节点故障**章节中的“IDC B 内节点故障”场景的处理逻辑一致。 |
| 仲裁节点 C 与 IDC A、IDC B 的存储网络中断，IDC A 与 IDC B 之间网络正常 | 该场景的处理逻辑与**节点故障**章节中的“仲裁节点 C 故障”场景的处理逻辑一致。 |
| IDC A 与 IDC B 之间存储网络中断，但仲裁节点 C 正常连接 IDC A 与 IDC B | - 集群优先可用域（IDC A）可正常提供存储服务。 - IDC A 中所有节点上的虚拟机 I/O 会观测到一次高延迟事件（通常为毫秒级，最大不超过 7 秒）。 - 在资源充足且放置组规则允许的情况下，IDC B 上的虚拟机将触发 HA 并在 IDC A 中重建。 - 对于此前运行于 IDC B、且在 IDC A 内仅保留单副本的数据，系统将尝试在 IDC A 内补齐副本数至 2 副本。 |
| 仲裁节点 C 与 IDC A 或 IDC B 失去网络连接，但 IDC A 与 IDC B 连接正常 | - 双活集群整体功能不受影响，可正常提供存储服务。 - 所有节点上的虚拟机 I/O 正常。 - 不会触发数据恢复或副本调整行为。 |
| 仲裁节点 C、IDC A、IDC B 之间的所有存储网络连接同时中断 | - ZooKeeper 服务无法完成选主。 - 集群无法正常提供存储服务。 - 所有节点上的虚拟机 I/O 中断。 |

## 网络延时较高

| **故障场景** | **双活集群处理方式及影响** |
| --- | --- |
| IDC A 与 IDC B 之间网络延时升高（10 ms < Ping 延迟 < 1 s）。但仲裁节点 C 与 IDC A、B 间的网络延迟正常。 | - 集群仍可维持运行，但 I/O 性能，尤其是写入性能明显下降，操作响应延迟增大。 - 当 Ping 延迟达到百毫秒级别后，虚拟机内对延迟敏感的应用可能出现异常或停止工作。 |
| IDC A 与 IDC B 之间网络严重拥塞（1 s < Ping 延迟 < 7 s），但仲裁节点 C 与 IDC A、B 间的网络延迟正常。 | - IDC B 内大量节点处于间歇性异常状态。部分虚拟机可能触发 HA 并迁移至 IDC A。 - 集群可能出现数据需要恢复但无法完成的情况。 - IDC A 内部分虚拟机 I/O 可能表现为极慢状态。 |
| IDC A 与 IDC B 之间网络极度拥塞（Ping 延迟 > 7 s）或发生网络中断，但仲裁节点 C 与 IDC A、B 间的网络延迟正常。 | - ZooKeeper Leader 所在可用域（绝大多数情况下为 IDC A）保持可用，另一可用域节点处于失连状态。 - 若 ZooKeeper Leader 位于仲裁节点 C，则 IDC A 可用，IDC B 失连。虚拟机将仅在 IDC A 内写入数据，I/O 延迟恢复至正常水平。 - 若 ZooKeeper Leader 位于次级可用域 IDC B，ZooKeeper 不会触发迁移， IDC B 失联且其内的虚拟机将触发 HA 并迁移至优先可用域 IDC A 。 |
| 仲裁节点 C 与 IDC A 和 IDC B 之间网络延迟较高（Ping 延迟 > 1 s），但 IDC A 与 IDC B 之间网络正常。 | 该场景等同于仲裁节点发生故障。优先可用域与次级可用域中的虚拟机均可正常运行，但无法使用双活特性。当其中一个可用域发生故障后，另一个可用域无法及时接管业务。 |

## 双活集群故障 > 业务整体停止

# 业务整体停止

当出现以下故障时，双活集群将无法维持业务运行，直至相关故障被排除：

- IDC A 或 IDC B 整体下线，且仲裁节点无法恢复。
- IDC A 与 IDC B 之间网络隔离，且仲裁节点无法恢复。
- IDC A 与 IDC B 之间网络隔离，同时 IDC A 与仲裁节点、IDC B 与仲裁节点的网络均中断。
- 集群中 5 个关键成员（IDC A 内 2 个主节点、IDC B 内 2 个主节点及仲裁节点）中有 3 个及以上同时异常。异常情况包括但不限于：

  - 节点整体不可用；
  - 存储网络断开；
  - 节点元数据服务异常。
