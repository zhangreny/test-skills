## 概述

# 概述

随着虚拟化平台下用户业务规模的持续扩大，虚拟化网络的数据访问量也不断增加，许多应用系统基于虚拟化平台构建，如果服务中断将会导致巨大的商业损失，用户对应用系统可用性的要求日益提高。与此同时，后端服务器的处理速度已远远落后于网络带宽的增长速度，网络带宽的急剧提升，使得后端服务器的资源消耗呈现递增态势，导致服务器成为整个网络架构中的瓶颈。此外，服务器的单点故障也严重影响着应用系统的稳定性和可靠性。如何在虚拟化环境下，有效满足应用系统对于性能和可用性的要求，成为了亟待解决的问题。

基于上述问题，负载均衡技术应运而生。负载均衡技术旨在平衡网络中各个服务器或计算资源之间的工作负载，以实现更高的性能和可用性。

负载均衡技术通常有硬件负载均衡器和软件负载均衡器两种实现形式。传统硬件负载均衡器存在价格昂贵、缺乏弹性、配置更改不够灵活、扩容周期长等问题，在企业应用的云化与数字化过程中仍有很多的局限性。相比硬件负载均衡器，软件负载均衡器灵活性高、资源利用率高，且购买和维护的成本较低，成为了虚拟化环境下提升应用系统性能的最佳选择。

在早期的 Everoute 负载均衡中，负载均衡实例仅支持以单组 LB 虚拟机（2–4 个）的方式部署在单个 SMTX OS 集群内。虽然能够满足基本业务转发需求，但在多集群或跨集群场景下存在局限：用户流量可能存在跨集群转发，单组 LB 虚拟机无法提供集群级别的灾备能力，也无法满足业务对高可用性的更高要求。如果通过部署多个 Everoute 服务来解决该问题，则会显著增加运维管理成本。

为了满足用户在多集群或跨集群场景下的业务需求，Everoute 自本版本起提供了多集群和跨集群负载均衡能力，使负载均衡服务能够覆盖多个 SMTX OS 集群，提升灵活性和可靠性。同时，本版本也支持 VPC 网络场景下的负载均衡，实现了 VPC 内部（东西向）和 VPC 外部到内部（南北向）流量的负载均衡调度。

## 概述 > 产品特点

# 产品特点

Everoute 网络负载均衡器支持第 4 层网络负载均衡，可以与虚拟化平台以及容器平台进行协同，提供统一的网络管理与业务扩展能力。

具有如下关键特点：

- 采用管理、控制和数据平面解耦的分层架构方式，提供了更加灵活的配置管理方式、稳定的流量转发能力、以及较好的可移植性。
- 采用 Keepalived + LVS（Linux Virtual Server）模式实现虚拟服务高可用，避免由于负载均衡器单点故障导致服务不可用。
- 支持高并发的业务流量，可以根据业务需求灵活调整后端服务器数量，以便更合理地利用服务器资源。
- 支持丰富的调度策略、多种主动健康检查机制、以及客户端的流量控制、访问控制，进一步提升了服务器的可靠性和可用性。

Everoute 网络负载均衡器不仅可以满足用户对应用的高性能和高可用性的要求，也解决了硬件负载均衡器的缺点。用户可以借助灵活的软件负载均衡器与虚拟化或超融合平台结合，以加速企业业务的数字化转型。

## 概述 > 产品功能

# 产品功能

Everoute 网络负载均衡器的主要功能如下：

- **支持虚拟服务及业务高可用**

  - 支持将流量通过负载均衡的方式转发到多台后端服务器，即使某台后端服务器故障，其它服务器仍可以继续提供服务。
  - 支持将两个服务器池组成服务器池组，主服务器池中所有成员故障时，可由备服务器池中的成员继续提供服务。
  - 每个虚拟服务由 LB 实例中的两个 LB 虚拟机以主备方式提供高可用，避免由于单点故障导致负载均衡服务不可用。如需进一步提升可靠性，可使用 LB 实例组，以避免由于 LB 实例整体故障而导致服务不可用。
- **提供丰富的负载均衡调度策略**

  支持轮询、加权轮询、最少连接、加权最少连接、源地址哈希、目标地址哈希六种调度策略，能够灵活地适应不同场景的应用负载均衡需求。
- **支持主动健康检查**

  支持 TCP、HTTP、UDP、ICMP 四种主动健康检查方式，可周期性检查后端服务器的健康状况。

  当检测到后端服务器出现异常时，可以将客户端请求自动转发至其他健康的后端服务器；当异常的后端服务器恢复正常运行后，可以将其自动恢复到负载均衡服务中，继续承载业务流量。
- **支持会话保持机制**

  支持在特定的时间段内，将所有源地址相同的连接送往同一个后端服务器，以保证会话的连续性和稳定性。
- **支持业务流量控制**

  支持灵活控制虚拟服务处理的出入峰值流量，避免单个虚拟服务占用过多的网络流量资源，实现合理的资源分配。
- **支持客户端访问控制**

  支持通过白名单或黑名单限制允许访问或禁止访问的客户端 IP 地址，有效保护虚拟服务资源，避免恶意请求的破坏，提高负载均衡器的安全性和健壮性。
- **支持客户端并发连接数控制**

  支持灵活限制同一时间下客户端允许与虚拟服务建立的连接数量，避免恶意攻击消耗过多资源。

## 概述 > 典型应用

# 典型应用

## 虚拟化场景

对于使用 SMTX OS（ELF）集群或 SMTX ELF 集群的用户，使用负载均衡功能时，可以选择部署外置的硬件负载均衡服务器或者软件负载均衡器。然而采用外置的负载均衡器，存在操作成本高、运维不方便等问题，因此在 SMTX OS（ELF）集群或 SMTX ELF 集群中使用集成的 Everoute 网络负载均衡器成为最佳选择。

在虚拟化平台中，可能存在多种不同的业务场景，比如音视频相关应用、网页相关应用、物联网相关应用等。每种业务可能部署在相同或不同的虚拟机网络中，不同应用类型都有其特定的流量行为，这对于整个系统的性能、弹性、稳定性和可用性带来了巨大的挑战。此时，即可使用网络负载均衡器进行业务流量分发，使用后端服务器弹性扩缩支撑业务变化。

![](https://cdn.smartx.com/internal-docs/assets/c7ba2ed7/whitepaper_lb_03-1.png)

## 容器化场景

继虚拟化技术成熟后，容器技术成为对云计算领域具有深远影响的变革技术，越来越多的用户选择在容器化环境中部署应用。

在容器化环境中使用负载均衡功能时，主要选择为开源的原生 LB 方案，比如 MetalLB。然而，开源的原生 LB 功能简单、并且没有企业级的服务支持，出现问题后很难定位和调试。

Everoute 网络负载均衡器支持容器化环境的四层负载均衡，无论是用户自建的 Kubernetes 环境还是 SMTX Kuberentes Service 管理的 Kubernetes 环境，均可以灵活方便地使用负载均衡功能，为集群内部应用提供对外的四层负载均衡能力。

> **说明**：
>
> 在容器化场景下，不支持集群灾备模式、VPC 网络场景以及服务器池组等新特性。

![](https://cdn.smartx.com/internal-docs/assets/c7ba2ed7/whitepaper_lb_03-3.png)

## 集群灾备场景

随着用户业务量的增加，对于负载均衡器的高可用性提出更高的要求。SMTX OS 集群级别的负载均衡器灾备能力能够确保在集群级别的故障发生时，快速将流量切换至备用集群，保障业务的连续性与高可用性。

如下图所示，描述了在 VPC 网络下支持负载均衡器跨 SMTX OS 集群的灾备能力。当 LB 实例-1 发生故障时，所有 LB 实例-1 上的虚拟服务将被自动迁移至 LB 实例-2 继续提供服务。此外，您还可以按需选择是否将迁移后的虚拟服务关联的活动服务器池从服务器池-1 切换为服务器池-2。

![](https://cdn.smartx.com/internal-docs/assets/c7ba2ed7/whitepaper_lb_03-4.png)

## 产品架构

# 产品架构

Everoute 网络负载均衡器采用分层架构的方式实现，包含管理平面、控制平面和数据平面。三个平面之间完全独立，通过 Rest API 或者 API Server 模式进行交互。

分层的结构设计具有如下优点：

- **高扩展性**：每个平面的独立性和通过 API 进行交互的能力，用户可以根据需要添加更多的数据平面和控制平面实例，以适应日益增长的负载。
- **高可靠性**：平面之间的解耦合保证了运维管理不会干扰当前的数据平面操作。即使在维护或配置更改的情况下，数据平面仍然可以持续稳定地处理流量，确保应用系统的可用性。
- **高性能**：每个平面可以专注于其特定任务，从而优化了整体性能。数据平面专注于流量转发，控制平面负责策略制定和动态调整，以确保系统在高负载情况下依旧保持出色的性能表现。

![](https://cdn.smartx.com/internal-docs/assets/c7ba2ed7/whitepaper_lb_05.png)

## 产品架构 > 管理平面

# 管理平面

管理平面主要由 Everoute operator、CloudTower 和 API 组成，用于管理所有配置相关的 API 请求，以及所有负载均衡器运维的相关操作。

- **CloudTower**：用于与 UI、集群控制平面以及运维管理平面进行交互，负责将业务配置以及运维管理配置通过相关 Rest API 进行下发。
- **Everoute operator**：用于 LB 虚拟机的安装部署，负责接收来自 CloudTower 的请求并且触发相应操作。

管理平面具有以下主要功能：

- 通过北向提供的 REST API，将用户的配置进行持久化存储和下发。
- 管理所有增、删、改、查任务的生命周期，为后端服务器预先处理高并发的请求。
- 过滤所有用户配置，校验相关参数，减轻后端服务器系统压力。
- 提供监控、认证、数据采集等基本能力，简化系统运维的复杂性。

## 产品架构 > 控制平面

# 控制平面

控制平面包括全局控制平面和本地控制平面，两者通过 API server 进行解耦和数据交互，以增加灵活性和容错性。

- **全局控制平面**：负责处理 LB 虚拟机组成的负载均衡集群的相关信息同步、资源管理和配置管理。
- **本地控制平面**：负责当前 LB 虚拟机相关的配置转换、状态同步以及资源管理。

![](https://cdn.smartx.com/internal-docs/assets/c7ba2ed7/whitepaper_lb_07.png)

## 全局控制平面

全局控制平面包含 API proxy、API server、etcd 和 Everoute Controller。

- **API server 和 etcd**：用于全局控制平面与本地控制平面之间，以及全局控制平面与外部进行通信的核心组件。etcd 用于存储集群中所有的数据，而 API server 则是唯一可以与 etcd 数据库进行交互的组件。所有数据的增删改查均通过 API server 与 etcd 集群的交互完成。
- **API proxy**：用于对外提供统一的 REST API 接口，对内进行数据转换和请求路由，最终完成与 API server 的交互。
- **Everoute Controller**：以单节点、三节点或者五节点集群的形式存在，用于处理负载均衡集群相关的全局信息。

  Everoute Controller 负责处理来自管理平面的所有配置信息，根据全局信息维护和当前的运行时数据（例如当前的集群资源利用情况）决定如何分配和部署负载均衡虚拟服务，然后将这些配置传递给本地控制平面。

## 本地控制平面

本地控制平面作为全局控制平面与数据平面之间的桥梁，仅关心当前的 LB 虚拟机，具有良好的扩展性。

本地控制平面具有以下主要功能：

- 监听来自全局控制平面的配置信息，主动拉取最新的数据，生成负载均衡器当前配置并作用于数据平面。
- 监控和上报负载均衡服务以及后端服务器的健康状态。
- 负责本地资源相关的注册和更新，比如虚拟机状态、虚拟网卡状态等。

## 产品架构 > 数据平面

# 数据平面

数据平面是处理请求和执行请求转发的核心，以虚拟机实例形式存在，Everoute 网络负载均衡器则以多实例模式部署于虚拟机上。

数据平面接收来自控制平面的配置信息，并根据具体规则进行相应的处理。它具备同时处理来自多个客户端请求的能力，并可以将这些请求转发至后端服务器。

数据平面支持高可用和高并发，具体介绍如下。

## 高可用

- **应用高可用**：正常情况下，服务器池中的所有后端服务器均参与处理客户端的请求。当服务器池中某台服务器产生故障时，客户端的请求将不会再转发到该服务器，而是由其它无故障的后端服务器继续处理请求。即使存在服务器故障，也不会影响应用正常提供服务。

  ![](https://cdn.smartx.com/internal-docs/assets/c7ba2ed7/whitepaper_lb_08-1.png)
- **虚拟服务高可用**：采用 Keepalived + LVS（Linux Virtual Server）模式实现高可用的四层负载均衡器，避免由于 LB 虚拟机单点故障而导致服务不可用，以确保应用系统的稳定性和可用性。当前支持配置 2 个、3 个或 4 个 LB 虚拟机。

  - 每一个虚拟服务均会对应一个负载均衡实例，每个实例由 Active 和 Standby 两个角色构成，分别作为主、备虚拟服务，同一个时刻只有主虚拟服务所在的 LB 虚拟机接收来自客户端的请求。
    - 当配置 2 个 LB 虚拟机时，主、备虚拟服务将分别被部署于两个虚拟机。
    - 当配置 3 个或 4 个 LB 虚拟机时，则按照当前 LB 虚拟机的负载情况，将主、备虚拟服务分别部署于当前负载较低的两台 LB 虚拟机。
  - 心跳检测机制通过周期性地发送心跳信号来确认 LB 虚拟机是否正常运行。当主虚拟服务所在的 LB 虚拟机故障时，VIP 将自动漂移至备虚拟服务所在的 LB 虚拟机，以保障虚拟服务正常使用。

  如图所示为部署 3 个 LB 虚拟机时高可用的实现示例，通过该模式很好地解决了 LB 虚拟机单点故障的问题。LB 虚拟机也可利用高可用机制滚动升级。当某台 LB 虚拟机升级时，可将 VIP 漂移至正常运行的虚拟机，以确保虚拟服务不受升级影响，提高了虚拟服务的可用性。

  ![](https://cdn.smartx.com/internal-docs/assets/c7ba2ed7/whitepaper_lb_08-2.png)
- **集群高可用**：支持主、备模式的集群灾备。即在 LB 实例组的两个 LB 实例中，同一时刻仅有一个实例中的一个虚拟机存在活跃虚拟服务。控制平面通过检测所有 LB 实例的健康状态判断是否触发灾备切换，一旦触发，VIP 将自动漂移至健康的 LB 实例，以确保业务继续正常运行。

  ![](https://cdn.smartx.com/internal-docs/assets/c7ba2ed7/whitepaper_lb_08-4.png)

## 高并发

可通过增加后端服务器数量的方式提升服务器的并发处理能力，以应对客户端日益增长的并发请求量。

如图所示，当客户端请求量为 10 K/s 时，服务器池中仅两台服务器即可满足客户端的并发访问需求，此时服务器的 CPU 利用率为 30%。当客户端请求量增长至 100 K/s 后，两台服务器的 CPU 利用率从 30% 突增至 90%，后端服务器的资源已严重耗尽。

此时，可以通过增加新的后端服务器的方式减轻现有服务器压力，以更好地应对客户端的并发请求。新增一台后端服务器后，CPU 利用率由 90% 降低至 50%。

这种通过水平扩容支持高并发的能力，极大地降低了由于突增流量而导致系统崩溃的风险。

![](https://cdn.smartx.com/internal-docs/assets/c7ba2ed7/whitepaper_lb_08-3.png)

## 技术实现

# 技术实现

使用 Everoute 网络负载均衡器时主要涉及如下资源：

- **虚拟服务**：每个虚拟服务通常对应一个应用。每个虚拟服务需要关联一个服务器池或服务器池组，LB 实例或 LB 实例组中的 LB 虚拟机在虚拟服务中负责通过 `VIP + Port` 的方式接收客户端请求，经过 NAT 地址转换后，根据服务器池中配置的调度策略将请求分发至后端服务器，实现流量负载均衡。
- **服务器池或服务器池组**：服务器池是为应用提供服务的一组后端服务器，负责接收并响应负载均衡器分发的客户端请求；服务器池组是将两个服务器池建立主、备关系，其中一个作为主服务器池、另一个作为备服务器池，以提供后端服务器的高可用。
- **健康检查监视器**：负责检查服务器池中一组后端服务器（即服务器池中的成员）的健康状态。

如图所示为资源间的基本关系示例，其中 VS 表示虚拟服务，Pool 表示服务器池，Monitor 表示健康检查监视器。同一个服务器池可以被多个虚拟服务关联，同一个健康检查器可以同时监控多个服务器池里的服务器。

## 技术实现 > NAT 模式

# NAT 模式

Everoute 网络负载均衡器支持 FullNAT 和 DNAT 两种模式。其中，FullNAT 模式包括 SNAT 和 DNAT。

- **DNAT**（仅适用于 VLAN 网络）：对客户端请求的目标地址进行转换。使用 VIP 与客户端通信，接收请求后，自动将请求的目标地址从 VIP 地址转为后端服务器的 RIP 地址。
- **SNAT**：对客户端请求的源地址进行转换。使用 LIP 与后端服务器通信，接收请求后，自动将客户端地址转换为固定的 LIP 地址，转换后服务器端不会感知真实的客户端地址。

---

## 技术实现 > NAT 模式 > FullNAT 模式

# FullNAT 模式

FullNAT 模式下将进行源地址转换（SNAT）和目标地址转换（DNAT），即与后端服务器通信时不仅将目标地址从 VIP 转换为 RIP，同时也将客户端源地址从 CIP 转换为 LIP。

**适用场景**

FullNAT 模式适用于后端服务器无须感知到真实的客户端地址的场景。

**网络要求**

FullNAT 模式对物理网络无特殊要求，只需保证网络连通性即可，可以灵活适配各种网络模型。

**地址转换流程**

FullNAT 模式的地址转换示意图如下：

![](https://cdn.smartx.com/internal-docs/assets/c7ba2ed7/whitepaper_lb_15.png)

1. 客户端发起请求报文，数据包中源地址为客户端地址（CIP），目的地址为虚拟服务地址（VIP）。
2. LB 虚拟机收到客户端请求后，经过 FullNAT 转换，数据包的源地址由客户端地址（CIP）转换为 LB 虚拟机的地址（LIP），目的地址由虚拟服务地址（VIP）转换为后端服务器真实 IP 地址（RIP）。此时，后端服务器无法感知客户端真实的 IP 地址。
3. 后端服务器接收并处理请求后，发送响应报文。数据包中源地址为后端服务器真实 IP 地址（RIP），目的地址为 LB 虚拟机的地址（LIP）。
4. LB 虚拟机收到后端服务器响应后，数据包的源地址由后端服务器真实 IP 地址（RIP）转换为虚拟服务地址（VIP），目的地址由 LB 虚拟机的地址（LIP）转换为客户端地址（CIP）。此时，客户端也无法感知后端服务器真实的 IP 地址。

---

## 技术实现 > NAT 模式 > DNAT 模式

# DNAT 模式

DNAT 模式下与后端通信时将仅对目标地址进行转换，从 VIP 转换为 RIP，但不对客户端源地址进行转换。

**适用场景**

DNAT 模式适用于希望后端服务器能够感知到真实的客户端地址并进行相关分析的场景。

**网络要求**

对于网络负载均衡器的 DNAT 模式，存在以下网络限制：

- 后端服务器与 VIP 之间需要二层可达，后端服务器与客户端之间需要二层不可达，以免后端服务器直接将响应信息发送至客户端。
- 后端服务器缺省路由的网关需配置为虚拟服务的 VIP 地址。

**地址转换流程**

DNAT 模式的地址转换示意图如下：

![](https://cdn.smartx.com/internal-docs/assets/c7ba2ed7/whitepaper_lb_16.png)

1. 客户端发起请求报文，数据包中源地址为客户端地址（CIP），目的地址为虚拟服务地址（VIP）。
2. LB 虚拟机收到客户端请求后，经过 DNAT 转换，数据包的目的地址由虚拟服务地址（VIP）转换为后端服务器真实 IP 地址（RIP），源地址保持不变。此时，后端服务器可以感知客户端真实的 IP 地址。
3. 后端服务器接收并处理请求后，发送响应报文。数据包中源地址为后端服务器真实 IP 地址（RIP），目的地址为客户端地址（CIP）。
4. LB 虚拟机收到后端服务器响应后，数据包的源地址由后端服务器真实 IP 地址（RIP）转换为虚拟服务地址（VIP），目的地址为客户端地址（CIP）。此时，客户端无法感知后端服务器真实的 IP 地址。

## 技术实现 > 调度策略

# 调度策略

调度策略，即负载均衡器选择后端服务器转发客户端请求时使用的策略。

![](https://cdn.smartx.com/internal-docs/assets/c7ba2ed7/whitepaper_lb_10.png)

Everoute 网络负载均衡器支持六种调度策略，具体介绍如下：

- **轮询**

  将客户端请求依次分发至服务器池中的每台后端服务器。例如，服务器池中有 3 台后端服务器 A、B、C，则客户端请求会依次分发给 A、B、C，然后重新开始循环。
- **加权轮询**

  采用轮询的方式，根据权重大小决定每台后端服务器处理的请求数量。例如，服务器池中有 3 台后端服务器 A、B、C，权重分别为 A（2）、B（3）、C（1），如果此时有 6 个来自客户端的请求，则将会有 2 个分发给 A、3 个分发给 B、1 个分发给 C。
- **最少连接**

  将客户端请求分发到连接数最少的服务器。例如，服务器池中有 3 台后端服务器 A、B、C，当前连接数分别为 A（5）、B（3）、C（7），则新收到的客户端请求会优先分发给 B。
- **加权最少连接**

  将客户端请求分发到连接数较少且服务器权重较大的服务器上。例如，服务器池中有 3 台后端服务器 A、B、C，权重分别为 A（2）、B（4）、C（1），当前连接数分别为 A（5）、B（3）、C（7），则会计算连接数和权重的比例，将新收到的客户端请求分发给连接数较少且服务器权重较大的 B。
- **源地址哈希**

  根据客户端请求的源地址，作为散列键从静态分配的散列表找出对应的服务器，若该服务器是可用的且未超载，将请求发送到该服务器。

  例如，服务器池中有 3 台后端服务器 A、B、C，客户端请求的源地址分别为 192.168.1.1、192.168.1.2、192.168.1.3，则将会根据源 IP 地址通过哈希函数计算得到一个数值，用该数值对服务器进行取模运算，得到访问的服务器序号。

  采用哈希法进行负载均衡，当后端服务器列表不变时，来自同一 IP 地址的客户端请求，每次都会映射到同一台后端服务器进行访问。
- **目标地址哈希**

  根据客户端请求的目标地址，作为散列键从静态分配的散列表找出对应的服务器，若该服务器是可用的且未超载，将请求发送到该服务器。

  与源地址哈希类似，当后端服务器列表不变时，访问同一 IP 地址（即 VIP）的客户端请求，每次都会映射到同一台后端服务器进行访问，适用于多个虚拟服务共享服务器池的场景。

## 技术实现 > 服务器池成员模式

# 服务器池成员模式

- **负载均衡模式**

  服务器池中的所有成员均为主成员，根据调度策略将流量均衡分发至后端服务器上，实现负载均衡。
- **主备模式**

  服务器池中包含一个主成员和一个备成员。在主成员健康的情况下，流量不会转发至备成员；当主成员不可用时，备成员将接管并处理流量。当主成员恢复健康后，流量将自动回迁至主成员。

  备成员是否生效依赖于服务器池是否配置健康检查监视器。如未配置健康检查监视器，则始终认为主成员健康，不会触发主备切换。
- **多主一备模式**

  服务器池中包含多个主成员和一个备成员。仅当所有主成员均发生故障时，备成员才会启用。该模式下的备成员主要用于兜底，例如提示服务器池发生故障，通常不处理实际流量。若服务器池已配置健康检查监视器，则备成员始终被默认为健康状态，不参与健康检查。

## 技术实现 > 服务器池组

# 服务器池组

服务器池组由两个服务器池组成，您既可以手动切换活动服务器池来进行流量转发，也可以为每个服务器池配置健康检查监视器，以启用服务器池的自动切换模式。该模式常用于虚拟服务跨集群灾备场景。您可以在主、备集群中分别配置对应的服务器池。当集群发生故障、触发灾备切换时，可按需同步触发服务器池的切换；也可以选择保持不切换，从而提供更高的配置灵活性。

服务器池切换逻辑如下：

- 当活动服务器池内的所有成员在 1 分钟内持续不健康时，系统会触发服务器池自动切换。
- 当服务器池在短时间内频繁切换时，则判定所有服务器池均不健康，系统会停止切换活动服务器池。
- 当服务器池恢复为健康状态时，系统会自动选择健康的服务器池作为虚拟服务的活动服务器池。如果两个服务器池均健康，则优先选择主服务器池作为活动服务器池。

在服务器池组中，服务器池的成员模式仅支持**负载均衡模式**和**主备模式**，不支持**多主一备模式**。

## 技术实现 > 健康检查

# 健康检查

健康检查监视器负责检查后端服务器的健康状况。当检测到存在后端服务器出现异常时，会将客户端请求自动转发至其他健康的后端服务器；当异常的后端服务器恢复正常运行后，可将其自动恢复到负载均衡服务中，继续承载业务流量。

当一个服务器池同时配置多个健康检查监视器时，只有所有健康检查监视器检查后端服务器的状态为健康，才认为后端服务器状态健康。

Everoute 网络负载均衡器支持四种检查模式，具体介绍如下：

- **TCP**：向后端服务器发起一个 TCP 连接请求，判断后端服务器是否响应或超时。

  具体流程如下：

  1. 根据健康检查配置，向后端服务器（IP + 健康检查端口）发送 TCP SYN 报文。
  2. 后端服务器收到请求后，如果相应端口正在正常监听，则会返回 SYN + ACK 数据包。
  3. 如果在超时时间之内，成功收到后端服务器返回的数据包，则认为服务正常运行，判定健康检查成功；如果在超时时间之内没有收到后端服务器返回的数据包，则认为服务无响应，判定健康检查失败。
- **UDP**：向后端服务器发起一个 UDP 请求，依赖于 ICMP 的错误消息，判断后端服务器是否健康。

  具体流程如下：

  1. 根据健康检查配置，向后端服务器发送 ICMP Echo Request 报文。
  2. 如果在超时时间内没有收到 ICMP Echo Reply 报文，则判定健康检查失败。
  3. 如果在超时时间内收到了 ICMP Echo Reply 报文，则向后端服务器发送 UDP 探测报文。
  4. 如果在超时时间内没有收到后端服务器返回的 ICMP Port Unreachable 报文，则判定健康检查成功；否则，判定健康检查失败。
- **HTTP**：向指定的 URL 发起一个 HTTP GET 请求，根据返回的结果或者状态码，判断后端服务器是否健康。

  具体流程如下：

  1. 根据健康检查配置，向后端服务器（IP + 端口 + 检查路径）发送 HTTP GET 请求。
  2. 后端服务器收到请求后，根据服务的情况返回相应的 HTTP 状态码。
  3. 如果在响应超时时间内收到了后端服务器的响应，且 HTTP 状态码符合期望，则认为健康检查成功；如果在响应超时时间内没有收到后端服务器的响应或 HTTP 状态码不符合期望，则判定健康检查失败。
- **ICMP**：向后端服务器发起 Ping 包检测，判断主机是否存活。

  具体流程如下：

  1. 根据健康检查配置，向后端服务器发送 ICMP Echo Request 报文。
  2. 如果在超时时间内收到了 ICMP Echo Reply 报文，则判定健康检查成功；如果在超时时间内没有收到 ICMP Echo Reply 报文，则判定健康检查失败。

## 技术实现 > 会话保持

# 会话保持

Everoute 网络负载均衡器在接到客户端访问请求后，以客户端的源地址作为判断关联会话的依据。在特定时间段内，所有源地址相同的连接将会被送往同一个后端服务器，以使某些需要保持会话的应用正常工作，保障会话的连续性和稳定性。当前默认的会话保持时间为 300 s。

设置会话保持后，若客户端本次的连接请求与上一次的连接请求之间间隔时间小于等于会话保持时间，则被认为是同一个会话，客户端请求仍会被分发至处理上一次请求的后端服务器，并刷新会话保持时间；若间隔时间大于会话保持时间，则被认为是新的会话，负载均衡器将根据调度策略重新选择后端服务器处理请求。

## 技术实现 > 流量控制

# 流量控制

流量控制用于针对某个虚拟服务限制出入峰值流量，即对 LB 虚拟机的虚拟网卡上的某个 VIP 进行出入方向的流量控制，避免其中一个 VIP 占用过多的带宽资源。

如图所示，同一台 LB 虚拟机上部署了两个虚拟服务，VIP 分别为 VIP 1 和 VIP 2。针对两个虚拟服务分别设置了 10 Mbps 和 20 Mbps 的峰值带宽，则在任何时刻，客户端访问 VIP 1 的总带宽峰值不会超过 10 Mbps，访问 VIP 2 的总带宽峰值不会超过 20 Mbps。

![](https://cdn.smartx.com/internal-docs/assets/c7ba2ed7/whitepaper_lb_17-1.png)

当多个虚拟服务复用同一个 VIP 时，则将以最后一次设置的峰值带宽为准，多个虚拟服务共享该带宽，总流峰值带宽不会超过最后一次设置的峰值。比如虚拟服务 1 和虚拟服务 2 复用同一个 VIP，虚拟服务 1 先设置 10 Mbps 的峰值带宽，配置虚拟服务 2 时，再次设置 20 Mbps 峰值带宽，则客户端访问 VIP 的总峰值带宽不会超过 20 Mbps。

流量控制采用 Linux TC 实现，基于 qdisc-class-filter 结构，如图所示。

![](https://cdn.smartx.com/internal-docs/assets/c7ba2ed7/whitepaper_lb_17-2.png)

具体实现步骤如下：

1. 创建 qdisc 队列，Linux 通过数据包的排队进行流量控制。
2. 创建 class 分类，即划分流量策略分类。例如，创建两种流量限速策略，分别为 10 Mbps 和 20 Mbps。
3. 创建 filter 过滤，将指定的 IP 和 Port 绑定至 class，以便流量控制能够对特定的虚拟服务生效。

此外，由于数据包的排队规则基于出方向，无法对入口方向的流量进行限制。因此，需要先将入方向流量重定向到 IFB 设备（用于流量控制的虚拟设备）上，再对 IFB 的出方向流量使用 qdisc 队列进行限制，以最终达到控制虚拟服务入方向流量的目的。

## 技术实现 > 访问控制

# 访问控制

访问控制用于限制哪些客户端 IP 禁止或者允许访问某个虚拟服务。该功能提供了更灵活的负载均衡业务级别的访问控制，能够有效地保护虚拟服务资源，防止恶意请求破坏正常请求的转发，提高负载均衡器的安全性和健壮性，同时也保障了对后端服务器的安全访问。

Everoute 网络负载均衡器支持黑名单和白名单两种访问控制类型。对于某个虚拟服务，只能设置其中一种类型。

- **黑名单**：只有名单内的 IP 无法访问该虚拟服务，其他所有 IP 都可以访问。
- **白名单**：只有名单内的 IP 可以访问该虚拟服务，其他所有 IP 都不可以访问。

如图所示，对于虚拟服务 1 设置了访问控制白名单，只有客户端 1 可以访问虚拟服务 1， 而对虚拟服务 2 设置了访问控制黑名单，只有客户端 1 不能访问虚拟服务 2。

- 当客户端 1 向虚拟服务 1 发出访问请求时，由于白名单设置仅允许客户端 1 访问，因此该访问请求会被接受；当客户端 1 向虚拟服务 2 发出访问请求时，由于黑名单设置不允许客户端 1 访问，因此该访问请求会被拒绝。
- 同理，当客户端 2 向虚拟服务 1 发出访问请求时，由于白名单设置仅允许客户端 1 访问，因此该访问请求会被拒绝；当客户端 2 向虚拟服务 2 发出访问请求时，由于黑名单仅设置不允许客户端 1 访问，客户端 2 可以访问，因此该访问请求会被接受。

![](https://cdn.smartx.com/internal-docs/assets/c7ba2ed7/whitepaper_lb_18.png)

## 技术实现 > 并发连接控制

# 并发连接控制

并发连接控制用于限制每个客户端能够与某个虚拟服务同时建立的连接总数，或者所有客户端能够与某个虚拟服务同时建立的连接总数。

- **从每个客户端 IP 接收的连接数限制**

  限制从每个客户端 IP 接收的连接数量以避免遭受恶意攻击。例如，应用的某些接口可能消耗较多资源，当攻击者发起大量对该接口的请求时，可能致使服务器负载过重而瘫痪。此时，如果设置从每个客户端 IP 接收的连接数限制，在连接数超过限制值时系统将拒绝该客户端的新增连接，已建立连接及其他客户端的连接不受影响，以确保服务器正常运行。
- **虚拟服务整体接收的连接数限制**

  对于某个虚拟服务，连接数的增加会增大对后端服务器 CPU 及内存的消耗。当资源消耗无限增加并达到服务器上限时，将影响当前业务。因此，用户可以根据业务需求对虚拟服务整体接收的连接数进行一定限制，以保证服务器正常运行。

如图所示，虚拟服务 1 从每个客户端 IP 接收的连接数限制为 10 K，虚拟服务 2 整体接收的连接数限制为 100 K。

- 客户端 1 访问虚拟服务 1 的并发连接数小于 10 K，符合从每个客户端 IP 接收的连接数限制，连接请求会被接受；客户端 2 访问虚拟服务 1 的并发连接数大于 10 K，超过从每个客户端 IP 接收的连接数限制，超出部分的连接请求会被拒绝。
- 客户端 1 和客户端 2 访问虚拟服务 2 的总并发连接数大于 100 K，超过虚拟服务整体接收的连接数限制，连接请求均会被拒绝。

# 支持 VPC 网络

如下图所示，以单个 LB 实例为例，展示了支持的负载均衡流量模式。其中 RS 表示后端服务器，Client 表示客户端。

在当前架构下，存在如下三种网络场景：

- **VLAN 网络场景**：与客户端通信、与服务器池通信均使用 **VLAN 网络**。
- **VPC 南北向场景**：与客户端通信使用 **VLAN 网络**、与服务器池通信使用 **VPC 子网**，用于处理 VLAN 网络与 VPC 网络之间的流量转发。
- **VPC 东西向场景**：与客户端通信、与服务器池通信均使用 **VPC 子网**，用于处理 VPC 内部的流量转发。
