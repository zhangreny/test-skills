# backup/2.3.0/SMTX 备份与容灾备份与恢复技术白皮书
## 关于本文档

# 关于本文档

SMTX 备份与容灾是由北京志凌海纳科技股份有限公司（以下简称 “SmartX”）自主研发的数据保护产品，可为运行在 SMTX OS（ELF）集群的虚拟机和 SMTX ZBS 块存储集群的存储卷提供全面的数据保护和灾难恢复解决方案。

SMTX 备份与容灾包含**备份与恢复模块**以及**复制与恢复模块**。本文档主要介绍**备份与恢复模块**的设计方案、架构与组件、工作原理以及高级特性。同步复制技术白皮书请参考 **《SMTX 备份与容灾-同步复制与恢复技术白皮书》**；异步复制与恢复模块的技术白皮书请参考 **《SMTX 备份与容灾-异步复制与恢复技术白皮书》**。

## 概述 > 功能简介

# 功能简介

备份与恢复模块主要提供以下功能：

- **备份虚拟机**

  备份服务以虚拟机的快照作为数据源，可以将虚拟机的配置信息和虚拟卷数据备份至集群外部的 NAS 服务器中。

  备份服务支持全量备份和增量备份两种备份类型，并可灵活定义备份周期和恢复点保留策略，满足多样的数据保护需求。
- **恢复虚拟机**

  支持通过读取备份存储库中的备份文件，对虚拟机进行原机恢复或重建一个新的虚拟机。当出现数据丢失或需要利用备份数据时，可以快速恢复虚拟机，保障业务的正常运行。

  ![](https://cdn.smartx.com/internal-docs/assets/ac489fd8/whitepaper_02.png)

## 概述 > 应用场景

# 应用场景

- **数据保护**

  防止因误操作、病毒感染、逻辑错误等因素导致的数据损坏和丢失。
- **数据复用**

  利用备份数据进行开发/测试、故障排除等，避免影响生产环境，最大化备份数据的价值。
- **数据安全**

  通过可靠的数据备份防止因勒索软件攻击导致大规模数据丢失和停机。

## 相关概念

# 相关概念

本章解释了手册中使用的专业术语。在阅读过程中，如果遇到不熟悉的概念，可在此快速查找其定义和详细解释。

**备份服务**

备份服务负责管理备份计划，并执行虚拟机的备份和恢复任务。

**备份计划**

备份计划用于详细规划和安排数据的备份操作。通过备份计划，用户可以定义备份对象、备份服务、备份存储库、备份周期、备份窗口、恢复点保留策略等关键信息，以确保数据的可靠备份，实现故障场景的数据恢复，从而保障业务的连续性和可靠性。

**备份对象**

备份的虚拟机。

**备份周期**

备份计划自动执行的频率和时间。用于指定备份服务对虚拟机进行备份的时间间隔。较短的备份周期可以实现更小的 RPO（Recovery Point Objective，故障恢复点目标）。例如，如果备份周期设置为 15 分钟，那么在发生灾难性事件时，最多可能丢失 15 分钟的数据。

**备份窗口**

允许备份计划自动执行的时间范围。

**全量备份**

全量备份是一种数据备份方式，旨在备份虚拟机的所有数据。在全量备份过程中，虚拟机的所有数据都会被备份到存储库。全量备份通常发生在初次备份时，并为后续的增量备份提供基准。

**增量备份**

增量备份是备份上次全量或增量备份后发生变化的数据。在增量备份过程中，只有上次备份以来发生变化的数据会被传输到存储库，这样可以大大减少数据传输的量和时间，提高备份的效率。

**指纹**

指纹是使用哈希算法对数据块内容进行计算得出的唯一标识符。在备份过程中，备份服务将对虚拟机数据分块，并对每一数据块计算指纹。在进行增量备份时，会对数据块的指纹进行比对，若同一数据块的指纹对比上次备份时相同，备份服务判断该数据块没有发生变化，不再传输该数据块。

**恢复点**

恢复点是在每次执行备份操作时生成的虚拟机数据状态，以 vmdk 文件的形式存储在备份存储库中。恢复点完整记录了虚拟机的数据和配置信息，反映了虚拟机在备份操作执行时的特定状态。

恢复点可以使虚拟机在需要时能够迅速恢复到特定的数据状态。例如在灾难恢复场景中将虚拟机恢复到某个先前的时间点，以恢复虚拟机的正常运行。通过定期生成恢复点，可以确保虚拟机的数据是最新的，保障业务的连续性和可用性。

**压缩**

对虚拟卷数据先执行压缩后再传输到备份存储库并存储为备份文件。

**原机恢复**

一种恢复模式，可将虚拟机的虚拟卷数据恢复到和生成恢复点时一致。

**重建**

一种恢复模式，可创建一个新的和生成恢复点时一致的虚拟机。

**RPO (Recovery Point Objective)**

数据恢复点目标，主要指在故障恢复场景中允许的数据损失的时间范围。RPO 以时间为度量单位，标示了从最近执行的备份时间点到发生故障时的时间跨度。较小的 RPO 值表示系统要求在发生故障时允许的数据损失较少。较大的 RPO 值意味着在故障发生时系统可以容忍较长时间范围内的数据丢失。

**ELF**

ELF 平台即指 SMTX OS 原生虚拟化计算平台，是当 SMTX OS 以超融合形态直接部署在 x86\_64/AArch64 架构的服务器上时，该超融合软件中基于 KVM 的虚拟化引擎组件。ELF 除提供虚拟机生命周期管理、电源操作、高可用、冷热迁移等基本功能外，通过与存储组件 ZBS 相结合，还能提供秒级快照、模板和克隆等高级的虚拟机服务。

ELF 简单易用，对 SMTX OS 超融合集群的所有操作均可通过 CloudTower 完成。

## 设计方案 > 方案概要

# 方案概要

SMTX 备份与容灾的设计方案采用了 Kubernetes Operator 模式，遵循声明式编程（declarative programming）范式，只需对备份相关资源设定期望的状态，而无需描述其实现流程，系统即能自行确定需要执行的操作以达到用户期望的状态。

Kubernetes Operator 模式包含了资源（Resource）和控制器（Controller）两个基本概念：

- **资源**

  Kubernetes 将被管理的对象抽象为“资源”。资源包含两种状态：期望状态和实际状态。期望状态通常由用户进行定义，实际状态通常由控制器负责更新和维护。Kubernetes 系统通过控制器监视资源对象，并执行业务逻辑，让资源的实际状态最终达到期望状态，从而达成用户的预期结果。

  Kubernetes 提供了多种默认的资源类型，也支持用户创建自定义资源（Custom Resource，简称 CR）。

  备份与恢复模块包含以下 8 种自定义资源：

  | 英文名称 | 中文名称 | 含义 |
  | --- | --- | --- |
  | BackupGroup | 备份组 | 代表一个备份对象的所有恢复点的集合。一个备份对象对应一个 BackupGroup，一个 BackupGroup 包含多个 BackupSlice。 |
  | BackupSlice | 恢复点链 | 代表一个备份对象的一条恢复点链。一条恢复点链由一个全量恢复点和多个增量恢复点组成。 |
  | BackupPlan | 备份计划 | 包含备份对象、备份周期和保留策略。 |
  | BackupExecution | 备份执行 | 手动或自动触发的一次备份计划的执行操作，包含一个或多个 BackupJob。 |
  | BackupJob | 备份任务 | 代表一个备份对象的备份任务。 |
  | RestoreJob | 恢复任务 | 代表一个恢复点的恢复任务。 |
  | RestorePoint | 恢复点 | 一个可以用于恢复的基本对象。 |
  | Repo | 备份存储库 | 保存备份数据的存储目录。 |
  | RepoRecoveryJob | 备份存储库恢复 | 代表一次从备份存储库中恢复数据的任务。 |
- **控制器**

  针对每一种资源，备份服务均有与之对应的控制器。控制器会监听资源的创建、编辑、删除等事件，并且根据资源的期望状态和实际状态，触发对应的执行逻辑，直至资源达到期望状态。如果对应资源的相关操作失败，且在有限的重试后无法达到期望状态，则对应的资源对象最终会处于失败或异常状态，且控制器不会再重试操作。

  例如，用户通过界面设置备份计划的执行周期后，系统会更新 BackupPlan 资源的期望状态（即执行备份计划的时间点）。负责 BackupPlan 资源的控制器，将根据 BackupPlan 资源的期望状态，自动在设置的备份时间点执行备份操作，从而使备份任务按照设定的周期执行。

## 设计方案 > 方案特点

# 方案特点

备份与恢复模块的设计方案具有如下特点：

- **无代理**

  无需在虚拟机中安装代理，减少管理和运维负担，且备份过程不影响虚拟机性能。
- **备份数据量优化**

  通过增量备份、跳过未分配的数据和压缩数据的方式，减少备份所需数据量，节省带宽和存储空间。
- **一致性保证**

  可借助虚拟机工具创建文件系统一致性的虚拟机快照，避免出现缓存数据和磁盘数据不一致，进而导致虚拟机恢复后部分文件可能损毁的情况。
- **高可用**

  依赖于 SMTX OS 的高可用功能，当备份服务所在节点发生故障时，可自动在其他节点继续运行并提供服务。

## 架构与组件 > CloudTower

# CloudTower

CloudTower 作为管理平台帮助用户完成备份与恢复的相关操作。CloudTower 是 SmartX 的多集群管理平台，可以帮助用户高效统一地管理多个数据中心、多个集群内的虚拟机，实现分布式云架构的构建。

CloudTower 包含 CloudTower UI 和 CloudTower Server 两个组件。

- **CloudTower UI**

  CloudTower UI 为备份模块提供操作界面，是用户进行相关操作的请求入口。用户可以在操作界面进行管理备份安装包、管理备份许可、安装、升级备份服务、管理备份计划等操作，CloudTower UI 会将操作请求发送至 CloudTower Server 进行处理。
- **CloudTower Server**

  用户在操作界面执行的备份相关操作时，CloudTower UI 会将操作请求先发送至 CloudTower Server，然后由 CloudTower Server 将请求发送至 Backup Service ，由 Backup Service 执行具体的操作。在执行备份和恢复操作时，Backup Service 也会向 CloudTower Server 请求 GraphQL API 以查询要备份的虚拟机的配置信息和创建虚拟机快照等。CloudTower Server 和 Backup Service 两者可相互调用。

## 架构与组件 > Backup Service

# Backup Service

Backup Service 是 备份与恢复模块的核心组件，负责执行备份和恢复任务，管理关联的备份计划和备份存储库等资源，并且与 CloudTower、ZBS 和 Repository 进行数据的查询、读取和写入等交互操作。

Backup Service 以虚拟机的形式运行在 SMTX OS (ELF) 集群中，在 Backup Service 虚拟机中，以容器形式运行了三个核心的服务模块：**kube-apiserver**、**ETCD** 和 **backup-controller-manager**。

## kube-apiserver

kube-apiserver 是 Backup Service 的数据总线和数据中心，是 Backup Service 和其他组件进行数据交互和通信的枢纽。kube-apiserver 提供了管理元数据的 HTTP REST API 接口。

在创建备份计划、执行备份和恢复任务等操作时，CloudTower Server、backup-controller-manager 会通过 kube-apiserver 查询、创建、修改或删除资源对象。kube-apiserver 会将获取的所有 Backup Service 的元数据信息持久化存储至 ETCD。

## ETCD

ETCD 是一个分布式键值存储软件，也是 Kubernetes 默认的数据存储引擎。在 Backup Service 中，ETCD 将 kube-apiserver 获取的 Backup Service 的所有元数据信息，如备份计划、备份任务、恢复点等，进行持久化存储。

## backup-controller-manager

backup-controller-manager 是 Backup Service 负责备份计划调度、执行备份和恢复逻辑的模块。

backup-controller-manager 包含了多个控制器，每个控制器负责对应的一类资源的调度、管理和任务执行工作。具体包括：

- 依照设定的备份周期和备份窗口，定时触发备份计划的执行；
- 管理备份存储库，定时探测备份存储库的网络连通性；
- 依照设定的恢复点保留策略，定时回收过期的恢复点；
- 执行备份任务，在任务成功或失败时清理临时资源；
- 执行恢复任务，在任务成功时清理临时资源，在任务失败时回滚虚拟机到恢复前的状态或者删除临时创建的虚拟机。

在执行备份和恢复任务时，backup-controller-manager 会与其他组件进行交互，包括：

- 监听 kube-apiserver 中资源对象的创建、编辑、删除事件，在执行逻辑后更新 CR 对象的状态等。
- 访问 CloudTower，查询并获取虚拟机元数据信息和虚拟机快照信息，创建和删除虚拟机快照等。
- 访问 ZBS 存储，通过 RPC 协议读取或写入 ZBS 存储卷数据。
- 访问 Repository，通过 NFS 协议写入或者读取备份文件数据。

## 架构与组件 > Repository

# Repository

Repository 是用于存储虚拟机备份文件的逻辑存储库，通常使用 SMTX OS（ELF）集群外独立的物理服务器作为备份存储库，例如 NAS 服务器。

Backup Service 支持通过 NFSv3 和 NFSv4 两种协议连接远程备份存储库。Backup Service 的 backup-controller-manager 会将备份存储库的远程目录通过 NFS 协议挂载到本地，在执行备份任务时将虚拟机数据保存到该目录中，在执行恢复任务时从该目录中读取数据。

---

## 架构与组件 > ZBS

# ZBS

ZBS 是 SMTX OS 集群的分布式块存储组件，用于为虚拟机提供存储服务。虚拟机的每一块虚拟磁盘，都对应 ZBS 中一个存储卷。备份服务对虚拟磁盘备份，即是对 ZBS 存储卷做备份。

备份服务对虚拟机进行备份时，Backup Service 的 backup-controller-manager 会创建虚拟机快照，然后查询 ZBS 的存储卷元数据信息，计算出需要备份的虚拟机数据块，并通过 RPC 协议从 ZBS 中读取虚拟盘的快照数据，将数据备份至备份存储库中。当需要恢复虚拟机时，Backup Service 将使用同样的方式将备份数据写回到 ZBS 的存储卷中，从而恢复虚拟机的虚拟盘数据。

## 工作执行流程 > 基本执行逻辑

# 基本执行逻辑

备份服务的 backup-controller-manager 模块中包含了一组控制器，用于控制每种资源的状态。当用户对某项资源执行操作后，可视为调整了该资源的期望状态。控制器在观察到用户的操作后，会根据资源当前的实际状态和期望状态，不断执行调谐（Reconcile）逻辑，使得资源的实际状态最终达到期望状态。

备份服务的基本执行逻辑如下：

1. 用户在 CloudTower UI 上进行某项操作后，CloudTower UI 将操作请求发送至 CloudTower Server。
2. CloudTower Server 将操作请求发送至 kube-apiserver，由 kube-apiserver 对相应的资源对象进行创建、编辑和删除操作。
3. backup-controller-manager 中相应的控制器观察到 kube-apiserver 中增加、修改、删除事件的发生，尝试查询最新的资源对象，进行相应的逻辑处理，并更新资源对象的实际状态。
4. CloudTower Server 观察最新的资源对象状态，并在 CloudTower UI 界面上展示。

![](https://cdn.smartx.com/internal-docs/assets/ac489fd8/whitepaper_04.png)

下面以备份计划的创建过程为例说明备份服务基本的执行逻辑，编辑和删除操作的逻辑基本相同。

1. 用户在 CloudTower UI 执行创建备份计划操作后，CloudTower UI 将操作请求发送至 CloudTower Server。
2. CloudTower Server 将该操作请求发送至 Backup Service 的 kube-apiserver 模块，kube-apiserver 将创建一个 BackupPlan 资源对象。
3. backup-controller-manager 模块的 BackupPlan 控制器观察到 kube-apiserver 创建了新的BackupPlan 后，将根据用户的设置，为每个待备份的虚拟机创建一个 BackupGroup 资源对象，同时计算备份任务的下一次执行时间，并在备份触发时间点创建一个 BackupExecution 资源对象，触发 BackupExecution 控制器执行相关备份操作。
4. CloudTower Server 观察到 BackupGroup 的创建后，会在备份计划详情页面展示备份的虚拟机，并且根据 BackupExecution 的状态，展示备份任务的执行结果。

## 工作执行流程 > 备份任务的执行流程

# 备份任务的执行流程

创建备份计划后，系统会根据备份计划设定的备份周期自动执行备份任务，用户也可选择手动执行，两种执行方式的内部处理流程基本相同。

![](https://cdn.smartx.com/internal-docs/assets/ac489fd8/whitepaper_05.png)

以手动执行备份计划为例，备份虚拟机的流程如下：

1. 用户在 CloudTower UI 界面上手动执行一个备份计划，前端将向 CloudTower Server 发送创建 BackupExecution 资源对象的请求。
2. CloudTower Server 向 backup-controller-manager 下发创建 BackupExecution 的请求。
3. backup-controller-manager 中的 BackupExecution 控制器在观察到 BackupExecution 的创建后，将会创建一个或多个 BackupJob 资源对象。BackupJob 控制器观察到 BackupJob 创建后，会通过 CloudTower 的 API 接口，为待备份的虚拟机创建虚拟机快照，该虚拟机快照会作为本次备份的数据源和下一次增量备份的对比源。
4. BackupJob 控制器读取虚拟机快照数据中的虚拟机配置信息，并将配置信息保存到备份存储库中。
5. BackupJob 控制器读取虚拟机快照数据，获取该虚拟机每块虚拟盘对应的 ZBS Snapshot 位置，并和对应 ZBS Server 建立连接，开始读取 ZBS Snapshot 的数据。
6. 备份服务不断将读取的 ZBS Snapshot 流式写入到备份存储库的 vmdk 文件中，直到完成所有虚拟盘的备份。
7. 根据备份任务的类型和结果，BackupJob 控制器会通过 CloudTower 的 API 接口删除虚拟机快照：

   - 如果当前备份成功，将会尝试删除上次成功的备份任务保留的虚拟机快照。手动备份与自动备份的虚拟机快照不会混用，因此手动备份第 7 步删除的是上次成功的手动备份虚拟机的快照，自动备份也是同理。如果此时虚拟机快照删除失败，该虚拟机快照将在下次删除恢复点时再尝试删除。而本次备份任务第 3 步中创建的虚拟机快照会保留至下一次备份任务成功后删除。
   - 如果当前备份失败，则会删除本次备份任务第 3 步中创建的虚拟机快照。

## 工作执行流程 > 恢复任务的执行流程

# 恢复任务的执行流程

用户可以通过恢复点对虚拟机进行恢复，并支持原机恢复和重建虚拟机两种恢复方式。两种方式的内部处理流程基本相同。

![](https://cdn.smartx.com/internal-docs/assets/ac489fd8/whitepaper_06.png)

以重建虚拟机为例，恢复任务的执行流程如下：

1. 用户在 CloudTower UI 界面上选择一个虚拟机恢复点，并单击“恢复”。CloudTower UI 将向 CloudTower Server 发送请求，由 CloudTower Server 向备份服务的 kube-apiserver 获取该恢复点的虚拟机配置信息。
2. 用户在操作界面选择“重建虚拟机”，并指定重建虚拟机的配置信息（如虚拟机名称、所在主机、网络映射等），CloudTower UI 将把带有虚拟机快照元数据信息和用户自定义配置信息的 RestoreJob 发送到 CloudTower Server。
3. CloudTower Server 将收到的 RestoreJob 中的参数信息转换为一个合法的 Create VM payload，并将带有该 payload 和恢复点名称的 RestoreJob 下发到 backup-controller-manager 中 的 RestoreJob 控制器。
4. RestoreJob 控制器收到 RestoreJob 后，使用其中的 payload 请求 CloudTower Server 创建一个新的虚拟机，并获取新虚拟机的虚拟盘信息。
5. RestoreJob 控制器开始读取备份存储库中该恢复点保存的备份数据。
6. RestoreJob 控制器不断将读取到的备份数据流式写入到 ZBS 存储卷中，直到完成所有虚拟盘的数据恢复。

## 工作执行流程 > 备份存储库恢复流程

# 备份存储库恢复流程

当备份服务出现异常且不可恢复，或 CloudTower 发生故障需要重新部署，同时需要恢复原有的备份管理功能时，可以通过恢复备份存储库功能，使用新的备份服务接管原来的备份存储库，利用存储库中原有的数据和元数据，快速重建备份计划并关联历史恢复点。

![](https://cdn.smartx.com/internal-docs/assets/ac489fd8/whitepaper_07.png)

备份存储库的恢复步骤如下：

1. 用户在 CloudTower UI 界面上选择恢复备份存储库，选择备份服务，输入原备份存储库地址，并点击“恢复”。CloudTower UI 将向 CloudTower Server 发送请求，发起备份存储库恢复任务。
2. CloudTower Server 将 RepoRecoveryJob 下发到 backup-controller-manager 中，其中的 RepoRecoveryJob 控制器将启动备份存储库恢复任务。
3. RepoRecoveryJob 控制器读取备份存储库中的元数据，重建出 BackupPlan、BackupGroup、BackupSlice 和 RestorePoint，并且建立对应的 Repo。
4. 当 RepoRecoveryJob 完成后，CloudTower Server 将读取上一步重建出的所有资源信息，将其同步到 CloudTower Server 中。
5. CloudTower UI 将展示出恢复出的所有备份计划和恢复点。

## 备份文件存储 > 备份存储库目录组织

# 备份存储库目录组织

BackupRepo 被挂载到备份服务器后，将会创建一个 FileStore 目录用于保存备份文件。FileStore 的目录结构如下：

```
.
└── repo-01/                           某个备份存储库
    ├── .backup-lock
    │   └── repo.lock                  锁文件，标识备份存储库已经被使用
    ├── backup-dr.metadata             备份存储库的元数据目录，用于服务故障时的数据恢复
    ├── backupgroup-01/             某台虚拟机
    │   ├── crs/                       备份对象相关的元数据目录，用于服务故障时的数据恢复
    │   ├── snapshot-01.tar/
    │   │   ├── vm-01.json             虚拟机配置信息
    │   │   └── vm-01.nvram            UEFI 启动项文件
    │   ├── snapshot-01-disk1.vmdk     第一次备份时，虚拟机的第一块磁盘数据
    │   ├── snapshot-01-disk1.fp       第一次备份时，虚拟机的第一块磁盘的指纹数据
    │   ├── snapshot-01-disk2.vmdk     第一次备份时，虚拟机的第二块磁盘数据
    │   ├── snapshot-01-disk2.fp       第一次备份时，虚拟机的第二块磁盘的指纹数据
    |   ...
    └── backupgroup-02/
```

FileStore 包含以下子目录：

- **`.backup-lock` 目录**

  .backup-lock 目录用于存储 `repo.lock` 锁文件。锁文件用于标识该 FileStore 已经被一个备份存储库所挂载，以避免被其他的备份存储库二次挂载。
- **`backup-dr.metadata` 目录**

  包含了备份服务和备份存储库相关的元数据。原备份服务故障时，可由其他备份服务读取元数据，进行虚拟机恢复。
- **`backupgroup` 目录**

  backupgroup 用于保存虚拟机的恢复点文件，每个虚拟机都有一个独立的 backupgroup 目录和 crs 子目录。每次对该虚拟机执行备份任务后，都会在此目录下创建一个 `tar` 文件和若干个 `vmdk` 文件：

  - `tar` 文件：包含了的虚拟机的配置文件（`.json` 格式）和包含 UEFI 引导信息的 `.nvram` 文件。
  - `vmdk` 文件：压缩后的虚拟机磁盘数据。每个 `.vmdk` 文件对应一块虚拟磁盘。
  - `crs` 目录：包含了该备份对象相关的服务元数据。用于原备份服务故障时，由其他服务读取元数据，进行虚拟机恢复。

## 备份文件存储 > 文件存储格式

# 文件存储格式

运行备份服务采用了 VMDK Stream Optimized 格式对虚拟盘快照数据进行存储。

VMDK Stream Optimized 格式具有以下特点：

- 原生支持稀疏（Sparse）存储格式，无需文件系统支持；
- 采用流式格式，适合顺序读写以及磁带等只支持顺序 IO 的设备；
- 支持文件内压缩；

VMDK 文件组织结构参考文档：[VMware Virtual Disks: Virtual Disk Format 1.1](https://www.vmware.com/app/vmdk/?src=vmdk)

## 备份文件存储 > 备份数据量优化

# 备份数据量优化

为了减少传输时间，并减少备份存储库占用空间，备份服务采用如下策略组合来减少备份的数据量：

- **增量备份**

  备份服务在备份时，会对备份数据分块计算指纹，指纹将作为元数据存储在备份存储库中。进行增量备份时，备份服务将会打开备份链中所有的指纹数据，并将其与当前虚拟机状态计算的指纹进行比较。若发现数据块的指纹相同，说明该数据块实际没有发生更新，备份时将跳过这些数据块。这一机制确保在执行备份任务时，只传输和保存实际发生变化的增量数据，从而提高备份效率并节省存储空间。
- **跳过未分配的数据块**

  当存储卷使用精简置备的存储策略时，备份服务会通过查询 ZBS 的存储卷元数据信息确认是否存在未分配的数据块，并在备份时跳过所有未实际分配的数据块，以减小备份文件的大小。
- **跳过全零数据块**

  执行备份任务时，备份服务将通过指纹识别已分配的数据块中是否有全零数据块。全零数据块将不会被写入备份文件中，以减小备份文件大小。
- **压缩数据**

  创建备份计划时默认开启压缩功能。执行备份任务时，备份服务会对读取到的 ZBS 存储卷的数据先进行压缩，再由备份服务传输到备份存储库。压缩算法为 zlib，压缩等级为 level 2（压缩比低，压缩速度较快）。

## 高级特性 > 一致性保证

# 一致性保证

备份运行中的虚拟机时，若虚拟机未安装虚拟机工具，备份服务将为虚拟机创建崩溃一致性快照。此类快照仅记录磁盘数据，不会强制将缓存数据刷写到磁盘。因此，若虚拟机存在缓存数据尚未全部写入磁盘的情况，创建的虚拟机快照可能会出现缓存数据和磁盘数据不一致，进而可能导致虚拟机恢复后出现部分文件损毁的情况。

若待备份的虚拟机中安装了虚拟机工具，在执行备份时，备份服务将借助虚拟机工具创建文件系统一致性的虚拟机快照，保证缓存数据和磁盘数据的一致性，从而确保恢复后的虚拟机可以正常工作。因此，建议用户为待备份的虚拟机安装虚拟机工具。

在不同操作系统中，通过虚拟机工具创建文件系统一致性快照的原理如下：

- **Linux 虚拟机**

  Linux 使用 FIFREEZE 来冻结文件系统，首先阻塞文件系统的写操作，然后 sync 文件系统的 inodes 信息和脏页。待冻结文件系统后，调用 ZBS 接口创建卷快照，创建成功后再解冻文件系统。
- **Windows 虚拟机**

  Windows 操作系统借助卷影复制服务（VSS）创建文件系统一致性快照，具体实现原理可参考 Windows [官方文档](https://docs.microsoft.com/zh-cn/windows-server/storage/file-server/volume-shadow-copy-service)。

## 高级特性 > 高可用

# 高可用

备份服务以虚拟机的形式运行在 SMTX OS（ELF） 集群中，因此，备份服务具有 SMTX OS 虚拟机的高可用特性。当备份服务虚拟机所在的主机宕机时， SMTX OS 会在集群内另一台健康的主机上重新创建一台备份服务虚拟机，以保证备份服务可以继续运行并提供服务。

备份服务发生 HA 时正在执行的备份和恢复任务，在虚拟机 HA 后会重新开始执行。
