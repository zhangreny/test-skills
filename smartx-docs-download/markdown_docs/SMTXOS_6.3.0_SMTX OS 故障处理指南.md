## 故障处理原则

# 故障处理原则

SMTX OS 集群在出现故障后，应以尽量减小对集群业务的影响，尽快恢复集群业务为第一原则。同时，您还需注意如下要求：

- 先确认故障现象并定位原因，避免在原因不明时盲目操作（如随意关机、重启或插拔磁盘），以免扩大问题。
- 若本文档提供了解决方案，请按文档执行；若本文档未涵盖，请联系 SmartX 售后工程师进行处理。
- 在故障处理过程中遇到的任何问题，应详细记录各种原始信息，请勿随意删除数据或日志。
- 除 SMTX Halo 一体机外，如遇服务器的硬件类故障，请尽快联系硬件供应商的售后工程师，并与 SmartX 售后工程师共同确认解决方案后，再实施硬件的更换。
- 集群恢复运行后需持续观察一段时间，确认故障已完全排除。

## 收集现网信息

集群在发生故障后，请收集并整理如下信息，并及时反馈给 SmartX 售后工程师。

| **现网信息分类** | **收集的具体内容** |
| --- | --- |
| 集群版本信息 | 软件版本号。例如 SMTX OS 6.3.0。  - 软件版本。例如基础版、标准版或企业版。 - 许可类型。例如订阅许可或永久许可。 |
| 故障发生时间 | 记录集群发生故障时的准确时间。 |
| 故障现象 | 详细描述集群发生故障时的具体表现。 |
| 故障前执行的操作 | 回忆并记录故障发生前，在集群上执行的具体操作及结果。 |
| 故障后执行的操作 | 记录故障发生后，在集群上执行的具体操作及结果。 |

## 采集集群日志

为了尽快定位出集群故障的原因，建议参考如下要求，采集集群的日志和节点信息。

- 日志采集类型：集群操作系统日志、SMTX OS 服务日志、Coredump 日志；
- 节点采集范围：集群全部节点；
- 服务采集范围：采集所有服务的日志；
- 采集时间范围：故障前 12 小时，以及故障发生后的 12 小时。

请参考《SMTX OS 管理指南》中的[管理日志](/smtxos/6.3.0/os_administration_guide/os_administration_guide_311)章节完成集群日志的采集和下载，并在故障申报时及时提交给 SmartX 售后工程师。

## 排障辅助工具

# 排障辅助工具

## 监控分析

SMTX OS 集群提供了监控分析功能，通过对集群内的多种资源对象的指标进行监控，并以视图和图表的方式组织和呈现数据。

运维人员可通过监控分析了解 CPU、内存、网络、存储等关键资源的使用情况。在故障场景中，可以通过指标的波动，确认问题发生的时间窗口，为进一步分析提供依据。

关于监控分析功能的详细介绍，请参考《SMTX OS 管理指南》的[监控分析](/smtxos/6.3.0/os_administration_guide/os_administration_guide_169)章节。

## 日志

日志是定位故障原因的重要依据。集群会记录系统运行过程中的详细事件，包括服务状态、错误信息、操作记录等。通过收集和分析故障时间点的相关日志，可以还原问题发生时的系统行为，帮助高效分析故障原因。

关于日志功能的详细介绍，请参考《SMTX OS 管理指南》中的[管理日志](/smtxos/6.3.0/os_administration_guide/os_administration_guide_311)章节。

## 报警

报警功能会在集群运行出现异常时触发提示。通过查看报警信息，可以第一时间感知风险，并可结合本文档对应的故障场景说明，指引采取后续的处理措施。

关于报警功能的详细介绍，请参考《CloudTower 使用指南》的[管理报警](/cloudtower/4.7.1/cloudtower_user_guide/cloudtower_user_guide_111)章节。

## 事件审计

事件审计可自动记录和展示集群事件，包括**用户事件**和**系统事件**

- **用户事件**：记录用户手动的操作，例如修改配置，资源的增删等。
- **系统事件**：记录系统自动触发的事件，例如虚拟机的 HA 重建，虚拟机重启等。

运维人员可在 CloudTower 的**事件**界面查看并追溯事件详情，用于确认问题原因、区分人为操作与系统异常，并为后续故障排查提供依据。

关于事件审计功能的详细介绍，请参考《CloudTower 使用指南》的[管理事件](/cloudtower/4.7.1/cloudtower_user_guide/cloudtower_user_guide_099)章节。

## 命令行

在进行故障原因排查时，相关命令行工具也可以提供辅助排障的信息。例如，常见的可辅助排障的命令行有：

- `zbs-node show_disk_status <disk_path>`

  该命令可查询物理盘的健康详情。
- `zbs-tool abnormal_detect find_dead_extent`

  该命令可查找 Dead 数据块以及其影响的虚拟机和虚拟卷
- `smartctl -x <disk_path>`

  收集物理盘的 S.M.A.R.T. 信息和自检日志等，可用于故障物理盘的报修等。

  ······

更多的命令行介绍，请参考[《SMTX OS CLI 命令参考》](/smtxos/6.3.0/cli_guide/cli_guide_summary)。

## 巡检中心

巡检中心可以对集群的配置、使用情况、性能和服务状态进行检查，由此识别集群当前存在的问题及潜在的风险，并为每条风险项提供针对性的解决方案，并生成检查报告供运维人员查看和分析，可帮助运维人员发现集群存在或潜在的问题，保障集群的健康运行。

巡检中心的具体使用方法，请参考[《巡检中心用户指南》](/inspector_cn/1.2.0/user_guide/user_guide_preface_generic)。

## 许可与资源限制类 > 软件许可过期

# 软件许可过期

## 故障说明

集群当前的许可将要过期或者已经过期，触发相关报警。

## 报警信息

- `软件试用许可距离过期不足 {.last_value} 天。`
- `软件试用许可已经过期。`
- `软件订阅许可距离过期不足 { .last_value } 天。`
- `软件订阅许可已经过期。`

## 故障影响

- **SMTX OS 6.2.0 以下，或 5.1.5 以下版本的 SMTX OS 集群**

  许可过期后，虚拟卷的 I/O 请求可能会被系统拒绝，虚拟机或其他直连存储服务的主机上将出现存储异常，并可能引发进一步的应用或系统异常。
- **SMTX OS 6.2.0 或 5.1.5 及以上版本的 SMTX OS 集群**

  许可过期后，系统将无法创建新的虚拟卷、iSCSI LUN 等资源，但已创建的资源不受影响。

## 故障原因

许可已过期或即将过期。

## 解决方案

请将集群当前所有许可信息提供给 SmartX 客户经理，并购买新的许可。在完成商务流程后，客户经理将提供新的许可码。

请按照如下流程更新许可：

1. 登录 CloudTower，进入集群的**设置** > **软件许可**页面。
2. 将许可码粘贴在**新许可码**处。
3. 单击输入框外部的空白区域，载入新许可信息。
4. 确认无误后，单击**更新**完成许可更新。

> **特别说明**：
>
> - 若 CloudTower 本身部署在许可过期的集群中，此时可能无法正常进入 CloudTower 完成更新许可操作。此时，可登录集群中任一主机，执行如下命令，完成许可更新:
>
>   `zbs-meta license set <license_number>`
>
>   其中，`license_number` 为新许可序列号。
> - 对于低于 5.1.5 版本的 SMTX OS 5.x.x 集群，低于 6.2.0 的 SMTX OS 6.x.x 集群，若集群发生了许可过期，虚拟卷和虚拟机可能因 I/O 长时间无响应而出现异常。通常在更新许可后，重启虚拟机即可恢复正常运行。

## 许可与资源限制类 > 集群数据空间超过授权限制

# 集群数据空间超过授权限制

## 故障说明

当集群当前挂载的物理盘空间超过许可限制后将触发相关报警。

## 报警信息

`集群数据空间超过授权限制。`

## 故障影响

超过授权许可的挂载的物理盘容量将不会被系统视为可用空间，可能会影响集群“空间利用率”类指标和报警的准确性。

该报警不会影响业务的正常运行，但建议及时处理，以确保资源计量与报警的准确性。

## 故障原因

系统检测到集群中实际挂载的物理盘总容量超过了当前授权许可中允许的容量上限。

## 解决方案

- **方案一**：联系 SmartX 客户经理购买新的许可以支持更大的许可容量，并完成许可更新。（推荐）
- **方案二**：参考《SMTX OS 运维指南》的[卸载物理盘](/smtxos/6.3.0/os_operation_maintenance/os_operation_maintenance_115)章节，卸载部分物理盘，以确保挂载容量在许可范围内。

## 许可与资源限制类 > 集群已使用逻辑容量超过限制

# 集群已使用逻辑容量超过限制

## 故障说明

ZBS 使用了固定大小的元数据 ID 空间，最大可以表示 2 PiB 的逻辑空间。虚拟卷在创建时，无论是否实际写入数据，即会占用相应的逻辑空间。快照和快速克隆卷在对应的数据区域未发生真实地数据写入之前，不额外消耗逻辑空间。当集群的逻辑空间接近耗尽时，将无法创建新的虚拟卷，且新数据无法写入。

## 报警信息

`集群已使用逻辑容量超过 { .threshold }。`

## 故障影响

- 当逻辑容量距离 2 PiB 尚有一定余量（TiB 级别）时，除无法创建超出剩余逻辑空间的新卷外，对系统运行没有明显影响。
- 当逻辑容量接近或达到 2 PiB 上限时，系统将：

  - 无法创建新的虚拟卷；
  - 已运行虚拟卷出现 I/O 延迟显著增加，甚至 I/O 错误。

## 故障原因

集群已使用逻辑空间已接近系统上限。

## 解决方案

- 删除部分较老快照或其他低优先级的虚拟卷；
- 将虚拟机或虚拟卷迁移到其他集群。

> **注意**：
>
> 删除快照或虚拟卷后，逻辑空间通常需要等待 3 分钟左右才会释放。

## 集群性能与负载类 > 主机的 I/O 带宽超过阈值

# 主机的 I/O 带宽超过阈值

## 故障说明

集群的 I/O 性能上限会因软件版本和硬件条件不同而存在差异。当主机的读带宽、写带宽或总带宽持续接近或超过阈值，系统将触发 I/O 带宽类报警，以提示可能存在的性能瓶颈风险。

为保障业务稳定运行，建议在业务上线前使用性能测试平台对集群进行压力测试，并将测试所得 I/O 上限约 70% 作为日常监控的报警阈值。

## 报警信息

- `主机 { .labels.name } 的 IO 总带宽超过 { .threshold }，状态持续超过 5 分钟。`
- `主机 { .labels.name } 的读带宽超过 { .threshold }，状态持续超过 5 分钟。`
- `主机 { .labels.name } 的写带宽超过 { .threshold }，状态持续超过 5 分钟。`
- `主机 { .labels.name } 的物理盘池 { .labels.instance_id } I/O 总带宽超过 { .threshold }，状态持续超过 5 分钟。`
- `主机 { .labels.name } 的物理盘池 { .labels.instance_id } 读带宽超过 { .threshold }，状态持续超过 5 分钟。`
- `主机 { .labels.name } 的物理盘池 { .labels.instance_id } 写带宽超过 { .threshold }，状态持续超过 5 分钟。`

## 故障影响

主机可能存在性能瓶颈，主机上的虚拟机和使用该主机存储资源的外部计算端可能会因受限于存储能力，而出现响应变慢或延迟增加的情况。

## 故障原因

主机的当前 I/O 流量超过设定的报警阈值且持续时间超过 5 分钟。

## 解决方案

- **场景一：预期内高负载（无需处理）**

  若相关报警是由于集群压力测试、计划内的高负载业务等引起的，带宽占用仅在短时出现峰值，且业务运行正常，可忽略报警，无需额外操作。
- **场景二：长期高负载（建议扩容）**

  若业务常态化占用节点的最大带宽能力，此时建议通过增加节点或者升级集群的网络、物理盘等硬件配置，来提高节点的最大吞吐能力，确保系统的性能冗余以应对突发负载需求或保障发生故障时稳定的性能表现。
- **场景三：异常高带宽（需排查分析）**

  若集群没有预期内的高带宽消耗业务，此时请通过 CloudTower 或可观测性平台，查看虚拟机的流量 TOP N 视图，判断是否有单个对象流量异常偏高。

  若有，建议检查虚拟机或关联虚拟卷的业务应用是否存在异常任务而产生了非预期流量。若存在非预期流量，建议先通过 CloudTower 对目标虚拟机或虚拟卷配置限流策略，以避免对其他正常业务的影响，然后再排查异常流量的具体原因。
- **场景四：节点负载不均（建议热迁移）**

  若流量相对均匀地分散在报警节点不同的虚拟机或虚拟卷上，且集群中其他节点上的负载压力相对较小，可考虑通过热迁移的方式将部分虚拟机迁移到压力较低的节点。

  需要说明的是，由于物理盘中的数据不会立即发生流动，所以迁移虚拟机或其他引发数据链路变化的动作，初期可能还会让计算端虚拟机的性能发生轻微下降。但运行数据逐渐在新节点聚集，集群整体的性能表现将比压力集中在某个主机上时候有所提高。

## 集群性能与负载类 > 主机的网络负载过高

# 主机的网络负载过高

## 故障说明

当主机网卡的网络带宽使用率持续超过设定阈值时，将触发相关报警。

## 报警信息

- `主机{.labels.hostname}上{.labels.bandwidth} GbE 网口{.labels._eth}的平均带宽使用率超过 {.threshold} 持续 5 分钟。与该网口关联的虚拟分布式交换机：{.labels._vds}`
- `主机{.labels.hostname}上{.labels.bandwidth} GbE 网口{.labels._eth}的发送带宽使用率超过 {.threshold} 持续 5 分钟。与该网口关联的虚拟分布式交换机：{.labels._vds}`
- `主机{.labels.hostname}上{.labels.bandwidth} GbE 网口{.labels._eth}的接收带宽使用率超过 {.threshold} 持续 5 分钟。与该网口关联的虚拟分布式交换机：{.labels._vds}`

## 故障影响

当网卡处于高负载状态时，依赖该网卡的轻负载业务可能出现性能下降的情况。

## 故障原因

网卡流量持续超过指定阈值。

## 解决方案

请根据发生报警的网卡的用途采取相应的处理措施：

- **存储网卡**

  请检查该网卡所在节点上的虚拟机 I/O 带宽使用情况，建议将高带宽负载的虚拟机迁移至其他节点，或者对其进行带宽限制。
- **业务网卡**

  请检查节点上的虚拟机业务网卡流量，确实其状态是否存在异常。若存在异常流量，请排查虚拟机内部的业务原因；若为正常业务流量，可通过在业务流量较低的节点上分散部署高网络需求的虚拟机，降低单网卡的负载。
- **接入网卡**

  在 CloudTower 查看该网卡所在主机的网络链路，定位哪些业务主机接入了该节点，可尝试通过业务迁移来降低对单个节点的接入带宽需求。

## 集群性能与负载类 > 主机的存储空间使用率过高

# 主机的存储空间使用率过高

## 故障说明

当主机的存储空间利用率超过设定阈值（默认值为 90%）时，将触发相关报警。

## 报警信息

- `主机 { .labels.name } 的存储空间使用率超过 { .threshold }。`
- `主机 { .labels.name } 的物理盘池 { .labels.instance_id } 存储空间使用率超过 { .threshold }。`

## 故障影响

- 若集群存在空间利用率高的主机时，空间分配策略将更倾向于在各节点间均衡数据，节点上接入的虚拟卷可能没有本地副本，集群可能会更加频繁地触发跨节点的数据均衡任务，对带宽和物理盘资源的消耗会更大。
- 若单个节点的空间使用率达到 100% 时，涉及新空间分配的 I/O 请求将会失败，影响数据写入和安全性。
- 当集群内多个节点处于高空间负载状态时，若发生故障，其他节点剩余空间可能不足以完成数据恢复，存在数据安全风险。

## 故障原因

节点上的空间利用率超过设定阈值。

## 解决方案

请根据具体的场景采取相应措施：

- **部分节点空间利用率明显高于其他节点**

  产生该现象的原因可能为：

  - 集群拓扑不够合理，此时通常伴随拓扑容量均衡的报警。请参考相应的报警说明处理；
  - 短时间内从容量较大的非模板虚拟机克隆出较多的虚拟机并快速写入数据，导致少量节点的数据压力聚集。此时，可通过将此类虚拟机通过热迁移的方式分散至其他节点，并适当限制写带宽以缓解系统压力，待系统容量均衡完成后再解除限速。
  - 节点上有大容量物理盘发生故障，导致空间利用率瞬间上升。此时请更换故障物理盘，恢复节点的正常容量后，报警将自动消失。
- **集群整体的空间利用率较高**

  此时会同步触发集群空间利用率相关报警，请参考相应报警说明处理。

## 集群性能与负载类 > 主机的 CPU 和内存使用率过高

# 主机的 CPU 和内存使用率过高

## 故障说明

当主机的 CPU 和内存使用率过高，超过设定阈值时，将触发相关报警。

尤其在**双活集群中**，仲裁节点仅承担关键的选举投票和元数据备份功能，预期其对 CPU 和内存资源的消耗应长期维持在低位，一旦出现高资源占用，即可能存在异常。

## 报警信息

- `主机 { .labels.hostname } 的 CPU 使用率超过 { .threshold } 持续 5 分钟。`
- `主机 { .labels.hostname } 的内存使用率超过 { .threshold } 持续 5 分钟。`
- `仲裁节点 { .labels.hostname } 的 CPU 使用率超过 { .threshold } 持续 5 分钟。`
- `仲裁节点 { .labels.hostname } 的内存使用率超过 { .threshold } 持续 5 分钟。`
- `SCVM { .labels.hostname } 的 CPU 使用率超过 { .threshold } 持续 5 分钟。`
- `SCVM { .labels.hostname } 的内存使用率超过 { .threshold } 持续 5 分钟。`

## 故障影响

当服务占用资源超过阈值时，系统可能会自动重启相关服务。若服务在重启后仍然迅速恢复到高资源占用状态，则可能会出现反复重启，影响集群的稳定性。具体影响请请结合对应服务异常的描述进行评估。

## 故障原因

常见的触发原因包括：

- **网络异常**

  当节点或集群的网络出现故障，导致大量请求无法被及时处理，从而造成部分服务积压大量请求，服务内存占用异常升高。
- **存储配置异常**

  对于内存占用过高的存储服务，有可能是通过命令行挂载了超过节点预设规格的物理盘，导致对数据的索引内存占用超出预期。

## 解决方案

- 若服务资源占用相关报警同时伴随着网络相关报警，通常在网络问题修复后，资源异常会自动恢复，无需额外处理。
- 若存储服务内存占用异常，且已确认挂载的缓存盘或数据盘容量超出节点的默认规格，请联系 SmartX 售后工程师调整内存配额以解决。
- 其他场景下，建议收集日志并直接联系 SmartX 售后工程师处理。

## 集群性能与负载类 > 集群存储空间使用率过高

# 集群存储空间使用率过高

## 故障说明

集群已经使用的存储空间比例超过设定阈值时，将触发相关报警。

## 报警信息

`集群存储空间使用率超过 { .threshold }`

## 故障影响

- 发生节点或物理盘故障时，集群剩余空间可能不足以完成数据恢复，导致数据安全等级可能会长期处于降级状态；
- 在集群发生新数据大量写入时，集群可能因存储空间耗尽而出现写入失败，从而影响业务的运行。

## 故障原因

集群中的已分配空间或有效数据空间比例超过设定的阈值。常见原因包括：

- 业务负载（含快照）占用空间持续增长；
- 节点或物理盘故障导致可用空间变少。

## 解决方案

- 若由硬件或服务故障引起集群有效空间降低而间接导致的空间使用率异常，请优先修复故障的硬件或服务，以恢复正常容量。
- 若异常硬件或服务无法在短期内修复，或者没有硬件或服务故障，而是集群正常的业务需求导致的空间不足，可考虑通过扩容物理盘或增加节点来提升集群整体容量。
- 若无法立即扩容，可考虑根据业务的实际需求，通常可以清理历史快照，优先删除时间较为久远，没有业务依赖的快照。

## 集群性能与负载类 > 集群的写缓存空间使用率过高

# 集群的写缓存空间使用率过高

## 故障说明

写缓存或热数据空间使用率超过设定的阈值时，将触发相关报警。

## 报警信息

- `集群的写缓存空间使用率超过 { .threshold }。`
- `主机 { .labels.name } 的热点数据大于该主机总缓存空间的 { .threshold }，状态持续超过 12 小时。`
- `主机 { .labels.name } 的物理盘池 { .labels.instance_id } 热点数据大于该物理盘池总缓存空间的 { .threshold }，状态持续超过 12 小时。`

## 故障影响

集群写缓存不足时，可能会引发集群的 I/O 性能下降：

- 写入请求在可直接写入容量层的情况下（中等负载时 256 KiB 大块对齐，和高负载时 8 KiB 对齐的副本模式写入），将不经过写缓存直接写入容量层。在缓存层性能显著高于容量层时，此时集群性能将会下降；
- 通常报警触发时，集群很有可能已经进入了写缓存空间高利用率引发的高负载状态。为保证必须写入写缓存的 I/O （纠删码写入以及 8 KiB 非对齐副本写入）有缓存空间可用，系统会同时启用缓存数据的加速下刷和业务数据的限流机制。此时业务层可用的系统资源会明显的小于低负载时的状态，出现较为明显的性能下降。

## 故障原因

常见原因包括：

- 集群中有大容量缓存盘异常或所在节点发生异常；
- 系统中出现了大容量卷的高强度随机写入数据流。如果同时存在高写入带宽的大容量卷，以及相对频繁的快照计划时，会更容易触发此问题。

## 解决方案

- 检查是否存在硬件故障或导致部分存储服务失效的故障发生，若存在，应优先修复相关故障；
- 通过监控功能，检查报警触发前是否存在突发的非预期大流量写入行为，若存在这种情况，可对异常的虚拟机或虚拟卷实施限流或关闭相关虚拟机来降低当前集群的负载。
- 若业务需求长期超过集群的硬件能力，可采取以下措施：

  - 若突发流量的规模固定，且其余时间容量层的 I/O 负载均保持在较低状态，可增加缓存盘空间以满足突发流量需求；
  - 若突发流量的业务优先级较低，可持续对此类业务实施限流策略，降低对集群其他高优先级业务的冲击；
  - 若大流量且大容量的业务优先级高且对单位时间内的吞吐能力敏感，可考虑将业务迁移至全闪集群。

## 集群性能与负载类 > 常驻缓存卷占用的缓存空间已达到指定阈值

# 常驻缓存卷占用的缓存空间已达到指定阈值

## 故障说明

集群的常驻缓存卷的占用的空间通常被限制用户指定的总常驻缓存空间内。然而，当物理盘活节点发生故障时，常驻缓存总空间将会减少，而已经分配出去的常驻缓存空间保持不变，从而可能导致实际占用比例超过指定阈值的情况。

## 报警信息

`集群内启用常驻缓存模式的卷占用的缓存空间已达到 { .labels.name }。`

## 故障影响

- 无法为新的卷开启常驻缓存模式；
- 现有常驻缓存卷的 I/O 请求可能无法完全落在缓存层上；
- 若该场景因缓存盘异常引发，则非常驻缓存卷的 I/O 性能可能会因为异常节点上剩余的缓存盘被大量占用而出现性能下降。

## 故障原因

提供常驻缓存的物理盘或节点发生故障或被错误移除，导致常驻缓存的总可用空间减少。

## 解决方案

- 优先修复物理盘和节点等硬件故障。在硬件故障被排除之后，常驻缓存的利用比例会自动恢复到阈值内，报警随即自然解除；
- 若短期内无法完成故障修复，且业务 I/O 已经受到影响，可以根据业务系统的优先级，临时将部分常驻缓存卷转化为非常驻缓存卷，以降低对缓存空间的占用，保证业务系统正常运行。

## 集群性能与负载类 > 集群发生数据恢复

# 集群发生数据恢复

## 故障说明

当集群内的数据的冗余等级低于预期，且节点上仍然有可用的物理盘容量空间时。系统会自动触发数据恢复操作，以尽快恢复到期望的冗余等级。

## 报警信息

`集群数据正在恢复中。`

## 故障影响

数据恢复期间，副本数低于预期的卷的安全等级会略有下降。对于新写入的数据，系统通常会启用临时副本机制维持与健康状态一致的数据安全等级；但对于已有数据，如果该数据恢复是由于故障引发副本丢失或异常，其安全等级相较于健康状态仍然存在一定的风险。

此外，处于恢复状态的数据访问性能会略有降低。

## 故障原因

数据恢复的触发原因可分为两类：

- **计划内的运维操作**

  例如对节点或存储服务进行关机、重启或维护时，无论是否启用维护模式，都可能触发对热数据的即时恢复。此外，若用户主动提升副本数，也会触发数据恢复将副本数补齐至期望状态。
- **故障事件引发**

  物理盘的数据恢复通常都是由于故障引发。此类数据恢复通常会伴随其他报警信息，可以结合相关报警进一步确认原因。 常见的故障原因包括：

  - 物理盘故障；
  - 网卡故障；
  - 内存故障
  - 节点故障；
  - 存储服务异常。

## 解决方案

- **对于计划内运维操作引发的数据恢复**

  此类场景通常无需额外处理，待运维操作完成后，系统会自动恢复至正常状态。
- **同时伴随节点/物理盘/网络/存储服务报警信息的数据恢复**

  此类场景下，请优先处理相关的节点、物理盘、网络或存储服务故障。待故障排除，数据恢复完成后也不再有新的报警出现，可以无需进一步处理。
- **数据恢复未伴随其他报警**

  此类场景大概率是由于临时的网络波动或者突发的硬件异常引起，可尝试通过查看主机的内核日志来确认是否存在相关异常，必要时可联系 SmartX 售后工程师定位具体原因。

## 集群的节点配置与拓扑类 > 集群主节点数量不足

# 集群的主节点数量不足

## 故障说明

主节点是指运行了 ZooKeeper 和 Meta 服务的节点。ZooKeeper 提供了分布式一致性协议，用于在分布式系统中协作多个任务；Meta 提供了元数据管理服务，主要负责管理存储对象、管理数据副本、控制访问权限和保障数据一致性等。

主节点的数量需要满足最小冗余要求：

- 当集群节点数 ≤ 4 时，预期应有 **3**个主节点；
- 当集群节点数 ≥ 5 时，预期应有 **5**个主节点；
- 对于双活集群，通常期望**优先可用域和次可用域各放置 2 个主节点**。

若主节点数量低于预期，将导致集群的容错能力下降，此时将触发相关报警。

## 报警信息

`集群中当前有 { .labels.master_num } 个主节点，少于预期的 { .labels.master_num_expected } 个主节点。`

## 故障影响

单个主节点发生故障，一般不会对集群的可用性产生影响。多个主节点发生故障可能对集群的可用性产生影响。若集群超过一半的主节点不可用，则元数据服务不可用，集群无法正常工作。

以 3 节点和 5 节点集群为例，故障主节点数量对集群的影响如下表所示：

| **集群节点数量** | **故障主节点数量** | **故障影响** |
| --- | --- | --- |
| 3 | 1 | 集群可正常提供服务 |
| 3 | ≥ 2 | 集群整体不可用 |
| 5 | ≤ 2 | 集群可正常提供服务 |
| 5 | ≥ 3 | 集群整体不可用 |

## 故障原因

集群当前配置的主节点数量低于期望值。常见于以下场景：

- 节点发生故障或手动移除了节点；
- 从 5 节点以下的集群第一次扩容至 5 节点时，主节点数量尚未完成调整。

## 解决方案

- 若该报警是由于节点移除导致：

  - 若集群内存在符合拓扑安全要求的存储节点，可将其转化为主节点，直至达到预期的主节点数量。节点角色转换操作，请参考《SMTX OS 运维指南》的[转换节点角色](/smtxos/6.3.0/os_operation_maintenance/os_operation_maintenance_55)章节。
  - 若集群处于 2 节点模式，需尽快扩容 1 个主节点，否则将持续运行在安全降级模式。节点扩容操作，请参考《SMTX OS 运维指南》的[添加节点](/smtxos/6.3.0/os_operation_maintenance/os_operation_maintenance_08)章节。
- 若该报警发生在集群的扩容过程中，扩容完成后，请将存储节点转化为主节点。主节点恢复至期望数量后，报警会自动消失。

## 集群的节点配置与拓扑类 > 集群拓扑结构不均衡

# 集群拓扑结构不均衡

## 故障说明

集群的拓扑感知能力，会在不同的拓扑结构（机箱、机架、可用域）之间分散存放数据副本，以提升容灾能力。然而，当集群内不同拓扑元素的资源分布严重不均衡时，单一节点的存储容量超过了其他节点的容量总和，在该节点发生障后，其他节点可能无法保证所有数据都可以恢复到期望的安全等级。

## 报警信息

- `如果 { .labels.zone_name } 整体失效，{ .labels.reserved_zone_names } 的内存资源不足以承载当前所有正在运行或暂停的虚拟机。`
- `不同可用域空间差大于集群总数据空间的 { .threshold }%。`
- `如果其他可用域整体失效，则可用域： { .labels.name } 数据空间不足以恢复当前全部数据至 { .threshold } 副本。`
- `机架 { .labels.name } 内机箱 { .labels.brick } 的总数据空间大于当前机架总数据空间的 { .threshold }。`
- `机架 { .labels.rack } 的总数据空间大于当前集群总数据空间的 { .threshold }。`
- `以下任一主机发生故障后，集群剩余空间不足以恢复数据至期望的冗余级别。主机：{ .labels.chunks }。`

## 故障影响

当报警涉及的拓扑元素（主机、机箱、机架或可用域）发生故障或失效时，可能出现部分数据无法完整恢复的风险，且部分数据可能出现无法访问的情况。

例如，当单一机架的总存储容量显著高于其他机架总容量时，尽管系统会避免将多个副本存放在同一主机上，但仍可能因资源分布不均，将多个副本集中在同一机架的不同主机上。在该机架整体失效时，相应数据将不可访问，直至该机架恢复正常运行。

## 故障原因

某一拓扑元素提供的资源能力（存储空间、内存或 CPU 核心）显著超过其他同级拓扑元素，导致资源分布失衡。

## 解决方案

建议优化资源分布，包括且不限于以下方式：

- 调整节点的物理部署位置，使各可用域、机架、机箱提供的资源更为均衡；
- 对能力较低的节点进行扩容（如增加内存或物理盘），提升整体均衡性。

## 集群的节点配置与拓扑类 > 集群拓扑不符合最佳实践

# 集群的拓扑不符合最佳实践

## 故障说明

集群的节点拓扑信息配置不符合最佳实践。

## 报警信息

- `集群存在未放入机架中的机箱：{ .labels.bricks }。`
- `集群已经开启了双活，但是硬件拓扑未正确设置，或主节点分布的拓扑不满足安全约束。`

## 故障影响

在集群中部分节点配置拓扑，部分节点未能配置正确的拓扑信息时。集群的数据分布可能无法满足拓扑安全的需求。在发生 TOR 或者电源等涉及到拓扑结构的故障时，集群可能无法以预期的可用性工作。

## 故障原因

部分节点配置拓扑信息，部分节点或机箱拓扑信息未配置。对于双活集群，要求每个可用域内都至少有 2 个主节点。

## 解决方案

登录 CloudTower，进入集群的**全部** > **机架配置**界面为每个节点配置正确的拓扑信息。

## 集群的节点配置与拓扑类 > 集群与 VMware vCenter 或 VMware vSphere 关联失败

# 集群与 VMware vCenter 或 VMware vSphere 关联失败

## 故障说明

SMTX OS 部署在 VMware ESXi 平台时，通常需要将 SMTX OS 集群所在的 VMware vSphere 集群的 VMware vCenter/vSphere 关联，以获取 VMware vSphere 集群的虚拟化信息用于问题定位和管理。

在 ESXi 主机部署 SMTX OS 软件时，需要配置每个 ESXi 主机的访问权限以实现 I/O 重路由的部署和升级。

当集群与 VMware vCenter 关联失败时，将触发相关报警。

## 报警信息

- `集群与 vCenter 关联失败。`
- `下列 { .labels.count } SCVM 与其所在 ESXi 主机的 vSphere 账号关联失败：{ .labels.invalid_account_hostname }。`

## 故障影响

- ESXi 关联账号无效， CloudTower 无法获取硬件信息并在主机详情中展示。
- ESXi 关联账号无效，导致无法正常部署和升级 I/O 重路由功能（ I/O 重路由仅在执行变更操作时需要 ESXi 账号，正常运行时不依赖）。若 I/O 重路由未成功部署，对应 ESXi 主机上的虚拟机在存储网卡失联、SCVM 异常或存储服务故障时，将无法依赖 I/O 重路由进行容错，仅能通过 HA 方式恢复业务。

## 故障原因

可能的原因如下：

- 未在 CloudTower 上正确配置集群的 VMware vCenter 连接信息；
- VMware vCenter 服务本身异常；
- CloudTower 无法连通 VMware vCenter；
- ESXi 主机的账号配置错误。

## 解决方案

- **VMware vCenter 关联异常场景**：

  通常可能由于 vCenter 密码变更后未在 CloudTower 中同步导致。请进入集群的**设置** > **vCenter Server** 页面确认配置信息是否正确。

  - 若配置不正确，请在更新配置后观察报警信息是否在 1 分钟内消失。
  - 若配置正确但报警仍然存在，请：

    - 检查 vCenter 是否配置了防火墙规则组织外部访问；
    - 使用命令行登录 CloudTower 虚拟机，确认能否正常 ping 通 vCenter。如果网络不通，请排查网络连通性问题。
- **ESXi 账号异常场景**

  - 进入集群的**设置** > **ESXi** 页面确认 ESXi 主机的配置信息是否正确。
  - 在 ESXi 主机上确认已开启**允许 SSH 登录**选项。

## 虚拟机类 > 虚拟机操作系统状态异常

# 虚拟机操作系统状态异常

## 故障说明

当虚拟机已安装并正常运行 VMTools 后，集群会定期探测虚拟机操作系统的运行状态。若系统检测到操作系统出现无响应或异常重启行为，将触发严重报警。具体表现包括：

- **未启用虚拟机操作系统 HA**：若操作系统无响应，将触发严重报警；
- **启用了虚拟机操作系统 HA**：若虚拟机在短时间内发生多次重启，但仍未恢复正常，也将触发严重报警。

## 报警信息

- `虚拟机 { .labels.vm_name } 发生操作系统故障。`
- `虚拟机 { .labels.vm_name } 在 { .labels.os_ha_duration } 小时内连续发生 { .labels.os_ha_count } 次操作系统无响应。`

## 故障影响

虚拟机操作系统可能已发生卡死、崩溃、异常重启等问题，导致虚拟机无法对外提供服务，可能造成业务中断或响应缓慢。

## 故障原因

虚拟机操作系统无响应或连续发生故障。

## 解决方案

1. 使用 SSH、VNC 等方式尝试登录虚拟机，若可以登录虚拟机，请检查系统负载、进程、日志等判断异常原因。
2. 若无法登录虚拟机，或登录后无法执行任何操作，请登录虚拟机所在的节点，执行以下命令导出虚拟机当前的内存快照（Memeory Dump）文件。

   `virsh dump --memory-only --live $vm_uuid $vm_uuid.dump`

   > **注意**：
   >
   > 快照文件大小通常接近虚拟机已分配内存大小，对于大规格虚拟机，请确保物理盘空间充足，建议将文件保存至 NFS 挂载目录（如 `/usr/share/smartx/images/tmp/`）。
3. 导出 Memeory Dump 文件后，可尝试在 CloudTower 上对该虚拟机执行强制重启操作，以重新引导系统恢复。若虚拟机仍然无法恢复正常，请联系 SmartX 售后工程师，并提交 Memeory Dump 文件。

## 虚拟机类 > 虚拟机无法重建

# 虚拟机无法重建

## 故障说明

主机故障触发 HA 时，由于集群中其他主机的内存不足，导致虚拟机无法再其他主机重建，将触发相关报警。

## 报警信息

`主机 { .labels.hostname } 上部分开启 HA 的虚拟机因其他主机内存不足无法重建。`

## 故障影响

如果该主机不可用，部分开启 HA 的虚拟机将无法在其他主机成功重建。

## 故障原因

集群内其他可用主机可分配内存不足，无法满足该主机上已开启 HA 的虚拟机重建所需的资源需求。

## 解决方案

- 为集群内主机增加可用内存资源；
- 释放部分虚拟机已分配的内存，以提高可用内存容量。

## 虚拟机类 > 主机无法提供为虚拟机 QoS 策略预留的 CPU 资源

# 主机无法提供为虚拟机 QoS 策略预留的 CPU 资源

## 故障说明

集群支持为虚拟机配置 QoS 策略，以限制或者保证虚拟机在运行时可以得到预期的 vCPU 调度时间片。当为所有虚拟机的 QoS 策略预留的 CPU 资源总量超过所在主机上物理 CPU 的最大能力时，将触发报警。

## 报警信息

`虚拟机 { .labels.vm_name } 的 CPU QoS 预留因节点资源不足而失效。`

## 故障影响

受影响的虚拟机无法获得期望配置的最小 CPU 资源，可能导致性能受到影响。

## 故障原因

虚拟机触发 HA 后，其所在主机无法满足其配置的 CPU QoS 资源需求。

## 解决方案

1. 进入 CloudTower，选择受影响的虚拟机。在虚拟机的详情页，选择**编辑基本信息** > **计算资源** > 高级属性，选择 **QoS策略**，确认预留的 CPU 资源是否符合预期。若预留 CPU 资源为期望值，请尝试将该虚拟机热迁移至 CPU 资源充足的其他主机。
2. 若集群 CPU 资源整体紧张，且无法通过虚拟机热迁移为主机腾挪出足够资源，可考虑先关停部分低优先级的虚拟机。
3. 若上述方案在实际场景中均不可行，可考虑扩容节点。

---

## 虚拟机类 > 为虚拟机分配的 vCPU 已显著超过主机可提供的数量

# 为虚拟机分配的 vCPU 已显著超过主机可提供的 vCPU 数量

## 故障说明

虚拟机未配置 QoS 策略或 CPU 独占模式时，默认将与主机上其他采用相同 CPU 分配策略的虚拟机共享 CPU 资源。如果集群中分配给虚拟机的 vCPU 总数显著超过所在主机上的物理 CPU 所能提供的 vCPU 数量，虚拟机可能因竞争 CPU 资源而出现性能下降的情况。

## 报警信息

- `主机 { .labels.hostname } 上的运行和暂停状态虚拟机分配的 vCPU 总个数已超过主机 CPU 个数的 { .threshold } 倍。`
- `主机 { .labels.hostname } 上未开启 CPU 独占且处于运行和暂停状态虚拟机分配的 vCPU 总个数已超过主机可共享 CPU 个数的 { .threshold } 倍。`

## 故障影响

虚拟机可能因频繁无法获得足够的 CPU 资源而出现性能波动或下降。

## 故障原因

主机上处于 CPU 共享模式的虚拟机需求的 CPU 资源总量显著超过物理 CPU 能力。

## 解决方案

- 若集群中其他主机 CPU 资源充足，可通过虚拟机热迁移均衡负载。
- 关闭部分低优先级虚拟机或降低其 vCPU 配置。
- 通过增加主机节点或提升主机 CPU 配置扩容集群资源。

## 虚拟机类 > 虚拟机 CPU 兼容性并非最优设置

# 虚拟机 CPU 兼容性并非最优设置

## 故障说明

虚拟化服务可以将物理主机上的 CPU 模拟为代数**小于或等于**当前物理 CPU 实际代数的虚拟 CPU，供虚拟机使用。此机制可确保虚拟机在不同代数 CPU 的主机之间实现平滑的热迁移。

**需要注意的是**：只有高代数的 CPU 可以模拟低代数的 CPU，反之不行。

当集群默认的虚拟机 CPU 兼容性不能兼容集群内所有主机，或并非最佳的兼容性设置时，将触发相关报警。

## 报警信息

- `集群虚拟机 CPU 兼容性为 { .labels.current_model }，不能兼容集群内所有主机。`
- `集群虚拟机 CPU 兼容性为 { .labels.current_model }，不是最优的兼容性设置。`

## 故障影响

- 若 CPU 兼容性代数**高于**某些主机的物理 CPU 代数，则使用该兼容性设置的虚拟机无法在这些主机上运行。
- 若 CPU 兼容性代数**低于**集群所有物理 CPU 代数，虚拟机可能无法充分利用物理 CPU 提供的高级指令集。

## 故障原因

集群默认的虚拟机 CPU 兼容性等级与集群中物理 CPU 的实际最小代数不一致。

## 解决方案

通常建议将集群的默认 CPU 兼容性设置为集群所有节点中物理 CPU 代数的最低值，以确保在大多数情况下，缺省配置的虚拟机也可以在集群中自由地进行启动、热迁移和 HA 以保证高可用性。对于有特殊性能需求的虚拟机，可以为其单独设置 CPU 兼容性。

集群的默认 CPU 兼容性设置，请参考《SMTX OS 管理指南》的[为集群中的所有虚拟机设置 CPU 兼容性](/smtxos/6.3.0/os_administration_guide/os_administration_guide_048)章节。

## 虚拟机类 > 虚拟机放置组异常

# 虚拟机放置组异常

## 故障说明

虚拟机放置组用于约束虚拟机的放置与迁移，确保其在合适的主机上运行。放置组规则会影响虚拟机在开机、迁移及 HA 重建过程中的目标主机选择。若配置不当，可能导致虚拟机在发生故障时无法按预期重建。

## 报警信息

- `虚拟机放置组 { .labels._pg_name } 中的下列 { .labels.vms_count } 台虚拟机当前所属主机不符合必须的放置组规则：{ .labels.vms}。`
- `虚拟机 { .labels.vm_name } 在下列 { .labels.placegroup_count } 个虚拟机放置组中的放置策略存在冲突：{ .labels.placegroups }。`

## 故障影响

- 若虚拟机所在主机不满足放置组规则，预期的虚拟机放置策略将被破坏。
- 若虚拟机存在多个冲突的放置组规则，在 HA 触发时，虚拟机可能无法在预期的目标节点上重建，影响业务可用性。

## 故障原因

- 虚拟机当前所在主机不满足放置组规则。
- 放置组本身的规则定义存在冲突。

## 解决方案

- **虚拟机所在主机不满足放置组规则**

  登录 CloudTower，进入集群，选择**全部** > **虚拟机放置组**，选择报警中涉及到的放置组，在详情页面查看检查虚拟机是否被错误地分配到不合适的放置组。若存在错误分配的情况，请根据实际需求，重新分配正确的放置组。若放置组规则是预期内的，请将虚拟机迁移至符合策略的主机，报警将自动消除。
- **放置组本身的规则定义存在冲突**

  登录 CloudTower，进入集群，选择**全部** > **虚拟机放置组**，选择报警中涉及到的放置组，检查并调整存在冲突的放置组规则，规则不再冲突后，报警将自动消除。

## 虚拟机类 > 虚拟机 CPU 独占功能未生效

# 虚拟机 CPU 独占功能未生效

## 故障说明

虚拟机启用 CPU 独占功能后，其 vCPU 不再与其他虚拟机共享物理 CPU 资源，预期可获得更稳定的性能表现。当虚拟机所在主机可调度的 CPU 资源不足，无法满足独占需求时，将触发相关报警。

## 报警信息

`处于开机或暂停状态的虚拟机 { .labels.vm_name } 的 CPU 独占未生效。`

## 故障影响

虚拟机的性能稳定性可能低于预期。

## 故障原因

虚拟机在 HA 后，目标主机的可用 CPU 核心数量不足，无法满足该虚拟机的独占需求。

此外，系统服务（如存储、虚拟化调度等）也需要占用部分核心，并且集群会为非独占虚拟机保留一定比例的 CPU 资源，导致单个虚拟机可独占的 vCPU 数量小于主机的物理核心总数。

## 解决方案

在确认目标主机扣除系统预留核心后，仍不足以满足虚拟机独占需求时，可采取以下措施：

- **迁移其他独占虚拟机**

  将部分独占虚拟机热迁移至其他主机，释放当前主机 CPU 资源，使报警虚拟机能够获得所需独占核心。
- **迁移报警虚拟机**

  若集群中存在具备足够独占核心数的主机，可将报警虚拟机迁移至该主机，满足其独占需求。
- **关闭独占功能**

  在无需强制 CPU 独占的情况下，可关闭报警虚拟机的 CPU 独占功能，以消除报警。

## 虚拟机类 > 使用了常驻缓存卷的虚拟机长期空闲

# 使用了常驻缓存卷的虚拟机长期空闲

## 故障说明

在混闪集群中，常驻缓存模式可确保虚拟卷的数据始终保存在缓存物理盘中，以获得稳定的性能表现。由于缓存资源在混闪集群中较为珍贵，如果使用常驻缓存卷的虚拟机长时间处于关机状态，将无法发挥该模式的性能优势，造成缓存资源浪费，因此会通过报警的形式提示。

## 报警信息

`虚拟机 { .labels.vm_name } 的虚拟卷 { .labels.vdisk_name } 启用了常驻缓存模式，但虚拟机关机已超过 { .threshold } 天。`

## 故障影响

该报警对系统功能和业务运行无直接影响，但由于存在常驻缓存空间浪费的情况，可能导致其他希望启用常驻缓存的虚拟卷没有足够的空间。

## 故障原因

开启常驻缓存的虚拟机关机时间超过设定阈值。

## 解决方案

- 如该虚拟机需继续保持常驻缓存，可忽略报警，或在 CloudTower 的报警配置中对所属集群关闭此类报警；
- 如该虚拟机不再是活跃的关键业务虚拟机，可在虚拟机的编辑页面关闭虚拟卷的常驻缓存模式，报警将自动清除。

## 主机类 > 主机未配置 IPMI 信息

# 主机未配置 IPMI 信息

## 故障说明

集群会定期通过 IPMI 接口收集主机上的硬件传感器信息，以便在检测到异常时及时生成报警并通知用户。如果主机未正确配置 IPMI，硬件传感器监控将无法正常运行。

## 报警信息

`下列 { .labels.count } 节点的 IPMI 信息未设置或设置不正确：{ .labels.invalid_account_hostname }。`

## 故障影响

主机的硬件传感器（比如 CPU 温度、风扇转速等）监控功能失效，可能导致硬件故障（电源故障）无法被及时发现。

## 故障原因

未正确配置主机的 IPMI 账号信息。

## 解决方案

- 若集群管理网络与 IPMI 网络之间无法互通，且也不需要进行硬件传感器数据采集，可以选择禁用此报警。
- 若主机未配置 IPMI 信息，请进入集群的**设置**> **IPMI 信息** 界面，配置主机的 IPMI 信息。
- 若主机已配置了 IPMI 信息但依旧触发了上述报警，请在主机上执行以下命令验证连接是否正常：

  `ipmitool -H <IP> -U '<user_name>' -P '<password>' -I lanplus user list`

  若无法正常返回结果，根据报错排查以下方向：

  - 确保节点可正常 Ping 通 IPMI IP，并确认端口 623 可用；
  - 确认 IPMI 账号是否拥有管理员访问权限；
  - 确认 IPMI 账号密码是否填写正确；
  - 检查服务器配置中的 **IPMI** 或 **DCMI overLAN 口**是否已开启；
  - 若同批次型号的服务器中部分节点无法连接，请确认其 BMC 固件版本是否一致。

## 主机类 > 主机的 LSM 元数据空间不足

# 主机的 LSM 元数据空间不足

## 故障说明

在每个节点上存储引擎（LSM）都会占用部分物理盘空间放置自身的元数据。元数据空间大小通常与节点所支持的物理盘总量相关，当元数据分区可用空间低于当前集群版本针对节点规格的最低要求时，将触发相关报警。

## 报警信息

- `SCVM {.labels.hostname} 的可用 LSM 元数据空间低于 {.threshold}。`
- `主机 {.labels.hostname} 的可用 LSM 元数据空间低于 {.threshold}。`

## 故障影响

在当前 LSM 元数据区域未满的情况下，执行 df /var/lib/zbs/lsm2db 命令查看当前元数据空间使用情况。示例输出如下：

```
/dev/loop0  
20466256     3162288    17077872  16% /var/lib/zbs/lsm2db
```

- **空间占用率低（<80%）**：对集群当前运行无影响。
- **空间占用率高（≥80%）**：后续 I/O 可能导致元数据分区写满，引发存储引擎异常，导致节点的数据安全等级降级。

## 故障原因

LSM 元数据空间在节点部署阶段通常会自动配置合适的大小。但如果集群初始安装版本较低（如 SMTX OS 4.0.8 及以下），初始规划的元数据空间可能不足以满足新版本要求。版本升级后在新的检查规则下，会出现此报警。

## 解决方案

请联系 SmartX 售后工程师解决。

## 主机类 > 密钥管理服务证书到期或连接异常

# 密钥管理服务证书到期或连接异常

## 故障说明

当密钥管理服务（KMS）的证书过期，或集群无法与 KMS 建立可信连接时，依赖外部加密服务的加密卷将无法正常工作。

## 报警信息

- `密钥管理服务{.kms.name}的证书距离到期已不足{.threshold}天。`
- `密钥管理服务{.kms.name}的证书已到期。`
- `集群无法与密钥管理服务{.kms.name}的服务器{.ip}建立可信连接。`

## 故障影响

密钥管理服务证书过期，或集群与密钥管理服务连接异常期间，集群内无法创建新的加密卷。已存在且正在 I/O 的加密卷短期内通常不会立即受到影响。但若集群的 Meta Leader 节点发生切换时，所有加密卷可能因无法获取密钥而停止 I/O。

## 故障原因

可能的原因包括：

- 密钥管理服务证书过期；
- 密钥管理服务与集群的连接异常；
- 密钥管理服务自身异常。

## 解决方案

- 如果是证书过期，在通过密钥管理服务获取新的证书后，在 CloudTower 的**系统配置** > **安全** > **密钥管理服务**中，选定对应的密钥管理服务，选择**上传证书**更新证书即可。
- 如果是密钥服务的链路异常，则优先检查密钥服务自身是否正常：

  - 如果有异常修复即可观察报警是否自动消失；
  - 如果密钥服务正常，但是报警持续存在。则登录使用密钥服务的集群中的 Meta Leader 节点。按照如下顺序进行排查：

    - 节点与密钥服务之间网络是否可以正常通信；
    - 从 Meta Leader 的日志中获取密钥服务连接异常的相关详细错误内容供进一步排查。

## 主机类 > 主机的 MegaRAID 控制器重置失败

# 主机的 MegaRAID 控制器重置失败

## 故障说明

系统在主机内核日志中检测到 MegaRAID 控制器重置失败，触发相关报警。

## 报警信息

`硬件故障：主机{.labels.hostname}的 MegaRAID 控制器重置失败。操作系统内记录的错误日志信息：{ .labels.message }`

## 故障影响

- 主机对物理盘的访问存在风险；
- 若操作系统未部署在独立的 M.2 盘上，主机的整体运行也存在风险；
- 在 MegaRAID 控制器异常期间，主机服务可能会出现整体失效的情况。

## 故障原因

当主机对物理盘的 I/O 请求长时间无响应，或者数据链路返回某些错误时，可能会触发 MegaRAID 控制器重置以清理未能正常完成的 I/O。在执行重置前，控制器上关联的物理盘可能出现长时间的 I/O 阻塞。

常见的故障原因包括：

- 某个物理盘不响应 I/O 请求；
- MegaRAID 固件或驱动异常。

## 解决方案

检查日志，确认是否存在诱发 MegaRAID 控制器重置的物理盘错误信息。若有，请移除该物理盘并更换为新盘。

若错误信息涉及到多个物理盘，可能为控制器固件或驱动异常，请联系相关硬件厂商确认是否升级固件或更换控制器。

# 主机硬件发生固件错误

## 故障说明

## 报警信息

`硬件故障：主机 { .labels.hostname } 发生固件错误，错误码为{ .labels.fault_code }。操作系统内记录的错误日志信息：{ .labels.message }`

## 故障影响

异常固件对应的硬件可能处于异常工作状态，存在潜在风险。

## 故障原因

可能原因包括：

- 固件自身代码存在缺陷；
- 固件在检测到硬件异常后上报操作系统。

## 解决方案

- 若异常固件对应的硬件是节点上的冗余设备（如物理盘或网卡），可临时隔离设备以避免对业务造成影响：

- 若异常固件对应的硬件是节点上的唯一硬件，不具备隔离条件，建议将主机置于维护模式，并将虚拟机迁移至其他健康节点。待故障解决后，再退出维护模式并根据需要将虚拟机迁回。
- 若要从根源解决硬件问题，建议联系对应的硬件供应商，确认是否为已知固件版本问题或硬件本身故障，并根据建议升级固件或更换硬件。

## 主机类 > 主机存储延迟过大

# 主机存储延迟过大

## 故障说明

主机上的存储延迟会随着硬件条件和业务负载的不同而有所波动。默认情况下，集群设置了一个远高于正常业务波动值（2 秒）作为报警触发阈值。当报警触发时，通常意味着主机关联的某些硬件或者网络环境存在异常。

## 报警信息

- `主机 { .labels.name } 的存储平均延时超过 { .threshold }。`
- `主机 { .labels.name } 的物理盘池 { .labels.instance_id } 存储平均延时超过 { .threshold }。`

## 故障影响

- 主机业务性能显著下降
- 虚拟机内部服务可能频繁触发存储高延迟报警

## 故障原因

常见原因如下，请结合 Cloud Tower 或可观测行平台上的主机物理盘延迟和存储网络延迟监控数据，确定具体的故障原因：

- 主机的网卡或光模块异常，导致丢包率升高，引发网络延迟骤增；
- 主机的 HBA 卡 或 RAID 卡异常，导致所有物理盘的 I/O 延迟剧烈上升，甚至出现 I/O 无响应的现象；
- 主机的物理盘异常， HBA 卡或 RAID 卡内部在处理物理盘异常时扩大了 I/O 影响范围。

## 解决方案

此类问题通常伴随着网络异常、物理盘异常或高延迟报警，请参考对应的报警信息解决方案处理，当故障排除后，主机的存储延迟报警通常将自动被解决。

## 主机类 > 主机的存储健康状态异常

# 主机的存储服务健康状态异常

## 故障说明

主机上的存储服务处于异常状态时将会触发报警。

## 报警信息

- `主机 { .labels.name } 的存储健康服务状态异常。`
- `SCVM { .labels.name } 的存储健康服务状态异常。`
- `主机 { .labels.name } 的物理盘池 { .labels.instance_id } 存储服务健康状态异常。`

## 故障影响

- 当前节点上的虚拟机可以继续运行，虚拟机的 I/O 请求将被转发至集群内的其他节点处理，性能相较于健康状态会有小幅下降；
- 集群整体的存储能力下降，下降比例通常为异常的节点数量占集群所有节点数量的比例；
- 当前节点上的数据将处于无法访问的状态，同时会触发数据恢复，在其他健康节点重建数据，并伴随数据恢复相关报警。

## 故障原因

存储服务异常多由硬件缺失或物理盘故障导致，常见情况包括：

- **网卡异常**：启用 RDMA 时，系统会在存储服务的配置文件中注入 RDMA 网卡信息。若发现网卡缺失，功能将无法启动并对外呈现为存储服务异常；
- **物理盘异常**：存储引擎会将已挂载给存储服务的物理盘记录在自身的元数据中。若该物理盘在存储服务停止服务的状态（如关机）被移除、损坏或无法被读取，存储服务将因无法确认数据的完整性而拒绝启动。
- **系统盘故障**：操作系统所在的物理盘出现异常可能导致存储服务因根分区只读或者操作系统本身的错误而被强制关闭。重启时若物理盘保持缺失无法读取状态，也会无法启动。

## 排查思路

检查存储服务日志 `/var/log/zbs/zbs-chunkd.*`，确认是其中否存在 NIC 或 BDEV 初始化异常信息，并定位关联的存储设备。

## 解决方案

- 对于可恢复或非预期的硬件移除情况，例如物理盘或网卡被误拔出，在将对应的硬件设备恢复后，存储服务会自动启动并解除报警。
- 若因网卡故障导致，请联系 SmartX 售后工程师进行网卡更换，然后重新启动服务即可。
- 若因物理盘故障导致，此时对应节点上的数据将因无法自证数据完整而无法做到无损地恢复服务。此时若节点上的数据在其他节点已经恢复完成，集群中无待恢复数据或无法访问的异常数据，请来联系 SmartX 售后工程师重置存储引擎，存储服务即可恢复健康，再将节点上其余健康的物理盘挂载回节点即可正常工作。

## 主机类 > 主机的工作状态未知

# 主机的工作状态未知

## 故障说明

当集群中存在与存储网络处于隔离状态的主机，或者有未能正常开机的主机，其工作状态将无法被感知，从而触发相关报警。

## 报警信息

- `主机 { .labels.hostname } 的工作状态未知。`
- `SCVM { .labels.hostname } 的工作状态未知。`

## 故障影响

- 主机与存储网络失去连接；
- 若主机上的虚拟机启用了 HA，虚拟机将自动迁移至其他可用节点以保持业务的高可用性。

## 故障原因

常见原因通常为如主板、电源、BIOS 固件、内存等硬件故障导致。若主机处于关机或重启状态也有可能因无响应导致此报警。

对于 SCVM，还可能因为 SCVM 在部署阶段未配置为随主机自启动，导致主机重启后 SCVM 依旧保持关闭状态导致。

- 对于 SCVM，请登录 VMware vCenter 或 SCVM 所在 ESXi 主机，检查 SCVM 是否处于运行状态。若 SCVM 未开机，请选择开启 SCVM，正常启动后报警将自动被解决。建议将 SCVM 配置为随主机自动启动，以避免类似情况再次发生。

## 主机类 > 主机处于断电关机状态

# 主机处于断电关机状态

## 故障说明

物理主机处于断电关机状态，触发相关报警。

## 报警信息

- `主机 { .labels.target_hostname } 当前电源状态处于断电关机状态。`
- `SCVM { .labels.target_hostname } 所在物理主机当前电源状态处于断电关机状态。`

## 故障影响

故障主机无法提供服务。如运行在该主机上的虚拟机启用了 HA，将触发 HA 迁移至其他满足资源需求和放置组策略的主机上。该节点关联的数据，在集群剩余空间充足且节点数量满足要求的情况下，也会在其他节点上恢复。

## 故障原因

主机未处于开机状态，可能原因包括：

- 计划内的运维动作（正常关机、维护）。
- 非计划性的关机，主机可能遭遇了非预期的故障，例如：

  - 主机断电，UPS 故障或供电超出 UPS 承载能力；
  - CPU、内存、主板等硬件故障。

## 解决方案

- **计划内关机**

  通常无需额外处理，待运维操作完成后重新开机，报警将在主机正常启动约 1 分钟后自动解除。
- **非预期关机**

  1. 通过串口或 IPMI 登录控制台，检查是否存在电源失效时事件或硬件相关报警：

     - 若有明确的硬件故障，请联系硬件厂商处理；
     - 若无异常情况，请检查集群的上下电日志，确认是否存在人工关机操作。
  2. 若未发现明硬件报错，请尝试开机：

     - 若系统能正常引导并启动服务，报警会自动解除。此时建议同时检查主机的内核日志（`/var/log/message`）确认是否存在硬件异常记录；
     - 若系统无法正常引导，通常是硬件缺失（例如发生系统故障或硬件已被非预期移除）导致，需要联系硬件厂商进一步排查。

## 主机类 > 主机的系统分区冗余度不足

# 主机的系统分区冗余度不足

## 故障说明

主机上包含系统分区的物理盘数量不足 2 块，导致系统分区冗余度不足。

## 报警信息

- `主机 { .labels.hostname } 的系统分区冗余度不足。`
- `SCVM { .labels.hostname } 的系统分区冗余度不足。`

## 故障影响

SMTX OS 系统安装在软 RAID 1 时，采用两块物理盘构建 RAID 1 存放系统分区。当其中一块系统盘故障时，操作系统会自动切换至另一块系统盘，业务不受影响，但系统的安全级别下降，若另一块系统盘也发生故障，节点将无法提供服务。

## 故障原因

某块包含系统分区的物理盘发生故障，导致物理盘所在主机的系统分区冗余度不足。

## 排查思路

登录主机后：

1. 执行 `lsblk` 命令列出所有的 md 设备，常见的设备名为 md0、md126、md127，下列命令以 md127 为例；
2. 执行 `mdadm --detail /dev/md127` 命令检查 md 设备下的成员盘数量和状态是否正常；
3. 使用 `ls -lah /sys/block/md127/md/` 命令检查对应 md 设备路径下的成员盘数量。

若 mdadm 输出中，成员盘数量少于 2，或成员盘存在 `faulty` 状态，则说明系统分区冗余度不足。

## 解决方案

请在 CloudTower 界面卸载对应的物理盘，然后重新挂载一块 SSD 物理盘，并选择含元数据分区的缓存盘或含元数据分区的数据盘以建立系统分区，从而保证该主机包含系统分区的物理盘达到 2 块。

## 主机类 > 主机的系统分区进入只读状态

# 主机的系统分区进入只读状态

## 故障说明

主机上操作系统分区所在物理盘发生故障，文件系统进入自我保护的**只读**状态。

## 报警信息

`节点 { .labels.hostname } 的系统分区进入只读状态。`

## 故障影响

- 节点上运行的存储服务将在一定时间后自动切断，无法再对外提供服务；
- 其他节点可能因无法写入日志或数据而运行异常；
- 集群中的虚拟机将触发 HA。

## 故障原因

主机根分区所在的物理盘发生故障，出现了 I/O 错误或者是超长时间没有响应。

## 排查思路

1. 登录异常节点。

   > **说明**：
   >
   > 若因为操作系统日志无法写入而造成无法使用 SSH 远程登陆，可尝试使用 IPMI 等带外方式登录
2. 执行 `mount |grep " / "` 命令（注意：斜杠 `/` 前后都有空格）。示例如下：

   ```
   [root@zbs17-73 10:34:54 ~]$ mount  |grep " / " 
   /dev/md127 on / type ext4 (ro,relatime)</td>
   ```

   若输出中`/`目录挂载状态为 ro (readonly) 模式，或在根分区中任意创建一个文件，发现系统返回失败，并且提示 Read-Only file system，即可判断根分区已经进入了只读状态。

> **注意**：
>
> 由于根分区进入只读状态后，系统日志可能无法写入。在进行下一步操作前，请先尝试查看 `/var/log/message`，尽量截图日志文件的最新内容，如果网络可用，也可将 message 文件复制到其他健康节点以备后续调查。

## 解决方案

1. 由于根分区进入只读状态后，节点上的所有服务预期都无法稳定运行。对配置了 HA 的虚拟机，确认虚拟机已在其他节点正常 HA 后再重启节点已恢复服务。
2. 若节点重启后系统无法引导，说明物理盘已出现严重故障，请通过 IPMI 登录节点，定位故障盘，并联络硬件供应商进行处理。
3. 若节点重启后系统能正常引导，说明物理盘上的操作系统核心数据正常，此时可通过日志和物理盘错误计数确认盘符，并联系硬件供应商处理。

## 主机类 > 主机最后一个系统分区或元数据分区所在的物理盘发生故障

# 主机最后一个系统分区或元数据分区所在的物理盘发生故障

## 故障说明

故障物理盘包含主机上最后一个系统分区或元数据分区。

## 报警信息

- `主机 { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 出现故障，且物理盘上存在主机最后一个系统分区或元数据分区。`
- `SCVM { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 出现故障，且物理盘上存在 SCVM 最后一个系统分区或元数据分区。`

## 故障影响

因该物理盘包含了主机最后一个系统分区或元数据分区，因此故障物理盘所在的节点可能无法正常工作。

## 故障原因

故障的原因可参考同时出现的物理盘故障类型报警信息进行分析。

## 解决方案

1. 在 CloudTower 中将故障盘所在主机设置进入维护模式；
2. 在主机上插入新的 SSD 并挂载，并设定为包含元数据分区：

   - 不分层模式下，设置新 SSD 用途为**包含元数据分区的数据盘**；
   - 分层模式下，新 SSD 用途为**包含元数据分区的缓存盘，或包含元数据分区的数据盘（单一 SSD 分层模式）**。
3. 在新的物理盘挂载成功后，此条报警将自动消除。
4. 将故障物理盘卸载，替换为新的 SSD 并挂载，同样设定为包含元数据区；
5. 主机上的物理盘报警都消除之后将主机退出维护模式。

## 主机类 > 主机系统分区空间使用率过高

# 主机系统分区空间使用率过高

## 故障说明

系统分区的空间使用率过高，存在系统服务因为无法获得存储空间而发生异常的风险。

## 报警信息

- `主机 { .labels.hostname } 的系统分区空间使用率已经超过 { .threshold }。`
- `仲裁节点 { .labels.hostname } 的系统分区空间使用率已经超过 { .threshold }。`
- `SCVM { .labels.hostname } 的系统分区空间使用率已经超过 { .threshold }。`

## 故障影响

当空间利用率达到 100% 时，对应主机可能无法正常工作。

## 故障原因

可能是日志文件等占用了系统分区过多的空间，或对应的服务进程出现了异常。系统服务占用的日志空间都已经配置了限额，会定期压缩回收。出现此类空间报警的原因通常是某些人工运维动作（例如临时上传了大体积的 ISO 文件，进行其他问题排查时在非指定目录生成了虚拟机或者程序的 Core Dump 文件后未清理等动作造成的根分区空间不足）。

## 排查思路

1. 对应节点上，使用 `df -h` 命令查看 `/` 挂载点的空间使用率；
2. 使用 `du -shx | sort -h` 在各层级目录下检查占用空间较大的文件，首先检查 /home 目录下是否有可疑的临时文件，此外可重点关注 /var/log/ 和 /var/lib 目录；
3. 如果确认是 `/var/log/zbs` 目录下的日志占用空间过大，则可以在咨询 SmartX 售后工程师的情况下删除同名文件中生成时间较早的日志。

## 解决方案

清理占用空间较大的文件，或相应的服务恢复正常运行。

## 物理盘类 > 物理盘发生硬件故障

# 物理盘发生硬件故障

## 故障说明

物理盘发生物理性损坏，导致物理盘发生命令超时、I/O 错误或者严重的设备错误等。

## 报警信息

物理盘发生硬件损坏的常见报警信息如下:

- `主机 { .labels._hostname } 的一块物理盘{ .labels._dev }扇区 { .labels.sector } 发生 I/O 错误。操作系统内记录的最新错误日志信息：{ .labels.message }`
- `主机 { .labels._hostname } 的一块物理盘{ .labels._dev }发生命令超时。操作系统内记录的最新错误日志信息：{ .labels.message }`
- `主机 { .labels._hostname } 的一块物理盘{ .labels._dev }扇区 { .labels.sector } 发生严重设备错误。操作系统内记录的最新错误日志信息：{ .labels.message }`

## 故障影响

当物理盘发生硬件故障时，根据硬件损坏的严重程度，可能导致命令行超时和发生 I/O 错误，系统可能无法正常识别损坏的物理盘，导致物理盘离线。物理盘离线后，也将触发物理盘的隔离和数据恢复。

## 故障原因

物理盘发生硬件损坏。

## 解决方案

- 物理盘通常预留有一定冗余空间，用于替换少量损坏扇区或数据块。当错误数量未达到系统阈值时，物理盘固件会自动启用冗余扇区继续工作。若仅出现少量 I/O 错误且错误计数未达到不健康或亚健康的状态的触发阈值，物理盘可继续使用，无需立即处理。
- 若物理盘确已严重损坏，建议参考《SMTX OS 块存储运维指南》的[物理盘运维](/smtxos/6.3.0/os_operation_maintenance/os_operation_maintenance_115)章节，对故障物理盘进行卸载和更换。完成换盘后，请在 CloudTower 报警页面手动解决相关报警。

## 物理盘类 > 物理盘寿命不足

# 物理盘寿命不足

## 故障说明

物理盘的寿命不足通常指物理盘接近其设计或预期的使用寿命，可能即将出现故障或性能下降。这是一种预警信号，提示需要采取措施以防止数据丢失或业务中断。

## 报警信息

- `主机 { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 寿命不足 { .threshold }`
- `SCVM { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 寿命不足 { .threshold }。`

## 故障影响

物理盘寿命到期后，故障风险和性能下降的概率将显著提高。

## 故障原因

直接原因是物理盘固件上报物理盘的寿命不足，通常情况根因为物理盘累计总写入数据量达到物理盘设计阈值的 80%。

## 排查思路

执行 `smartctl -A` 命令确认对应物理盘的剩余寿命相关数值。

- **HDD 和 SATA/SAS SSD**

  执行 `smartctl -A` 命令获取物理盘寿命相关的指标，示例如下：

  ```
  $smartctl -A /dev/sdc
  ```

- **NVMe SSD**

  执行 `smartctl -A` 命令获取 NVMe SSD 盘寿命相关的指标，示例如下：

  ```
  $ smartctl -A /dev/nvme0n1
  ```

  - **Available Spare**：备用空间，数值越低，则 NVMe SSD 剩余寿命越低，当前报警采用本指标数值。当剩余寿命低于（默认为 20%）时，将触发报警。
  - **Percentage Used**：损耗，数值越高，则 NVMe SSD 剩余寿命越低。

2. 完成故障处理后，请在 CloudTower 报警页面手动解决相关报警。

## 物理盘类 > 物理盘 S.M.A.R.T. 检测不通过

# 物理盘 S.M.A.R.T. 检测不通过

## 故障说明

S.M.A.R.T. (Self-Monitoring Analysis and Reporting Technology，自我监测、分析及报告技术）是一种自动的状态检测与预警系统和规范。通过在物理盘内的检测指令对物理盘的硬件如、盘片、电路的运行情况进行监控、记录并与厂商所设定的预设安全值进行比较，若监控情况将要或已超出预设安全值的安全范围，将触发此报警。当物理盘的 S.M.A.R.T. 检测不通过时，此时虽然仍可工作，但已经变得不再可靠，随时可能出现故障。如检查确认并非外部环境引发的临时异常,例如机房的临时高温等，都应进行换盘处理。

## 报警信息

- `主机 { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) S.M.A.R.T. 检测不通过。`
- `SCVM { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) S.M.A.R.T. 检测不通过。`

## 故障影响

当物理盘的 S.M.A.R.T. 检测不通过时，通常意味着物理盘存在潜在性能问题，后续发生故障的风险将显著提高。

## 故障原因

物理盘固件报告物理盘处于异常状态，具体原因需要提供 S.M.A.R.T. 报告给物理盘供应商确认。

> **特殊情况说明**：
>
> 当机箱内温度过高会导致触发物理盘 S.M.A.R.T. 检测不通过，当机箱内温度恢复正常后，`smartctl -H` 的输出结果会恢复正常。该情况下无需更换物理盘，但报警会仍然处于触发状态，可以在登录物理盘所在的物理节点后使用 `zbs-node set_disk_healthy` 命令重置物理盘故障记录，一小段时间后报警将自动解决。

## 排查思路

您可以通过监控系统中的 `host_disk_temperature_celsius` 或 `scvm_disk_temperature_celsius` 物理盘温度指标佐证该情况来判断是否发生 S.M.A.R.T. 检测不通过的情况。

执行 `smartctl -a` 命令获取物理盘的 S.M.A.R.T. 检测结果。正常情况下自检结果为通过，异常时可能显示对应故障信息或 FAILED。

## 解决方案

1. 若确认物理盘 S.M.A.R.T. 检查未通过，建议参考《SMTX OS 块存储运维指南》[物理盘运维](/smtxos/6.3.0/os_operation_maintenance/os_operation_maintenance_115)章节，对故障物理盘进行卸载和更换。
2. 完成故障处理后，请在 CloudTower 报警页面手动解决相关报警。

## 物理盘类 > 物理盘发生 I/O 阻塞

# 物理盘发生 I/O 阻塞

## 故障说明

物理盘频繁出现 I/O 超时或终止，或物理盘上的 I/O 请求进入异常 I/O 队列。

## 报警信息

- `主机 { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 发生 I/O 阻塞，已将该物理盘下线。`
- `SCVM { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 发生 I/O 阻塞，已将该物理盘下线。`

## 故障影响

系统会自动下线发生 I/O 阻塞的物理盘并触发数据恢复，被下线的物理盘会停止处理任何 I/O 请求。被下线的物理盘在解除下线状态前，无法从 CloudTower 界面卸载物理盘。

在异常物理盘被自动下线前，集群中部分卷的 I/O 可能会受到短暂影响而变得缓慢。在物理盘被自动下线后会恢复正常。

## 故障原因

该故障通常情况下为物理盘本身的固件或内部介质错误导致。

## 排查思路

在物理盘所在的节点上使用 `sdm -l | less` 查看对应节点上的 sd-offline 服务的日志，检查对应物理盘如下相关计数：

- I/O 超时计数
- SCSI 指令意外终止计数
- I/O 错误处理队列处理计数

当计数满足一定判定条件后，物理盘将被标记为 offline 并触发报警。集群中最多允许一块物理盘被标记为 offline。

## 解决方案

1. 建议参考《SMTX OS 块存储运维指南》[物理盘运维](/smtxos/6.3.0/os_operation_maintenance/os_operation_maintenance_115)章节，对故障物理盘进行卸载和更换。

## 物理盘类 > 物理盘处于亚健康状态

# 物理盘处于亚健康状态

## 故障说明

物理盘的“亚健康状态”是指物理盘虽然尚未发生故障，但其性能已经下降，可能对系统稳定性和业务连续性构成潜在威胁。

一般来说，当系统探测到如下情况时，物理盘将被标记为亚健康盘：

- 物理盘的 I/O 延时增高，但是尚未达到超时状态。
- 物理盘内部异常扇区累计超过阈值（400）。

## 报警信息

- `主机 { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 处于亚健康状态，请卸载此盘。`
- `主机 { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 处于亚健康状态，系统会自动隔离该盘，并将数据恢复至其他健康物理盘。请勿拔盘。`
- `SCVM { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 处于亚健康状态，系统会自动隔离该盘，并将数据恢复至其他健康物理盘。请勿拔盘。`
- `SCVM { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 处于亚健康状态，请卸载此盘。`

## 故障影响

物理盘有损坏或者长期处于性能异常状态风险，可能会影响底层的存储系统的运行表现，进而影响上层业务的流畅性。当系统检测到物理盘处于亚健康状态时，将对该物理盘上的相关数据执行数据迁移和恢复。建议及时卸载或更换亚健康盘。

## 故障原因

以下原因会触发物理盘的亚健康报警。当前系统处理完数据恢复/迁移后，物理盘将不再处于使用中的状态，报警将转化为物理盘出现故障，现处于未挂载状态，可安全拔出。

| **故障原因** | **检查信息** |
| --- | --- |
| disk-healthd 服务汇报硬盘的 I/O 延迟增高，但是尚未达到超时状态。 | 在物理盘所在的节点上查看 disk-healthd 日志： `less /var/log/zbs/disk-healthd/disk-healthd.log` 或通过 `zbs-node show_disk_status /dev/sdx` 查看磁盘的异常信息。 |
| S.M.A.R.T 报告： Reallocated Sector Count + Current Pending Sector Count ≥ 400 | 查看 S.M.A.R.T 信息 |

## 解决方案

1. 建议参考《SMTX OS 块存储运维指南》[物理盘运维](/smtxos/6.3.0/os_operation_maintenance/os_operation_maintenance_115)章节，对故障物理盘进行卸载和更换。
2. 完成故障处理后，请在 CloudTower 报警页面手动解决相关报警。

## 物理盘类 > 物理盘处于不健康状态

# 物理盘处于不健康状态

## 故障说明

当物理盘出现读写错误，以及发生读写请求超时或者不返回时，基本可判断该物理盘已处于不健康状态。

## 报警信息

- `主机 { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 处于不健康状态，系统会自动隔离该盘，并将数据恢复至其他健康物理盘。请勿拔盘。`
- `主机 { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 出现故障，现处于未挂载状态，可安全拔出。`
- `主机 { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 出现故障，已完成隔离。请卸载其上剩余的分区。`
- `SCVM { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 出现故障，已完成隔离。请卸载其上剩余的分区。`
- `SCVM { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 处于不健康状态，系统会自动隔离该盘，并将数据恢复至其他健康物理盘。请勿拔盘。`
- `SCVM { .labels.hostname } 的物理盘 { .labels._serial_number }({ .labels._device }) 出现故障，现处于未挂载状态，可安全拔出。`

## 故障影响

系统会自动隔离不健康盘，且将该不健康盘上的数据迁移至其他健康物理盘。完成隔离操作后，该故障盘提供的存储容量会计入失效空间。

## 故障原因

以下原因会触发物理盘的不健康报警。当前系统处理完数据恢复/迁移后，物理盘将不再处于使用中的状态，报警将转化为物理盘出现故障，现处于未挂载状态，可安全拔出。

| **故障原因** | **检查信息** |
| --- | --- |
| 因 I/O 超时或中止，zbs-chunk 服务触发 errflag。 | - 在物理盘所在的节点上执行命令：`zbs-node show_disk_status /dev/sdxx` 检查对应物理盘的 I/O 错误计数和物理盘状态标记。 - 执行 `less /var/log/messages` 命令查看 messages 日志。 |
| 因 I/O 错误（累计超过 300 次）或数据校验错误累计过多（累计超过 100 次）， zbs-chunk 服务触发 errflag |

## 解决方案

1. 建议参考《SMTX OS 块存储运维指南》[物理盘运维](/smtxos/6.3.0/os_operation_maintenance/os_operation_maintenance_115)章节，对故障物理盘进行卸载和更换。
2. 完成故障处理后，请在 CloudTower 报警页面手动解决相关报警。

## 物理盘类 > SMTX 系统盘发生故障

# SMTX 系统盘发生故障

## 故障说明

当选择将 SMTX OS 系统独立部署在两个物理盘时，这两个物理盘将作为系统盘并组成硬件 RAID 1。当出现如下情况时，说明发生了 SMTX 系统盘发生了故障。

- 组成硬件 RAID 的成员系统盘读写出现错误或者高延时，被标记为失灵的部件。
- 组成硬件 RAID 的成员物理盘 S.M.A.R.T. 检查不通过。
- 组成硬件 RAID 的成员物理盘目前没有表现出读写超时、IO 延时增高或者坏盘的迹象，但根据系统的判断，其寿命不足，即将产生影响。

## 报警信息

- `主机 { .labels.hostname } 硬件 RAID 下的物理盘 { .labels._serial } S.M.A.R.T. 检测不通过。`
- `主机 { .labels.hostname } 硬件 RAID 下的物理盘 { .labels._serial } 寿命不足 { .threshold }。`
- `主机 { .labels.hostname } 硬件 RAID 下的虚拟盘 { .labels._vd_name } 冗余度不足。`
- `主机 { .labels.hostname } 硬件 RAID 下的虚拟盘 { .labels._vd_name } 状态异常：{ .labels.status }。`

## 故障影响

当两块系统盘中任意一块发生故障时，操作系统将自动切换到节点另一块系统盘进行读写，整个过程中集群业务不受影响。

当两块系统盘同时发生故障时，操作系统分区和元数据分区将彻底无法使用，进而导致该节点不可用。若故障节点上的虚拟机配置了高可用（HA），虚拟机可迁移至其他节点正常运行。由于节点导致部分数据块可用副本数量减少，少于预期副本数，将触发数据恢复。

## 故障原因

请参考相应报警信息中的提示。

## 排查思路

您可以通过 CloudTower 或 BMC 系统查看 SMTX 系统盘及其成员物理盘的相关信息，判断是否发生硬件 RAID 故障。

- **CloudTower**

  在 CloudTower 的主机概览页面以及 SMTX 系统盘详情页面查看 SMTX 系统盘及其成员物理盘的状态。
- **BMC 系统**

  通过服务器的 BMC 界面确认存储控制器型号以及其下的 RAID 和物理盘的状态和信息。

  SMTX OS 支持的存储控制器可以通过[硬件兼容性查询工具](https://www.smartx.com/smtxos-compatibility/)查询，下表仅列出几款常见的、用于 SMTX 系统盘的存储控制器及其对应的 BMC 系统，不同 BMC 系统的界面各不相同，请参照各厂商的说明文档进行操作。

  | **品牌** | **BMC 系统** | **RAID 控制器** |
  | --- | --- | --- |
  | Dell PowerEdge | iDRAC | Marvell SATA M.2（Dell BOSS-S1/S2） |
  | Dell PowerEdge | iDRAC | Marvell NVMe M.2（Dell BOSS-N1） |
  | Lenovo ThinkSystem | XClarity | Marvell SATA M.2（ThinkSystem M.2 SATA 2-Bay RAID） |
  | Lenovo ThinkSystem | XClarity | Marvell NVMe M.2（ThinkSystem M.2 NVMe 2-Bay RAID） |
  | xFusion | iBMC | Broadcom MegaRAID（LSI SAS3XXX） |

## 解决方案

当出现相关故障时：

- 多数情况下您只需对故障的物理盘进行更换，RAID 控制器会自动处理好相关配置；
- 部分特殊情况可能需要您通过服务器的 BMC 或者 BIOS 界面进行 RAID 配置。

更换 SMTX 系统盘的操作请参考《SMTX OS 运维指南》的[更换 SMTX 系统盘](/smtxos/6.3.0/os_operation_maintenance/os_operation_maintenance_90)章节。

## 物理盘类 > 硬件 RAID 的 Write cache 策略未设置为 Write through

# 硬件 RAID 的 Write cache 策略未设置为 Write through

## 故障说明

主机系统盘所在的硬件 RAID 下的虚拟盘未启用 Write Through 模式，而采用了 Write Back 模式。在主机发生意外断电时，可能导致未落盘的数据丢失，造成系统盘数据发生不可逆的丢失。

## 报警信息

`主机 { .labels.hostname } 系统盘所在的硬件 RAID 下的虚拟盘 { .labels._vd_name } 的 Write cache 策略非 Write through，请尽快进行调整，以便正常完成后续操作。`

## 故障影响

在 **Write Back** 模式下，物理盘会先将数据写入缓存并立即返回写入成功。若在缓存尚未下刷到物理盘时节点突然断电，数据可能直接丢失。这可能导致：

- 主机系统无法正常启动；
- 节点上的数据永久丢失。

## 故障原因

主机在部署 SMTX OS 阶段未正确设置物理盘写入模式，导致默认使用 **Write Back**。

## 排查思路

进入主机 BIOS 或 IPMI 管理界面，检查系统盘所在硬件 RAID 的写入策略。

## 解决方案

在 BIOS 或 IPMI 中将物理盘写入模式调整为 **Write Through**，并重启主机。

## 物理盘类 > 物理盘发生插拔操作

# 物理盘发生插拔操作

## 故障说明

当物理盘在主机被插入或拔出时，将触发相关报警。

## 报警信息

- `于 { .labels.created_time } 侦测到物理盘 { .labels.disk_id_serial }({ .labels._device }) 从主机 { .labels._hostname } 拔出。`
- `{ .labels.created_time } 侦测到物理盘 { .labels.disk_id_serial }({ .labels._device }) 从主机 { .labels._hostname } 拔出，该盘原位于机箱 { .labels._brick_name } 的插槽位置 { .labels._physical_slot }。`
- `{ .labels.created_time } 侦测到物理盘 { .labels.disk_id_serial }({ .labels._device }) 从主机 { .labels._hostname } 插入。`

## 故障影响

- 若拔插的为**空闲物理盘**，对系统运行无实际影响。
- 若拔出的为**正在使用中的物理盘**，可能导致：

  - 该物理盘上的数据副本进入异常状态，触发数据恢复，导致集群存储性能下降；
  - 若该物理盘包含操作系统分区，节点运行的安全性显著降低，存在节点失效风险；
  - 若所有包含操作系统的物理盘均被拔出，节点将直接崩溃。
- 在未完成数据恢复的情况下，若多个节点同时拔出正在使用的物理盘，极有可能造成数据在移除期间不可访问，并在后续节点或服务故障重启时导致**永久性数据丢失**。

## 故障原因

操作系统探测到物理盘的插拔事件触发报警。

## 解决方案

- 若为计划内的空闲物理盘插拔操作，手动将报警标记为已解决即可。
- 若错误拔出使用中的物理盘，应立即将物理盘重新插回节点。同时，由于系统无法自动判断该物理盘离线原因（人为拔出或物理盘自身故障），因此不会自动挂载该物理盘，以避免将新数据反复写入故障区域以造成更大风险。若确认物理盘本身无异常，可执行 `zbs-node set_disk_healthy /dev/xxx` 命令将物理盘标记为健康状态，然后通过 CloudTower 管理界面重新挂载物理盘。
- 若物理盘实际并未被移除，但仍出现此类报警。此时请采集内核与 IPMI 日志并提交给 SmartX 售后工程师或硬件供应商进行排查。常见的硬件故障原因包括：

  - 物理盘损坏
  - 物理盘背板或供电模块异常
  - HBA 卡或 RAID 卡异常

## 服务异常类 > 高级监控服务资源不足

# 高级监控服务资源不足

## 故障说明

高级监控服务以独立虚拟机的形式部署在 SMTX OS（VMware ESXi）集群中，用于存储集群中的监控数据并触发报警。当高级监控自身的资源配置（CPU、内存、数据盘容量）低于所管理集群规模需要的规格时，可能出现服务降级的现象。

## 报警信息

- `运行高级监控的虚拟机的内存使用率过高。`
- `运行高级监控的虚拟机的数据盘空间不足。`
- `运行高级监控的虚拟机的 CPU 占用过高。`

## 故障影响

- 可能会出现监控视图展示缓慢的现象；
- 数据盘空间不足时，可能无法采集和保存最新的监控数据；
- 可能会出现较早的历史监控数据被提前清理，导致服务降级。

## 故障原因

高级监控虚拟机配置的规格不能满足所管理的集群的需求。

## 解决方案

- 检查高级监控虚拟机的资源配置是否低于集群规模对应的推荐规格。若不足，请提升虚拟机规格。推荐配置如下：

  | **集群节点数（N）** | **CPU** | **内存** | **物理盘** |
  | --- | --- | --- | --- |
  | N ≤ 4 节点 | 4 | 6 GB | 60 GB |
  | 5 ≤ N ≤ 31 节点 | N | 1.5 × N GB（需向上取整） | 15 × N GB |
  | N ≥ 32 节点 | 32 | 48 GB | 480 GB |
- 若配置已满足推荐要求，但报警仍然存在，请联系 SmartX 售后工程师，排查可能存在的服务内部异常。

## 服务异常类 > I/O 重路由服务异常

# I/O 重路由服务异常

## 故障说明

SMTX OS 部署在 VMware vSphere 虚拟化平台时，在每个 ESXi 主机上均会部署 I/O 重路由服务，用于实时感知 ESXi 主机上 ZBS 存储服务的健康状态。一旦探测到存储服务发生异常，ZBS 会自动将 I/O 请求重定向至集群中其他健康节点，避免因存储异常直接触发虚拟机 HA，降低计划内维护操作（如升级、重启等）或其他存储故障对业务的影响。

## 报警信息

- `存储 IP 为 {.labels._data_ip} 的 SCVM 所在主机的 I/O 重路由服务停止工作。`
- `存储 IP 为 {.labels._data_ip} 的 SCVM 所在主机的 I/O 重路由未指向当前节点。`

## 故障影响

- 若 SCVM 所在主机的 I/O 重路由服务未能指向当前节点，主机的 I/O 性能可能会出现下降。
- 当 I/O 重路由服务完全停止时，若该主机上的 SCVM 出现异常或存储服务出现较长时间异常（具体异常时长取决于 VMware vSphere 中配置的 HA 策略，通常为分钟级别 ）时，将导致虚拟机触发 HA。

## 故障原因

可能的故障原因包括：

- I/O 重路由服务未正确部署；
- 报警 SCVM 上的存储服务处于异常状态。

## 解决方案

- 若报警出现在集群升级或存储服务维护期间，属于正常现象。待运维操作完成后报警自然消失，则无需额外处理。
- 若报警并非由于计划内的操作触发，则通常会伴随其他存储、网络或 SCVM 异常报警。请优先排查并修复相关故障，此报警预期也会在故障解决后自动消失。
- 若排查无果或报警持续存在。请联系 SmartX 售后支持以进一步处理。

## 服务异常类 > 内部服务进程异常

# 内部服务进程异常

## 故障说明

当集群中的内部服务无法正常运行时，将触发相关报警。

## 报警信息

- `主机 { .labels.hostname } 的服务进程 { .labels._service } 未运行。`
- `主机 { .labels.hostname } 的服务进程 { .labels._service } 服务状态异常，为 { .labels.state_str }。`
- `SCVM { .labels.hostname } 的服务进程 { .labels._service } 未运行。`
- `SCVM { .labels.hostname } 的服务进程 { .labels._service } 服务状态异常，为 { .labels.state_str }。`
- `仲裁节点 { .labels.hostname } 的服务进程 { .labels._service } 未运行。`
- `仲裁节点 { .labels.hostname } 的服务进程 { .labels._service } 服务状态异常，为 { .labels.state_str }。`

## 故障影响

不同服务异常对系统的影响范围不一，请结合对应服务的报警说明进行分析和判断。

## 故障原因

常见原因包括：

- **运维操作**

  在计划内的重启或升级过程中，大部分服务会经历关闭再启动，此时可能短暂触发此类报警。若运维完成后数分钟内报警自动消失，无需处理。
- **操作系统异常**

  若安装操作系统的物理盘发生故障，或系统无响应时，服务可能会进入异常状态。若系统根分区进入只读或者超慢状态，节点上的监控程序也会主动关闭部分服务以保证集群模式工作的服务的稳定性。
- **内存超限**

  在网络异常时，大量的请求堆积可能导致服务进程占用内存超过规划的配置，引发系统的 OOM 探测，从而主动关闭服务。
- **硬件故障**

  处于直通状态的硬件组件异常（如 RDMA 网卡、加密卡、显卡等）可能导致相关服务异常。

## 解决方案

通常情况下，系统中的内部服务都具备状态检查和自动重启功能，能够在异常发生后自动拉起并恢复运行。

- 若报警出现在计划内运维（如节点重启或集群升级）过程中，通常无需处理，待操作完成并观察数分钟，若报警自动消失则无需额外操作。
- 若报警持续存在或频繁复现，请通过 CloudTower 收集异常期间的集群日志，并提交至 SmartX 售后工程师进行进一步排查和处理。

## 服务异常类 > 集群监控报警无法工作

# 集群监控报警无法工作

## 故障说明

当集群监控与报警相关服务运行异常时，将触发对应报警。这些异常可能导致指标采集、报警发送、Prometheus 服务或 Siren 服务无法正常工作，从而影响 CloudTower 上的监控与报警功能。

## 报警信息

- `集群的指标采集服务无法正常工作。`
- `节点 { .labels.hostname } 报警发送功能异常。`
- `集群的 Prometheus 服务状态异常。`
- `节点 { .labels.hostname } 的 Siren 服务状态异常。`

## 故障影响

- **指标采集服务异常**

  该故障将会导致监控报警系统整体失效。CloudTower 的监控视图将无法加载监控数据，报警无法加载等错误。
- **节点的报警发送功能异常**

  CloudTower 无法按预期显示或转发报警信息。
- **Prometheus 服务异常**

  集群本地监控数据采集与报警发送功能异常，表现为 CloudTower 上未关联可观测性服务的集群无法显示监控数据或触发报警。
- **Siren 服务异常**

  集群的报警无法显示和对外发送，已配置的集群报警通知渠道无法正常收到报警通知；未关联可观测性服务的集群无法触发报警，已关联可观测性服务的集群无法收到集群本地监控转发的报警等。

## 故障原因

各报警项对应的故障原因分别为：

- **指标采集服务异常**

  集群 Octopus Leader 节点上的 vmagent 服务未运行或持续重启。
- **节点的报警发送功能异常**

  主节点（Octopus Leader 或高级监控宿主机）上的 siren 服务在 2 分钟内未收到 Prometheus 心跳。
- **Prometheus 服务异常**

  节点的 Siren 服务每 1 分钟检查一次主节点或高级监控虚拟机内的 Prometheus 服务，若持续 5 分钟无法响应请求，则服务状态可能已经异常，将触发此报警。
- **Siren 服务异常**

  集群 Siren Leader 节点的 Siren 服务状态异常，或其他节点通过管理或存储网络检测到 Siren Leader 节点的 Siren 运行状态存在异常。

## 解决方案

- **指标采集服务异常**

  1. 执行 `zbs-cluster show_election_servers -s octopus` 命令定位 Octopus Leader 节点。
  2. 在 Octopus Leader 节点执行 `systemctl status vmagent` 命令检查服务状态；如服务未运行或持续重启，请尝试执行 `systemctl restart octopus` 命令重启 vmagent。
  3. 若服务仍无法恢复，建议通过 CloudTower 收集集群日志并联系 SmartX 售后工程师处理。
- **节点的报警发送功能异常**

  1. 登录当前 Octopus Leader 节点或高级监控所在节点。

     - 对于低于 SMTX OS 6.0.0 版本的集群， Octopus Leader 和 Meta Leader 相同。
     - 对于SMTX OS 6.0.0 及以上版本的集群，请执行 `zbs-cluster show_election_servers -s octopus` 命令定位 Octopus Leader 节点。
  2. 查看节点 siren 日志中是否存在持续 2 分钟未收到 "alerting\_is\_normal" 的情况。
  3. 若已部署高级监控：检查虚拟机文件系统是否只读，可尝试重启虚拟机修复；再检查虚拟机容器内的 Prometheus 日志，排查是否为网络抖动。
  4. 若未部署高级监控：检查是否同时存在 “Prometheus 服务异常”，若有，先修复该报警。
  5. 若问题持续，请通过 CloudTower 收集集群日志并联系 SmartX 售后工程师处理。
- **Prometheus 服务异常**

  1. 检查 Meta Leader 节点、Octopus Leader 节点和高级监控虚拟机内的 Prometheus 服务状态是否正常。
  2. 在 Siren 日志中通过关键字 prometheus\_is\_normal 过滤出 Prometheus 服务异常的相关日志，结合错误原因执行针对性操作。
- **Siren 服务异常**

  1. 在提示异常的主机执行 `systemctl status siren` 命令检查 Siren 服务的运行状态。
  2. 尝试执行 `systemctl restart siren` 命令重启 Siren 服务，观察服务是否恢复正常。
  3. 若问题持续，请通过 CloudTower 收集集群日志并联系 SmartX 售后工程师处理。

# 邮件报警通知配置已失效

## 故障说明

## 服务异常类 > NTP 服务器配置异常

# NTP 服务器配置异常

## 故障说明

当集群配置使用外部 NTP 服务器进行时间同步时，系统会定期检测与 NTP 服务器的连通性及时间同步状态。如果出现异常，将触发相关报警。

## 报警信息

- `集群无法与外部 NTP 服务器建立连接。`
- `外部 NTP 服务器与当前集群无法同步时间。`
- `主机 { .labels.hostname } 与 NTP 服务器无法同步时间。`
- `主机 { .labels.hostname } 无法与 NTP 服务器建立连接。`
- `主机 { .labels.hostname } 与 NTP 服务器时间偏移量大于 { .threshold }。`
- `SCVM { .labels.hostname } 与 NTP 服务器无法同步时间。`
- `SCVM { .labels.hostname } 无法与 NTP 服务器建立连接。`
- `SCVM { .labels.hostname } 与 NTP 服务器时间偏移量大于 { .threshold }。`
- `SCVM { .labels.hostname } 和集群系统时间相差大于 { .threshold }。`
- `仲裁节点 { .labels.hostname } 与 NTP 服务器时间偏移量大于 { .threshold }。`
- `仲裁节点 { .labels.hostname } 和集群系统时间相差大于 { .threshold }。`

## 故障影响

若发生 NTP 配置异常，集群主机的时钟可能与外部时间源不一致，这不会影响集群服务本身的正常运行，但会影响日志、报警、监控数据等与时间相关的信息显示。

## 故障原因

常见原因包括：

- 外部 NTP 服务器时钟异常，与主机时间偏差过大。
- 外部 NTP 服务器配置错误，如域名拼写错误、IP 地址配置错误、未配置 DNS 导致无法解析域名等。
- 网络异常导致主机无法访问 NTP 服务器。
- 主机或 NTP 服务器的防火墙策略阻断了相关端口。

## 解决方案

请根据实际情况逐项排查：

- **检查配置**

  确认 NTP 地址是否正确，必要时在集群的**设置** > **集群时间**中修改为正确的 NTP 地址。
- **检查网络**

  检查主机与 NTP 服务器之间的连通性，排查可能的防火墙或路由策略问题。
- **修正时间偏差**

  若主机与 NTP 服务器时间偏差过大，请将主机时间主动同步至 NTP 服务器。具体操作请参考《SMTX OS 运维指南》的[调整集群系统事件](/smtxos/6.3.0/os_operation_maintenance/os_operation_maintenance_11)章节。

## 服务异常类 > MongoDB 服务异常

# MongoDB 服务异常

## 故障说明

MongoDB 是系统运维和虚拟化调度依赖的关键服务。当 MongoDB 服务整体异常时，将导致：

- 虚拟化调度任务失效；
- 运维操作无法正常执行。

## 报警信息

`集群的 MongoDB 服务状态异常。`

## 故障影响

当 MongoDB 服务整体异常时，集群将无法正常对外提供服务。所有虚拟机的生命周期管理操作均会收到影响，包括但不限于虚拟机的创建、开机、快照和热迁移等。但对运行中的虚拟机没有影响，不会触发虚拟机 HA。

## 故障原因

MongoDB 是分布式集群服务，只有当超过半数主（Master）节点上的 MongoDB 服务同时异常，才会导致整体失效。常见原因包括：

- **集群启动过程未完成**

  因不同节点和服务启动时间存在差异，MongoDB 尚未完全启动，此时等待节点都启动完成报警自然会消失，无需特别处理。
- **存储网络异常**

  主节点间存储网络不通或存在严重延迟或抖动。
- **网络分区**

  集群节点之间出现网络隔离。

## 解决方案

1. **检查主节点间连通性**

   - 登录主节点，执行 `ping -s 1400 <目标IP>` 命令，测试节点间存储网络连通性。
   - 如因系统异常无法确定主节点，可通过 `cat /etc/zbs/zbs.conf` 命令查看 mongo 字段中的 IP 地址组（即需运行 MongoDB 的节点）。
   - 若检测到严重丢包，请逐一排查网络链路（网卡、交换机、存储网络等）。
2. **验证 MongoDB 服务状态**

   - 网络问题修复后，检查 MongoDB 集群实例的服务状态。预期应存在一个 PRIMARY 实例，其余为 SECONDARY。
   - 若状态异常，可在对应节点执行 `systemctl restart mongod` 命令重启服务，并再次检查状态。

# 主机的存储网络进入亚健康状态

## 故障说明

## 网络类 > 主机网口收发包异常

# 主机网口收发包异常

## 故障说明

系统会定期采集和统计物理网口的丢包数和错误包数，并在统计值超过阈值时触发报警。

## 报警信息

- `主机 { .labels.hostname } 的物理网口 { .labels._device } 2 分钟内，接收到的包中 Dropped 的数量超过 { .threshold }。`
- `主机 { .labels.hostname } 的物理网口 { .labels._device } 2 分钟内，发送的包中 Dropped 的数量超过 { .threshold }。`
- `主机 { .labels.hostname } 的物理网口 { .labels._device } 2 分钟内，接收到的包中 Error 的数量超过 { .threshold }。`
- `主机 { .labels.hostname } 的物理网口 { .labels._device } 2 分钟内，发送的包中 Error 的数量超过 { .threshold }。`
- `SCVM { .labels.hostname } 的物理网口 { .labels._device } 2 分钟内，接收到的包中 Dropped 的数量超过 { .threshold }。`
- `SCVM { .labels.hostname } 的物理网口 { .labels._device } 2 分钟内，发送的包中 Dropped 的数量超过 { .threshold }。`
- `SCVM { .labels.hostname } 的物理网口 { .labels._device } 2 分钟内，接收到的包中 Error 的数量超过 { .threshold }。`
- `SCVM { .labels.hostname } 的物理网口 { .labels._device } 2 分钟内，发送的包中 Error 的数量超过 { .threshold }。`

## 故障影响

通常情况下不会对业务产生影响，但仍需排查确认原因。

## 故障原因

物理网卡丢包和错误包计数增加，可能由以下原因引起：

- **突发大流量**：流量高峰时网卡无法及时处理，导致丢包。常见于使用流量测试工具进行大规模吞吐测试场景。
- **队列过小**：物理网卡默认的接收或发送队列深度过小，在流量突发场景容易导致队列溢出导致丢包。
- **交换机和网卡的 MTU 设置不匹配**：若交换机和网卡的 MTU 配置不一致，将会导致超过网卡 MTU 大小的包被丢弃，或者在分片重组过程中因校验失败而被计为错误包。
- **包格式错误**：如 CRC 校验失败、物理层干扰等问题，也会增加错误包计数。

## 排查思路

1. 检查 MTU 设置：确认交换机和主机物理网卡配置的 MTU 值是否一致。
2. 检查队列配置：登录主机，执行 `ethtool -g <物理网卡名称>` 命令查看队列设置。示例如下：

   ```
   [root@node31 11:11:10 ~]$ ethtool -g enp94s0f0np0
   Ring parameters for enp94s0f0np0:
   Pre-set maximums:
   RX:             8192
   RX Mini:        0
   RX Jumbo:       0
   TX:             8192
   Current hardware settings:
   RX:             1024
   RX Mini:        0
   RX Jumbo:       0
   TX:             1024
   ```

   输出包含两部分：

   - `Pre-set maximums`：物理网卡支持的最大队列深度。
   - `Current hardware settings`：当前的实际队列配置。

   若当前实际队列配置深度明显小于支持的最大深度，可适当调整队列深度配置。

## 解决方案

根据排查结果采取对应的调整方案：

- **MTU 配置不一致**

  若交换机与物理网卡的 MTU 值不一致，则需要调整交换机或者物理网卡的 MTU，配置成相同的值。请确保统一网络路径上的所有设备（交换机、物理网卡、系统网络、虚拟机网络等）的 MTU 值保持一致，否则可能导致报文被丢弃或数据分片异常。
- **网卡队列配置过小**

  如果物理网卡的接收和发送队列配置过小，请通过执行 `ethtool -g <物理网卡名称> tx <队列大小>` 和 `ethtool -g <物理网卡名称> rx <队列大小>` 命令分别调整物理网卡发送和接收队列的深度。

  示例如下：


## 网络类 > 主机网口出现异常

# 主机网口出现异常

## 故障说明

系统通过 **l2ping** 组件周期性检测网口健康状况。当检测到网口不健康时，系统会记录异常并触发报警。

## 报警信息

- `主机 { .labels.hostname } 的接入网口 { .labels._nic } 出现异常，重置后仍未恢复。`
- `主机 { .labels.hostname } 的接入网口 { .labels._nic } 出现异常，重置后仍未恢复，已自动对网口进行隔离。`
- `主机{ .labels.hostname } 的接入网口{ .labels._nic }出现异常，重置后恢复正常。`
- `主机 { .labels.hostname } 的存储网口 { .labels._nic } 出现异常，重置后仍未恢复。`
- `主机 { .labels.hostname } 的存储网口 { .labels._nic } 出现异常，重置后仍未恢复，已自动对网口进行隔离。`
- `主机{ .labels.hostname } 的存储网口{ .labels._nic }出现异常，重置后恢复正常。`
- `主机 {.labels.hostname} 的网口 {.labels._nic} 出现异常。`
- `主机 { .labels.hostname } 的物理网口 { .labels._eth } 关联了虚拟分布式交换机 { .labels._vds }，但是发生了故障。`

## 故障影响

当 l2ping 组件检测到网口不健康时，会尝试对网口进行重置。如果多次重置后网口仍未恢复，且网口属于 active-backup 或 balance-slb bond 成员，则会对故障网口进行隔离。

- **单网口场景**

  该网口异常将直接导致其承载的网络（如存储网络或接入网络）无法连接。
- **网口绑定场景**

  若网口 Bonding 中某个成员网口异常，由于流量可以自动切换至其他成员网口，因此不会对业务产生影响。但是 Bonding 的冗余度降低，一旦其他成员网口再发生异常，风险将显著增加。因此在网口绑定场景下，也应及时修复异常网口，以恢复冗余能力。

## 故障原因

物理网卡或者物理网络设备（交换机，网线，光模块等）异常。

## 解决方案

替换无法正常工作的网卡、网线、交换机、光模块等。

## 网络类 > 主机间的存储网络出现高延迟

# 主机间的存储网络出现高延迟

## 故障说明

系统会周期性通过 **fping** 检测主机之间的网络连通性和时延情况。当监测到延迟指标超过设定阈值时，将触发报警。

## 报警信息

- `主机 { .labels.hostname } 和主机 { .labels.to_hostname } 之间的存储网络 ping 延时 5 分钟内 P{ .quantile } 值超过 { .threshold}。`
- `SCVM { .labels.hostname } 和 SCVM { .labels.to_hostname } 之间的存储网络 ping 延时 5 分钟内 P{ .quantile } 值超过 { .threshold}。`

## 故障影响

- 当延迟仅轻微超过 5 毫秒（如十几毫秒到几十毫秒）时，集群整体性能会略有下降，具体影响程度取决于延迟大小与持续时间。
- 当延迟上升至上百毫秒时，集群可能出现**偶发性的数据恢复**。
- 当延迟达到数百毫秒甚至更高时，集群节点间的通信将严重受阻，可能导致集群**无法正常工作**。

## 故障原因

- 较低的延迟，通常由网络抖动或者突发的流量压力引起；
- 若长时间的高延迟或频繁抖动，则需结合以下因素综合分析，例如：

  - 系统整体负载过高或者虚拟网络组件异常；
  - 物理网络设备（交换机、网线、光模块等）故障或配置不合理。

## 排查思路

- 使用延迟测量工具（如 fping、ping）确认延迟波动情况。
- 在物理网口抓包分析具体原因。

## 解决方案

- 检查物理网络链路（交换机、网卡、网线等）是否异常。
- 排查 OVS、网卡驱动等虚拟化网络组件的配置与负载情况。
- 基于测量与分析结果，优化网络架构或进行硬件替换。

## 网络类 > 主机间网络断开

# 主机间网络断开

## 故障说明

系统会定期通过 fping 检测主机间的网络连通性。当对端主机在一定时间内持续无法 ping 通时，将触发网络断开报警。

## 报警信息

- `从下列 { .labels.count } 台主机无法 ping 通 { .labels.hostname } 关联的管理 IP { .labels.ip }: { .labels.can_not_ping_hostnames }。`
- `从下列 { .labels.count } 台主机无法 ping 通 { .labels.hostname } 关联的存储 IP { .labels.ip }: { .labels.can_not_ping_hostnames }。`
- `从下列 { .labels.count } 台主机无法 ping 通 { .labels.hostname } 关联的接入 IP { .labels.ip }：主机 { .labels.can_not_ping_hostnames }。`
- `主机 { .labels.name } 至如下主机的通信链路存在异常：{ .labels.chunks }。`
- `从下列 { .labels.count } 台 SCVM 无法 ping 通 { .labels.hostname } 关联的管理 IP { .labels.ip }: { .labels.can_not_ping_hostnames }。`
- `从下列 { .labels.count } 台 SCVM 无法 ping 通 { .labels.hostname } 关联的存储 IP { .labels.ip }：SCVM { .labels.can_not_ping_hostnames }。`

## 故障影响

当网络断开时，相关存储或接入服务将无法正常访问，导致业务中断。

## 故障原因

可能原因包括：

- 系统宕机或进程卡死
- 网卡故障
- 交换机或上联链路故障
- 网线或物理连接异常

## 排查思路

检查 OVS、网卡驱动等虚拟化网络组件的配是否有异常，更换网络相关设备重新测试。

## 解决方案

请根据排查出的具体原因，调整虚拟网络相关组件的配置，或替换故障的物理网络设备（网卡、网线、交换机等）。

## 网络类 > 双活集群可用域间网络异常

# 双活集群可用域间网络异常

## 故障说明

在双活集群中，各可用域内的主机会持续探测与其他可用域主机之间的连接状态。当一个可用域内所有主机均无法连通另一个可用域内的所有主机时，即判定为域间网络异常，并触发报警。

## 报警信息

- `{ .labels.zone_name }与{ .labels.to_zone_name }存储网络断开。`
- `{ .labels.src_zone_name } 的主机 { .labels.hostname } 到 { .labels.target_zone_name } 的最差往返延迟在 5 分钟内的 P{ .quantile } 值超过 { .threshold }。`

## 故障影响

可用域间网络延迟增高将直接影响集群的 I/O 写入性能和进行虚拟机创建/编辑等操作的响应时间。当网络往返延迟超过 1s 或者发送中断时:

- 已开启了高可用且放置组规则为“允许在优先可用域运行”的虚拟机，会在优先可用域资源足够的情况下触发 HA，迁移至优先可用域中运行。
- 次级可用域的主机和存储服务可能异常；
- 集群出现大量待恢复数据，且恢复速率可能长期保持在 0，导致数据恢复无法完成。

## 故障原因

可用域间存储网络出现高延迟或中断。

## 排查思路

由于双活集群通常节点数量较多，且单一可用域内至少包含 2 台主机，因此域间网络整体断开事件往往不是由单个网卡异常导致的，更可能涉及网络设备或链路层面的故障。常见可能原因包括：

- 两个可用域节点的上联交换机异常；
- 两个可用域之间的物理链路中断（如光纤故障）；
- 用于域间通信的交换机或其他中继设备故障；
- 网络规则（如 ACL、防火墙策略）配置错误，导致通信被阻断。

## 解决方案

根据排查结果，针对性修复相关问题，例如：

- 更换或修复异常的交换机、链路或中继设备；
- 调整或恢复错误的网络配置；
- 确认并优化域间通信链路的冗余，避免单点故障再次发生。

## 网络类 > 仲裁节点网络连接异常

# 仲裁节点网络连接异常

## 故障说明

在双活集群中，仲裁节点会持续监控自身到可用域存储网络之间的联通状态。当检测到网络连接异常或异常中断时，将触发相关报警。

## 报警信息

- `仲裁节点到 { .labels.target_zone_name } 的最差往返延迟5分钟 P{ .quantile } 值超过 { .threshold }。`
- `仲裁节点与{ .labels.zone_name }存储网络断开。`
- `仲裁节点的工作状态未知。`

## 故障影响

- 在以下条件均满足的情况下，仲裁节点的网络连接失效对集群的运行**无直接影响**：

  - 优先可用域与次级可用域内的节点均正常工作；
  - 优先可用域与次级可用域之间的网络连通保持正常。
- 若仲裁节点持续处于网络连接异常状态，同时叠加以下任一情况，可能导致**集群整体失效**：

  - 某个可用域整体失效；
  - 优先可用域和次级可用域之间网络连接异常。

## 故障原因

网络状态检测服务发现仲裁节点网络质量异常或仲裁节点整体失连后触发相关报警。

## 排查思路

仲裁节点通常以虚拟机的形式部署，且位置独立于优先可用域和次级可用域。建议按照以下顺序进行故障原因确认：

- 确认仲裁节点是否处于启动状态；
- 登录仲裁节点，确认仲裁节点是否可以与可用域内主机的存储网络进行正常连通；
- 在仲裁节点上对可用域内的存储网络地址进行大包探测，检测大包传输时的延迟、抖动和丢包情况。例如执行命令：`ping -s 1400 1.1.1.1`。
- 若确认网络无法连通，请根据仲裁节点的部署位置，联络相应网络部门进一步确认网络状态。

## 解决方案

当仲裁节点自身恢复正常运行，或与可用域之间的网络连接恢复正常后，报警将自动消除。
