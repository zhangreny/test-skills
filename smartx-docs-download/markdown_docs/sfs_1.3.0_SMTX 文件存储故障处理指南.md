# sfs/1.3.0/SMTX 文件存储故障处理指南
## 关于本文档

# 关于本文档

本文档主要围绕文件存储集群运行时常见的报警信息展开，逐一说明相关报警对应的故障场景。运维人员在遇到报警时，可查阅本文档中对应的条目，快速理解相关报警下的故障场景的含义、可能的影响和风险，并参考给出的排查思路和解决方案，进行高效可靠的处理。

## 故障处理原则

# 故障处理原则

文件存储集群在出现故障后，应以尽量减小对集群业务的影响，尽快恢复集群业务为第一原则。同时，您还需注意如下要求：

- 先确认故障现象并定位原因，避免在原因不明时盲目操作（如随意关机、重启或插拔磁盘），以免扩大问题。
- 若本文档提供了解决方案，请按文档执行；若本文档未涵盖，请联系 SmartX 售后工程师进行处理。
- 在故障处理过程中遇到的任何问题，应详细记录各种原始信息，请勿随意删除数据或日志。
- 集群恢复运行后需持续观察一段时间，确认故障已完全排除。

集群在发生故障后，请收集并整理如下信息，并及时反馈给 SmartX 售后工程师。

| **现网信息分类** | **收集的具体内容** |
| --- | --- |
| 集群版本信息 | - 软件版本号。例如 SMTX 文件存储 1.3.0。 - 许可类型。例如订阅许可或永久许可。 |
| 故障发生时间 | 记录集群发生故障时的准确时间。 |
| 故障现象 | 详细描述集群发生故障时的具体表现。 |
| 故障前执行的操作 | 回忆并记录故障发生前，在集群上执行的具体操作及结果。 |
| 故障后执行的操作 | 记录故障发生后，在集群上执行的具体操作及结果。 |

## 文件控制器状态异常

# 文件控制器状态异常

## 故障说明

文件控制器未运行、文件控制器内部服务组件异常，或文件存储网络异常时，将触发状态异常报警。

## 报警信息

`文件存储集群 {{ $labels.sfscluster }} 的文件控制器 {{ $labels.fsc }} 状态异常。`

## 故障影响

该故障可能带来严重影响，需要尽快处理。

- 文件存储集群的管控服务可能受到影响，例如：无法对文件集群进行编辑和升级操作，无法新建或编辑文件系统，或者相应操作耗时较长；
- 文件存储集群的存储服务可能受到影响，例如：无法进行文件 I/O 或者延迟较高。

## 故障原因

- 文件控制器未运行；
- 文件存储集群的存储网络可能中断或延迟较高；
- 文件控制器内部组件可能未正常运行。

## 解决方案

1. **检查文件存储集群状态**

   登录 CloudTower，进入**文件存储**的**概览**页面，根据文件存储集群的状态进行相关操作：

   - 若文件存储集群处于**维护中**状态，在文件存储集群可退出维护模式时（例如文件存储集群所属的 SMTX OS（ELF）集群已退出维护模式），进入文件存储集群的**设置** > **文件控制器**页面，对已下线的文件控制器，单击右侧 **...**，选择**上线**。
   - 若文件存储集群未处于**维护中**状态，进入文件存储集群的**设置** > **文件控制器**页面，对报警的文件控制器，单击右侧 **...**，选择**查看详情**。确认该文件控制器处于**运行中**状态。若未处于**运行中**状态，请联系 SmartX 售后工程师获取技术支持。
2. **检查网络连通性**

   通过 SSH 或 VNC 方式登录报警文件控制器，使用 ping 命令检查各文件控制器间文件存储 IP 的连通性。若网络不连通，请进行如下排查：

   - 检查文件存储网络及交换机等网络设备的状态与连通性；
   - 进入文件存储集群的**设置** > **网络配置**页面，单击文件存储网络对应的虚拟机网络名称，进入相关虚拟机网络的详情页面，单击该虚拟机网络所属的虚拟分布式交换机，查看该虚拟分布式交换机，确认其已正确关联所有物理主机的网口。
3. 若上述步骤均未解除报警，请联系 SmartX 售后工程师获取技术支持。

## 文件存储集群中 API 组件状态异常

# 文件存储集群中 API 组件状态异常

## 故障说明

文件存储集群中的 API 组件未运行，文件管理网络异常，文件存储网络异常，或文件控制器状态为未运行。

## 报警信息

`文件存储集群 {{ $labels.sfscluster }} 中的 API 组件 {{ $labels.service }} 状态异常。`

## 故障影响

- 文件存储集群的管控服务将受到影响，例如：无法对文件存储集群进行编辑和升级操作，无法新建或者编辑文件系统；
- 文件存储集群的存储服务将受到影响，例如：无法读写文件。

该故障将带来严重问题，请尽快处理。

## 故障原因

- 文件存储集群使用的管理网络可能中断或者延迟过高；
- 文件存储集群使用的存储网络可能中断或者延迟过高；
- 文件控制器可能没有运行；
- 文件存储集群中的 API 组件可能没有正常运行。

## 解决方案

1. **检查文件管理网络与 CloudTower 的连通性**

   通过 SSH 或 VNC 方式登录 CloudTower 虚拟机，使用命令 ping 检查文件存储集群的文件管理 IP 和文件管理虚拟 IP 与 CloudTower 的连通性。若网络无法连通，请进行如下排查：

   - 检查文件管理网络、交换机等网络设备的状态与连通性；
   - 进入文件存储集群的**设置** > **网络配置**页面，单击文件管理网络对应的虚拟机网络名称，进入相关虚拟机网络的详情页面，单击该虚拟机网络所属的虚拟分布式交换机，查看该虚拟分布式交换机，确认其已正确关联所有物理主机的网口。
2. **检查各文件控制器之间存储网络的连通性**

   通过 SSH 或 VNC 方式登录报警文件控制器，使用 ping 命令检查各文件控制器间文件存储 IP 的连通性。若网络不连通，请进行如下排查：

   - 检查文件存储网络、交换机等网络设备的状态与连通性；
   - 进入文件存储集群的**设置** > **网络配置**页面，单击文件存储网络对应的虚拟机网络名称，进入相关虚拟机网络的详情页面，单击该虚拟机网络所属的虚拟分布式交换机，查看该虚拟分布式交换机，确认其已正确关联所有物理主机的网口。
3. 若上述步骤均未解除报警，请联系 SmartX 售后工程师获取技术支持。

## 文件控制器系统盘未启用常驻缓存

# 文件控制器系统盘未启用常驻缓存

## 故障说明

文件控制器的系统盘未启用常驻缓存。

## 报警信息

`文件存储集群 {{ $labels.sfscluster }} 的文件控制器 {{ $labels.fsc }} 的系统盘 {{ $labels.volume_uuid }} 未启用常驻缓存。`

## 故障影响

文件存储集群的管控服务可能会在缓存击穿时受到影响，建议尽快处理。

## 故障原因

在 CloudTower 4.5.0 及以上版本部署低于 1.2.0 版本的文件存储集群时，系统盘的常驻缓存功能不会自动启用，需要手动启用。

## 解决方案

请联系 SmartX 售后工程师手动为文件控制器开启系统盘常驻缓存。

## 文件系统剩余空间过少

# 文件系统剩余空间过少

## 故障说明

文件系统的数据空间实际使用率较大超过设置阈值，触发相关报警。

## 报警信息

`文件存储集群 {{ $labels.service_name }} 的文件系统 {{ $labels.file_system_name }} 存储使用率已经达到 {{ $value }}%。`

## 故障影响

文件系统的分配容量即将耗尽，新数据将无法写入。

## 故障原因

文件系统的数据空间实际使用量较大，剩余空间不足。

## 解决方案

- 扩容文件系统：登录 CloudTower，进入**文件存储**页面，选择报警信息涉及的文件存储集群，在**文件系统**页面，在相应的文件系统右侧单击 …，选择**编辑基本信息**。在许可范围内扩容文件系统容量。
- 删除不再需要的数据以释放空间。

## 文件控制器的 CPU 占用过高

# 文件控制器的 CPU 占用过高

## 故障说明

当文件控制器的 CPU 占用率过高，超过设定阈值时，将触发告警。

## 报警信息

`运行文件存储服务 {.service_name} 的虚拟机 {.vm_name} 的 CPU 占用过高。`

## 故障影响

可能导致 I/O 响应慢，影响文件存储集群的稳定性。

## 故障原因

该文件控制器系统负载增过高，当前 vCPU 数量已不足以支持服务平稳运行。

## 解决方案

进入 CloudTower 的监控页面，检查该文件存储集群内其他文件控制器的 CPU 使用情况：

- 若其他文件控制器的 CPU 占用较低，可将某些客户端挂载到其他文件控制器的文件接入 IP，以分担负载。
- 若其他文件控制器的 CPU 占用同样较高，或客户端挂载方式不可调整，请联系 SmartX 售后工程师，为文件控制器增加 vCPU 数量。

## 文件控制器的内存使用率过高

# 文件控制器的内存使用率过高

## 故障说明

当文件控制器的内存使用率过高，超过设定阈值时，将触发告警。

## 报警信息

`运行文件存储服务 {.service_name} 的虚拟机 {.vm_name} 的内存使用率过高。`

## 故障影响

内存使用率过高可能导致文件服务异常重启。

## 故障原因

文件控制器为尽量保持缓存的有效性以提升性能，通常会保持较高的内存占用率。同时文件服务会监控操作系统的可用内存，若可用容量低于阈值，会主动加速回收缓存。

当文件控制器存在某些服务或者任务快速占用大量内存时，易触发此告警。

## 解决方案

进入 CloudTower 的监控页面，检查该文件存储集群内其他文件控制器的内存使用率：

- 若其他文件控制器的内存占用较低，可将某些客户端挂载到其他文件控制器的文件接入 IP，以分担负载。
- 若其他文件控制器的内存占用同样较高，或客户端挂载方式不可调整，请联系 SmartX 售后工程师，为文件控制器增加内存。

## 文件控制器的系统盘存储空间不足

# 文件控制器的系统盘存储空间不足

## 故障说明

文件控制器的系统盘可用空间低于阈值时，将触发报警。

## 报警信息

`运行{.service_type} {.service_name} 的虚拟机 {.vm_name} 的存储空间不足。`

## 故障影响

若系统盘存储空间耗尽，可能导致系统服务无法正常提供服务。

## 故障原因

常见原因是运维过程中遗留了大容量的 coredump 文件。

## 解决方案

通过 SSH 或 VNC 方式登录报警文件控制器，执行 du 命令逐层查找占用空间大的目录或文件，若确认文件或目录可以安全删除，请手动清理以释放空间；若无法判断文件是否可删除，请联系 SmartX 售后工程师处理。

## 文件存储集群无法与 NTP 服务器建立连接

# 文件存储集群无法与 NTP 服务器建立连接

## 故障说明

文件存储集群依赖 CloudTower 配置的 NTP 服务器来保持各文件控制器的时钟同步。若集群无法与 NTP 服务器建立连接，将触发告警。

## 报警信息

`文件存储服务 { .service_name } 无法与 NTP 服务器 { .ntp_server } 建立连接。`

## 故障影响

当集群与 NTP 服务器时间不同步时，文件控制器之间可能产生时间漂移，进而影响文件存储服务的稳定性。

## 故障原因

可能的故障原因包括：

- 当前配置的 NTP 服务器 IP 地址无效；
- 集群到 NTP 服务器之间存在网络异常；
- NTP 服务器配置为域名，且域名解析失败。

## 解决方案

- **检查网络连通性**

  - 确认当前 NTP 服务器的 IP 地址是否有效；
  - 确认集群和 NTP 服务器之间的网络连接是否正常。
- **若 NTP 服务器使用域名配置，且域名解析失败**

  - 在 CloudTower 使用 IP 地址配置 NTP 服务器，避免域名解析问题。
  - 如需继续使用域名配置 NTP 服务器，可执行临时方案：

    - 通过 SSH 或 VNC 登录每台文件控制器；
    - 编辑 /etc/resolv.conf，配置正确的 DNS 服务器；
    - 执行 docker restart ntpm 命令重启 ntpm 容器；
    - 执行 docker exec ntpm cat /etc/resolv.conf 命令确认容器内 DNS 配置已生效。
- **若仍然无法解决，请重新配置一个可用的 NTP 服务器。**

## CloudTower 与文件控制器时间偏移量过大

# CloudTower 与文件控制器时间偏移量过大

## 故障说明

文件存储集群通过 CloudTower 配置的 NTP 服务器保持时钟同步。预期 CloudTower 与各文件控制器均保持时间一致。当 CloudTower 与某个文件控制器的时间偏移超过阈值时，将触发此告警。

## 报警信息

`CloudTower 与系统服务虚拟机 { .vm_name } 的时间偏移量过大。`

## 故障影响

告警文件控制器可能与其他控制器时钟不一致，进而导致文件服务异常。

## 故障原因

可能的故障原因包括：

- CloudTower 时间未与 NTP 服务器同步；
- 文件控制器时间未与 NTP 服务器同步。

## 解决方案

- **检查时间**

  通过 SSH 或 VNC 登录 CloudTower 和告警文件控制器，确认时间是否正确。
- **若 CloudTower 时间异常**

  请联系 SmartX 售后工程师同步 CloudTower 时间。
- **若文件控制器时间异常**

  - 请登录 CloudTower，进入 **系统配置** > **CloudTower 时间**，确认 NTP 服务器已正确设置。

    > **注意**：
    >
    > 若使用域名配置 NTP 服务器，可能同时触发“文件存储集群无法与 NTP 服务器建立连接”告警。
  - 登录告警文件控制器并切换至 root 用户：

    - 执行 `docker exec -it ntpm /app/ntpm --addr <fsc-ip>:20003 server show` 查看当前同步状态（其中 `<fsc-ip>` 为文件控制器的文件管理 IP）；
    - 若未显示 `sync` 状态的 NTP 服务器，执行 `docker exec -it ntpm /app/ntpm --addr <fsc-ip>:20003 server sync` 命令强制同步时间。
