## 概述

# 概述

ELF 是 SmartX 基于 KVM 开发的一套虚拟化计算平台，能与 SmartX 设计研发的分布式块存储服务（ZBS）整合，从而提供完整的超融合虚拟化解决方案。下文中，为方便叙述和理解，会使用 ELF 来指代基于 KVM 的虚拟机服务的计算平台，使用 ZBS 代替 SmartX 自研的分布式块存储服务。

ELF 实现了常见虚拟化计算场景里的需求，如虚拟机管理、VLAN 网络管理等。 ELF 运维简单，裁剪了大量低频次使用的功能特性，从而大大降低了系统的学习和运维成本，用户通过简单 Web 交互就能实现对超融合虚拟化计算平台的管理。 此外，由于底层的存储使用 ZBS，所以运行在 ELF 平台的虚拟机 I/O 性能非常卓越，而且数据高可靠高可用。

- 全分布式的架构设计

  管理平台不会出现类似 vCenter 的单点失效。
- Web UI 简单易用

  Hypervisor 基于开源的 KVM，提供绝大部分虚拟化场景的虚拟化计算的需求。
- VM 高可用

  当物理服务器发生故障的时候，其上的虚拟机会自动迁移到健康的物理服务器，确保业务高可用。
- VLAN 网络支持

  支持 VLAN 二层网络隔离，配置运维简单，满足绝大部分网络使用场景。

## RESTful API Service

RESTful API Service 是所有 HTTP API 的入口，由一组相互独立的服务组成，用于给 Web 控制台管理界面提供 API 服务，还有部分命令行的入口服务。提供的 API 服务不仅限于 ELF，也提供其它例如 ZBS 或者监控数据等 API。

性能上服务基于协程开发，可以提供数以百计的并发量以达到高性能的需求。可扩展性上，模块可以通过插件的形式加载到服务器中使用。

服务会被部署在每一个节点，用户访问时只需要访问其中任意一个节点即可。单独一个节点故障不会影响其它节点的入口，每一个节点的入口功能是一样并且是无状态服务。用户调用任意一个节点的 API 服务得到的结果均是相同。

## VNC Proxy Service

VNC Proxy Service 是提供给 Web 控制台管理界面访问虚拟机 VNC 接口服务的组件，部署在所有节点上。服务通过管理网络进行通讯，并提供 Proxy 服务，只需要访问任意一个节点的 VNC Proxy 服务，就可以访问到该节点中任意一个虚拟机的 VNC。

## VM Scheduler

VM Scheduler 用于按照放置策略和自动调度规则来调度虚拟机，部署在主节点，各节点上的 VM Scheduler 有且仅有 1 个处于活动（Active）状态，并随 ZBS-Meta Leader 切换，其它的处于待机（Standby）状态。

更多详情，请参阅[虚拟机自动调度](/smtxos/6.3.0/whitepaper_elf/whitepaper_elf_11#%E8%99%9A%E6%8B%9F%E6%9C%BA%E8%87%AA%E5%8A%A8%E8%B0%83%E5%BA%A6)章节。

## DHCP Service

DHCP Service 用于为虚拟机网络配置 DHCP 服务，响应虚拟机的 DHCP 请求，部署在主节点，各节点上的 DHCP Service 有且仅有 1 个处于 Active 状态，并随 ZBS-Meta Leader 切换，其他的处于 Standby 状态。

## VM VMTools Agent

VM VMTools Agent 部署运行在每一个节点，支持宿主机与虚拟机中部署的 VMTools 服务通过 VirtIO Serial 进行通信，主要功能包括但不限于：

- 定期采集虚拟机 Guest OS 静态信息并持久化到 MongoDB 数据库
- 提供 gRPC 接口查询虚拟机 Guest OS 相关性能数据，且支持配置虚拟机网络、用户名、密码等
- 提供静默文件系统的接口以支持虚拟机文件系统一致性快照

## Job Center Worker

Job Center Worker，简称 JC Worker，是 Job Center 的一个组件，部署在每一个节点，用于执行各类异步任务。各节点上的 JC Worker 没有主从之分，都处于 Active 状态。每个节点上的 JC Worker 可以产生最多 100 个并发执行单元。每个执行单元可对应于一个 Leader 任务或者 Follower 任务。

Job Center 以 MongoDB 作为任务队列组成一个 JC Worker 集群。Job 提交到 Job Center 后经由 MongoDB 路由到任意可用节点，由节点上的 JC Worker 处理：Job 会由一个 Leader 任务开始，随后被拆分成多个 Follower 任务，每个 Follower 任务又会被作为异步任务被提交到 Job Center 执行。Follower 任务可以指定目标节点，则 Job Center 会将该任务分发到指定目标节点，否则会根据集群负载随机选择一个相对空闲的节点作为目标节点。

更多详情，请参阅 [Job Center Worker](/smtxos/6.3.0/whitepaper_elf/whitepaper_elf_04) 章节。

## Job Center Scheduler

Job Center Scheduler，简称 JC Scheduler，是 Job Center 的另一个组件，部署在主节点，用于向 Job Center Worker 提交定时 Job。各节点上的 JC Scheduler 没有主从之分，都处于 Active 状态。集群中只要有一个 JC Scheduler 正常便可以保证集群的定时 Job 工作。

目前定时触发的 Job 主要有：

- SMTX OS 虚拟机状态同步与恢复
- Job Center 高可用
- 定时搜集和清理工作

更多详情，请参阅 [Job Center Scheduler](/smtxos/6.3.0/whitepaper_elf/whitepaper_elf_05) 章节。

## VM Monitor

VM Monitor 是一个轻量级的监控服务，部署在每个节点上，用于监控存储和网络，以支持高可用性（HA，High Availablity）功能。各节点上的 VM Monitor 有且仅有 1 个 Leader，随 ZBS-Meta Leader 切换，其它的作为跟随者（Follower）的身份运行。不论 Leader 还是 Follower，当监控到存储和网络都不可用时，会关闭所在节点上处于运行状态的虚拟机，以防止健康集群触发 HA 后出现重复虚拟机。另外，作为 Leader 的 VM Monitor 还负责将各个节点的状态更新到 MongoDB，以在健康的节点上触发 HA。

以下任一条件满足时，即会触发 HA。

- 基于 NFS 的 ZBS 服务超过 90 秒不可用，并且基于 MongoDB 的 Job Center 心跳检查超过 90 秒不可用
- 宿主机文件系统进入只读状态

更多详情，请参阅[虚拟机高可用](/smtxos/6.3.0/whitepaper_elf/whitepaper_elf_14)章节。

## Job Center

# Job Center

![](https://cdn.smartx.com/internal-docs/assets/b6349ae1/image5.png)

上图反映 Job Center 和与其交互的重要组件的关系。在 SMTX OS 内虚拟机服务中虚拟机、虚拟卷、虚拟机快照、虚拟卷快照、虚拟机模板等被定义为 “资源”，被存储在 MongoDB 中。资源具有状态，在不同的状态之间可以通过执行某些操作进行状态转换，这些操作称为 Follower 任务。Job Center 通过分析资源的当前状态和目标状态生成 Follower 任务，这个操作也被定义为任务，称为 Leader 任务。通常每个 Job 都会包含一组资源集合，可以包含前述资源中的 一种或者多种。此外，Job Center 还支持定时触发的周期性 Job，该类 Job 可以不包含资源集合，但通过 Leader 处理后也会被拆分为 Follower 任务，如定时收集节点信息。

MongoDB 在 ELF 中以主从副本集的（ master/secondary replica set）方式部署，通常根据需要部署在 3 个或者 5 个节点。Job Center 以 MongoDB 作为任务队列，将 Job 相关的任务路由到目标节点执行。Job 及其任务状态也会被存储到 MongoDB，Job Center 会追踪任务的状态，决定何时终止一个任务（超时）以及是否需要重新提交任务，以保证每个任务至少被执行一次，任务可重入性（reentrancy）由任务自身实现来保证。

Job Center 由 Job Center Worker 和 Job Center Scheduler 两个组件构成。

## Job Center > Job Center Worker

# Job Center Worker

Job Center Worker ，简称 JC Worker，被部署到所有节点，没有主从之分，都处于 Active状态。从结构上讲，JC Worker 由 Leader、Follower 和 Splitter 3 个组件构成。从 Job 执行过程上讲，JC Worker 作为任务的实际执行者，分为 Leader 和 Follower 两个角色，每个 JC Worker 都可以在两个角色间切换，但在一次任务执行周期内只可能是 Leader 或者 Follower。

## Leader

第一个接手处理 Job 的 JC Worker 被称为 Leader，执行 Leader 任务，负责对 Job 进行任务拆分、构建拓扑结构和分发。当一个 Job 被提交到 Job Center，将由一个 Leader 任务开始。Leader 使用 Splitter 来对 Job 进行任务拆分，生成一组子任务（即 Follower 任务），并对子任务构建拓扑结构后进行分发。当所有子任务执行完毕，最后的子任务会再次提交到 Leader，由 Leader 根据所有子任务执行结果来更新 Job 执行结果。其关系描述如下所示：

![](https://cdn.smartx.com/internal-docs/assets/b6349ae1/image10.png)

Job Center 不允许多个 Job 同时操作同一个资源，并发操作一个资源可能损坏资源，造成数据不一致。为禁止发生这种情况，Leader 也负责锁定 Job 包含的资源，并在 Job 执行完成后释放资源。Leader 开始处理 Job 时将对包含资源挨个进行标记，如果发现任何一个资源已经被其他 Job 标记，则放弃当前 Job，将其状态更新为失败后结束处理。仅当所有包含资源被成功标记，才会进行任务拆分。由此可见，Job Center 中对同一资源的操作是序列化的，并发提交包含大量资源的 Job 可能会在资源锁定处造成性能瓶颈。

## Follower

真正执行 Job 子任务的 JC Worker 被称为 Follower。在实现上，每一个 Follower 任务对应一个函数，这个函数只做一件简单的事情。每一个 Follower 任务完成的状态和时间都会被记录下来，以减低机器故障时候的影响和提高恢复任务的成功率。例如创建、启动、关闭虚拟机等，都是单独的 Follower 任务。Follower 任务之间可能有依赖，那么被依赖的 Follower 任务完成后，将按照 Leader 构建的子任务拓扑结构提交下游任务，直到所有 Follower 任务完成。

## Splitter

Splitter 是 Leader 运行过程中使用的组件，负责将一个 Job 拆分成一个或者多个 Follower 任务。通常 Job 都会包含一组资源，资源被设计为有状态的对象。资源可以在不同状态之间进行合法转换，转换过程被分解成一系列操作步骤，这些操作步骤即被描述为 Follower 任务。资源状态之间转换的合法性（注：不是所有状态之间都能相互转换）及转换步骤被预先定义，并映射到 Follower 任务，构成一张状态转换图。Job Center 中每一类资源都有对应的 Splitter， 例如虚拟机对应会有 KvmVMSplitter，负责运行时按照状态转换图将 Job 拆分成 Follower
任务。假定目前有一个虚拟机的状态是关闭状态，现在提交一个 Job，希望 Job 执行后该虚拟机是运行状态。按照虚拟机状态转换图，虚拟机由关闭状态进入运行状态需要经过开机操作，则 Splitter 变化生成一个虚拟机开机的 Follower 任务。

![ELF-4.8.1.png](https://cdn.smartx.com/internal-docs/assets/b6349ae1/image15.png)

此外，Job Center 还支持定时触发的周期性 Job，该类 Job 可以不包含资源集合，如定时收集节点信息。这类 Job 将由一个特定的 Splitter 处理，通过分析 Job 附带的数据将 Job拆分为 Follower 任务。

## Job Center > Job Center Scheduler

# Job Center Scheduler

Job Center Scheduler，简称 JC Scheduler，根据需要被部署到 3 个或者 5 个节点上，仅用于向 Job Center 提交 Job。各节点上的 JC Scheduler 没有主从之分，都处于 Active 状态。在每个定时 Job 周期内，各节点上的 JC Scheduler 都要向 Job Center 提交 Job，但有且只有一个 JC Scheduler 提交成功，其他 JC Scheduler 都会因为竞争失败而放弃提交，此过程由 MongoDB　的原子操作来保证。因此集群中只要有一个 JC Scheduler 正常便可保证整个集群的定时 Job 工作.

## Job Center > Job 调度机制

# Job 调度机制

Job Center 的调度机制基于有向无环图，而图中的每一个节点是 Follower 任务，带箭头的直线表示任务之间的依赖关系。

任务调度有以下几个准则：

- 所有 Follower 任务构成的有向无环图中，入度为 0 的节点任务称为就绪任务，表明该任务没有依赖任务或者依赖任务已经执行完成，任务就绪等待执行；
- 当 Leader 提交任务到 Follower 时，提交的是所有就绪任务；
- 当 Follower 任务完成后，需要 Follower 检查依赖此任务的下游任务是否有可以执行的任务并提交到下一个或多个 Follower；
- Follower 检查是否有新任务可以执行的准则为：删掉当前 Follower 的任务节点后，是否有新的就绪任务节产生；
- 每个 Follower 任务的数据库更新操作，在整个 Job 执行周期内需要保证其原子性，以确保 Follower 任务的多个上游任务之间在同一时刻看到的下游任务状态是一致的，从而保证按照流程正确执行任务；
- 失败的任务不会被删除节点，并且此 Follower 会把所有直接或者间接依赖这个任务的任务标记为失败；
- 最后一个完成或者失败的 Follower 重新提交回 Leader。

## 正常成功 Job

如上图所示，简述并发执行步骤：

1. Leader 接收任务并构建 DAG，判断可以并发执行任务是三个，分别是①②③，提交任务；
2. ①执行完后，依赖 a、b 会消失，但④⑦因为还有依赖所以无法执行，退出；
3. ②执行完后，依赖 c、d、e、f 消失，④⑤⑥可以作为就绪任务执行，所以④⑤⑥的任务由②的Follower 触发发送；
4. ③执行完后，没有产生依赖而消失。所有可执行的就绪任务目前还处于运行中，退出；
5. ⑤任务执行完，依赖 i 消失，但⑧还依赖⑥所以无法触发任务，退出；
6. ④任务执行完，依赖 g、 h 消失，⑦成为就绪任务，由④的 Follower 触发发送，⑨还依赖⑥，退出；
7. ⑥任务执行完，依赖 j、k 消失，⑧⑨成为就绪任务，由⑥的 Follower 触发发送；
8. ⑧任务执行完，没有可执行任务，退出；
9. ⑨任务执行完，没有可执行任务，退出；
10. ⑦任务执行完，所有任务执行完毕，触发消息重新发送到 Leader 进行总结完成；
11. Leader 检查所有任务成功，标记 Job 为 Done 并退出；

## 中途失败 Job

如上图所示，简述并发执行步骤，并在任务④执行失败：

1. Leader 接收任务并构建 DAG，判断可以并发执行任务是三个，分别是①②③，提交任务；
2. ①执行完后，依赖 a、b 会消失，但④⑦因为还有依赖所以无法执行，退出；
3. ②执行完后，依赖 c、d、e、f 消失，④⑤⑥可以作为就绪任务执行，所以④⑤⑥的任务由②的 Follower 触发发送；
4. ③执行完后，没有产生依赖而消失。所有可执行的就绪任务目前还处于运行中，退出；
5. ⑤任务执行完，依赖 i 消失，但⑧还依赖⑥所以无法触发任务，退出；
6. ④任务执行失败，标记所有和④有关系的下属任务为失败，也就是⑦⑨退出；
7. ⑥任务执行完，依赖 j、k 消失，⑧⑨作为就绪任务，但⑨被标记为失败，只提交⑧任务；
8. ⑧任务执行完，检查到所有任务已经完结，触发消息重新发送到 Leader 进行总结完成；
9. Leader 标记 Job 为 Failed 并退出；

所有的图更新均需要是原子性操作，7 执行时 ④ 有关联的下游任务 ⑦⑨ 一定都为失败状态。上述任务中 6 - 7 交换顺序并不会造成不一样的结果。若 6 - 7 同时完成，Job Center 会定期检测到这种情况，并提交就绪任务 ⑧ 。

## 执行途中节点断电导致 Follower/Leader 失败

所有执行途中的 Follower 假如因为断电或网络中断等原因导致无法执行，Scheduler 触发的定时任务会检查所有运行中的 Job 的子任务状态，包含子任务的执行者 Handler（[请见 Job Recovery 机制](#job-recovery-%E6%9C%BA%E5%88%B6)）、开始时间、还有提交后没有执行的超时时间等。发现不正常的任务会重新提交，然后等待最终 Follower 触发 Leader 完成 Job。假如因为断电或死机等原因在所有 Follower 完成时中断而且还没来得及提交回 Leader，则由 Scheduler 重新提交任务到 Leader 进行处理。

## Job Recovery 机制

Job Recovery 的设计目的是为了处理任何时候下的分布式任务处于“完成”或者“失败”两状态之一，而不会出现中间状态。例如电源、网络、磁盘、操作系统等故障不会导致任何任务（例如创建虚拟机、创建磁盘等）处于中间状态，以致资源被锁。

Job Recovery 是通过一个定时任务定期检查来实现的。当到预定时间 Job Center Scheduler 会触发一个任务，然后发送到 Job Center Worker 检查是否有 Job 需要恢复。发送到 Job Center Worker 的是集群任务，并没有指定节点，所以只需集群至少一个节点是正常的，就可以正常工作。

此外，Job Recovery 任务因为是集群任务，所以为了防止同时执行因延时或者瞬发任务量太多而积聚过多的同类型任务，利用了 Job Queue 的特性，来控制同一时间只能有一个集群的恢复任务。

Job 的描述内，有以下一些信息用于恢复的判断：

- Handler

  用于记录 Job 或者 Job 下子任务的目前处理者，内容为每一个 Job Center Worker 启动时候的临时 UUID。
- Job 和每个子任务的创建时间

  Job 和 Job 下子任务的创建时间。Job 的创建时间为提交 Job 到数据库的时间；而 Job 下的子任务的创建时间则为 Splitter 分析完后，写入数据库的时间。
- Job 和每个子任务的最后修改时间
- Job 和每个子任务的完成时间

  Job 下的子任务的完成时间应该为单个任务完成的时间；而 Job 的完成时间是整个 Job 所有任务完成后的完成时间。

Job Recovery 规则描述：

- 每一分钟检查运行中的和未运行的所有 Job；
- 每一个 Job 和 Job 下的子任务都需要记录分配时间、执行时间和 Handler ID；
- Handler 已经不在运行中，则重新提交 Job 或者子任务到 Job Center 处理；
- 当超过5分钟 Handler 没有被写入时：如果是 Leader 任务则重新提交；如果是 Follower 任务则提交对应的任务到 Job Center 处理。假如子任务是指定节点但节点无响应，则标记该子任务失败；
- Job 所有子任务已经执行完毕，但 Job 超过1分钟没有标记完成或者失败，则重新提交 Job 到 Job Center 处理；
- 发送到 Job Center 的任务消息重复时，不再提交；当任务被标记失败后，需要清理在等待或者未完成的消息；
- 判断上述需要重新提交任务的 Job 的类型是 Scheduler 时候，不会提交 Job 并置整个 Job 和未完成的子任务为 Failed，以防止产生大量定时任务不断重试的情况。

## Job Center > Job 类型

# Job 类型

## 资源类型

资源类型是通过描述资源的状态，并通过类似状态机的方式处理来达到目标状态的任务。其核心思想是资源状态保证。也就是无论目前状态如何，系统都会保证在提交当前的资源描述后，得到的结果就是和提交的是一致的。

![ELF-4.8.1.png](https://cdn.smartx.com/internal-docs/assets/b6349ae1/image15.png)

优点：

- 非操作类描述，避免状态不一致而导致的出错。
- 对前端入口友好，输入简单，不需要前端分析行为。
- 能完成一连串复杂的类事务任务。

## 一次性任务类型

一次性的任务通常用于耗时比较长而又明确简单的任务，例如挂载磁盘、移除主机等。

## 定时任务类型

定时任务是由 Job Center Scheduler 触发的定时任务，用来处理需要不断循环的任务，例如报警检查、虚拟机 HA 功能。

整个定时任务流程主要分两部分：

- Job Center Scheduler 定时触发

  定时任务由只在固定的三或五个节点部署的 Job Center Scheduler 触发，而只要有一个节点服务是正常的，便可以让定时任务正常执行，而多节点部署是为了灾备更完善。

  ![ELF-4.8.3.png](https://cdn.smartx.com/internal-docs/assets/b6349ae1/image14.png)
- 执行明确的任务

  定时任务并没有资源类型的类似状态机的设定，所以定时任务在提交的时候便需要输入要执行的任务和需要的细节。

# Job Resource Group

Job Resource Group 是服务于资源类型 Job，对同一个 Job 内的资源分组，以便在处理依赖和构成有向无环图的时候进行操作隔离。分组关系需要在提交 Job 的时候指定。假如有多个分组，则有向无环图有多个，因为图之间没有依赖，因此可以并行执行。目前此特性被引用于批量处理中，例如从虚拟机模板批量创建虚拟机。

## 支持的组件及功能

# 支持的组件及功能

## 虚拟机

虚拟机是整个 ELF 的核心，为客户提供虚拟计算机服务。ELF 使用 KVM 做基础，libvirt 辅助来实现虚拟机的构建和管理。此外，为保证虚拟机能得到高可靠、高可用等集群特性，特别加入虚拟机 HA、虚拟机迁移、虚拟机自动调度等功能。

虚拟机提供的主要功能有：

- （批量）创建
- （批量）开机
- （批量）关闭
- （批量）强制关闭
- （批量）重启
- （批量）强制重启
- （批量）暂停
- （批量）恢复
- （批量）迁移
- （批量）删除
- 克隆

## 虚拟机自动调度

虚拟机自动调度是虚拟机的一个重要组成部分，其目的是根据集群的使用情况来自动调度创建或者启动虚拟机，以达到平衡虚拟机在集群内部的分配。

ELF 中由 VM Scheduler 实现调度算法，配合 Job Center 完成虚拟机自动调度。VM Scheduler 根据需要部署在 3 个或者 5 个节点上，其中只有 1 个节点上的 VM Scheduler 处于 Active 状态，其它的都处于 Standby 状态。Active 状态的 VM Scheduler 随 ZBS-Meta Leader 切换而切换，始终与 ZBS-Meta Leader 在相同的节点上。另外，VM Scheduler 会使用 MongoDB 存储计算中的节点资源数据。当其不可用时，为虚拟机分配资源的相关功能都不可用，典型的如虚拟机创建、开机和迁移等。

VM Scheduler 对节点的筛选基于过滤器链来实现，然后对满足调度的节点计算权重，最后选择最优的节点作为虚拟机调度的目标节点。其过程如下图所示：

![](https://cdn.smartx.com/internal-docs/assets/b6349ae1/image7.png)

过滤器链由多个过滤器组成，筛选出可用的主机，其完成的功能主要是：

1. 过滤不满足虚拟机硬性物理条件要求的节点

   通过一些硬性准则，例如是否为存活节点、网络环境配置、CPU 兼容性、虚拟机内存分配在单节点的总量不超过物理内存、是否开启主机维护模式等，来过滤掉不合适的节点。
2. 按照虚拟机放置组规则过滤掉不满足策略要求的节点

   虚拟机放置组策略是一组对虚拟机放置位置作出规定的策略。一旦启动，就成为强制性策略。它的优先级高于上述的两个原则。目前 ELF 的放置组原则有如下三类：

   1. 必须放置或不放置在选定的主机上：双活环境中，当启用了此类策略和高可用功能时，推荐在两个可用域内至少各选择一台主机。否则，有可能会导致业务连续性中断。如果此处所选主机都在优先可用域，当整个优先可用域都不可用时，高可用功能受限于“必须”，无法把虚拟机自动迁移到次可用域的主机上。
   2. 优先放置或不放置在选定的主机上：双活环境中，当启用了此类策略和高可用功能时，不推荐选择次可用域的主机，这样虚拟机会优先迁移到优先可用域的主机上。即便整个优先可用域不可用，然而因为是“优先”策略，不是“必须”策略，所以系统还是会自动把虚拟机迁移到次可用域上，使得业务的连续性得以保障。
   3. 优先放置或不放置在相同或不相同的主机上：上述两类规定了虚拟机和主机之间的放置关系，而此类规定了虚拟机和虚拟机之间的放置关系。
3. 双活环境中，优先选择优先可用域节点

   若优先可用域中有可用节点则优先选择优先可用域节点，当且仅当优先可用域中无可用节点时才选择次级可用域节点。
4. 根据调度场景选择支持虚拟机启用 CPU 独占的节点

   - 若调度场景是用户主动发起调度需求（例如开机、热迁移等场景），则调度器会选择可独占 CPU 数量支持虚拟机启用 CPU 独占的节点。
   - 若调度场景是虚拟机 HA，则调度器会优先选择可独占 CPU 数量支持虚拟机启用 CPU 独占的节点，如果没有满足该条件的节点，则调度器不会过滤节点，虚拟机重建后不会启用 CPU 独占。

当过滤掉不合适的节点后，如果余下的节点多于一个，则根据当前虚拟机是否处于**运行中**状态，通过下列算法进行算分排序：

- 当虚拟机处于**运行中**状态时：

  - CPU 分值 =（主机上处于**运行中**状态的虚拟机的 vCPU 总数 + 当前虚拟机需要的 vCPU 个数）/（集群已分配的 vCPU 总数 + 当前虚拟机需要的 vCPU 个数）
  - 内存分值=（主机上处于**运行中**状态的虚拟机的内存总数 + 当前虚拟机需要的内存）/ 物理节点总内存
  - 主机分值=（CPU 分值+ 内存分值）/2
- 当虚拟机处于**关机**状态时：

  - CPU 分值 =（主机上所有虚拟机的 vCPU 总数 + 当前虚拟机需要的 vCPU 个数）/ 主机上所有虚拟机的 vCPU 总数
  - 内存分值=（主机上所有虚拟机的内存总数 + 当前虚拟机需要的内存）/ 物理节点总内存
  - 主机分值=（CPU 分值+ 内存分值）/2

主机分值最低的节点权重最高，最终将被选择为目标调度节点。

## 虚拟机快照

虚拟机快照是对虚拟卷和虚拟机配置信息的备份。虚拟机和虚拟机快照是独立的存在，它们并不互相依赖。

每一份虚拟机快照包含下面信息：

- 所有虚拟卷的快照
- 虚拟机的配置信息，例如 vCPU 内存、磁盘启动顺序、网络配置等

此外，虚拟机快照支持下面的功能：

- 快照回滚
- 从快照重建虚拟机

## 虚拟机模板

虚拟机模板和虚拟机快照类似，也是对虚拟卷和虚拟机配置的备份，但不一样的是模板在制作完成后可以批量生产出新的虚拟机，而且也无法对虚拟机进行回滚。此外，虚拟机快照下的虚拟卷快照是基于 ZBS 的快照特性开发的，而虚拟机模板下的虚拟卷模板则是通过 SMTX OS
虚拟机服务控制的克隆方式处理。虚拟机模板和虚拟机是独立的存在，互不依赖。每一个虚拟机模板下的虚拟卷，是
ZBS 的一个被克隆出来的新的卷，只是这个特殊的虚拟卷在 ELF 不会被显示和不会被使用。

每一份虚拟机模板包含下面的信息：

- 根据源虚拟机下所有虚拟卷所克隆出来的一份新虚拟卷
- 虚拟机的配置信息，例如 vCPU、内存、盘启动顺序、网络配置等

虚拟机模板支持的功能

- 从关闭的虚拟机创建模板
- 从模板批量创建虚拟机

## 虚拟卷

在 ZBS 的基础上，ELF 使用 ZBS 的 iSCSI 特性构建供虚拟机使用的虚拟卷。ZBS 的优势使得虚拟卷具有高性能、高可用等特性。

在 ELF 使用 ZBS 的存储功能时候，一个虚拟卷会对应一个 iSCSI 的 LUN。ELF 会帮助使用者更方便地使用 ZBS，所以一些 ZBS 概念例如 iSCSI Target 在 ELF 会被隐藏，不需要感知，用户只需要创建虚拟卷，并设置对应的存储策略，那么 ELF 便会处理所有底层和 ZBS 的交互和关联。

## 虚拟卷快照

虚拟卷快照是基于 ZBS 卷快照来处理的。ELF 的虚拟卷所使用的 iSCSI LUN 属于 ZBS 卷，所以只需要对 SMTX 卷做快照，则可以完成对 iSCSI LUN 的快照创建。

## 虚拟机镜像

虚拟机镜像是基于 ZBS 的 NFS 服务而构建的。每一个上传的镜像，都会做成 NFS 的一个文件供虚拟机挂载成 CDROM 读使用。上传的镜像都会被放到 NFS 的一个特定的 Export，并赋予唯一的 UUID。

目前支持创建镜像的方式只有浏览器上传的形式，浏览器会自动拆分文件并分块上传。

## 网络管理

# 网络管理

## 虚拟分布式交换机

ELF 虚拟分布式交换机（Virtual Distributed Switch - VDS）是一个抽象的虚拟交换机概念，用于管理多台主机上的虚拟交换机（ELF 中基于开源的 Open vSwitch 实现）。这样一方面可以对多台主机的虚拟交换机进行统一配置和管理，另一方面可以保证虚拟机在不同主机之间迁移时网络的一致性。

实际而言，ELF 虚拟分布式交换机也可视为物理交换机在服务器主机上的延伸。如下图所示，ELF 虚拟分布式交换机会对主机物理网口和虚拟机的虚拟网口进行管理，虚拟机的一个虚拟网口对应虚拟交换机的一个虚拟端口，主机的一个或者多个物理网口作为虚拟交换机的上联端口（Uplink-Port）。

![](https://cdn.smartx.com/internal-docs/assets/b6349ae1/elf-network-overview.png)

虚拟交换机可以创建多个虚拟端口（为保证服务质量建议不要超过 1024），用于连接虚拟机的虚拟网口，可以同时绑定一个或者多个虚拟网口，用于虚拟机和外部网络通信。主机的一个物理网口只能属于一个虚拟交换机，但是虚拟交换机可以绑定多个物理端口，此时各个物理端口之间是绑定的关系。ELF 虚拟分布式交换机基于 OVS（Open vSwitch） 实现，因此 ELF 虚拟机分布式交换机在缺省情况下使用 OVS Bond 绑定类型。如果存储网络开启了 RDMA 功能，则存储网络关联的虚拟机分布式交换机使用 Linux Bond 绑定类型。

- **OVS Bond**

  OVS Bond 支持以下绑定模式：

  - active-backup：主备模式。
  - balance-slb：根据包的 Source MAC + VLAN Tag 来均衡流量。
  - balance-tcp：根据包的 L2、L3、L4 Header 来均衡流量，该模式要求物理交换机设置 802.3ad(LACP) 才可以使用。
- **Linux Bond**

  Linux Bond 支持以下绑定模式：

  - active-backup：主备模式。
  - balance-xor: 根据由报文的源 MAC 地址和目的 MAC 地址计算出的哈希值将报文通过不同物理端口转发。该模式要求物理交换机配置静态 EtherChannel，否则会存在 MAC 地址漂移的问题。
  - 802.3ad: 根据包的 L2、L3、L4 Header 来均衡流量，该模式要求物理交换机设置 802.3ad(LACP) 才可以使用。

## 虚拟网络

依据虚拟分布式交换机和 VLAN ID 共同定义的连通性，在 ELF 中抽象出 “虚拟网络” 的概念。针对管理、存储和虚拟机的不同需求，ELF 中定义了三种虚拟网络，分别是：

1. 管理网络：有且只有一个。主要用于连接 Web 和 SSH 的网络，对性能没有特别需求，百兆就足够。物理上可能和业务网络混合，但更多的时候是和业务网络分开的。
2. 存储网络：有且只有一个。在 ZBS 中也称为数据网络。专门用于支持存储，是物理隔离的，不能对外网通信，不能访问其它网络，有绑定的需求。因为是物理隔离的，一般不需要 VLAN。
3. 虚拟机网络：虚拟机运行的网络。根据业务需求可能存在多个虚拟机网络。一台虚拟机可以通过多个不同的虚拟网口连接不同的虚拟机网络，有 VLAN 和绑定的需求。当虚拟机在使用同一存储的不同主机之间进行迁移时，虚拟机网络保证在迁移过程中网络不中断，不需要重新配置网络数据库。

从技术上来讲，这三种网络是相同的，只是逻辑分工不同，管理网络和存储网络上不支持用户创建虚拟机。

## 管理虚拟分布式交换机

网络管理支持虚拟分布式交换机的创建、编辑和删除，但管理网络和存储网络所属的分布式虚拟交换机受以下规则约束：

1. 由于存储网络的重要性，其所属的虚拟分布式交换机是只读的，即不支持编辑或删除操作，但不限制在该虚拟分布式交换机下创建虚拟机网络（不建议）。
2. 对管理网络所属的虚拟分布式交换机可编辑但不可删除。

## 管理虚拟网络

任何虚拟分布式交换机下都可以创建虚拟机网络，并支持对其进行编辑和删除。删除一个虚拟机网络的前提是该网络没有被虚拟机、虚拟机模板或虚拟机快照使用，否则无法执行删除操作。虚拟机网络的可编辑项包括：网络名称、VLAN ID 和 QoS 策略。

管理网络由于其管理属性，仅支持编辑操作，与虚拟机网络稍有不同，其可编辑项包括：VLAN ID、网关、子网掩码、各主机的管理 IP、MTU 和 QoS 策略。

存储网络仅支持编辑 MTU 和 QoS 策略。

## 存储策略

# 存储策略

## 数据存储与存储策略

数据存储是一组虚拟卷的集合，在 ZBS 的设定中，数据存储与存储策略是耦合在一起的，也就是说一个数据存储中包含的虚拟卷具有相同的存储策略。

## ELF 与数据存储的单向感知关系

ELF 中与存储相关的资源包括虚拟机对应的虚拟卷、虚拟卷快照、虚拟机快照以及用到的 ISO 映像，这些资源都对应到 ZBS 的存储对象，但是这些存储对象是被 ZBS 独立管理的，无法被 ELF 自动感知或者直接使用，除非被主动导入到 ELF（实际是记录在 ELF 的 MongDB 中）。二者的关系如下图所示：

![](https://cdn.smartx.com/internal-docs/assets/b6349ae1/image11.png)

## ELF 的存储策略

在 ELF 中用户只需要关注计算业务，而不需要关心存储资源属于哪一个 ZBS 数据存储。但在 ZBS 中数据存储与存储策略是耦合的，存储策略作为数据存储的属性会给 ELF 用户带不必要的复杂度。为简化用户操作，ELF 中也引入“存储策略”的概念。ELF 通过自身层面的存储策略来管理虚拟卷的创建，决定其所属的数据存储。

目前 ELF 的存储策略默认仅支持 iSCSI 数据存储，可以根据需要自动创建或删除数据存储，
让用户更专注计算业务层。用户可以根据自身业务需要在 ELF 中创建不同的存储策略。

### 存储策略选项

ELF 存储策略除名称、描述外，支持的选项包括基础策略和高级策略两部分。

1. 基础策略：

   - 副本数
   - 是否精简配置
2. 高级策略：

   - 条带数量
   - 条带大小

ELF 默认提供一个名为 “default” 的存储策略，选项设置为：

- 副本数：2
- 是否精简配置：是
- 条带数量：8
- 条带大小：256 KiB

### 自动管理虚拟卷

自动管理虚拟卷是 ELF 存储策略一个主要组成部分，用户在创建虚拟卷时只需要关心虚拟卷对应的存储策略。ELF 会按照指定的存储策略创建虚拟卷并自动将其放入符合策略的数据存储。

ELF 中一个存储策略存在 1 个或多个对应的 iSCSI 数据存储。由于一个 iSCSI 数据存储最多可容纳 256 个 LUN，创建虚拟卷时，如果该策略下可用的 iSCSI 数据存储 LUN 数量达到上限，则 ELF 会自动为该存储策略创建一个新的 iSCSI 数据存储，并将该虚拟卷放入到新建的 iSCSI 数据存储。当一个虚拟卷被删除时，ELF 会主动回收其对应 iSCSI 数据存储下的 LUN ID，以供后续新建虚拟卷使用。

## 虚拟机高可用

# 虚拟机高可用

虚拟机高可用是一个虚拟机的高级功能，目的是为了保证在极端环境下虚拟机都是可恢复和可用的。

下图描述了整个虚拟机高可用的构成组件和实现原理。最核心的判断思想是分别通过 Job Center 和 VM Monitor 两个组件检查存储的可用性和 MongoDB 心跳来给虚拟机高可用提供依据。

![](https://cdn.smartx.com/internal-docs/assets/b6349ae1/image2.png)

**保证**

- 已启用 HA 保护的虚拟机在一定条件下，会被重建到集群健康的节点并重新启动。
- 触发了 HA 恢复的虚拟机不可因为网络抖动等任何原因，出现两个相同的虚拟机同一时间分布在同一个集群，以避免数据损坏。
- HA 机制并不依赖物理时钟。

**触发标准**

- 集群健康情况下，其中一个主机发生断电、拔掉存储网线、ifdown 存储网络、文件系统进入只读状态或者关机时，会触发 HA。
- 拔掉非存储网络网线，例如管理网络网线，不会触发 HA。
- 已启用 HA 保护的虚拟机运行状态与数据库中记录状态不一致时，会触发 HA 将虚拟机恢复到数据库中记录的状态，例如 qemu 进程被 kill。Guest OS 内部主动关机不触发。
- 若集群启用 Boost 模式，存储服务故障时会触发 HA。
- 若在集群的高可用故障场景中将虚拟机网络故障场景是否触发 HA 选择为是，当虚拟机网络故障时，会触发 HA。
- 若在集群的高可用故障场景中将虚拟机操作系统故障场景是否触发 HA 选择为是，当虚拟机操作系统蓝屏、内核崩溃或触发 HA 后操作系统启动失败时，会触发 HA。

**行为**

- 已启用 HA 保护的虚拟机会在集群中自动触发重建、恢复、热迁移或重启，以保证高可用。
- 如果节点因网络故障或特殊情况被隔离，则这个节点上的所有虚拟机将首先被暂停，而如果暂停失败，则虚拟机将被强制关机。
- 如果节点因存储网络故障被隔离，节点上所有的虚拟机将先被暂停，并触发 HA 调度至其他节点开机，部分虚拟机可能因集群剩余资源不足而未成功调度到其他节点上开机，将一直处于暂停状态。在存储网络恢复正常后，该节点上启动 HA 的虚拟机（暂停状态）将自动恢复为触发 HA 之前的状态，而不会重新启动。
- 如果因存储交换机故障或者其他原因致使整个集群的存储网络不可用，导致所有的节点被触发隔离，则节点上的所有虚拟机将会被暂停，而由于集群上无正常运行的节点，虚拟机也不会调度到其他节点；待存储网络恢复正常后，每个节点上启动了 HA 的虚拟机（暂停状态）将自动恢复为触发 HA 之前的状态，而不会重新启动。
- 如果节点因存储网络故障被隔离，节点上所有的虚拟机被暂停，节点上的虚拟机在触发 HA 后成功恢复至其他节点开机，则在该故障节点恢复正常后，集群会自动清理节点本地已经暂停的虚拟机。
- 已启用 HA 保护的虚拟机状态恢复失败 13 次后，将按照虚拟机当前的实际状态进行标记，并停止恢复操作。虚拟机在前 5 次进行恢复操作的时间间隔为 10s，第 5 次之后为 60s，第 10 次之后为 100s。虚拟机的恢复状态最长持续时间为 650s。
- 如果是 ELF 双活集群，在未专门配置虚拟机放置组规则情况下，将按照优先可用域优先的模式进行 HA 调度，即在优先可用域内节点资源满足虚拟机启动需求时，虚拟机触发 HA 时将始终选择优先可用域内节点作为 HA 目的地。
- 如果因虚拟机网络故障触发 HA，将根据所选 HA 动作触发相应行为，支持**热迁移虚拟机**和**重建虚拟机**两种动作。当选择**虚拟机热迁移**时，若集群中无可调度主机，则该虚拟机将保留在所属主机上运行，相应的虚拟网卡将展示网络故障提示。
- 如果虚拟机在触发 HA 后操作系统启动失败，则该虚拟机将在所属主机上重启。

**性能**

当集群内启用高可用的虚拟机故障时，将在一定时间内触发高可用，具体时间如下表所示。

| **故障场景** | | **故障检测敏感度** | | |
| --- | --- | --- | --- | --- |
| **标准（默认）** | **较高** | **高** |
| 主机异常 | 断电、存储网络异常 | 130 s | 100 s | 60 s |
| 文件系统只读 | 85 s | 85 s | 85 s |
| 虚拟机状态异常 | | - | - | - |
| 虚拟机网络故障 | | 65 s | 45 s | 35 s |
| 虚拟机操作系统故障 | | 120 s | 60 s | 30 s |

> **说明**：
>
> 当系统检测到虚拟机状态与预期不符时会立刻进行虚拟机状态恢复，故不考虑高可用触发时间。

**组成部分**

- Job Center Scheduler

  Job Center Scheduler 会定时触发检查任务和 Job Center 心跳任务并发送消息到 Job Center
  Worker 执行。
- Job Center Worker

  Job Center Worker 在 HA 中，检查集群中所有虚拟机状态和主机状态，和触发虚拟机重建任务。
- VM Monitor

  VM Monitor 是用于监控本地虚拟机和存储网络的程序。一旦达到了判断条件，则负责关闭虚拟机，以避免网络恢复后，本地虚拟机和远程 HA 恢复重建后的虚拟机发生冲突。此外，此服务还负责定时触发 NFS 存储心跳和 MongoDB 心跳，而属于 Zookeeper Leader 的节点则还需要负责定时检查其它节点的心跳状态是否有更新，无更新则累计次数，当次数累计到一定后，则更新虚拟机状态供 Worker 进行重建任务判断。

**场景**

| **场景** | **是否触发** |
| --- | --- |
| 单节点 MongoDB 不可用，且相同节点 NFS 可用 | 不触发。 |
| 所有节点 MongoDB 不可用，且 NFS 可用 | 不触发。 |
| 单节点 NFS 不可用，且相同节点 Mongo 可用 | 不触发。 |
| 所有节点 NFS 不可用，且 Mongo 可用 | 不触发。 |
| 单节点存储网络故障（或拔存储网络网线） | 触发。单节点中运行虚拟机被关闭，节点中打开 HA 的虚拟机在健康节点重建并维持原有状态。 |
| 所有节点存储网络故障（或拔存储网络网线） | 不触发。 |
| 单节点断电 | 触发。 |

**对比**

下面是 ELF HA 和其它厂商的 HA 之间的差异：

| 场景 | VMware | ELF |
| --- | --- | --- |
| 集群任务如何判断集群中节点故障 | 网络心跳故障，存储心跳正常，进入 Isolated 状态，触发 Isolated 操作。网络心跳故障，且存储心跳故障，进入 Failed 状态 | MongoDB 未更新状态，且 NFS 未更新状态 |
| 故障节点如何判断自己出现故障 | 网络心跳故障，存储心跳正常，进入 isolated 状态，触发 isolated 操作。网络心跳故障，且存储心跳故障，进入 failed 状态 | MongoDB 无法连接，且 NFS 无法访问 |
| 集群对故障节点上的虚拟机的处理 | 虚拟机有三种配置：不重启（No restart）；  如果可能，则重启（Restart if possible）；重启（restart） | 如果可能，则重启 |
| 故障节点对自己节点上虚拟机的处理 | 取决于用户配置 | 关机 |

## 数据存储

# 数据存储

数据存储是数据保护和存储策略的组成单元，每一个数据存储包含一组具有相同数据保护策略的对象。数据存储通过标准协议提供服务，目前 ZBS 支持
iSCSI 和 NFS 存储通信协议，ELF 对这两种协议提供了相应的 RESTful API 支持。在 SMTX OS
虚拟机服务存储策略部分已经提到过在 ELF 中 ZBS 存储对象是不被 SMTX OS
虚拟机服务自动感知的。在大多数情况下，用户可以通过 ELF 提供的 RESTful API
来对集群数据存储进行独立管理而不用直接访问 ZBS 的可编程接口 。

---

## 数据存储 > iSCSI 数据存储

# iSCSI 数据存储

针对 iSCSI 协议，按照存储资源结构，ELF 分别提供了针对 Target、LUN 和快照的 RESTful API。这些 RESTful API 支持的功能如下：

1. Target：创建、删除、编辑、查询。支持通过指定允许访问 Target 的 IP 地址列表，在 ZBS 接入层校验客户端的 IP 地址，对于不符合白名单中匹配规则的来源 IP，将拒绝该请求；
2. LUN：创建、删除、编辑、查询、移动、克隆、上传、从快照重建及 LUN 读写性能查询；
3. 快照：创建、删除、编辑、查询、移动、回滚。

---

## 数据存储 > NFS 数据存储

# NFS 数据存储

与 iSCSI 相同，按照 NFS 资源结构，ELF 分别提供了针对 Export、Inode 和快照的 RESTful
API。这些 RESTful API 支持的功能如下：

1. Export ：创建、删除、编辑、查询。支持通过指定允许访问 Export 的 IP 地址列表，在 ZBS 接入层校验客户端的 IP 地址，对于不符合白名单中匹配规则的来源 IP，将拒绝该请求；
2. Inode：创建、删除、编辑、查询、 移动、重命名、克隆、上传、从快照重建及文件（NFS File）读写性能查询。另外，额外提供了一个批量查询 Inode 名称的 API；
3. 快照：创建、删除、编辑、查询、移动、回滚。
