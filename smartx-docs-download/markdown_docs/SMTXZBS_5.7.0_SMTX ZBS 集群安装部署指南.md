# SMTXZBS/5.7.0/SMTX ZBS 集群安装部署指南
## 关于本文档

# 关于本文档

本文档介绍了 SMTX ZBS（读作 SmartX ZBS）产品的架构，以及在 Intel x86\_64、AMD x86\_64、Hygon x86\_64 或鲲鹏 AArch64 服务器上安装部署块存储集群和文件存储集群的流程。

SMTX ZBS 对外提供标准版和企业版，两个版本的功能和服务均有所不同，但安装部署流程完全相同。因此，本文档描述的安装部署流程均适用于 SMTX ZBS 的标准版和企业版。

阅读本文档需了解分布式存储软件，了解块存储、文件存储、iSCSI 协议和 NVMe-oF 协议等相关技术背景知识，并具备数据中心操作的丰富经验。

## SMTX ZBS 简介

# SMTX ZBS 简介

SMTX ZBS（读作 SmartX ZBS）是由北京志凌海纳科技股份有限公司（以下简称“SmartX” ）研发的分布式存储软件，提供分布式块存储、分布式文件存储及运维管理服务，可帮助企业快速构建稳定可靠的 IT 基础架构。SMTX ZBS 功能丰富且性能卓越，可运行在 Intel x86\_64、AMD x86\_64、Hygon x86\_64 或鲲鹏 AArch64 服务器上，目前已被金融、制造等行业的相关企业使用。

## SMTX ZBS 简介 > 相关概念

# 相关概念

在安装部署 SMTX ZBS 前，您需要先了解与其相关的一些概念，以更好地帮助您理解 SMTX ZBS 的安装部署流程和功能。

**SMTX ZBS 集群**

SMTX ZBS 集群包含块存储集群和文件存储集群，分别提供块存储服务和文件存储服务。文件存储集群需要在块存储集群安装部署完成后再进行部署。

**块存储**

块存储是一种将数据划分为固定大小的“块”并将这些块独立存储的存储方式。块存储将集群中的存储设备划分为多个卷，每个卷以裸设备的方式映射给计算端使用。

**文件存储**

文件存储是一种将数据组织为文件的形式进行存储的存储方式。文件存储使用树状的目录结构，在根目录下有若干子目录和文件。计算端通过路径挂载文件系统进行访问和使用。

**块存储集群**

块存储集群属于逻辑概念，用于提供块存储服务。在实际生产环境中，一个块存储集群至少由 3 个节点通过网络互连组成。

**文件存储集群**

一个文件存储集群提供一套完整的文件服务，由至少 3 个文件控制器组成。

**节点**

块存储软件可运行在物理服务器上，而节点特指运行了块存储软件的单个物理服务器。节点是组成块存储集群的基本单位。

**文件控制器**

文件存储服务组件以容器的形式运行在 SMTX ZBS 集群的系统服务虚拟机中，这些虚拟机在文件存储集群中被称为文件控制器。

**主节点**

块存储集群中额外运行元数据服务的节点。

**存储节点**

块存储集群中未运行元数据服务的普通节点。

**副本数**

使用多副本技术提供数据冗余，一份数据会在不同主机上保存多个副本，以提高数据的可靠性和安全性。

**物理盘池**

由块存储集群中某个节点的一块或多块物理盘组成，不支持跨节点构建。SMTX ZBS 块存储支持在单个节点上配置多个物理盘池，以提升节点的性能上限并充分利用物理盘的性能。每增加一个物理盘池都会增加一定的 CPU 核心和内存消耗。根据节点容量规格不同，可配置 1、2 或 4 个物理盘池。

**纠删码**

使用一定的算法对 K 个原始数据块计算出 M 个校验块，在节点或硬盘故障后，使用其中 K 个数据块和校验块来重建数据块，从而提供对应的容错能力。纠删码仅支持在存储分层模式下使用。

**CloudTower**

SmartX 多集群资源集中管理平台。CloudTower 运⾏在虚拟机内，通过使⽤少量的计算和存储资源便可管理上百台主机。除存储管理、⽹络管理、硬件管理、监控和报警等基本服务外，CloudTower 还提供全局搜索、事件审计和报表等功能，同时它也提供完整和标准的 RESTful API 和多种语⾔的 SDK，帮助运维管理员轻松管理和使⽤多个 SMTX ZBS 块存储集群。

**容量规格**

SMTX ZBS 块存储根据单主机物理盘容量上限分为三种规格：标准容量规格（128 TB）、大容量规格（256 TB）和加大容量规格（512 TB）。

**存储分层模式**

把集群内的存储设备分为缓存层和容量层，其中容量层是所有节点数据分区构成的空间，用于存放冷数据，冷数据根据设定的数据冗余策略，以副本或纠删码的形态存储。不同存储配置下缓存层的构成不同。

当节点为混闪配置、多种类型 SSD 全闪配置或单一类型多种属性 SSD 全闪配置时，缓存盘与数据盘将独立部署，高速介质为缓存，低速介质为容量。数据会以副本形态先存入写缓存，待数据变冷后下沉到容量层。

- 对于混闪配置和多种类型 SSD 全闪配置，缓存层将分为写缓存和读缓存；写缓存用于存放新写入的数据，读缓存用于存放被频繁访问的已下沉的数据，写缓存与读缓存的容量比为 8:2。
- 对于单一类型多种属性 SSD 全闪配置，缓存层均为写缓存，用于存放新写入的数据。

当节点均为单一类型单一属性 SSD 全闪配置时，缓存与数据将共享所有物理盘，每块物理盘的部分容量作为缓存使用，缓存分区与数据分区的容量比为 1:9。此时缓存层均为写缓存，用于存放新写入的数据，副本卷将仅使用容量层，不使用缓存层。

**存储不分层模式**

不设置缓存层，除了含有系统分区的物理盘，剩余的所有物理盘都作为数据盘使用。使用存储不分层模式时只能用全闪配置。

**RDMA 功能**

RDMA（Remote Direct Memory Access）技术是一种直接内存访问技术，节点间进行数据传输时，可以绕过双方的操作系统内核，通过网络直接访问其内存数据。由于不经过操作系统，可以节省大量 CPU 资源，提高系统吞吐量，并降低系统的网络通信延时。

SMTX ZBS 块存储通过引入 RDMA 能力来达到降低远程节点的写入延时，提高集群的整体性能的目的，成功改进延时和吞吐能力两个性能指标。

**NVMe-oF 功能**

NVMe-oF（NVMe over Fabrics）是 NVMe 协议基于网络协议的拓展，NVMe 专为使用具有非易失性内存的更快存储介质而设计，这类存储可实现对数据的高效并行访问。

SMTX ZBS 块存储支持 NVMe-oF 协议，允许外部的计算端通过更高速的网络协议连接到块存储集群，集群中的存储设备被划分为多个 Namespace，每个 Namespace 以块设备的方式呈现给计算端，提供更高性能、低延时的对外 IO 链路。

NVMe-oF 支持多种类型的网络协议，包括 FC、RDMA 和 TCP/IP 协议。SMTX ZBS 块存储的 NVMe-oF 功能支持 RDMA 协议（NVMe over RDMA）和 TCP 协议（NVMe over TCP），使用户在配置相对较低的环境下也能利用 NVMe-oF 的能力。

**常驻缓存模式**

SMTX ZBS 块存储支持常驻缓存模式，将 LUN 或 Namespace 中的数据保留在缓存层，从而获得更稳定的高性能。

**Boost 模式**

SMTX ZBS 块存储集群的高性能模式，用于提升文件存储集群的性能，会在部署块存储集群时默认开启，可通过 vhost 共享协议来实现 Guest OS、QEMU、Chunk 三者之间内存共享，从而降低 IO 延时。

- 在数据传输方面，Chunk 可以直接与 Guest OS 共享数据，不再需要依靠 iSCSI 协议来传输数据，从而避免了使用 iSCSI over TCP 模式在本地网络协议栈传输带来的性能衰减和使用 iSCSI 协议造成的封装开销，使得虚拟机读写的性能都得到优化。
- 在 IO 请求处理方面，Chunk 可以直接访问到 Guest OS 的内存，通过 vhost-user target 来直接处理 IO 请求，不再需要 QEMU 的参与，绕开了由 QEMU 处理 IO 请求的线程资源有限所造成的瓶颈，从而大幅提升了处理 IO 请求的能力。

**LACP 动态链路聚合**

LACP（Link Aggregation Control Protocol，链路聚合控制协议）是一种基于 IEEE802.3ad 标准的实现链路动态聚合与解聚合的协议，属于在链路聚合中常用的协议之一。链路聚合组中启用了 LACP 协议的成员端口通过发送 LACP PDU 报文进行交互，双方对能够发送和接收报文的端口明确并达成一致，确定承担业务流量的链路。此外，当聚合条件发生变化时，如某个链路发生故障，LACP 模式会自动调整聚合组中的链路，组内其他可用成员链路接替故障链路维持负载平衡。因此在不进行硬件升级时，使用 LACP 协议也可以增加设备之间的逻辑带宽，提高网络的可靠性。

动态链路聚合模式需要本端和对端设备同时启用 LACP 协议，所选择的活动接口必须保持一致，然后才能建立链路聚合组 LAG（Link Aggregation Group）。LAG 是指将若干条以太链路捆绑在一起形成一条逻辑链路，也称为 Eth-Trunk 链路。

SMTX ZBS 支持 LACP 动态链路聚合功能。在虚拟分布式交换机中，需选择与对端交换机配置 LACP 端口相连的两个物理网口，并且网口绑定模式选择为 OVS bonding 中的 `balance-tcp` 模式或 Linux Bonding 中的 `802.3ad` 模式。设置完成后即可启用 LACP 动态链路聚合功能。

## SMTX ZBS 简介 > 安装方式

# 安装方式

SMTX ZBS 块存储支持使用独立物理盘安装，在安装 SMTX ZBS 映像文件时，您可根据系统提示选择将 SMTX ZBS 块存储安装在两块物理盘中配置软 RAID 1 或使用独立的物理盘安装 SMTX ZBS 块存储。

- **将 SMTX ZBS 块存储安装在独立的硬 RAID 1 中**

  如果希望系统盘故障时不影响集群系统运行并且能够通过硬件 RAID 重建故障盘，或者希望将系统分区与缓存和数据物理分隔、降低系统盘的 IO 频率以降低故障率，可选择此安装方式。该方式将使用两块 SSD 与存储控制器组成的硬 RAID 1 作为 SMTX 系统盘，用于引导和安装 SMTX ZBS 块存储，无需单独的启动盘。安装完成后，节点内其他物理盘作为缓存盘或数据盘用于存放缓存或数据。

  使用此模式安装 SMTX ZBS 前，请先阅读[将 SMTX ZBS 安装在硬 RAID 1 中的配置推荐](/smtxzbs/5.7.0/zbs_installation_guide_INTERNAL_EXTERNAL/zbs_installation_guide/zbs_installation_guide_82)了解推荐配置，以更好地保证系统稳定性和数据安全。
- **将 SMTX ZBS 块存储安装在软 RAID 1 中**

  如果不需要将系统分区与缓存和数据做物理隔离，或者因硬件限制无法配置单独的硬 RAID 1，可选择此部署模式。该方式需要准备一块单独的物理盘作为启动盘用于引导加载程序。在安装 SMTX ZBS 块存储时，系统将使用两块 SSD 安装 SMTX ZBS 块存储并配置软 RAID 1，这两块 SSD 称为含元数据分区的缓存盘或含元数据分区的数据盘。安装完成后，含元数据分区的缓存盘和含元数据分区的数据盘的剩余存储空间将用于存放缓存或数据，节点内其他物理盘作为缓存盘或数据盘。

## 构建 SMTX ZBS 集群的要求 > 集群规模要求

# 集群规模要求

规划 SMTX ZBS 集群时，其规模需遵照以下要求进行设置，并参考不同集群规格下主节点的配置要求和可用的冗余策略。

- 集群节点数

  单个 SMTX ZBS 集群最少应包含 3 个节点，最多不超过 255 个节点。为了发挥集群的最高性能，我们推荐单个集群最多不超过 64 个节点。

  单个 SMTX ZBS 块存储集群中仅可部署一个文件存储集群。文件存储集群最少应包含 3 个文件控制器，最多支持 16 个文件控制器，但文件控制器的数量不可超过 SMTX ZBS 块存储集群的节点数。
- 物理盘池数

  SMTX ZBS 块存储支持为节点配置物理盘池，每个物理盘池对应一个存储服务，单个集群支持配置最多 255 个物理盘池。不同容量规格的节点支持的物理盘池数量如下：

  - 标准容量规格：仅支持配置 1 个物理盘池
  - 大容量规格：支持配置 1 或 2 个物理盘池
  - 加大容量规格：支持配置 1、2 或 4 个物理盘池。

  为实现资源均衡并提升集群稳定性，建议为集群内所有节点配置相同数量的物理盘池。
- 冗余策略

  SMTX ZBS 块存储支持使用副本或纠删码的冗余策略，集群节点数不同时，推荐的副本数或纠删码配比也有所不同。

  - 副本数

    SMTX ZBS 支持 2 副本或者 3 副本的冗余策略。

    如果使用副本的冗余策略，一般情况下，推荐使用 3 副本，您也可以根据实际情况设计冗余策略。
  - 纠删码

    纠删码要求集群至少包含 4 个节点。创建 iSCSI Target 或 NVMe Subsystem 时，如果选择纠删码作为冗余策略，系统会根据集群内的节点数量默认选择推荐的纠删码配比，您也可以根据实际需求自定义纠删码配比。不同节点数下的推荐配比和可自定义配比如下。

    | **节点数** | **推荐配比 K + M** | **自定义配比 K** | | | |
    | --- | --- | --- | --- | --- | --- |
    | M = 1 | M = 2 | M = 3 | M = 4 |
    | 4 | 2 + 1 | 2 | - | - | - |
    | 5 | 2 + 2 | 2 | 2 | - | - |
    | 6 | 2 + 2 | 2、4 | 2 | - | - |
    | 7 | 4 + 2 | 2、4 | 2、4 | - | - |
    | 8 | 4 + 2 | 2、4、6 | 2、4 | 4 | - |
    | 9 | 4 + 2 | 2、4、6 | 2、4、6 | 4 | 4 |
    | 10 | 4 + 2 | 2、4、6、8 | 2、4、6 | 4、6 | 4 |

## 构建 SMTX ZBS 集群的要求 > 网络要求 > 网络类型

# 网络类型

在部署块存储集群和文件存储集群时需要创建虚拟分布式交换机并关联网络。

**虚拟分布式交换机**

SMTX ZBS 利用虚拟化技术将一组物理连通的物理网口（可结合网口绑定实现网络高可用）转化为虚拟化层的分布式交换机（Virtual Distributed Switch, VDS），为主机之间提供网络连接，并为处理各类型系统流量提供基础。

SMTX ZBS 通过虚拟分布式交换机管理物理网口绑定和 VLAN ID 等，从而实现网络高可用和网络隔离。

**块存储集群网络类型**

SMTX ZBS 块存储直接部署在物理节点上时，通过虚拟分布式交换机需要创建三种类型的系统网络：管理网络、存储网络和接入网络。每种系统网络可使用其所属的虚拟分布式交换机所关联的物理网口进行通信。

- **管理网络**：用于管理块存储集群。
- **存储网络**：用于块存储集群内各节点间数据交换，该网络只能在块存储集群内部使用。
- **接入网络**：用于为块存储集群外部计算端提供块存储服务。

**文件存储集群网络类型**

在部署文件存储集群时，需要创建三种类型的网络：文件管理网络、文件存储网络和文件接入网络。每种网络使用其所属的虚拟分布式交换机所关联的物理网口进行通信，因此这三种网络所属的虚拟分布式交换机需关联 SMTX ZBS 集群每一个主机上至少一个网口。

- **文件管理网络**：用于管理文件存储集群。
- **文件存储网络**：用于文件存储集群内各文件控制器间的数据交换，该网络只能在文件存储集群内部使用。
- **文件接入网络**：用于为文件存储外部计算端提供文件存储服务。

## 构建 SMTX ZBS 集群的要求 > 网络要求 > 网络配置要求

# 网络配置要求

SMTX ZBS 集群的网络配置须满足以下要求：

- 网络带宽

  管理网络和文件管理网络带宽须为 1 Gbps 及以上；

  存储网络和文件存储网络带宽须为 10 Gbps 及以上；

  接入网络和文件接入网络带宽须为 10 Gbps 及以上。

  如果块存储集群的两个或以上的系统网络共用相同的网口，则网络带宽须为 10 Gbps 及以上。
- 网络连接

  集群中的所有主机必须连接到 L2 层网络。
- 网络延时

  对于 1400 字节及以上的数据包，集群的存储网络 ping 延时在 1 ms 以下。

# 系统总 CPU 占用说明

当 CPU 启用超线程时，块存储将占用逻辑核心。由于同一物理核心上的两个逻辑核心存在资源竞争，另一逻辑核心的性能会显著降低，因此在计算块存储的 CPU 占用时应按物理核心进行计算。您可参考本章节，了解 CPU 占用的计算方式，以便合理规划主机计算资源。

当文件存储部署在 Hygon x86\_64 架构的块存储集群上时，文件存储将独占 CPU 核心，不与其他系统服务共享。

综上所述，系统总 CPU 占用除取决于块存储和文件存储的 CPU 占用外，还取决于主机 CPU 架构、是否启用超线程以及其他系统服务的 CPU 占用。您可参考本章节，了解不同架构下 CPU 占用的计算方式，以便合理规划主机计算资源。

## Intel x86\_64、AMD x86\_64 或鲲鹏 AArch64 架构

当在文件存储集群部署在 Intel x86\_64、AMD x86\_64 架构或鲲鹏 AArch64 架构的块存储集群上时，文件存储可与其他系统服务共享 CPU 核心。此时，块存储之外的 CPU 占用以文件存储和其他系统服务的 CPU 占用最大值为准。

系统总 CPU 占用计算公式如下：

- 启用超线程：`总 CPU 占用 = 块存储 CPU 占用 × 2 + max(文件存储 CPU 占用，其他系统服务 CPU 占用)`
- 未启用超线程：`总 CPU 占用 = 块存储 CPU 占用 + max(文件存储 CPU 占用，其他系统服务 CPU 占用)`

下面以如下配置为例介绍如何计算系统的 CPU 占用：

| 组件 | 配置 | CPU 占用 |
| --- | --- | --- |
| 块存储 | - 标准容量规格 - 1 个物理盘池 - 单 NUMA node 包含 12 颗物理核心 | 15 核 |
| 文件存储 | 每个文件控制器分配 16 核 vCPU | 16 核 |
| CloudTower | 高配 | 8 核 |
| 可观测性服务 | 管理 8～32 个节点 | 8 核 |
| 复制服务 | - | 4 核 |

文件存储、CloudTower、可观测性服务和复制服务共享 CPU 核心，取四者中最大值 16 计算，此时：

- 启用超线程：系统占用主机的逻辑核心数 = 15 × 2 + 16 = 46 核。
- 未启用超线程：系统占用主机的物理核心数 = 15 + 16 = 31 核。

## Hygon x86\_64 架构

当文件存储集群部署在 Hygon x86\_64 架构的块存储集群上时，文件存储将独占 CPU 核心，不与其他系统服务共享，块存储和文件存储之外的 CPU 占用以其他系统服务的 CPU 占用最大值为准。

系统总 CPU 占用计算公式如下：

- 启用超线程：`总 CPU 占用 = 块存储 CPU 占用 × 2 + 文件存储 CPU 占用 + max(其他系统服务 CPU 占用)`
- 未启用超线程：`总 CPU 占用 = 块存储 CPU 占用 + 文件存储 CPU 占用 + max(其他系统服务 CPU 占用)`

下面以如下配置为例介绍如何计算系统的 CPU 占用：

| 组件 | 配置 | CPU 占用 |
| --- | --- | --- |
| 块存储 | - 标准容量规格 - 1 个物理盘池 - 单 NUMA node 包含 12 颗物理核心数 | 15 核 |
| 文件存储 | 每个文件控制器分配 16 核 vCPU | 16 核 |
| CloudTower | 高配 | 8 核 |
| 可观测性服务 | 管理 8～32 个节点 | 8 核 |
| 复制服务 | - | 4 核 |

文件存储独占 16 核 CPU，CloudTower、可观测性服务和复制服务占用的 CPU 核心取三者中最大值 8 计算，此时：

- 启用超线程：系统占用主机的逻辑核心数 = 15 × 2 + 16 + 8 = 54 核
- 未启用超线程：系统占用主机的物理核心数 = 15 + 16 + 8 = 39 核

## 构建 SMTX ZBS 集群的要求 > 启用高级功能要求 > 启用机架拓扑感知

# 启用机架拓扑感知

为了确保在机架断电或损坏等极端状况下，系统还能在其他机架的主机上找到存活的副本或找到数据块和校验块重建数据，继续提供数据访问能力和增加数据可用性，SMTX ZBS 提供了机架感知功能。

启用机架拓扑感知功能并达到机箱或机架级别的容错能力需要在规划块存储集群时满足以下要求：

- 如果期望使用多副本的冗余策略，则机箱或机架数须大于等于副本数，并将服务器平均分布在不同的机架中，从而尽量使数据的多个副本均匀地分配到不同的机架、机箱和主机上。
- 如果期望使用纠删码的冗余策略，则机箱或机架数须大于等于 K + M + 1，并将服务器平均分布在不同的机架中，从而尽量使数据块和校验块均匀地分到不同的机架、机箱和主机上。

除上述要求外，还需在 CloudTower 管理平台正确配置集群的物理拓扑。即在块存储集群部署结束后，使用 CloudTower 的机架配置功能，如实呈现安装部署现场的主机、机箱与机架的位置和拓扑结构，然后系统将根据拓扑来调整数据的分配情况。详细的配置操作可参见《SMTX ZBS 管理指南》中**管理机架拓扑**章节的[配置机架拓扑](/smtxzbs/5.7.0/zbs_administration_guide/zbs_administration_guide_097)小节。

## 在物理服务器上安装 SMTX ZBS 块存储 > 检查 BIOS 设置 > 戴尔 Intel x86 服务器

# 戴尔 Intel x86 服务器

请对每个节点执行以下操作。

**操作步骤**

1. 打开浏览器，在浏览器的地址栏输入节点的 IPMI 管理台 IP，登录该节点的 IPMI 管理台。
2. 登录成功后，找到**虚拟控制台**，单击**启动虚拟控制台**，进入虚拟控制台界面。
3. 在界面顶端单击**启动**，在弹出的**开机控制**窗口中单击 **BIOS 设置**，然后在弹出的**确认启动操作**对话框中单击**是**。
4. 在界面顶端单击**功率**，在弹出的**电源控制**窗口中单击**复位系统（热启动）**，然后在弹出的**确认电源操作**对话框中单击**是**，完成后等待服务器重启。
5. 当服务器进入 **System Setup** 界面时，单击 **System BIOS**，进入 **System BIOS Settings** 界面。

   ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_55.png)
6. 在界面中单击 **Boot Settings**，将 **Boot Mode** 选项设置为 **UEFI**，以提升部署效率。

   ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_51.png)
7. 单击 **Back**，返回 **System BIOS Settings** 界面后，单击 **Processor Settings**，检查并确认 **Virtualization Technology** 选项为 **Enabled**，以确保启用 CPU 的虚拟化特性。

   ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_52.png)
8. 单击 **Back**，返回 **System BIOS Settings** 界面后，单击 **System Profile Settings**，检查并确认 **System Profile** 选项为 **Performance**，以确保关闭电源的节能模式。

   ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_53.png)
9. 检查并确认 **Workload Profile** 选项为 **Virtualization Optimized Performance Profile**，以确保系统针对虚拟化场景优化性能。

   ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_54.png)
10. （可选）若计划将 SMTX ZBS 块存储安装在独立的硬 RAID 1 中，则必须将 RAID 1 虚拟磁盘的写策略设置为 Write Through，下面以 **H730P Mini** RAID 卡为例，描述操作步骤。

    1. 1. 在 **System Setup** 界面单击 **Device Settings**，进入设备设置界面。

       ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_56.png)
    2. 在 **Device Settings** 界面选择 RAID 1 虚拟磁盘对应的存储控制器。

       ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_57.png)
    3. 单击 **Virtual Disk Management**，进入虚拟磁盘管理界面。

       ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_58.png)
    4. 选择所要安装 SMTX ZBS 块存储的 RAID 1 虚拟磁盘。

       ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_59.png)
    5. 选择虚拟磁盘的高级配置选项后，将 **VIRTUAL DISK POLICIES** 中的 **Default Write Cache Policy** 设置为 **Write Through** 策略后，单击 **Apply Changes**。

       ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_60.png)

# 部署须知

- 当接入块存储集群的管理网络、接入网络或存储网络的物理交换机启用了 LACP 动态链路聚合时，需要在部署块存储集群的[第 5 步：配置网络](/smtxzbs/5.7.0/zbs_installation_guide_INTERNAL_EXTERNAL/zbs_installation_guide/zbs_installation_guide_39)阶段，为该网络创建虚拟分布式交换机时，在**物理网口**区域，为绑定的网口选择绑定模式。

  | 网络类型 | 网络配置 | 网口绑定模式 |
  | --- | --- | --- |
  | 管理网络 | - | balance-tcp |
  | 存储网络 | 已启用 RDMA | balance-tcp |
  | 未启用 RDMA | balance-tcp |
  | 接入网络 | 已启用 NVMe over TCP 和 RDMA | 802.3ad(LACP) |
  | 未启用 NVMe over TCP 和 RDMA | balance-tcp |

  > **注意**：
  >
  > 当存储网络使用 Mellanox CX5 网卡且启用 RDMA 时，请将网口绑定模式配置为 `active-backup`；若配置为 `balance-tcp`，可能导致网络延迟升高。若需实现存储网络 RDMA 多路径，建议使用 Mellanox CX6 网卡。
- 当接入块存储集群管理网络的物理交换机启用了 LACP 动态链路聚合时，还需要在部署块存储集群前配置 LACP Bond 和管理网络，具体请参考**检查并配置网络环境**章节中的[配置管理网络](/smtxzbs/5.7.0/zbs_installation_guide_INTERNAL_EXTERNAL/zbs_installation_guide/zbs_installation_guide_98)和[配置 LACP Bond](/smtxzbs/5.7.0/zbs_installation_guide_INTERNAL_EXTERNAL/zbs_installation_guide/zbs_installation_guide_99) 小节。
- 当接入块存储集群存储网络的物理交换机启用了 LACP 动态链路聚合时，建议在物理交换机上配置 Port Channel 哈希策略，具体请参考[配置 Port Channel 哈希策略](/smtxzbs/5.7.0/zbs_installation_guide_INTERNAL_EXTERNAL/zbs_installation_guide/zbs_installation_guide_106)小节。

## 部署块存储集群 > 配置 Port Channel 哈希策略

# 配置 Port Channel 哈希策略

当接入块存储集群的存储网络的物理交换机启用了 LACP 动态链路聚合时，建议在物理交换机上配置 Port Channel 哈希策略，在默认参数的基础上增加根据源端口和目的端口作为哈希字段，以便充分利用多网口的带宽。下文分别以 Mellanox 交换机和华为交换机为例，介绍配置 Port Channel 哈希策略的操作。

## Mellanox 交换机

1. 执行以下命令，以管理员身份登录交换机，进入配置模式。

   ```
   switch > enable
   switch # configure terminal
   ```
2. 执行 `show interfaces port-channel load-balance` 命令，检查当前 Port Channel 哈希字段是否包含 `source-destination-port`。

   - 若输出结果中已包含 `source-destination-port`，则无需执行后续配置操作。
   - 若输出结果中不包含 `source-destination-port`，请继续参考步骤 3～5 进行配置。
3. 执行以下命令，在现有字段的基础上增加 `source-destination-port`。

   ```
   switch (config) # port-channel load-balance ethernet source-destination-mac source-destination-port
   ```
4. 执行 `show interfaces port-channel load-balance` 命令，检查 Port Channel 哈希字段。

   若输出结果中包含 `source-destination-port` ，说明配置成功。

   ```
   switch (config) # show interfaces port-channel load-balance
   source-destination-mac, source-destination-port
   ```
5. 执行 `configuration write` 命令，保存配置。

## 华为交换机

1. 执行以下命令，以管理员身份登录交换机，进入配置模式。

   ```
   <6865-2>system-view
   Enter system view, return user view with return command.
   ```
2. 执行 `display load-balance profile` 命令，查看负载分担模板的详细信息，确认输出结果中 IP 一行的哈希字段是否包含 `l4-src-port` 和 `l4-dst-port`。

   - 若输出结果中已包含 `l4-src-port` 和 `l4-dst-port`且不包含 `dst-qp`，则无需修改。输出示例如下：

     ```
     [~6865-2]display load-balance profile
     Load-balance Profile: default
     Packet    HashField
     ---------------------------------------------------
     IP             src-ip          dst-ip
                 l4-src-port     l4-dst-port
     IPv6           src-ip          dst-ip
                 l4-src-port     l4-dst-port
     L2             src-mac         dst-mac
     MPLS           top-label       2nd-label
     Eth-Trunk      universal-id(1) hash-mode(1)
     Stack          universal-id(1) hash-mode(1)
     ```
   - 若输出结果中不包含 `l4-src-port` 和 `l4-dst-port` 或包含 `dst-qp`，则需要手动修改，否则会影响 RDMA 的连接。请执行如下命令更新默认 profile，添加 `l4-src-port` 和 `l4-dst-port` 并删除 `dst-qp`。

     ```
     [~6865-2]load-balance profile default
     [~6865-2-load-balance-profile-default]ip src-ip dst-ip l4-src-port l4-dst-port
     [*6865-2-load-balance-profile-default]commit
     [~6865-2-load-balance-profile-default]quit
     ```

     更新完成后，再次执行 `display load-balance profile` 命令，确认配置结果符合预期。
3. 执行 `display interface Eth-Trunk 1 | include "Hash"` 命令查看 Eth-Trunk 接口的哈希策略。

   - 若输出结果中 Eth-Trunk 已经默认采用系统的 profile 作为哈希策略，则无需修改。输出示例如下：

     ```
     [~6865-2]display interface Eth-Trunk 1 | include "Hash"
     Switch Port, PVID :    1, TPID : 8100(Hex), Hash Arithmetic : profile default, The Maximum Frame Length is 9216
     ```
   - 若输出结果中 Eth-Trunk 未默认采用系统的 profile 作为哈希策略，请执行如下命令配置 Eth-Trunk。

     ```
     [~6865-2]interface Eth-Trunk 1
     [~6865-2-Eth-Trunk1]load-balance enhanced profile default
     [~6865-2-Eth-Trunk1]commit
     [~6865-2-Eth-Trunk1]quit
     ```

     配置完成后，再次执行 `display interface Eth-Trunk 1 | include "Hash"` 命令，确认配置结果符合预期。

# 配置管理网络

安装 SMTX ZBS 映像文件后，节点默认无网络配置，请参考以下步骤为节点配置管理网络。

**注意事项**

- 当管理网络使用的物理交换机启用了 LACP 动态链路聚合时，请先参考[配置 LACP Bond](/smtxzbs/5.7.0/zbs_installation_guide_INTERNAL_EXTERNAL/zbs_installation_guide/zbs_installation_guide_99) 小节创建绑定网口，再配置管理网络。
- 管理网络配置完成后不支持修改。如需变更，请参考[删除 LACP Bond 和管理网络配置](/smtxzbs/5.7.0/zbs_installation_guide_INTERNAL_EXTERNAL/zbs_installation_guide/zbs_installation_guide_102)小节，删除现有配置后重新配置。
- 在集群完成部署前，若重启已配置管理网络的节点，其管理网络配置将失效。请参考[删除 LACP Bond 和管理网络配置](/smtxzbs/5.7.0/zbs_installation_guide_INTERNAL_EXTERNAL/zbs_installation_guide/zbs_installation_guide_102)小节，清理残留配置后重新配置。

**操作步骤**

1. 在 **Network Configure Platform** 界面选择 **Management Network Configure**，单击 **Ok**，系统将显示当前节点上的所有网口及其速率（单位：Mb/s）。若节点已配置 LACP Bond，还会显示一个名称为 `pre_bond` 的网口。

   - 当管理网络、存储网络和接入网络采用分离部署模式时，速率为 1000 Mb/s 及以上的网口可以作为用于管理网络的网口。

3. 根据实际规划填写管理网络的 IPv4 地址、子网掩码和默认网关。如果物理交换机为 SMTX ZBS 管理网络配置了 Trunk VLAN，还需要填写实际规划的 VLAN ID，填写完成后单击 **Confirm**。

   ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_33.png)
4. 配置成功后，界面将显示 **Success**，单击 **Confirm** 完成配置。

   > **注意**：
   >
   > 如需启用常驻缓存模式或使用纠删码，必须选择存储分层模式。
2. （可选）当采用存储分层模式部署时，若集群内所有节点均为单一类型 SSD 全闪配置，且将 SMTX ZBS 块存储安装在软 RAID 1 中，请按照以下说明选择系统盘用途，否则请忽略。

   - 如果节点为单一属性 SSD 全闪配置，请选择系统盘用途为**含元数据分区的数据盘**。
   - 如果节点为多种属性 SSD 全闪配置，请选择系统盘用途为**含元数据分区的缓存盘**。
3. 当集群节点满足多物理盘池的配置要求时，系统会根据节点的存储配置自动选择推荐的物理盘池数量，并分配物理盘及其用途。您也可以手动修改物理盘池数量，建议各节点的物理盘池数量保持一致。

   > **注意**：
   >
   > - 单集群最多支持配置 255 个物理盘池。
   > - 单物理盘池的数据盘总容量上限为 256 TB，缓存盘总容量上限为 51 TB。
   > - 存储分层模式下：
   >   - 混闪配置、多种类型 SSD 全闪配置，或单一类型多种属性 SSD 全闪配置，单物理盘池的物理盘数量上限为 64。
   >   - 单一类型且单一属性 SSD 全闪配置，单物理盘池的物理盘数量上限为 32。
   > - 存储不分层模式下：单物理盘池的物理盘数量上限为 32。
   > - 集群部署完成后，不支持修改物理盘池数量。
4. 参考在确认[存储设备配置要求](/smtxzbs/5.7.0/zbs_installation_guide_INTERNAL_EXTERNAL/zbs_installation_guide/zbs_installation_guide_09)时已记录的每块物理盘的用途，确认系统自动分配的用途和所属的物理盘池。

   - **SMTX 系统盘**：即[将 SMTX ZBS 块存储安装在独立的硬 RAID 1 中](/smtxzbs/5.7.0/zbs_installation_guide_INTERNAL_EXTERNAL/zbs_installation_guide/zbs_installation_guide_104)的步骤 4 所选择的物理盘，不属于任何物理盘池，不可修改用途。
   - **含元数据分区的缓存盘**和**含元数据分区的数据盘**：即[将 SMTX ZBS 块存储安装在软 RAID 1 中](/smtxzbs/5.7.0/zbs_installation_guide_INTERNAL_EXTERNAL/zbs_installation_guide/zbs_installation_guide_26)的步骤 5 所选择的物理盘，不可修改用途，可移至其他物理盘池。
   - **其余物理盘**

     存储分层模式部署时：

     - 若节点为混闪配置，一般只有当系统判断的物理盘类型有误时，才需要手动修改用途，可将物理盘移至其他物理盘池或选择**不挂载**。
     - 若节点为多种类型 SSD 全闪配置，当计划预留常驻缓存比例高于 75% 时，需要手动将所有**数据盘**用途修改为**缓存盘**；其余情况下，只有当系统判断的物理盘类型有误时，才需要手动修改物理盘用途。可将物理盘移至其他物理盘池或选择**不挂载**。
     - 若节点为单一类型且多种属性的 SSD 全闪配置，物理盘默认为**数据盘**，需手动将高性能 SSD 的用途修改为**缓存盘**；当计划预留常驻缓存比例高于 75% 时，需要手动将所有**数据盘**的用途修改为**缓存盘**。可将物理盘移至其他物理盘池或选择**不挂载**。
     - 若节点为单一类型且单一属性的 SSD 全闪配置，物理盘默认为**数据盘**，无需修改用途，可将物理盘移至其他物理盘池或选择**不挂载**。

     存储不分层模式部署时：所有物理盘默认为**数据盘**，不可修改用途，可移至其他物理盘池或选择**不挂载**。
5. 单击**配置网络**，进入下一步。

## 部署块存储集群 > 第 5 步：配置网络 > 创建虚拟分布式交换机

# 创建虚拟分布式交换机

请根据管理网络、存储网络和接入网络三种系统网络是否需要独占虚拟分布式交换机进行创建。

## 系统网络独占虚拟分布式交换机

当某一系统网络需要独占一个虚拟分布式交换机时，请按以下步骤操作。

1. 填写虚拟分布式交换机的名称。
2. 在**关联网络**区域的**类型**选项中，选择该系统网络。
3. （可选）当接入该网络的物理交换机为该网络配置了 VLAN 时，请在 **VLAN ID** 字段下填写实际规划的 VLAN ID。
4. （可选）若该网络为存储网络，并且需要启用 RDMA 功能，请在**关联网络**区域的最右侧，勾选**启用 RDMA** 复选框，且 VLAN ID 必须填写为 0。
5. （可选）若该网络为接入网络，并且需要启用 NVMe-oF 功能，请在**关联网络**区域最右侧的 **NVMe-oF** 选项中，根据实际需要选择**启用 NVMe over TCP** 或者**启用 NVMe over TCP 和 RDMA**，且当启用 NVMe over TCP 和 RDMA 时，VLAN ID 必须填写为 0。
6. 在**物理网口**区域的**关联网口**字段下，给每个主机选择用于该网络的物理网口。
   - 可以根据显示的网口速率、连接状态和 MAC 地址等判断合适的网口。

     - 若该网络为管理网络，所选的网口速率需为 1 Gbps 及以上。
     - 若该网络为存储网络，所选的网口速率需为 10 Gbps 及以上。当启用 RDMA 功能时，若同一个主机选择了多个网口，还需确认所选的网口均来自支持 RDMA 功能的网卡。
     - 若该网络为接入网络，所选的网口速率需为 10 Gbps 及以上。当启用 NVMe over TCP 和 RDMA 功能时，若同一个主机选择了多个网口，还需确认所选的网口均来自同一张支持 NVMe over RDMA 功能的网卡。
   - 若各个主机的关联网口名称相同，可以在设置完一个主机的关联网口后，单击右侧的 **...** 图标，选择**选中其他主机上的同名网口**。
   - 当接入该网络的物理交换机启用了 LACP 动态链路聚合时，请参考下表为需要绑定网口的主机设置**网口绑定模式**。

     | **网络类型** | **网络配置** | **网口绑定模式** |
     | --- | --- | --- |
     | 管理网络 | - | balance-tcp |
     | 存储网络 | 已启用 RDMA | balance-tcp |
     | 未启用 RDMA | balance-tcp |
     | 接入网络 | 已启用 NVMe over TCP 和 RDMA | 802.3ad(LACP) |
     | 未启用 NVMe over TCP 和 RDMA | balance-tcp |

     > **注意**：
     >
     > 当存储网络使用 Mellanox CX5 网卡且启用 RDMA 时，请将网口绑定模式配置为 `active-backup`；若配置为 `balance-tcp`，可能导致网络延迟升高。若需实现存储网络 RDMA 多路径，建议使用 Mellanox CX6 网卡。

## 系统网络共用虚拟分布式交换机

当多种系统网络需要共用一个虚拟分布式交换机时，请按以下步骤操作。

1. 填写虚拟分布式交换机的名称。
2. 在**关联网络**区域的**类型**选项中，选择其中一种系统网络。
3. 单击 **+ 关联网络**，参考步骤 2 添加完所有要关联的系统网络。
4. （可选）当接入这些系统网络的物理交换机为其中的某一网络配置了 VLAN 时，请在对应网络的 **VLAN ID** 字段下填写实际规划的 VLAN ID。
5. （可选）若要保证某一系统网络的流量使用需求，可以为该网络启用 **QoS 策略**并进行以下设置。

   | **字段** | **说明** | **取值/示例** |
   | --- | --- | --- |
   | 优先级及权重 | 该网络在虚拟分布式交换机中的带宽分配优先级。 - 高（100） - 中（60） - 低（30） - 自定义权重：可设置 0 ~ 100 之间的权重 | “100” |
   | 预留带宽 | 该网络在虚拟分布式交换机中保证预留的带宽。  虚拟分布式交换机中所有虚拟网络的预留带宽之和不允许超过总带宽的 75%。 | “1 Gbps” |
   | 最大带宽 | 启用**限制带宽**后，将限制该网络可达到的最大带宽。 | “3 Gbps” |
   | 突发通信量上限 | 启用**限制带宽**后，设置该网络以超出最大带宽的速率进行通信时，最多可完成通信的数据量。 | “300 Mb” |
6. （可选）若这些系统网络包含接入网络，并且需要启用接入网络的 NVMe over TCP 功能，请在**关联网络**区域最右侧的 **NVMe-oF** 选项中，选择**启用 NVMe over TCP**。
7. 在**物理网口**区域的**关联网口**字段下，给每个主机选择这些系统网络共用的物理网口。

   - 可以根据显示的网口速率、连接状态和 MAC 地址等判断合适的网口，所选的网口速率需为 10 Gbps 及以上。
   - 当接入这些系统网络的物理交换机启用了 LACP 动态链路聚合时，请为需要绑定网口的主机设置**网口绑定模式**为 **balance-tcp**。

## 部署块存储集群 > 第 5 步：配置网络 > 配置管理、存储和接入网络信息

# 配置管理、存储和接入网络信息

1. 在**管理、存储、接入 IP** 面板中，填写每个主机实际规划的块存储集群管理 IP 地址、存储 IP 地址和接入 IP 地址。
2. 在**管理网络**面板中，填写为管理网络实际规划的网关和子网掩码。
3. 在**存储网络**面板中，填写为存储网络实际规划的子网掩码。
4. 在**接入网络**面板中，填写为接入网络实际规划的子网掩码。当使用存储的计算端和块存储集群的接入 IP 不在同一个子网时，需要启用**静态路由**，并填写该计算端的 IP 地址和子网掩码，以及路由器的网关；若需要添加多个计算端的信息，可以单击 **+ 添加路由配置**进行添加。

> **注意**：
>
> SMTX ZBS 支持为集群的系统网络设置 MTU，其中编辑存储网络和接入网络的 MTU 将造成业务 I/O 较长时间中断。若需修改存储网络或接入网络的 MTU，建议在集群部署完成后至业务上线前的窗口期进行，以免对业务造成影响，具体请参考《SMTX ZBS 管理指南》的[编辑系统网络](/smtxzbs/5.7.0/zbs_administration_guide/zbs_administration_guide_057)小节进行操作。

# 配置 DNS 服务器和 NTP 服务器

1. 在 **DNS 服务器**面板中，填写 DNS 服务器地址。

   - 若有域名配置需求，请填写实际规划的 DNS 服务器地址。
   - 若无域名配置需求，保留默认地址 127.0.0.1 即可。
2. 在 **NTP 服务器**面板中，选择 NTP 服务器。

   - 若无使用集群外部 NTP 服务器的需求，请选择**使用集群内主机作为 NTP 服务器**。
   - 若计划将集群关联至已有的 CloudTower 且 CloudTower 配置了 NTP 服务器，请确保集群的 NTP 服务器与 CloudTower 的 NTP 服务器的时钟保持同步。

## 设置参数 > 关联 CloudTower > 环境中已有 CloudTower

# 环境中已有 CloudTower

若当前环境中已部署 CloudTower，请在**关联 CloudTower** 对话框中选择**已有 CloudTower**，并参考下述步骤设置后，登录 CloudTower 和关联块存储集群。

**注意事项**

若现有的 CloudTower 管理平台版本低于当前 SMTX ZBS 所兼容的版本，则需升级 CloudTower 至兼容的版本。请提前获取 CloudTower 的软件升级包，相应升级方法请参见对应版本的《CloudTower 安装与升级指南》。

**操作步骤**

1. 在**关联 CloudTower** 对话框中设置块存储集群管理虚拟 IP 地址，完成后单击**下一步**。设置块存储集群的管理虚拟 IP 后，即可通过管理虚拟 IP 访问集群中可用的主机，从而保证管理界面的高可用。

   > **说明：**
   >
   > 必须确保 CloudTower 可以通过管理虚拟 IP 访问块存储集群，建议管理虚拟 IP 与节点的管理 IP 在同一个子网内。
2. 在浏览器中输入 CloudTower 的 IP 地址，使用超级管理员 `root` 账户登录 CloudTower。
3. 在 CloudTower 主界面右上角单击 **+ 创建** > **关联集群**，弹出**关联集群**对话框。
4. 在**关联集群**对话框中按照要求填写块存储集群的信息。完成后单击**下一步**。

   ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_49.png)

   - **集群管理虚拟 IP**：填写块存储集群的管理虚拟 IP。
   - **管理员用户名**：填写块存储集群的超级管理员账户的用户名。
   - **管理员密码**：填写管理员账户对应的密码。
5. 单击**下一步**，开始加载集群的数据。连接集群成功后，确认关联集群的信息，包括集群的名称、管理虚拟 IP、以及软件版本等信息。
6. 为集群选择所属的数据中心。也可以选择**不加入数据中心**。

   > **说明：**
   >
   > - 用户可以在关联集群的**确认关联集群**阶段，直接创建新的数据中心，然后再将此集群加入该数据中心。
   > - 若用户此时选择**不加入数据中心**，在关联集群后，也可以在 CloudTower 主界面创建一个新的数据中心，并重新选择加入。
7. 单击**关联集群**，成功将 CloudTower 与块存储集群完成关联。

   > **注意**：
   >
   > - 完成集群关联后，如果发现该集群有错误或遗漏的配置而需要重新安装部署，必须先取消其与 CloudTower 的关联。
   > - 若 CloudTower 时区与集群时区不一致，建议手动调整时区使两者一致。

**后续操作**

# 为 RDMA 或 NVMe over RDMA 功能配置流量控制

- 若在部署块存储集群时未开启 RDMA 或 NVMe over RDMA 功能，以下操作可忽略。
- 若用户已开启存储网络的 RDMA 功能或接入网络的 NVMe over RDMA 功能（即为接入网络创建虚拟分布式交换机时，选择了**启⽤ NVMe over TCP 和 RDMA** 选项），为了保证性能，需要分别在交换机和主机中针对存储网络或接入网络配置流量控制。

---

## 为 RDMA 或 NVMe over RDMA 功能配置流量控制 > 在交换机端配置流量控制（以 Mellanox Switch 为例）

# 在交换机端配置流量控制（以 Mellanox Switch 为例）

SMTX ZBS 支持在交换机中配置基于 L3 DSCP 的 PFC 流量控制和基于 Global Pause 流量控制，但推荐使用基于 L3 DSCP 的 PFC 流量控制。

**注意事项**

- 在块存储集群中使用上述任何一种流量控制的方式进行配置时，在存储网络交换机和接入网络交换机端配置的流程与步骤都完全相同。
- 若用户同时开启块存储集群存储网络的 RDMA 功能和接入网络的 NVMe over RDMA 功能，则需在两个网络交换机中分别进行配置。

## 基于 L3 DSCP 的 PFC 流量控制配置（推荐配置）

1. 将 RoCE 的流量类型设置为 **`3`**，同时开启 ECN, 以启用 DCQCN。将 ECN 的最低和最高触发阈值都设置为 **`3000`** KB。

   ```
   switch > enable
   switch # configure terminal
   switch (config) # interface ethernet 1/1-1/2 traffic-class 3 congestion-control ecn minimum-absolute 3000 maximum-absolute 3000
   ```
2. 设置 **QoS trust mode** 为 **`L3(DSCP)`**。

   ```
   switch (config) # interface ethernet 1/1-1/2 qos trust L3
   ```
3. 设置交换机的 PFC 在 priority 3 上开启。

   ```
   switch (config) # dcb priority-flow-control enable force
   switch (config) # dcb priority-flow-control priority 3 enable
   switch (config) # interface ethernet 1/1-1/2 dcb priority-flow-control mode on force
   switch (config) # interface ethernet 1/1-1/2 pfc-wd
   ```
4. 验证 PFC 是否已开启, 确认后保存配置。

   ```
   switch (config) # show dcb priority-flow-control   
   PFC: enabled
   Priority Enabled List: 3
   Priority Disabled List: 0 1 2 4 5 6 7   
   -------------------------------------------------
   Interface        PFC admin        PFC oper
   -------------------------------------------------
   Eth1/1           Auto             Enabled
   Eth1/2           Auto             Enabled
   ......   
   switch (config) # configuration write
   ```

## 基于 Global Pause 流量控制配置

设置 RDMA 网卡所连交换机端口的 `flowcontrol` 为 **`on`**, 验证并保存配置。

```
Mellanox Switch
switch > enable
switch # configure terminal
switch (config) # interface ethernet 1/1-1/2 traffic-class 0 congestion-control ecn minimum-absolute 3000 maximum-absolute 3000
switch (config) # interface ethernet 1/1-1/2 flowcontrol send on force
switch (config) # interface ethernet 1/1-1/2 flowcontrol receive on force
switch (config) # show interfaces ethernet 1/1 | include Flow-control
Flow-control                     : receive on send on
switch (config) # show interfaces ethernet 1/2 | include Flow-control
Flow-control                     : receive on send on
switch (config) # configuration write
```

## 为 RDMA 或 NVMe over RDMA 功能配置流量控制 > 在主机端配置流量控制

# 在主机端配置流量控制

SMTX ZBS 支持在主机中配置基于 L3 DSCP 的 PFC 流量控制和基于 Global Pause 流量控制，但在主机端和交换机端须使用同一种流量控制的方式进行配置。

## 基于 L3 DSCP 的 PFC 流量控制配置

由于在块存储集群部署阶段，系统已自动在主机端对存储网络或接入网络配置基于 L3 DSCP 的 PFC 流量控制，因此若在交换机端也是使用该方式配置，则此处主机端无需再做任何配置，请直接进入[验证流量控制的配置结果](/smtxzbs/5.7.0/zbs_installation_guide_INTERNAL_EXTERNAL/zbs_installation_guide/zbs_installation_guide_55)。

## 基于 Global Pause 流量控制配置

在块存储集群的主机端为存储网络的 RDMA 功能或接入网络的 NVMe over RDMA 功能配置流量控制时，必须使用超级管理员 `root` 账号登录每台主机执行命令。

1. 使用 `root` 账号登录每台主机，并在每台主机上执行以下命令。

   - 仅开启了存储网络的 RDMA 功能时

     ```
     zbs-deploy-manage config-rdma-fc --storage_qos_mode global
     ```
   - 仅开启了接入网络的 NVMe over RDMA 功能时

     ```
     zbs-deploy-manage config-rdma-fc --access_qos_mode global
     ```
   - 同时开启了存储网络的 RDMA 功能和接入网络的 NVMe over RDMA 功能时

     ```
     zbs-deploy-manage config-rdma-fc --storage_qos_mode global --access_qos_mode global
     ```
2. 检查并确认流量控制配置的结果。

   - 若已为存储网络的 RDMA 功能配置流控，请使用如下方式验证流控配置文件。

     1. 使用 `smartx` 账号登录每台主机，并在每台主机上执行以下命令：

        ```
        cat /etc/sysconfig/netguard | grep storageQosType
        ```
     2. 执行完命令后，系统将输出以下信息。若 `storageQosType` 字段后显示的值为 **`global`**，则表示流控配置已生效，否则表示配置错误。

        ```
        NETGUARD_OPTS=' -storageQosType global -accessQosType global'
        ```
   - 若已为接入网络的 NVMe over RDMA 功能配置流控，请使用如下方式验证流控配置文件。

     1. 使用 `smartx` 账号登录每台主机，并在每台主机上执行以下命令：

        ```
        cat /etc/sysconfig/netguard | grep accessQosType
        ```
     2. 执行完命令后，系统将输出以下信息。若 `accessQosType` 字段后显示的值为 **`global`**，则表示流控配置已生效，否则表示配置错误。

        ```
        NETGUARD_OPTS=' -storageQosType global -accessQosType global'
        ```
   - 若为存储网络的 RDMA 功能和接入网络的 NVMe over RDMA 功能均已配置流控，请参照上述两种方式分别进行验证。

## 为 RDMA 或 NVMe over RDMA 功能配置流量控制 > 验证流量控制的配置结果

# 验证流量控制的配置结果

在交换机和主机端部署完流控后，可以通过在任一节点中，执行 SMTX ZBS 块存储系统文件中自带的自动化测试脚本 `data_channel_bench.py`，以产生较大的 IO 流量，从而验证节点流控配置的正确性。

**测试验证原理**

以三节点的块存储集群进行举例说明。

假设块存储集群中有 A B C 三个节点，若在三个节点上均已正确配置流量控制，则每个节点运行下面描述的 3 种典型网络 IO 模型时，其表现应与 B 节点完全相同。B 节点的网络 IO 模型表现如下（-> 表示数据传输方向）：

- A -> B -> C: B 节点从 A 节点接收数据，同时向 C 节点发送数据，收发两条链路传输速度应满带宽运行；
- A <- B -> C: B 节点同时向 A C 两节点发送数据，两条发送链路的速度都应为满带宽的一半；
- A -> B <- C: B 节点同时从 A C 两节点接收数据，两条接收链路的速度都应为满带宽的一半。

在块存储集群的任意节点上执行自动化测试脚本 `data_channel_bench.py` 时，需指定集群所有节点的管理 IP 和测试的网络类型（存储网络/接入网络），脚本文件执行时将根据测试网络获取每个节点的 RDMA 网卡名称，然后依次对集群中的每个节点执行上述三种网络 IO 模型的测试, 从而验证每个节点的流控配置是否正确。

该脚本支持的参数如下：

```
$ python /usr/share/zbs/bin/data_channel_bench.py -h
usage: data_channel_bench.py [-h] [--mode MODE] [--no_rdma]
                             [--use_mpath {true,false}] [--io_op IO_OP]
                             [--start_cpu START_CPU]
                             nodes [nodes ...]

positional arguments:
  nodes                 node ips, at least 3 ips should be given. E.g.
                        192.168.1.1 192.168.1.2 192.168.1.3 or 192.168.1.[1-3]

optional arguments:
  -h, --help            show this help message and exit
  --mode MODE           test mode, can be data or access. Default is data.
  --no_rdma             do not use rdma for data network
  --use_mpath {true,false}
                        use multipath for data network, can only be used in
                        data mode
  --io_op IO_OP         io operation, can only be [pextent_read,
                        pextent_write, vextent_read, vextent_write]
  --start_cpu START_CPU
                        start cpu, default is 0
```

**注意事项**

- 执行该自动化测试脚本产生的 I/O 流量较大，很可能会影响正常业务的运行，请勿在客户的生产环境中执行此脚本文件和验证流量控制的配置结果。
- 必须使用 `smartx` 账号登录节点执行该自动化测试脚本。
- 该自动化测试脚本也适用于三个节点以上的块存储集群测试。脚本文件在执行时自动将三个节点分为一组，对每个组执行上述测试，总的测试时间约为 round\_up(N / 3) × (3 × 3) × 5 秒，约等于 N × 15 秒，其中 N 为块存储集群节点数量。

## 部署文件存储集群

# 部署文件存储集群

文件存储集群由文件控制器组成，部署在 SMTX ZBS 块存储集群中。一个 SMTX ZBS 块存储集群中仅允许部署一个文件存储集群。

**前提条件**

- 文件控制器的配置与所属块存储集群中节点可提供的资源量有关，且根据高可用原则，文件控制器将分布在块存储集群的不同节点上，所以规划文件控制器前请您提前了解块存储集群的节点数量、每个节点上可独占的 vCPU 数量和内存容量。
- 每个文件控制器将占用所在的块存储集群中 240 GiB 的存储空间。

  - 如块存储集群为混闪配置，请为其开启常驻缓存模式，并确保该块存储集群中空闲常驻缓存容量不少于 240 GiB \* 预期文件控制器数量；
  - 如块存储集群无法开启常驻缓存模式， 请确保该块存储集群中可用存储容量不少于 240 GiB \* 预期文件控制器数量。

**操作步骤**

1. 在 CloudTower 的**文件存储**页面，单机左侧导航栏中的**文件存储集群**。
2. 在右侧**文件存储集群**页面单击 **+ 部署文件存储集群**。
3. 请在弹出的**部署文件存储集群**对话框中进行配置。

**注意事项**

部署文件存储集群过程中，请不要为块存储集群中其他存储卷启用常驻缓存。

## SMTX ZBS 节点的本地账户与默认安全设置说明 > SMTX ZBS 节点的本地账户说明

# SMTX ZBS 节点的本地账户说明

SMTX ZBS 节点支持以下三种本地账户：

| **本地账户** | **说明** | **注意事项** |
| --- | --- | --- |
| root | 默认账户，默认禁止 SSH 远程登录。您可以使用该账户管理节点。 | - 请勿删除该账户。 - 强烈建议不要开启 root 账户通过 SSH 远程登录的权限。 - 以命令行方式升级集群时必须拥有 root 权限。 |
| smartx | 默认账户，默认支持 SSH 远程登录。您可以使用该账户通过 SSH 协议远程连接到节点。 | - 请勿删除该账户。 - 请勿更改 `/etc/sudoers` 里针对该账户的配置。 - 以该账户登录节点后，必须切换为 root 账户再执行运维操作。执行 `sudo su` 命令可免密码切换。 |
| 其他本地账户 | 由 root 账户创建的其他账户，账户权限由 root 账户决定。您可自行创建新的本地账户用于 SSH 远程登录。 | - |

## SMTX ZBS 节点的本地账户与默认安全设置说明 > SMTX ZBS 节点的默认安全设置说明

# SMTX ZBS 节点的默认安全设置说明

SMTX ZBS 块存储系统安装完成后，会默认对节点进行如下安全设置。

| **设置项** | **默认设置说明** |
| --- | --- |
| 连接超时设置 | 登录节点后，如果在 30 分钟内未进行操作，与节点的连接将自动断开。 |
| 密码强度设置 | 除 root 和初始 smartx 外的账户设置密码时：  - 密码长度必须至少为 8 个字符。 - 密码必须至少包含 1 个数字、1 个大写字母、1 个小写字母和 1 个特殊字符。 - 重设的密码不能与前 5 次的密码重复。 |
| 登录失败后锁定设置 | 连续 5 次登录失败后，账户将被锁定 10 分钟。 |
| SSH 设置 | - 禁止 root 账户直接 SSH 远程登录节点。 - SSH 服务的 MACs 和 Ciphers 配置支持 CTR 和 SHA2 算法，不支持 CBC、MD5 和 SHA1 算法。 |
