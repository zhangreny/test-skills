## 概述

# 概述

SMTX 备份与容灾是由北京志凌海纳科技股份有限公司（以下简称 “SmartX”）自主研发的数据保护产品，可为 SMTX OS（ELF）集群的虚拟机和 SMTX ZBS 块存储集群的存储卷提供全面的数据保护和灾难恢复解决方案。

## 概述 > 功能简介

# 功能简介

SMTX 备份与容灾包含两个功能模块：

- **备份与恢复**

  支持将 SMTX OS（ELF）集群中的虚拟机整机备份至集群外部的存储库（如 NAS 服务器），并可通过备份在存储库中的数据文件和配置文件，将虚拟机恢复至备份时的状态或者重建新的虚拟机。

  备份与恢复的基本架构如下：

  ![](https://cdn.smartx.com/internal-docs/assets/d668feb7/installation_guide_003.png)
- **复制与恢复**

  支持虚拟机同步复制、虚拟机异步复制和存储卷异步复制，并同时支持站点内复制和跨站点复制。

  站内复制的基本架构如下：

  ![](https://cdn.smartx.com/internal-docs/assets/d668feb7/installation_guide_004.png)

  跨站点复制的基本架构如下：

  ![](https://cdn.smartx.com/internal-docs/assets/d668feb7/installation_guide_005.png)

  - **虚拟机同步复制和异步复制**

    支持将 SMTX OS（ELF）集群中的虚拟机同步或异步复制到另一个 SMTX OS（ELF）集群中。当原虚拟机发生故障时，可实现故障转移、故障恢复和永久故障转移。源集群和目标集群可以位于同一个站点，也可以位于不同站点。
  - **存储卷异步复制**

    支持将 SMTX ZBS 块存储集群中的存储卷异步复制到另一个 SMTX ZBS 块存储集群中，并保持副本卷与原卷同步。当原卷发生故障时，可实现故障转移、故障恢复和永久故障转移。源集群和目标集群可以位于同一个站点，也可以位于不同站点。

## 概述 > 产品特性

# 产品特性

SMTX 备份与容灾具有如下特性：

- 灵活数据保护选项：

  - 支持虚拟机备份、虚拟机同步复制、虚拟机异步复制和存储卷异步复制。
  - 支持保护不同类型的工作负载，包括虚拟机和存储卷。
  - 支持自定义执行周期、时间窗口和保留策略，满足不同类型业务的 SLA 要求。
  - 支持系统服务负载控制，可自定义同时执行的任务数量、任务重试次数、任务传输速率限制。
- 便捷的运维和管理方式：

  - 支持单一管理平台统一管理集群资源和备份与容灾软件。
  - 支持一键部署和升级系统服务。
  - 支持任务策略自动化和执行记录可视化。
  - 支持自定义⻆色和权限，可对用户的操作权限进行细粒度控制。
  - 通过记录详细的事件信息，满足安全审计要求，可用于追溯故障原因或其它运维事件。

## 相关概念

# 相关概念

本章解释了手册中使用的专业术语。在阅读过程中，如果遇到不熟悉的概念，可在此快速查找其定义和详细解释。

## 备份与恢复

**备份服务**

备份服务负责管理备份计划，并执行虚拟机的备份和恢复任务。

**备份资源**

备份计划、加入备份计划的虚拟机、备份文件、恢复点和备份存储库。

**备份计划**

备份计划用于详细规划和安排数据的备份操作。通过备份计划，用户可以定义备份对象、备份服务、备份存储库、备份周期、备份窗口、恢复点保留策略等关键信息，以确保数据的可靠备份，实现故障场景的数据恢复，从而保障业务的连续性和可靠性。

**备份对象**

备份的虚拟机。

**备份周期**

备份计划自动执行的频率和时间。用于指定备份服务对虚拟机进行备份的时间间隔。较短的备份周期可以实现更小的 RPO（Recovery Point Objective，故障恢复点目标）。例如，如果备份周期设置为 15 分钟，那么在发生灾难性事件时，最多可能丢失 15 分钟的数据。

**备份窗口**

允许备份计划自动执行的时间范围。

**恢复点**

恢复点是在每次执行备份操作时生成的虚拟机数据状态，以备份文件的形式存储在备份存储库中。恢复点完整记录了虚拟机的数据和配置信息，反映了虚拟机在备份操作执行时的特定状态。

恢复点可以使虚拟机在需要时能够迅速恢复到特定的数据状态。例如在灾难恢复场景中将虚拟机恢复到某个先前的时间点，以恢复虚拟机的正常运行。通过定期生成恢复点，可以确保虚拟机的数据是最新的，保障业务的连续性和可用性；也可以使用恢复点对应的备份文件重建一个对应时间点状态的虚拟机，用于开发测试等。

**SMTX OS 集群**

SMTX OS 集群属于逻辑概念。在实际生产环境中，一个 SMTX OS 集群至少由 3 个节点通过网络互连组成。

**ELF**

SMTX OS 原生虚拟化计算平台，是当 SMTX OS 以超融合形态直接部署在 x86\_64 或 AArch64 架构的服务器上时，该超融合软件中基于 KVM 的虚拟化引擎组件。ELF 除提供虚拟机生命周期管理、电源操作、高可用、冷热迁移等基本功能外，通过与存储组件 ZBS 相结合，还能提供秒级快照、模板和克隆等高级的虚拟机服务。

ELF 简单易用，对 SMTX OS 超融合集群的所有操作均可通过 CloudTower 完成。

**CloudTower**

SmartX 多集群资源集中管理平台，旨在管理站点内数据中心及跨数据中心的多个 SmartX 集群和系统服务，并提供完整和标准的 RESTful API 和多种语言的 SDK。

## 复制与恢复

**复制服务**

复制服务负责管理复制计划，并执行虚拟机和存储卷的复制、故障转移、故障恢复和永久故障转移等任务调度。

根据关联的集群不同，复制服务分为目标端复制服务和源端复制服务。

- **目标端复制服务**

  与目标集群关联的复制服务，主要负责执行复制计划、故障转移、管理复制资源，并在故障恢复时从目标集群读取数据传输给源端复制服务。目标端复制服务负责确保复制数据的同步性，保障副本虚拟机和原虚拟机，以及副本卷和原卷的数据一致性。
- **源端复制服务**

  与源集群关联的复制服务，主要负责从源集群读取数据并传输到目标端复制服务，并在故障恢复时接收目标端复制服务传输的数据并将数据写入源集群。

**站点**

站点是指由单一 CloudTower 管理的一个故障域，可以包含一个或多个集群和数据中心。每个站点的资源都由其专属的 CloudTower 进行管理，且在地理位置、硬件设备和软件服务等方面完全隔离，不与其他站点共享任何资源。

站点可以分为以下两种类型：

- **源站点**：源站点通常是实际运行业务的站点。复制服务将源站点的虚拟机和存储卷数据复制到目标站点的副本对象，实现数据的同步。
- **目标站点**：目标站点通常是用于容灾的站点，用于存储从源站点复制过来的副本对象。

**复制计划类型**

- **站内复制**

  源集群和目标集群位于同一站点。当原虚拟机或原卷不可用时，可切换使用站点内其他集群中的副本对象，实现集群级别的容灾。若管理站点的 CloudTower 发生故障，无法发起故障转移。
- **跨站点复制**

  源集群和目标集群位于不同站点。当源站点内的原虚拟机或原卷不可用时，无论管理源站点的 CloudTower 是否发生故障，均可切换至目标站点的副本对象，实现站点级别的容灾。

  跨站点复制包括以下两种类型：

  - **入站复制**：目标站点为当前站点，源站点为其他关联站点。
  - **出站复制**：源站点为当前站点，目标站点为其他关联站点。

**源集群**

通常为生产集群，承载实际运行的业务应用和数据，是原虚拟机或原卷所属的集群，位于源站点。

**目标集群**

通常为用于容灾的集群，用于存储从源集群复制过来的副本对象，位于目标站点。

**虚拟机**

通过虚拟化技术实现，运行在物理服务器虚拟化平台上的计算机系统，具有与物理计算机相同的特性。物理服务器上的所有虚拟机共享底层的物理硬件资源，但每个虚拟机拥有独立的操作系统、应用程序和资源配置。虚拟机之间相互隔离，运行互不影响，可用于测试、开发和生产环境。

对虚拟机进行复制时，涉及三种虚拟机角色：

- **原虚拟机**

  需要进行容灾保护的虚拟机，负责承载实际的业务应用和数据。
- **副本虚拟机**

  通过复制计划创建的原虚拟机的完整副本。副本虚拟机通常用于容灾，当原虚拟机不可用时，可以接替原虚拟机继续提供服务。
- **测试虚拟机**

  通过故障转移测试任务创建的原虚拟机的完整副本，具有和原虚拟机相同的数据和配置，通常用于灾备演练，验证故障转移流程的正确性以及数据一致性。

**存储卷**

SMTX ZBS 块存储集群对外提供的最基本的存储单元。复制服务支持对 iSCSI LUN 进行复制和故障处理。

对存储卷进行复制时，涉及两种存储卷角色:

- **原卷**

  需要进行容灾保护的存储卷，负责实际承载业务数据的读写操作。
- **副本卷**

  通过复制计划创建的原卷的完整副本。副本卷通常用于容灾，当原卷不可用时，可以接替原卷提供业务数据。

**复制资源**

复制计划、复制对象、测试虚拟机和恢复点。

**复制对象**

进行虚拟机复制时，一个原虚拟机及其副本虚拟机称为一组复制对象；进行卷复制时，一个原卷及其副本卷称为一组复制对象。

**副本对象**

进行虚拟机复制时，指副本虚拟机；进行卷复制时，指副本卷。

**同步复制计划**

用于规划和安排数据的复制操作。通过同步复制计划，用户可以定义复制对象、复制方向，复制站点、目标集群、复制服务、同步策略、副本虚拟机和测试虚拟机设置等信息。

**异步复制计划**

用于规划和安排数据的复制操作。通过异步复制计划，用户可以定义复制对象、复制方向，复制站点、目标集群、复制服务、复制周期、复制窗口、保留策略、副本虚拟机和测试虚拟机设置等信息。

**恢复点**

每次为虚拟机或存储卷执行复制、故障转移（立即生成恢复点模式）或故障恢复时生成的数据状态，以快照形式保存在副本对象上。

恢复点可以使虚拟机或存储卷在需要时快速回滚到特定的数据状态，例如在灾难恢复场景中将虚拟机或存储卷恢复到某个先前的时间点，以恢复业务运行。通过定期生成恢复点，可以确保副本对象的数据是最新的，也可以使用恢复点对应的快照重建一个对应时间点状态的副本对象，在不影响生产业务的同时，进行开发测试等。

**故障处理**

故障处理操作包括故障转移、故障转移测试、故障恢复和永久故障转移。

- **故障转移**

  故障转移是一种应对原虚拟机或原卷不可用的操作。当原虚拟机或原卷不可用时，可以将副本对象回滚到与指定恢复点一致的状态，并切换至使用副本对象。通过故障转移，系统可以在短时间内将服务快速恢复到故障发生前的状态，保障业务的连续性和稳定性。
- **故障转移测试**

  故障转移测试是一种模拟容灾演练的操作，在不中断源端生产业务的前提下，通过生成测试虚拟机，在测试虚拟机验证恢复机制的有效性。演练过程中的增量数据不会同步回被保护的虚拟机。
- **故障恢复**

  故障恢复是指当原虚拟机或原卷恢复正常使用后，副本对象将故障转移期间变化的数据回传，然后切换回使用原虚拟机或原卷。故障恢复旨在将复制对象之间的数据进行同步，确保原虚拟机或原卷可以在故障解决后继续提供服务。
- **永久故障转移**

  永久故障转移是当原虚拟机或原卷无法从故障中恢复时采取的措施。在原虚拟机或原卷发生永久性故障时，副本对象将正式接替原虚拟机或原卷的服务角色，以确保业务的持续运行和可用性。

**SMTX OS 集群**

SMTX OS 集群属于逻辑概念。在实际生产环境中，一个 SMTX OS 集群至少由 3 个节点通过网络互连组成。

**ELF**

SMTX OS 原生虚拟化计算平台，是当 SMTX OS 以超融合形态直接部署在 x86\_64 或 AArch64 架构的服务器上时，该超融合软件中基于 KVM 的虚拟化引擎组件。ELF 除提供虚拟机生命周期管理、电源操作、高可用、冷热迁移等基本功能外，通过与存储组件 ZBS 相结合，还能提供秒级快照、模板和克隆等高级的虚拟机服务。

ELF 简单易用，对 SMTX OS 超融合集群的所有操作均可通过 CloudTower 完成。

**SMTX ZBS 集群**

SMTX ZBS 集群包含块存储集群和文件存储集群，分别提供块存储和文件存储服务。复制服务可以部署在 SMTX ZBS 块存储集群上，并为块存储集群提供存储卷复制和故障处理服务。

**CloudTower**

SmartX 多集群资源集中管理平台，旨在管理站点内数据中心及跨数据中心的多个 SmartX 集群和系统服务，并提供完整和标准的 RESTful API 和多种语言的 SDK。

## 部署备份服务 > 部署前的准备 > 规划备份服务的数量

# 规划备份服务的数量

在部署备份服务之前，请根据数据中心、集群和 CloudTower 的实际情况，确认需部署的备份服务数量。

- **是否需跨数据中心**

  由于跨数据中心的网络易发生传输不稳定的问题，建议为每个数据中心分别部署备份服务，并将虚拟机备份至位于同一个数据中心的一个或多个备份存储库中，保障备份和恢复的稳定性和传输效率。
- **待备份的多个集群是否互相连通**

  若需备份多个集群，请确认集群之间的存储网络是否连通，并据此规划需部署的备份服务数量。

  - 若集群之间的存储网络连通，部署一个备份服务即可对多个集群进行备份和恢复。
  - 若集群之间的存储网络不连通，则需为每个集群单独部署一个备份服务。

  ![](https://cdn.smartx.com/internal-docs/assets/18db1932/user_guide_001.png)

  > **说明**：
  >
  > 一个备份服务仅可以部署在一个 SMTX OS（ELF）集群中，但一个 SMTX OS（ELF）集群可以部署多个备份服务。
- **是否使用多个 CloudTower 管理**

  由于备份服务需通过 CloudTower 进行部署和关联操作，对于未被同一个 CloudTower 管理的集群，无法使用一个备份服务同时对这些集群进行备份。因此，若使用了多个 CloudTower 管理集群，那么每个 CloudTower 中都需部署备份服务，且使用单独的许可。
- **确认待备份的数据量**

  单个备份服务可对实际数据量不超过 100 TiB 的单个虚拟机进行备份，也可对总实际数据量之和不超过 100 TiB 的多个虚拟机进行备份。若多个虚拟机的实际数据量之和超过 100 TiB，建议部署多个备份服务以完成操作。

## 部署备份服务 > 部署前的准备 > 确认软件配套要求

# 确认软件配套要求

备份服务需部署在 SMTX OS（ELF）集群内，并通过关联 SMTX OS（ELF）集群为其提供备份和恢复功能。此外，SMTX 备份与容灾使用 CloudTower 作为管理平台，提供操作界面和资源管理。

因此，在部署前请参考《SMTX 备份与容灾发布说明》的[版本配套说明](/backup/2.3.0/backup_release_notes/release_notes_02)，确认以下配套软件是否满足要求：

## 部署备份服务 > 设置备份和恢复任务

# 设置备份和恢复任务

备份服务部署完成后，备份和恢复任务具有初始的默认设置，用户可按照实际需要进行修改。

**操作步骤**

1. 登录 CloudTower，在**备份与容灾**界面中，单击**设置**进入功能管理界面。
2. 在弹出的对话框中，选择已部署的备份服务，单击**备份和恢复任务设置**，对备份服务的备份和恢复任务进行设置。

   - **同时执行的任务数量限制**

     - 最大备份任务数量：设置当前备份服务同时可执行的最大备份任务数量。修改最大备份任务数量不会影响当前正在执行的备份任务。
     - 最大恢复任务数量：设置当前备份服务同时可执行的最大恢复任务数量。修改最大恢复任务数量不会影响当前正在执行的恢复任务。
   - **失败任务自动重试**

     选择任务失败后是否自动重试，该功能默认启用。启用后，若备份任务和恢复任务执行失败，系统将自动重试，直到任务成功或达到最大重试次数。
   - **负载控制**

     选择是否启用备份和恢复任务的传输速率限制，默认不启用，表示不限制。启用后，请设置备份任务和恢复任务的最大传输速率，最大传输速率不能低于 2 MiB/s。

     - 备份任务最⼤传输速率：设置备份任务最大传输速率，留空或置零表示不限制。设置后，将限制当前备份服务同时执行的所有备份任务的总传输速率。
     - 恢复任务最⼤传输速率：恢复任务最大传输速率，留空或置零表示不限制。设置后，将限制当前备份服务同时执行的所有恢复任务的总传输速率。
3. 单击**保存**，保存备份和恢复任务设置。

## 部署复制服务 > 部署前的准备 > 确认复制服务的部署位置和数量

# 确认复制服务的部署位置和数量

在部署复制服务之前，请从如下几方面考虑，确认复制服务部署的位置和数量。

- **确认复制计划类型**

  - **站内复制**

    选择站内复制策略时：

    - 若站点内源集群和目标集群的存储网络均互相连通，站点内只需部署一个复制服务供多个集群共同使用。**建议部署在目标集群以应对集群级别的故障**。
    - 若集群之间的存储网络不连通，那么需为不连通的集群单独部署复制服务。

    此外，若站点内存在多个跨地域的数据中心，且源集群和目标集群分布于不同的数据中心。由于跨数据中心的网络易发生传输不稳定的问题，建议为不同的数据中心分别部署复制服务，以保证复制和恢复的稳定性和传输效率。
  - **跨站点复制**

    选择跨站点复制策略时，需确保每个站点都部署复制服务。如果站点内有多个集群需要与另一个站点的集群进行数据复制，请确认这些集群之间的存储网络连通性。如果集群之间的存储网络不连通，则需为每个不连通的集群分别部署独立的复制服务，以确保数据能够顺利复制到对方站点的集群。否则，在站点内部署一个复制服务即可。
- **确认待复制的数据量**

  单个复制服务可对实际数据量不超过 320 TiB 的单个虚拟机或存储卷进行复制，也可对总实际数据量不超过 320 TiB 的多个虚拟机或存储卷进行复制。若需复制的实际数据量超出 320 TiB，建议部署多个复制服务以完成操作。

## 部署复制服务 > 部署步骤 > 部署复制服务虚拟机

# 部署复制服务虚拟机

复制服务以虚拟机的形式部署在 SMTX OS（ELF）集群或 SMTX ZBS 块存储集群中，部署复制服务需要设置复制服务虚拟机的基本信息、运行位置和网络信息。

**前提条件**

- CloudTower 已关联了将要部署复制服务的集群。
- 集群中有足够的计算和存储资源。

**注意事项**

复制服务部署完成后不可删除安装包，仅在所有使用该安装包部署的备份或复制服务升级到新版本或被卸载后才可删除。

**操作步骤**

1. 登录 CloudTower，在**备份与容灾**界面中，单击**设置**进入功能管理界面。
2. 在弹出窗口的左侧导航栏底部单击**部署服务**，选择**部署复制服务**。
3. 设置复制服务的名称和描述，并选择复制服务的版本。
4. 设置复制服务虚拟机所属的**集群**和**主机**。
5. 为复制服务的**管理网络**、**存储网络**和**复制网络**选择虚拟机网络。具体要求请参考[网络要求](/backup/2.3.0/backup_dr_installation_guide/backup_dr_installation_guide_13#%E7%BD%91%E7%BB%9C%E8%A6%81%E6%B1%82)章节。

   当复制服务部署在 SMTX OS（ELF）集群中时，仅支持使用已有虚拟机网络；当复制服务部署在 SMTX ZBS 块存储集群中时，可选择使用已有虚拟机网络或新建虚拟机网络。

   - 使用已有虚拟机网络：

     选择对应网卡连接的虚拟机网络，并输入对应网卡的 IP 地址、子网掩码，管理网卡还需输入网关。
   - 使用新建虚拟机网络：

     输入虚拟机网络名称、VLAN ID、虚拟分布式交换机以创建新的虚拟机网络，并输入对应网卡的 IP 地址、子网掩码，管理网卡还需输入网关。
6. 单击**部署**，系统将自动在选择的集群和主机上完成复制服务的部署。

## 部署复制服务 > 设置复制和故障恢复任务

# 设置复制和故障恢复任务

复制服务部署完成后，复制和故障恢复任务具有初始的默认设置，用户可按照实际需要，对异步复制任务（包括异步复制计划触发和同步复制过程中产生的任务），以及故障处理任务进行配置。进行修改。

**操作步骤**

1. 登录 CloudTower，在**备份与容灾**界面中，选择**设置**进入功能管理界面。
2. 在弹出的对话框中，选择已部署的复制服务，单击**复制任务设置**，对复制服务的复制和故障恢复任务进行设置。

   - **同时执行的任务数量限制**

     - 最大异步复制任务数量：设置当前复制服务同时可执行的最大异步复制任务数量。修改最大复制任务数量不会影响当前正在执行的异步复制任务。
     - 最大故障恢复任务数量：设置当前复制服务同时可执行的最大故障恢复任务数量。修改最大故障恢复任务数量不会影响当前正在执行的故障恢复任务。
   - **失败任务自动重试**

     选择任务失败后是否自动重试，失败任务自动重试功能适用于多种任务类型，包括复制、故障转移、故障恢复和永久故障转移。

     该功能默认启用。启用后，若任务执行失败，系统将自动重试，直到任务成功或达到最大重试次数。
   - **设置负载控制**

     选择是否启用复制和故障恢复任务的传输速率限制，默认不启用，表示不限制。启用后，请设置复制任务和故障恢复任务的最大传输速率，最大传输速率不能低于 2 MiB/s。

     - 异步复制任务最⼤传输速率：设置当前复制服务同时执行的所有异步复制任务的总传输速率，留空或置零表示不限制。
     - 故障恢复任务最⼤传输速率：设置当前复制服务同时执行的所有故障恢复任务的总传输速率，留空或置零表示不限制。
3. 单击**保存**，保存复制任务设置。

3. 进行邮件设置。

   | 参数 | 描述 |
   | --- | --- |
   | SMTP 服务器 | 选择一个已有的 SMTP 服务器或新创建一个 SMTP 服务器，通过该 SMTP 服务器向指定邮箱发送任务报表通知邮件。若要新创建 SMTP 服务器，您可在下拉列表中选择 **+ 创建 SMTP 服务器**，并根据《CloudTower 用户指南》的**配置 SMTP 服务器章节**完成配置。 |

   | 通知语言 | 邮件通知的语言，包括简体中文和英语。 |
   | 发送测试邮件 | 单击该按钮，系统将发送测试邮件至填写的邮箱地址。 |
4. 设置报表的发送周期。可选择在每天的特定时刻。
5. 单击**保存**。

## 管理操作权限 > 用户角色

# 用户角色

在 CloudTower 中，用户的权限由其角色决定，每个用户只能对应一个角色。不同角色拥有不同的操作权限，用户仅在具备相应的权限时才可对资源进行操作，在不具备相应的权限时，仅拥有基本的可读权限。

CloudTower 提供两种类型的角色：默认角色和自定义角色。其中，默认角色的权限是固定的，不可更改。自定义角色可根据需要自行创建，并为其自定义操作权限。

## 查看事件

# 查看事件

CloudTower 会自动记录备份和复制相关的操作并在界面以事件形式展示，以便追溯故障原因或其它运维事件。

事件类型包括用户事件和系统事件。

- **用户事件**

  记录用户对软件许可、安装包、备份服务、备份存储库、备份计划、恢复点、备份任务、恢复任务、复制服务、复制计划、恢复点、复制任务、复制对象、故障处理、邮件通知和站点执行的相关操作。

  > **说明**：
  >
  > 进行跨站点操作时将调用 API 接口，对应的操作记录将记录为用户事件，用户名为 API 密钥的名称。
- **系统事件**

  记录系统触发的关键性操作，包括自动执行备份计划、复制计划和自动删除恢复点。在跨站点场景下，系统事件仅记录在目标端站点所关联的 CloudTower 界面。

**前提条件**

CloudTower 的用户角色需为超级管理员、运维管理员或安全审计员。

**操作步骤**

1. 可通过以下三种方式查看备份服务和复制服务的事件。事件界面默认展示用户事件，可单击**系统事件**切换为系统事件。

   - 在 CloudTower 的**事件**界面，可以查看当前 CloudTower 记录的所有事件。
   - 在**备份与容灾**界面中，单击**事件**，可以查看当前 CloudTower 记录的所有备份服务和复制服务相关的事件。

## 许可使用

| **字段** | **说明** |

| 总额度 | SMTX 备份与容灾的许可单元数量。 |
| 备份已用 | 备份计划中虚拟机已使用的许可单元数量，每备份一个虚拟机消耗一个许可单元。  - 若一个虚拟机同时属于多个备份计划，仅计为一个许可单元。 - 若一个虚拟机同时属于备份计划和复制计划，将计为两个许可单元。 |
| 复制已用 | **异步复制**：每异步复制一个虚拟机消耗 2 个许可单元；每异步复制 1 TiB 的存储卷消耗 2 个许可单元。  - 若一个虚拟机同时属于多个异步复制计划，仅计算一次许可消耗。 - 若一个存储卷同时属于多个异步复制计划，仅计算一次许可消耗，且根据所有所属复制计划内最新恢复点记录的已使用容量中的最大值消耗许可。 - 若使用跨站点复制，出站复制计划里的虚拟机或存储卷仅消耗目标站点的许可单元，站内和入站复制计划里的虚拟机或存储卷仅消耗当前站点的许可单元。 |
| **同步复制**：每同步复制一个虚拟机消耗 4 个许可单元。  - 若一个虚拟机同时属于多个同步复制计划，仅计算一次许可消耗。 - 若使用跨站点复制，出站复制计划里的虚拟机或存储卷仅消耗目标站点的许可单元，站内和入站复制计划里的虚拟机或存储卷仅消耗当前站点的许可单元。 |
| 可用 | 未被使用的许可单元数量。 |

**注意**：

当许可过期或已达到可用的许可单元数量，部分操作将被限制。具体说明如下：

| 场景 | 操作限制 |
| --- | --- |
| 许可过期 | - 允许创建备份计划或添加新的虚拟机到备份计划中，但无法进行备份；备份计划已有的虚拟机不允许继续备份，已生成的恢复点允许恢复类操作。 - 允许创建复制计划或添加新的复制对象到复制计划中，但无法进行复制；复制计划已有的复制对象不允许继续复制，已生成的恢复点允许故障处理类操作。 - 允许使用已生成的恢复点进行恢复或故障处理类操作。 - 允许创建同步复制计划和异步复制计划。 - 允许恢复已暂停的同步复制。 |
| 达到可用的许可单元数量 | - 无法添加新的虚拟机到备份计划中；备份计划中已有的虚拟机可继续备份，已生成的恢复点允许进行恢复操作。 - 可以添加新的存储卷到异步复制计划中，但无法进行复制；异步复制计划中已有的存储卷不允许继续复制，已生成的恢复点允许进行故障处理类操作。 - 无法添加新的虚拟机到同步或异步复制计划中；复制计划中已有的虚拟机可继续复制，已生成的恢复点允许进行故障处理类操作。 |

## 升级备份服务

# 升级备份服务

备份服务支持一键在线升级。如果 CloudTower 中部署了多个备份服务，建议升级到相同的版本。

**注意事项**

- 执行升级前，请参考《SMTX 备份与容灾发布说明》的[版本配套说明](/backup/2.3.0/backup_release_notes/release_notes_02)，确保 CloudTower 版本与目标备份服务版本配套。若 CloudTower 版本不配套，请先升级至配套版本。
- 若当前备份服务有正在执行的备份或恢复任务，则不允许升级。
- 升级过程中可能备份服务会出现短暂的离线（约 10 秒），如果在离线过程中有计划中要执行的备份计划，在升级完成后将自动执行一次。
- 升级过程中无法对备份计划、恢复点和备份存储库执行操作，且无法编辑备份服务关联的 SMTX OS（ELF）集群，也无法设置备份和恢复任务。
- 升级过程中不允许删除原本使用的安装包。升级完成后，如果原本使用的安装包没有关联的备份服务或复制服务，允许删除。
- 升级完成后不允许删除升级时使用的安装包，直到备份服务升级到下一个新版本或备份服务被卸载。

## 上传升级安装包

1. 登录 CloudTower，在**备份与容灾**界面中，单击**设置**进入功能管理界面。
2. 单击**安装包管理**，再单击**上传安装包**，上传 `.tar.gz` 格式的 SMTX 备份与容灾安装包用于升级。
3. 单击**上传**。

## 开始升级

上传升级安装包后，备份服务的状态将显示为**可升级**，表示当前备份服务可以升级到更高版本。

**操作步骤**

1. 登录 CloudTower，在**备份与容灾**界面中，单击**设置**进入功能管理界面。
2. 选择待升级的备份服务，单击**备份服务升级**，可查看备份服务的基本信息及可升级版本。若 CloudTower 中存在多个适用 CPU 架构相同、版本号更高的安装包，将展示多个可升级版本。
3. 选择升级的版本，单击**升级**。
4. 在弹出的对话框中单击**升级**。

## 卸载备份服务

# 卸载备份服务

当备份服务部署失败、或不再需要备份服务继续运行时，允许卸载备份服务。

卸载备份服务前，建议先手动删除当前备份服务关联的所有备份存储库，避免数据残留。

**风险提示**

备份服务卸载后，除保留在备份存储库中的备份文件，其他数据将被删除，请谨慎操作。

**前提条件**

- 待卸载的备份服务没有关联的备份计划。
- 备份服务当前没有正在执行的备份或恢复任务。

**注意事项**

- 备份服务卸载过程中无法对关联的备份存储库执行操作。
- 备份服务卸载过程中不允许删除原本使用的安装包。卸载完成后，若原本的安装包没有关联的备份服务或复制服务，允许删除。

**操作步骤**

1. 登录 CloudTower，在**备份与容灾**界面中，单击**设置**进入功能管理界面。
2. 选择欲卸载的备份服务，单击**卸载**。
3. 在弹出的对话框中单击**卸载**。

# 升级复制服务

  - 复制对象处于`故障转移`状态中；
  - 存在测试虚拟机。

- 升级过程中不允许删除原本使用的安装包。升级完成后，如果原本使用的安装包没有关联的复制服务或备份服务，允许删除。
- 升级完成后不允许删除升级时使用的安装包，直到复制服务升级到下一个新版本或复制服务被卸载。

## 上传升级安装包

2. 选择**安装包管理**，再单击**上传安装包**，上传 `.tar.gz` 格式的 SMTX 备份与容灾安装包用于升级。

## 开始升级

**操作步骤**

1. 登录 CloudTower，在**备份与容灾**界面中，单击**设置**进入功能管理界面。
2. 选择待升级的复制服务，单击**服务升级**，可查看复制服务的基本信息及可升级版本。若 CloudTower 中存在多个适用 CPU 架构相同、版本号更高的安装包，将展示多个可升级版本。
3. 选择升级的版本，单击**升级**。
4. 在弹出的对话框中单击**升级**。
