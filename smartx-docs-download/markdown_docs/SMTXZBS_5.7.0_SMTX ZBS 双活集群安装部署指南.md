## 双活集群简介

# 双活集群简介

双活是 SMTX ZBS 的一种数据保护和容灾机制，提供 RPO（数据恢复点目标）为 0，RTO（恢复时间目标）为分钟级的站点失效容错能力。

SMTX ZBS 双活集群（以下简称“双活集群”）只支持块存储服务。双活集群以拉伸形态部署，由两个可用域和一个仲裁节点组成。两个可用域与仲裁节点间通过网络连接通信。当一个可用域失效后，另一个可用域仍可以继续提供块存储服务，从而获得可用域级别的容灾能力。

**可用域**

两个可用域一般位于同一个城市有一定距离的物理节点所在区域，分为优先可用域和次级可用域。每个可用域至少包含 2 个节点，并且都可以完全独立地提供存储服务，双活集群正常工作时客户端主要通过优先可用域访问块存储服务。

当两个可用域工作正常时，双活集群中存储卷的数据块会在两个可用域内分别保存副本，以保证当一个可用域失效时，客户端可从另外一个存活的可用域中获得完整的数据，从而恢复服务。

**仲裁节点**

设置仲裁节点主要是为了保证双活集群中选主类服务正常运行，并使得业务在优先或次级可用域中自动切换。仲裁节点仅仅参与选主算法，保存部分元数据（如 ZooKeeper 服务），而不存放任何用户数据。

仲裁节点可以是物理机或虚拟机，需要部署在与优先可用域和次级可用域不同的物理故障区。一般情况下，仲裁节点与优先可用域、次级可用域均位于不同的数据中心。如果仲裁节点与可用域位于同一故障区，例如同一个 IDC 机房内，在遭遇 IDC 整体下线的场景时，双活集群将无法自动恢复服务。

## 双活集群简介 > 相关概念

# 相关概念

在阅读本文档前，您需要先了解与 SMTX ZBS 双活集群相关的一些概念，以帮助您更好地理解双活集群的安装部署和运维操作。

**块存储**

块存储是一种将数据划分为固定大小的“块”并将这些块独立存储的存储方式。块存储将集群中的存储设备划分为多个卷，每个卷以裸设备的方式映射给计算端使用。

**节点**

块存储软件可运行在物理服务器上，而节点特指运行了块存储软件的单个物理服务器。节点是组成块存储集群的基本单位。

**主节点**

SMTX ZBS 集群或 SMTX ZBS 双活集群中额外运行元数据管理服务的节点。

**存储节点**

该概念主要相较于“主节点”而言，指 SMTX ZBS 集群或 SMTX ZBS 双活集群中没有运行 ZooKeeper 和 Meta 等元数据管理服务的节点。

**SMTX ZBS 双活集群**

SMTX ZBS 双活集群属于逻辑概念。在实际生产环境中，一个 SMTX ZBS 双活集群至少由 5 个节点通过网络互连组成。其中优先可用域 2 个节点，次级可用域 2 个节点，1 个仲裁节点。

**副本数**

使用多副本技术提供数据冗余，一份数据会在不同主机上保存多个副本，以提高数据的可靠性和安全性。

**物理盘池**

由块存储集群中某个节点的一块或多块物理盘组成，不支持跨节点构建。SMTX ZBS 块存储支持在单个节点上配置多个物理盘池，以提升节点的性能上限并充分利用物理盘的性能。每增加一个物理盘池都会增加一定的 CPU 核心和内存消耗。根据节点容量规格不同，可配置 1、2 或 4 个物理盘池。

**常驻缓存模式**

SMTX ZBS 双活集群支持常驻缓存模式，将 iSCSI LUN 或 NVMe Namespace 中的数据保留在缓存层，从而获得更稳定的高性能。

**容量规格**

SMTX ZBS 集群和 SMTX ZBS 双活集群根据单主机物理盘容量上限分为三种规格：标准容量规格（128 TB）、大容量规格（256 TB）和加大容量规格（512 TB）。

**CloudTower**

SmartX 多集群资源集中管理平台。CloudTower 运⾏在虚拟机内，通过使⽤少量的计算和存储资源便可管理上百台主机。除存储管理、⽹络管理、硬件管理、监控和报警等基本服务外，CloudTower 还提供全局搜索、事件审计和报表等功能，同时它也提供完整和标准的 RESTful API 和多种语⾔的 SDK，帮助运维管理员轻松管理和使用多个 SmartX 集群。

**存储分层模式**

把集群内的存储设备分为缓存层和容量层，其中容量层是所有节点数据分区构成的空间，用于存放冷数据，冷数据根据设定的数据冗余策略，以副本形态存储。不同存储配置下缓存层的构成不同。

当节点为混闪配置、多种类型 SSD 全闪配置或单一类型多种属性 SSD 全闪配置时，缓存盘与数据盘将独立部署，高速介质为缓存，低速介质为容量。数据会以副本形态先存入写缓存，待数据变冷后下沉到容量层。

- 对于混闪配置和多种类型 SSD 全闪配置，缓存层将分为写缓存和读缓存，写缓存用于存放新写入的数据；读缓存用于存放被频繁访问的已下沉的数据，写缓存与读缓存的容量比为 8:2。
- 对于单一类型多种属性 SSD 全闪配置，缓存层均为写缓存，用于存放新写入的数据。

由于双活集群仅支持副本策略，因此单一类型单一属性 SSD 全闪配置部署为不分层模式即可。

**存储不分层模式**

不设置缓存层。除了含有系统分区的物理盘，剩余的所有物理盘都作为数据盘使用。使用存储不分层模式时只能用全闪配置。

**VMware vSphere 集群**

VMware vSphere 是 VMware 公司推出的一套服务器虚拟化解决方案，可将数据中心转换为包括 CPU、存储和网络资源的聚合计算基础架构。vSphere 将这些基础架构作为统一的运行环境进行管理。

ESXi 是 VMware vSphere 的一个关键组件，可独立安装和运行在物理服务器上，是 VMware 的裸机虚拟机管理程序。

**LACP 动态链路聚合**

LACP（Link Aggregation Control Protocol，链路聚合控制协议）是一种基于 IEEE802.3ad 标准的实现链路动态聚合与解聚合的协议，属于在链路聚合中常用的协议之一。链路聚合组中启用了 LACP 协议的成员端口通过发送 LACP PDU 报文进行交互，双方对能够发送和接收报文的端口明确并达成一致，确定承担业务流量的链路。此外，当聚合条件发生变化时，如某个链路发生故障，LACP 模式会自动调整聚合组中的链路，组内其他可用成员链路接替故障链路维持负载平衡。因此在不进行硬件升级时，使用 LACP 协议也可以增加设备之间的逻辑带宽，提高网络的可靠性。

动态链路聚合模式需要本端和对端设备同时启用 LACP 协议，所选择的活动接口必须保持一致，然后才能建立链路聚合组 LAG（Link Aggregation Group）。LAG 是指将若干条以太链路捆绑在一起形成一条逻辑链路，也称为 Eth-Trunk 链路。

SMTX ZBS 块存储支持 LACP 动态链路聚合功能。在虚拟分布式交换机中，需选择与对端交换机配置 LACP 端口相连的两个物理网口，并且网口绑定模式选择为 `OVS bonding` 中的 `balance-tcp` 或 `Linux Bonding` 中的 `802.3ad` 模式。设置完成后即可启用 LACP 动态链路聚合功能。

**部署控制节点**

在部署 SMTX ZBS 双活集群时，需要通过浏览器访问双活集群中优先可用域或次级可用域中某个节点的管理 IP，进入集群部署界面。此节点即为集群部署时的控制节点，简称部署控制节点。

## 双活集群简介 > 部署形态与应用场景

# 部署形态与应用场景

将 SMTX ZBS 部署为双活集群时，SMTX ZBS 块存储软件直接运行在优先/次级可用域的物理服务器上，通过 iSCSI 接入协议为客户端（例如 VMware vSphere 集群）提供块存储服务。仲裁节点可以是物理服务器或者虚拟机。

在使用 SMTX ZBS 双活集群为客户端提供存储服务时，需对客户端进行相应的配置，使得客户端也具备跨站点的容灾能力，从而实现客户端和存储端的容灾。一般建议将 SMTX ZBS 双活集群优先可用域所在数据中心用作客户端的业务运行站点，将次级可用域所在数据中心用作客户端的容灾站点。

业务运行站点通常持有两份数据副本，容灾站点持有一份数据副本。当业务运行站点故障后，客户端可切换至容灾站点运行，并可以访问 SMTX ZBS 双活集群次级可用域的数据副本，保证故障发生后的业务连续性。

> **注意：**
>
> - 本文档仅描述安装部署 SMTX ZBS 双活集群的流程，以及双活集群搭配 VMware vSphere 集群使用的配置操作，其他场景的配置方案需根据实际需求确定。
> - VMware vSphere 集群作为客户端时需完成的配置请参考[配置 ESXi 主机](/smtxzbs/5.7.0/multiactive_user_guide/zbs_multiactive_user_guide/zbs_multiactive_user_guide_42)，其使用存储服务的方式请参考《SMTX ZBS 块存储配置指南》的 [iSCSI 存储服务](/smtxzbs/5.7.0/zbs_configuration/zbs_configuration_02)。

## 双活集群简介 > 安装方式

# 安装方式

SMTX ZBS 块存储支持使用独立物理盘安装，在安装 SMTX ZBS 映像文件时，您可根据系统提示选择将 SMTX ZBS 块存储安装在两块物理盘中配置软 RAID 1 或使用独立的物理盘安装 SMTX ZBS 块存储。

- **将 SMTX ZBS 块存储安装在独立的硬 RAID 1 中**

  如果希望系统盘故障时不影响集群系统运行并且能够通过硬件 RAID 重建故障盘，或者希望将系统分区与缓存和数据物理分隔、降低系统盘的 IO 频率以降低故障率，可选择此安装方式。该方式将使用两块 SSD 与存储控制器组成的硬 RAID 1 作为 SMTX 系统盘，用于引导和安装 SMTX ZBS 块存储，无需单独的启动盘。安装完成后，节点内其他物理盘作为缓存盘或数据盘用于存放缓存或数据。

  使用此模式安装 SMTX ZBS 前，请先阅读[将 SMTX ZBS 安装在硬 RAID 1 中的配置推荐](/smtxzbs/5.7.0/multiactive_user_guide/zbs_installation_guide/zbs_installation_guide_82)了解推荐配置，以更好地保证系统稳定性和数据安全。
- **将 SMTX ZBS 块存储安装在软 RAID 1 中**

  如果不需要将系统分区与缓存和数据做物理隔离，或者因硬件限制无法配置单独的硬 RAID 1，可选择此部署模式。该方式需要准备一块单独的物理盘作为启动盘用于引导加载程序。在安装 SMTX ZBS 块存储时，系统将使用两块 SSD 安装 SMTX ZBS 块存储并配置软 RAID 1，这两块 SSD 称为含元数据分区的缓存盘或含元数据分区的数据盘。安装完成后，含元数据分区的缓存盘和含元数据分区的数据盘的剩余存储空间将用于存放缓存或数据，节点内其他物理盘作为缓存盘或数据盘。

## 构建双活集群的要求 > 集群要求

# 集群要求

规划双活集群时，集群的节点数、优先/次级可用域、仲裁节点、副本数和拓扑需遵照以下要求进行设计。

- **集群节点数**

  单个双活集群最少应包含 5 个节点，其中优先可用域 2 个节点，次级可用域 2 个节点，1 个仲裁节点。推荐优先可用域和次级可用域分别至少 3 个节点。

  双活集群支持部署的最大节点数（优先可用域节点 + 次级可用域节点）要求不超过 255 个节点。为了发挥双活集群的最高性能，我们推荐单个双活集群的优先可用域节点和次级可用域节点的总个数不超过 64 个节点。
- **优先/次级可用域**

  优先可用域和次级可用域需最少包含 2 个节点，其中 2 个主节点，并且两个可用域的所有主节点应尽量分布在不同的机架和机箱上，以确保更高的可靠性。
- **仲裁节点**

  仲裁节点可以部署在虚拟机或者物理服务器中。
- **副本数**

  双活集群必须设置 3 副本的冗余策略。

请在规划双活集群时同时记录优先/次级可用域内的每个节点和仲裁节点的服务器序列号，同时标记出两个可用域内每个主节点的服务器序列号。

## 构建双活集群的要求 > 网络要求 > 网络类型与网络拓扑

# 网络类型与网络拓扑

在部署双活集群时需要创建虚拟分布式交换机并关联网络。

以一个典型的 7 节点（优先可用域 3 节点 + 次级可用域 3 节点 + 仲裁节点）组成的同城双活集群为例，其网络拓扑图如下图所示。

![](https://cdn.smartx.com/internal-docs/assets/bd6c2df9/zbs_multiactive_user_guide_02.png)

由上图可见，在优先可用域和次级可用域的节点上安装部署 SMTX ZBS 时，通过虚拟分布式交换机需要创建三种类型的系统网络：管理网络、存储网络和接入网络。每种系统网络可使用其所属的虚拟分布式交换机所关联的物理网口进行通信。

- **管理网络**：用于集群管理，该网络的带宽须为 1Gbps 及以上。
- **存储网络**：用于集群内各节点间数据交换，该网络只能由集群内部使用，带宽须为 10Gbps 及以上。一般两个可用域间的存储网络应为光纤直连。
- **接入网络**：用于为集群外部的客户端（计算平台）提供存储服务，该网络的带宽须为 10Gbps 及以上。两个可用域间的接入网络应为光纤直连。

## 构建双活集群的要求 > 网络要求 > 网络配置要求

# 网络配置要求

双活集群里两个可用域的数据是实时同步的。在正常情况下，所有的数据写入操作都会确保在两个可用域写入存储磁盘后，才会返回成功。而读取操作会优先选择从数据访问点所在可用域内的数据副本上获取数据。因此双活集群的网络配置对集群的写性能影响较大。

- **网络带宽**

  双活集群的管理网络带宽不低于 1 Gbps，可用域之间的存储网络带宽必须为 10 Gbps 及以上（通常两个可用域间的存储网络应为光纤直连），仲裁节点与优先/次级可用域的存储网络之间的带宽应不小于 100 Mbps。接入网络带宽必须为 10 Gbps 及以上。

  双活集群的写入带宽上限为两个可用域间的最大存储网络带宽。如果期望更高的吞吐量，则需要对可用域间带宽进行扩容。
- **网络连接**

  集群的优先可用域与次级可用域的主机必须连接到 L2 层管理网络、接入网络和存储网络。

  仲裁节点与优先/次级可用域的节点之间可通过 L2 层或 L3 层存储网络进行连接。

  > **说明：**
  >
  > 仲裁节点与优先/次级可用域节点通过存储网络连接时不支持 NAT 地址转换。
- **网络延时**

  集群内优先可用域与次级可用域之间的存储网络 ping 延时在 5 ms 以下。

  仲裁节点与优先可用域或次级可用域之间的存储网络 ping 延时在 100 ms 以下。

  数据写入将受到可用域间的链路往返延时影响，尽量降低可用域间存储网络 ping 延时将改善性能。当可用域间存储网络 ping 延时为 5 ms 时：

  - 对于队列深度为 1 的小块 IO（IO 大小 <= 4k），其影响最为明显，写性能约为 SMTX ZBS 集群的 10% 左右。
  - 对于高队列深度（64）或者大块 IO（IO 大小 >= 64k），其性能影响较小，写性能约为 SMTX ZBS 集群的 90% 左右。

## 构建双活集群的要求 > 网络要求 > IP 规划要求 > 双活集群节点 IP

# 双活集群节点 IP

> **说明：**

仲裁节点需要与优先/次级可用域节点的 L2 或 L3 层存储网络连通，因此需为其规划一个 IP 用于存储网络，以及对应的子网掩码和网关等信息。如果希望双活集群支持通过 SSH 方式登录仲裁节点，建议为其再配置一个 IP（块存储集群管理 IP）用于管理网络。

## 构建双活集群的要求 > 硬件要求 > 系统资源占用 > 系统对主机 CPU 和内存的占用 > 系统总 CPU 占用说明

# 系统总 CPU 占用说明

当 CPU 启用超线程时，块存储将占用逻辑核心。由于同一物理核心上的两个逻辑核心存在资源竞争，另一逻辑核心的性能会显著降低，因此在计算块存储的 CPU 占用时应按物理核心进行计算。

此外，节点中部署的系统服务也会占用 CPU 核心。综上所述，系统总 CPU 占用除取决于块存储的 CPU 占用外，还取决于是否启用超线程以及系统服务的 CPU 占用。计算公式如下：

- 启用超线程：`总 CPU 占用 = 块存储 CPU 占用 × 2 + max(其他系统服务 CPU 占用)`
- 未启用超线程：`总 CPU 占用 = 块存储 CPU 占用 + max(其他系统服务 CPU 占用)`

下面以如下配置为例介绍如何计算系统的 CPU 占用：

| 组件 | 配置 | CPU 占用 |
| --- | --- | --- |
| 块存储 | - 标准容量规格 - 1 个物理盘池 - 单 NUMA node 包含 12 颗物理核心 | 15 核 |
| CloudTower | 高配 | 8 核 |
| 可观测性服务 | 管理 8～32 个节点 | 8 核 |
| 复制服务 | - | 4 核 |

CloudTower、可观测性服务和复制服务共享 CPU 核心，取四者中最大值 8 计算，此时：

- 启用超线程：系统占用主机的逻辑核心数 = 15 × 2 + 8 = 38 核。
- 未启用超线程：系统占用主机的物理核心数 = 15 + 8 = 23 核。

## 构建双活集群的要求 > 启用其他高级功能要求 > 启用机架拓扑感知

# 启用机架拓扑感知

为了确保在机架断电或损坏等极端状况下，系统还能在其他机架的主机上找到存活的副本，继续提供数据访问能力和增加数据可用性，SMTX ZBS 提供了机架感知功能。

启用机架拓扑感知功能要求在规划双活集群时，尽量将服务器平均分布在不同的机架中，从而尽量使数据的多个副本均匀地分配到不同的机架、机箱和主机上。

此外，还需在 CloudTower 管理平台正确配置双活集群的物理拓扑。即在双活集群部署结束后，使用 CloudTower 的[机架配置](/smtxzbs/5.7.0/zbs_administration_guide/zbs_administration_guide_096)功能，如实呈现安装部署现场的主机与机架的位置和拓扑结构，然后系统将根据拓扑来调整数据的分配情况。

# 戴尔 Intel x86 服务器

**操作步骤**

1. 打开浏览器，在浏览器的地址栏输入节点的 IPMI 管理台 IP，登录该节点的 IPMI 管理台。
2. 登录成功后，找到**虚拟控制台**，单击**启动虚拟控制台**，进入虚拟控制台界面。
3. 在界面顶端单击**启动**，在弹出的**开机控制**窗口中单击 **BIOS 设置**，然后在弹出的**确认启动操作**对话框中单击**是**。
4. 在界面顶端单击**功率**，在弹出的**电源控制**窗口中单击**复位系统（热启动）**，然后在弹出的**确认电源操作**对话框中单击**是**，完成后等待服务器重启。
5. 当服务器进入 **System Setup** 界面时，单击 **System BIOS**，进入 **System BIOS Settings** 界面。

   ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_55.png)
6. 在界面中单击 **Boot Settings**，将 **Boot Mode** 选项设置为 **UEFI**，以提升部署效率。

   ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_51.png)
7. 单击 **Back**，返回 **System BIOS Settings** 界面后，单击 **Processor Settings**，检查并确认 **Virtualization Technology** 选项为 **Enabled**，以确保启用 CPU 的虚拟化特性。

   ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_52.png)
8. 单击 **Back**，返回 **System BIOS Settings** 界面后，单击 **System Profile Settings**，检查并确认 **System Profile** 选项为 **Performance**，以确保关闭电源的节能模式。

   ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_53.png)
9. 检查并确认 **Workload Profile** 选项为 **Virtualization Optimized Performance Profile**，以确保系统针对虚拟化场景优化性能。

   ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_54.png)
10. （可选）若计划将 SMTX ZBS 块存储安装在独立的硬 RAID 1 中，则必须将 RAID 1 虚拟磁盘的写策略设置为 Write Through，下面以 **H730P Mini** RAID 卡为例，描述操作步骤。

    1. 1. 在 **System Setup** 界面单击 **Device Settings**，进入设备设置界面。

       ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_56.png)
    2. 在 **Device Settings** 界面选择 RAID 1 虚拟磁盘对应的存储控制器。

       ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_57.png)
    3. 单击 **Virtual Disk Management**，进入虚拟磁盘管理界面。

       ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_58.png)
    4. 选择所要安装 SMTX ZBS 块存储的 RAID 1 虚拟磁盘。

       ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_59.png)
    5. 选择虚拟磁盘的高级配置选项后，将 **VIRTUAL DISK POLICIES** 中的 **Default Write Cache Policy** 设置为 **Write Through** 策略后，单击 **Apply Changes**。

       ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_60.png)
11. 单击 **Back**，在 **System BIOS Settings** 界面单击 **Finish**，然后在弹出的 **Warning** 对话框中单击 **Yes** 完成设置。

## 部署 SMTX ZBS 双活集群 > 部署须知 > 物理交换机启用 LACP 动态链路聚合

# 物理交换机启用 LACP 动态链路聚合

当接入 SMTX ZBS 双活集群的管理网络、接入网络或存储网络的物理交换机启用了 LACP 动态链路聚合时，需要根据实际情况在集群部署前和部署期间完成相应的配置。

## 部署集群前的配置

仅当接入双活集群管理网络的物理交换机启用了 LACP 动态链路聚合时，才需要执行以下操作；否则请忽略。

1. 使用默认账户 `root` 登录需要设置网口绑定的优先/次级可用域节点的 SMTX ZBS 块存储系统。
2. 使用 `cd /etc/sysconfig/network-scripts/` 命令，进入网口配置文件的访问路径。
3. 使用 `ip a` 命令查看所有网卡的接口信息，再使用 `ethtool <port>` 查看每个网卡接口的速率，以判断出两个用于管理网络的网口，其中 `<port>` 表示实际网口的名称。

   **Speed** 显示 1000 Mb/s 及以上的网口可以作为管理网络的网口。
4. 在网口配置文件路径下执行以下命令，将两个用于管理网络的网口绑定为一个网口（名称为 `pre_bond`），并生成 ifcfg-pre\_bond 网口配置文件。

   ```
   network-preconfig create-lacp --nics <nics>
   ```

   其中，`<nics>` 表示需要绑定的网口名称，网口名称之间需用空格隔开，且名称外侧需加引号（"）。例如，当需要绑定网口 eth0 和 eth1 时，则执行命令 `network-preconfig create-lacp --nics "eth0 eth1"`。

   当输出以下信息时，表明 pre\_bond 网口创建成功。

   ```
   create lacp bond port success
   ```
5. （可选）如果需要重新绑定网口，可先执行以下命令清除所有配置，然后根据输出信息执行下一步操作。

   ```
   network-preconfig clear-config
   ```

   当输出以下信息时，表明 pre\_bond 网口的配置清理成功，请参考步骤 3 ～ 4 重新绑定管理网口。

   ```
   delete pre_bond configuration
   ```

   如果为管理网口配置过 VLAN 子接口，还会输出以下信息，表明 VLAN 子接口的配置也已被清理。若需要重新创建 VLAN 子接口，请参考[配置 VLAN](/smtxzbs/5.7.0/multiactive_user_guide/zbs_multiactive_user_guide/zbs_multiactive_user_guide_25) 进行部署集群前的配置。

   ```
   delete pre_vlan configuration
   ```

在部署集群的[第 5 步：配置网络](/smtxzbs/5.7.0/multiactive_user_guide/zbs_multiactive_user_guide/zbs_multiactive_user_guide_35)，为某个系统网络创建虚拟分布式交换机时，在**物理网口**区域，为需要绑定的网口选择 **balance-tcp** 的绑定模式。

## 部署 SMTX ZBS 双活集群 > 部署须知 > 配置 VLAN

# 配置 VLAN

当物理交换机为 SMTX ZBS 双活集群的管理网络、接入网络或存储网络配置了 VLAN 时，需要根据实际情况在集群部署前和部署期间完成相应的配置。

## 部署集群前的配置

仅当物理交换机为管理网络配置了 VLAN 时，才需要在部署集群前执行以下操作；否则请忽略。

1. 使用默认账户 `root` 登录优先/次级可用域节点的 SMTX ZBS 块存储系统。
2. 使用 `cd /etc/sysconfig/network-scripts/` 命令，进入网口配置文件的访问路径。
3. 使用 `ip a` 命令查看所有网卡的接口信息，再使用 `ethtool <port>` 查看每个网卡接口的速率，以判断并确定用于管理网络的网口，其中 `<port>` 表示实际网口的名称。

   **Speed** 显示 1000 Mb/s 及以上的网口可以作为管理网络的网口。
4. 在网口配置文件路径下执行以下命令，为网口创建 VLAN 子接口，并配置 IP、网关和子网掩码。

   ```
   network-preconfig create-vlanif --iface <iface> --ipaddr <ipaddr> --netmask <netmask> --gateway <gateway> --vlanid <vlanid>
   ```

   其中，带有`<>`符号的字段表示管理网络的参数。

   | 参数 | 说明 |
   | --- | --- |
   | `iface` | 表示网口名称。 - 当计划接入双活集群管理网络的物理交换机启用了 LACP 动态链路聚合时，若当前节点用于管理网络的网口为 pre\_bond 网口，请替换为 `pre_bond`。 - 若当前节点用于管理网络的网口不为 pre\_bond 网口，请替换为实际的网口名称，如 `eth0`。 |
   | `ipaddr` | 表示实际规划的 IP 地址。 |
   | `netmask` | 表示实际规划的子网掩码。 |
   | `gateway` | 表示实际规划的网关。 |
   | `vlanid` | 表示实际规划的 VLAN ID。 |

   当输出以下信息时，表明 VLAN 子接口创建成功。

   ```
   create vlan interface success
   ```
5. （可选）如果需要重新为管理网口配置 VLAN 子接口，可先执行以下命令清除所有配置，然后根据输出信息执行下一步操作。

   ```
   network-preconfig clear-config
   ```

   当输出以下信息时，表明 VLAN 子接口的配置清理成功，请参考步骤 3 ～ 4 重新为管理网口配置 VLAN 子接口。

   ```
   delete pre_vlan configuration
   ```

   如果绑定过管理网口，还会输出以下信息，表明 pre\_bond 网口的配置也已被清理。若需要重新绑定管理网口，请参考[物理交换机启用 LACP 动态链路聚合](/smtxzbs/5.7.0/multiactive_user_guide/zbs_multiactive_user_guide/zbs_multiactive_user_guide_24) 进行部署集群前的配置。

   ```
   delete pre_bond configuration
   ```

## 部署 SMTX ZBS 双活集群 > 配置 Port Channel 哈希策略

# 配置 Port Channel 哈希策略

当接入块存储集群的存储网络的物理交换机启用了 LACP 动态链路聚合时，建议在物理交换机上配置 Port Channel 哈希策略，在默认参数的基础上增加根据源端口和目的端口作为哈希字段，以便充分利用多网口的带宽。下文分别以 Mellanox 交换机和华为交换机为例，介绍配置 Port Channel 哈希策略的操作。

## Mellanox 交换机

1. 执行以下命令，以管理员身份登录交换机，进入配置模式。

   ```
   switch > enable
   switch # configure terminal
   ```
2. 执行 `show interfaces port-channel load-balance` 命令，检查当前 Port Channel 哈希字段是否包含 `source-destination-port`。

   - 若输出结果中已包含 `source-destination-port`，则无需执行后续配置操作。
   - 若输出结果中不包含 `source-destination-port`，请继续参考步骤 3～5 进行配置。
3. 执行以下命令，在现有字段的基础上增加 `source-destination-port`。

   ```
   switch (config) # port-channel load-balance ethernet source-destination-mac source-destination-port
   ```
4. 执行 `show interfaces port-channel load-balance` 命令，检查 Port Channel 哈希字段。

   若输出结果中包含 `source-destination-port` ，说明配置成功。

   ```
   switch (config) # show interfaces port-channel load-balance
   source-destination-mac, source-destination-port
   ```
5. 执行 `configuration write` 命令，保存配置。

## 华为交换机

1. 执行以下命令，以管理员身份登录交换机，进入配置模式。

   ```
   <6865-2>system-view
   Enter system view, return user view with return command.
   ```
2. 执行 `display load-balance profile` 命令，查看负载分担模板的详细信息，确认输出结果中 IP 一行的哈希字段是否包含 `l4-src-port` 和 `l4-dst-port`。

   - 若输出结果中已包含 `l4-src-port` 和 `l4-dst-port`且不包含 `dst-qp`，则无需修改。输出示例如下：

     ```
     [~6865-2]display load-balance profile
     Load-balance Profile: default
     Packet    HashField
     ---------------------------------------------------
     IP             src-ip          dst-ip
                 l4-src-port     l4-dst-port
     IPv6           src-ip          dst-ip
                 l4-src-port     l4-dst-port
     L2             src-mac         dst-mac
     MPLS           top-label       2nd-label
     Eth-Trunk      universal-id(1) hash-mode(1)
     Stack          universal-id(1) hash-mode(1)
     ```
   - 若输出结果中不包含 `l4-src-port` 和 `l4-dst-port` 或包含 `dst-qp`，则需要手动修改，否则会影响 RDMA 的连接。请执行如下命令更新默认 profile，添加 `l4-src-port` 和 `l4-dst-port` 并删除 `dst-qp`。

     ```
     [~6865-2]load-balance profile default
     [~6865-2-load-balance-profile-default]ip src-ip dst-ip l4-src-port l4-dst-port
     [*6865-2-load-balance-profile-default]commit
     [~6865-2-load-balance-profile-default]quit
     ```

     更新完成后，再次执行 `display load-balance profile` 命令，确认配置结果符合预期。
3. 执行 `display interface Eth-Trunk 1 | include "Hash"` 命令查看 Eth-Trunk 接口的哈希策略。

   - 若输出结果中 Eth-Trunk 已经默认采用系统的 profile 作为哈希策略，则无需修改。输出示例如下：

     ```
     [~6865-2]display interface Eth-Trunk 1 | include "Hash"
     Switch Port, PVID :    1, TPID : 8100(Hex), Hash Arithmetic : profile default, The Maximum Frame Length is 9216
     ```
   - 若输出结果中 Eth-Trunk 未默认采用系统的 profile 作为哈希策略，请执行如下命令配置 Eth-Trunk。

     ```
     [~6865-2]interface Eth-Trunk 1
     [~6865-2-Eth-Trunk1]load-balance enhanced profile default
     [~6865-2-Eth-Trunk1]commit
     [~6865-2-Eth-Trunk1]quit
     ```

     配置完成后，再次执行 `display interface Eth-Trunk 1 | include "Hash"` 命令，确认配置结果符合预期。

## 部署 SMTX ZBS 双活集群 > 检查并设置双活集群部署的环境 > 为优先/次级可用域节点设置部署的环境

# 为优先/次级可用域节点设置部署的环境

**注意事项**

当物理交换机为管理网络配置了 VLAN 时，请使用 `root` 账户登录优先/次级可用域节点的 SMTX ZBS 块存储系统，然后从步骤 4 开始执行。

**操作步骤**

1. 使用默认账户 `root` 登录优先/次级可用域节点的 SMTX ZBS 块存储系统，确认用于管理网络和存储网络的网口。

   1. 使用 `cd /etc/sysconfig/network-scripts/` 命令，进入网口配置文件的访问路径。
   2. 使用 `ip a` 命令查看所有网卡的接口信息，再使用 `ethtool <port>` 查看每个网卡接口的速率，以判断并确定用于管理网络和存储网络的网口，其中 `<port>` 表示实际网口的名称。

      **Speed** 显示 1000 Mb/s 及以上的网口可以作为用于管理网络的网口，**Speed** 显示 10000 Mb/s 及以上的网口可以作为用于存储网络的网口。
2. 确认当前配置路径下是否存在管理网络网口的对应配置文件，如果网口名称是 `ens4f0np0` ，那么预期会存在 `ifcfg-ens4f0np0` 配置文件。如果当前配置路径下不存在管理网络网口的对应配置文件，可以从当前配置路径下已有的其他网口配置文件拷贝后进行修改。如果网口名称是 `ens4f0np0` 且配置文件不存在，而当前路径存在 `ifcfg-ens4f0` 配置文件，那么可以执行如下操作来生成网口对应配置文件：

   1. 使用 `cp ifcfg-ens4f0 ifcfg-ens4f0np0` 命令拷贝配置文件。
   2. 使用 `vi ifcfg-ens4f0np0` 命令打开 `ifcfg-ens4f0np0` 配置文件。
   3. 将 `DEVICE=ens4f0` 修改为 `DEVICE=ens4f0np0`。
3. 配置用于管理网络的网口。

   使用 `vi ifcfg-<port>` 命令，打开用于管理网络的网口的配置文件，并进行以下修改，其中 `<port>` 表示该网口的名称。

   - 修改 **BOOTPROTO=dhcp** 为 **BOOTPROTO=static**
   - 修改 **ONBOOT=no** 为 **ONBOOT=yes**
   - 在配置文件末尾添加以下内容：

     ```
     IPADDR=ManageIP_address
     NETMASK=ManageIP_netmask
     GATEWAY=ManageIP_gateway
     ```

     其中 *`ManageIP_address`* 、*`ManageIP_netmask`* 和 *`ManageIP_gateway`* 为实际规划的优先/次级可用域节点的块存储集群管理 IP 地址、子网掩码和网关。
4. 为双活集群的部署控制节点配置存储网络的网口。

   使用 `vi ifcfg-<port>` 命令，打开用于存储网络的网口的配置文件，并进行以下修改，其中 `<port>` 表示该网口的名称。

   1. 修改 **BOOTPROTO=dhcp** 为 **BOOTPROTO=static**
   2. 修改 **ONBOOT=no** 为 **ONBOOT=yes**
   3. 在配置文件末尾添加以下内容：

      ```
      IPADDR=control_storageIP_address
      NETMASK=control_storageIP_netmask
      ```

      其中 `control_storageIP_address` 和 `control_storageIP_netmask` 为实际规划的部署控制节点的块存储集群存储 IP 和子网掩码。
5. （可选）在优先/次级可用域节点的 SMTX ZBS 块存储系统中，输入以下命令，修改主机名称，使其更有辨识度，其中 *`<name>`* 表示待重命名的主机名称。在部署集群的[配置主机](/smtxzbs/5.7.0/multiactive_user_guide/zbs_multiactive_user_guide/zbs_multiactive_user_guide_33)期间勾选集群节点时，可以根据主机名称辨认集群节点和主节点。

   ```
   hostname <name>
   su
   ```
6. 在每个优先/次级可用域节点中执行 `systemctl restart network` 命令，重启服务。
7. 两个可用域中所有节点的网口都配置完成后，测试节点网络，确保测试节点间的管理网络能够互相连通。
8. 在每个优先/次级可用域节点中执行 `systemctl status nginx` ，确认 nginx 服务启动。
9. 在每个优先/次级可用域节点中执行 `systemctl status zbs-deploy-server` ，确认 zbs-deploy-server 服务启动。
10. 使用 date 命令将节点的时间调整为当前时间，例如 `date -s "2022-11-08 15:21:23"`。在每个优先/次级可用域节点中执行命令后，系统将同步所有节点的时间，避免因集群内节点的时间设置不同步，导致集群无法正常工作。

## 部署 SMTX ZBS 双活集群 > 检查并设置双活集群部署的环境 > 为双活集群配置静态路由（可选）

# 为双活集群配置静态路由（可选）

**前提条件**

在[为仲裁节点设置部署的环境](/smtxzbs/5.7.0/multiactive_user_guide/zbs_multiactive_user_guide/zbs_multiactive_user_guide_28)时，若仲裁节点与双活集群两个可用域里的主机之间 L2 层存储网络无法连通，此时需要手动配置仲裁节点与部署控制节点（在部署集群时准备通过其块存储集群管理 IP 访问集群进行部署的节点）间的静态路由。

**操作步骤**

1. 在仲裁节点上添加到部署控制节点的静态路由。

   1. 使用 **root** 账户登录 SMTX ZBS 块存储系统, 执行下述命令，添加仲裁节点到部署控制节点的静态路由。

      ```
      ip route add 10.10.1.100/32 via 10.10.2.2 dev eth0
      ```

      其中 `10.10.1.100` 为部署控制节点的块存储集群存储 IP，`10.10.2.2` 为规划的仲裁节点到部署控制节点所在可用域的网关地址，请用户根据实际情况替换。
   2. 执行命令 `ip route show`，查看仲裁节点的路由表，确认上述添加的路由条目已刷新至路由表。
2. 在部署控制节点上添加到仲裁节点的静态路由。

   1. 使用 **root** 账户登录 SMTX ZBS 块存储系统, 执行下述命令，添加部署控制节点到仲裁节点的静态路由。

      ```
      ip route add 10.10.2.1/32 via 10.10.1.253 dev eth1
      ```

      其中 `10.10.2.1` 为仲裁节点的块存储集群存储 IP，`10.10.1.253` 为规划的部署控制节点所在可用域到仲裁节点的网关地址。
   2. 执行命令 `ip route show`，查看部署控制节点的路由表，确认上述添加的路由条目已刷新至路由表。
3. 使用 **root** 账户登录仲裁节点或部署控制节点的 SMTX ZBS 块存储系统，确认仲裁节点与部署控制节点之间的存储网络可连通。其中 `storageIP_address` 为部署控制节点或者仲裁节点的块存储集群存储 IP。

   ```
   ping storageIP_address
   ```

**配置实例**

以下图中描述的同城双活集群为例，若集群两个可用域 IDC A、IDC B 和 IDC C 内仲裁节点的网络配置如下：

![](https://cdn.smartx.com/internal-docs/assets/bd6c2df9/zbs_multiactive_user_guide_01.png)

- IDC A 和 IDC B 位于同一子网内，各节点的存储网络均位于 `10.0.255.0/24` 网段； IDC C 位于另一子网，仲裁节点的存储网络位于 `40.0.0.0/24` 网段。
- 将 IDC A 的前两个节点设置为主节点，其存储网络分别配置为 `10.0.255.11/24`、`10.0.255.12/24`，第三个节点的存储网络配置为 `10.0.255.13/24`，三个节点到仲裁节点的路由网关为 `10.0.255.253`。
- 将 IDC B 的前两个节点设置为主节点，其存储网络分别配置为 `10.0.255.14/24`、`10.0.255.15/24`，第三个节点的存储网络配置为 `10.0.255.16/24`，三个节点到仲裁节点的路由网关为 `10.0.255.254`。
- IDC C 内仲裁节点的存储网络配置为 `40.0.0.11/24`。IDC C 到 IDC A 可用域网关配置为 `40.0.0.253`，IDC C 到 IDC B 可用域网关配置为 `40.0.0.254`。

可见，IDC C 内的仲裁节点与 IDC A、IDC B 的节点之间 L2 层存储网络不连通，需配置为 L3 层存储网络连通。

IDC A 内各节点至仲裁节点之间的静态路由应参考如下进行配置：

| 节点存储网络 | 静态路由配置 |
| --- | --- |
| 10.0.255.11/24 | 40.0.0.11/32 via 10.0.255.253 |
| 10.0.255.12/24 | 40.0.0.11/32 via 10.0.255.253 |
| 10.0.255.13/24 | 40.0.0.11/32 via 10.0.255.253 |

IDC B 内三个节点至仲裁节点之间的静态路由应参考如下进行配置：

| 节点存储网络 | 静态路由配置 |
| --- | --- |
| 10.0.255.14/24 | 40.0.0.11/32 via 10.0.255.254 |
| 10.0.255.15/24 | 40.0.0.11/32 via 10.0.255.254 |
| 10.0.255.16/24 | 40.0.0.11/32 via 10.0.255.254 |

仲裁节点 C 至两个可用域内每个节点之间的静态路由应参考如下进行配置：

| 序号 | 静态路由配置 |
| --- | --- |
| 1 | 10.0.255.11/32 via 40.0.0.253 |
| 2 | 10.0.255.12/32 via 40.0.0.253 |
| 3 | 10.0.255.13/32 via 40.0.0.253 |
| 4 | 10.0.255.14/32 via 40.0.0.254 |
| 5 | 10.0.255.15/32 via 40.0.0.254 |
| 6 | 10.0.255.16/32 via 40.0.0.254 |

## 部署 SMTX ZBS 双活集群 > 第 3 步：配置主机

# 第 3 步：配置主机

1. 勾选要添加到集群的主机，取消勾选其他主机。

   > **注意**:
   >
   > 系统将对主机的 CPU 和内存进行检查，只允许选择满足[计算资源配置要求](/smtxzbs/5.7.0/multiactive_user_guide/zbs_multiactive_user_guide/zbs_multiactive_user_guide_67)，并且容量规格相同的主机作为优先/次级可用域的节点，而不满足硬件要求的主机可作为仲裁节点。
2. 修改各节点对应的主机名，使集群内各个节点的名称更有辨识度。
3. 根据规划集群时记录的[集群要求](/smtxzbs/5.7.0/multiactive_user_guide/zbs_multiactive_user_guide/zbs_multiactive_user_guide_11)中的可用域和主节点的信息，通过管理 IP 或主机名称辨认节点，为每个节点指定其所属的可用域，并在每个可用域中勾选两个节点作为主节点。

   ![](https://cdn.smartx.com/internal-docs/assets/bd6c2df9/zbs_multiactive_user_guide_12.png)
4. 勾选双活集群的仲裁节点，并选择前期为仲裁节点设置的块存储集群存储 IP 作为其心跳 IP。
5. （可选）若在[为仲裁节点设置部署的环境](/smtxzbs/5.7.0/multiactive_user_guide/zbs_multiactive_user_guide/zbs_multiactive_user_guide_28)阶段配置了仲裁节点的块存储集群管理 IP，则还需选择该 IP 作为其管理 IP。
6. 为集群内主机统一设置 root 账户和 smartx 账户的密码。
7. 单击**配置存储**，进入下一步。

## 部署 SMTX ZBS 双活集群 > 第 4 步：配置存储

# 第 4 步：配置存储

1. 根据集群要采用的存储分层或不分层模式，选择**分层**或**不分层**。

   系统会根据所选的模式和物理盘类型自动选择物理盘用途。

   > **注意**：
   >
   > 只有选择存储**分层**模式，才能支持启用常驻缓存模式。
2. （可选）当采用存储分层模式部署时，若集群内所有节点均为单一类型 SSD 全闪配置，且将 SMTX ZBS 块存储安装在软 RAID 1 中，可选择系统盘用途。

   - 如果节点为单一属性 SSD 全闪配置，建议选择系统盘用途为默认的**含元数据分区的数据盘**。
   - 如果节点为多种属性 SSD 全闪配置，建议选择系统盘用途为**含元数据分区的缓存盘**。
3. 当集群节点满足多物理盘池的配置要求时，系统会根据节点的存储配置自动选择推荐的物理盘池数量，并分配物理盘及其用途，您也可以手动修改物理盘池数量，建议不同节点设置相同的物理盘池数量。

   > **注意**：
   >
   > - 单集群最多支持配置 255 个物理盘池。
   > - 单物理盘池的数据盘总容量上限为 256 TB，缓存盘总容量上限为 51 TB。
   > - 存储分层模式下：
   >   - 混闪配置、多种类型 SSD 全闪配置，或单一类型多种属性 SSD 全闪配置，单物理盘池的物理盘数量上限为 64；
   >   - 单一类型且单一属性 SSD 全闪配置，单物理盘池的物理盘数量上限为 32。
   > - 存储不分层模式下：单物理盘池的物理盘数量上限为 32。
   > - 集群部署完成后，不支持修改物理盘池数量。
4. 参考在确认[存储设备配置要求](/smtxzbs/5.7.0/multiactive_user_guide/zbs_multiactive_user_guide/zbs_multiactive_user_guide_83)时已记录的每块物理盘的用途，确认系统自动选择的物理盘用途。

   - **SMTX 系统盘**：即[将 SMTX ZBS 块存储安装在独立的硬 RAID 1 中](/smtxzbs/5.7.0/multiactive_user_guide/zbs_multiactive_user_guide/zbs_multiactive_user_guide_84)的步骤 4 所选择的物理盘，不属于任何物理盘池，不可修改用途。
   - **含元数据分区的缓存盘**和**含元数据分区的数据盘**：即[将 SMTX ZBS 块存储安装在软 RAID 1 中](/smtxzbs/5.7.0/multiactive_user_guide/zbs_multiactive_user_guide/zbs_multiactive_user_guide_85)的步骤 5 所选择的物理盘，不可修改用途，可移至其他物理盘池。
   - **其余物理盘**

     存储分层模式部署时：

     - 若节点为混闪配置，一般只有当系统判断的物理盘类型有误时，才需要手动修改用途，可将物理盘移至其他物理盘池或选择**不挂载**。
     - 若节点为多种类型 SSD 全闪配置，当计划预留常驻缓存比例高于 75% 时，需要手动将所有**数据盘**用途修改为**缓存盘**；其余情况下，只有当系统判断的物理盘类型有误时，才需要手动修改物理盘用途。可将物理盘移至其他物理盘池或选择**不挂载**。
     - 若节点为单一类型且多种属性的 SSD 全闪配置，物理盘默认为**数据盘**，需手动将高性能 SSD 的用途修改为**缓存盘**；当计划预留常驻缓存比例高于 75% 时，需要手动将所有**数据盘**的用途修改为**缓存盘**。可将物理盘移至其他物理盘池或选择**不挂载**。
     - 若节点为单一类型且单一属性的 SSD 全闪配置，物理盘默认为**数据盘**，无需修改用途，可将物理盘移至其他物理盘池或选择**不挂载**。

     存储不分层模式部署时：所有物理盘默认为**数据盘**，不可修改用途，可移至其他物理盘池或选择**不挂载**。
5. 单击**配置网络**，进入下一步。

## 部署 SMTX ZBS 双活集群 > 第 5 步：配置网络

# 第 5 步：配置网络

## 创建虚拟分布式交换机

请参考如下步骤，分别为双活集群的每个系统网络（管理网络、存储网络和接入网络）创建单独的虚拟分布式交换机。

1. 在**虚拟分布式交换机**面板中，单击**创建虚拟分布式交换机**，为管理网络、存储网络或接入网络创建虚拟分布式交换机。
2. 填写虚拟分布式交换机的名称。
3. 在**关联网络**区域的**类型**选项中，选择系统网络的名称，例如`管理网络`、`存储网络`或`接入网络`。
4. （可选）当接入 SMTX ZBS 双活集群的某个网络的物理交换机为其配置了 VLAN 时，请在 **VLAN ID** 字段下填写实际规划的 VLAN ID。
5. 选择不启用 Q0S 策略，保持**开启 Q0S 策略**按钮为关闭状态。
6. 在**物理网口**区域的**关联网口**字段下，给每个优先/次级可用域的主机选择用于该系统网络的物理网口。
   - 可以根据显示的网口速率、连接状态和 MAC 地址等判断合适的网口：

     - 管理网络所选的网口速率需为 1 GbE 及以上。
     - 存储网络、接入网络所选的网口速率需为 10 GbE 及以上。
   - 若各个主机的关联网口名称相同，可以在设置完一个主机的关联网口后，单击右侧的 **...** 图标，选择**选中其他主机上的同名网口**。
   - 当接入 SMTX ZBS 双活集群某个系统网络的物理交换机启用了 LACP 动态链路聚合时，请为需要绑定网口的主机设置**网口绑定模式**为 `balance-tcp`。

下图为管理网络虚拟分布式交换机的配置界面，此时管理网络未配置 VLAN，且未设置网口绑定模式。

![](https://cdn.smartx.com/internal-docs/assets/bd6c2df9/zbs_multiactive_user_guide_13.png)

## 配置系统网络信息

1. 在**管理、存储与接入 IP** 面板中，填写每个主机实际规划的块存储集群管理 IP 地址、存储 IP 地址和接入 IP 地址。
2. 在**管理网络**面板中，填写为管理网络实际规划的网关和子网掩码。
3. 在**存储网络**面板中，填写为存储网络实际规划的子网掩码。
4. 在**接入网络**面板中，填写为接入网络实际规划的子网掩码。
5. （可选）若仲裁节点与集群两个可用域里的主机之间 L2 层存储网络无法连通时，此处还需要填写规划好的存储网络的网关配置信息。存储网络的网关包含优先/次级可用域到仲裁节点网关，以及仲裁节点到优先/次级可用域网关。
6. （可选）当使用存储的客户端（计算端）和 SMTX ZBS 双活集群的接入 IP 不在同一个子网时，需要启用**静态路由**，并配置计算节点与双活集群接入网络的**静态路由**。

   如下图所示，需填写该客户端的 IP 地址和子网掩码，以及路由器的网关；若需要添加多个客户端的信息，可以单击 **+ 添加路由配置**进行添加。

   ![](https://cdn.smartx.com/internal-docs/assets/bd6c2df9/zbs_multiactive_user_guide_14.png)

> **注意**：
>
> SMTX ZBS 支持为集群的系统网络设置 MTU，其中编辑存储网络和接入网络的 MTU 将造成业务 I/O 较长时间中断。若需修改存储网络或接入网络的 MTU，建议在集群部署完成后至业务上线前的窗口期进行，以免对业务造成影响，具体请参考《SMTX ZBS 管理指南》的[编辑系统网络](/smtxzbs/5.7.0/zbs_administration_guide/zbs_administration_guide_057)小节进行操作。

## 配置 DNS 服务器和 NTP 服务器

1. 在 **DNS 服务器**面板中，填写 DNS 服务器地址。

   - 若有域名配置需求，请填写实际规划的 DNS 服务器地址。
   - 若无域名配置需求，保留默认地址 `127.0.0.1` 即可。
2. 在 **NTP 服务器**面板中，选择 NTP 服务器。

   - 若需要使用集群外部 NTP 服务器来同步集群内的主机时间，请选择**使用集群外部 NTP 服务器**，并填写实际规划的 NTP 服务器地址。
   - 若无使用集群外部 NTP 服务器的需求，请选择**使用集群内主机作为 NTP 服务器**。
   - 若计划将集群关联至已有的 CloudTower 且 CloudTower 配置了 NTP 服务器，请确保集群的 NTP 服务器与 CloudTower 的 NTP 服务器的时钟保持同步。
3. 单击**检查配置**，进入下一步操作。

## 设置参数 > 关联 CloudTower > 环境中已有 CloudTower

# 环境中已有 CloudTower

若当前环境中已部署 CloudTower，请在**关联 CloudTower** 对话框中选择**已有 CloudTower 环境**，并参考下述步骤设置后，登录 CloudTower 和关联 SMTX ZBS 双活集群。

**前提条件**

SMTX ZBS 要求双活集群的优先可用域与次级可用域分属两个不同的数据中心，请确保双活集群即将关联的 CloudTower 存在两个数据中心。可以在 CloudTower 主界面中单击右上方的 **+ 创建**，选择**创建数据中心**，即可为两个可用域创建新的数据中心。

**注意事项**

若现有的 CloudTower 管理平台版本低于当前 SMTX ZBS 所兼容的版本，则需升级 CloudTower 至兼容的版本。CloudTower 的升级方法，请参见对应版本的《CloudTower 安装与升级指南》。

**操作步骤**

1. 在**关联 CloudTower** 对话框中设置双活集群管理虚拟 IP 地址。设置双活集群的管理虚拟 IP 后，即可通过管理虚拟 IP 访问集群中可用的主机，从而保证管理界面的高可用。

   > **说明：**
   >
   > 必须确保 CloudTower 可以通过双活集群管理虚拟 IP 访问集群。管理虚拟 IP 与节点的块存储集群管理 IP 必须在同一个子网内。
2. 单击**下一步**。
3. 在浏览器中输入 CloudTower 的 IP 地址，使用超级管理员 **root** 账户登录 CloudTower。
4. 在 CloudTower 主界面右上角单击 **+ 创建 > 关联集群**，弹出**关联集群**对话框。

   ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_61.png)
5. 在**关联集群**对话框中按照要求填写 SMTX ZBS 双活集群的信息。完成后单击**下一步**。

   - **集群管理虚拟 IP**：填写集群的管理虚拟 IP。
   - **管理员用户名**：填写集群的超级管理员账户的用户名。
   - **管理员密码**：填写超级管理员账户对应的密码。
6. 单击**下一步**，开始加载集群的数据，获取到集群信息后，确认即将关联的集群的信息，包括集群的名称、管理虚拟 IP 和软件版本等。
7. 为双活集群的优先可用域和次级可用域分别选择其所属的数据中心。

   ![](https://cdn.smartx.com/internal-docs/assets/bd6c2df9/zbs_multiactive_user_guide_16.png)
8. 单击**关联集群**，成功将 CloudTower 与 SMTX ZBS 双活集群完成关联。

   > **注意**：
   >
   > - 完成集群关联后，如果发现该集群有错误或遗漏的配置而需要重新安装部署，必须先取消其与 CloudTower 的关联。
   > - 若 CloudTower 时区与集群时区不一致，建议手动调整时区使两者一致。

## 设置参数 > 关联 CloudTower > 安装全新的 CloudTower 并关联集群

# 安装全新的 CloudTower 并关联集群

若当前环境中没有 CloudTower，请在**关联 CloudTower** 对话框中直接选择**全新 CloudTower 环境**，系统将自动为 CloudTower 创建虚拟机，并启动安装 CloudTower。安装完后可以关联双活集群。

**前提条件**

- 根据服务器的 CPU 架构提前下载对应的 `tar.gz` 格式的 CloudTower 安装文件。
- 已规划 CloudTower 的 IP 地址、子网掩码和网关。确保 CloudTower IP 与双活集群的管理虚拟 IP 正常连通。

**注意事项**

在集群中安装完 CloudTower 后，请勿在 CloudTower 虚拟机中安装第三方插件，以免影响 CloudTower 虚拟机的稳定运行。

**操作步骤**

1. 在**关联 CloudTower** 对话框中设置集群管理虚拟 IP 地址。设置集群的管理虚拟 IP 后，即可通过管理虚拟 IP 访问集群中可用的主机，从而保证管理界面的高可用。

   > **说明：**
   >
   > 必须确保 CloudTower 可以通过集群管理虚拟 IP 访问集群。管理虚拟 IP 与节点的块存储集群管理 IP 必须在同一个子网内。
2. 单击**下一步**。
3. 上传 CloudTower 的安装映像文件。将本地 `tar.gz` 格式的 CloudTower 安装文件拖拽至文件区域，或者单击**选择文件**，选择从本地上传文件。
4. 选择 CloudTower 环境的配置。选择**低配**或**高配**时，需考虑待管理的集群数量和主机规模，并确保集群有足够的资源（CPU、内存和存储空间）可分配给部署 CloudTower 的虚拟机（如下表所示）。

   | **配置等级** | **CloudTower 管理能力** | **CloudTower 虚拟机配置** |
   | --- | --- | --- |
   | **低配** | - 10 个集群 - 100 台主机 - 1000 个虚拟机 | - 4 核 vCPU - 12 GiB 内存 - 100 GiB 存储空间 |
   | **高配** | - 100 个集群 - 1000 台主机 - 10000 个虚拟机 | - 8 核 vCPU - 19 GiB 内存 - 400 GiB 存储空间 |
5. 设置 CloudTower 的 IP 地址和子网掩码、网关信息。

   > **说明：**
   >
   > CloudTower 的 IP 地址必须与双活集群的管理虚拟 IP 正常连通。
6. 设置 CloudTower 的组织名称，以及超级管理员 **root** 的密码。
7. 单击**下一步**，开始安装 CloudTower。安装完成后，系统将弹出提示框，单击**完成**。大约 15 秒后系统提示即将离开此网站。单击**离开**后将自动跳转至 CloudTower 环境。
8. 在 CloudTower 中关联 SMTX ZBS 双活集群。

   1. 在浏览器中输入 CloudTower 的 IP 地址，使用超级管理员 **root** 账户登录 CloudTower。
   2. 在 CloudTower 主界面中单击右上方的 **+ 创建**，选择**创建数据中心**，分别为优先可用域和次级可用域分别创建其对应的数据中心。
   3. 在 CloudTower 主界面右上角单击 **+ 创建**，选择**关联集群**，弹出**关联集群**对话框。

      ![](https://cdn.smartx.com/internal-docs/assets/9dd8a4af/zbs_installation_guide_61.png)
   4. 在**关联集群**对话框中按照要求填写 SMTX ZBS 双活集群的信息。完成后单击**下一步**。

      - **集群管理虚拟 IP**：填写集群的管理虚拟 IP。
      - **管理员用户名**：填写集群的超级管理员账户的用户名。
      - **管理员密码**：填写超级管理员账户对应的密码。
   5. 单击**下一步**，开始加载集群的数据，获取到集群信息后，确认即将关联的集群的信息，包括集群的名称、管理虚拟 IP、软件版本、以及使用的虚拟化平台等。
   6. 为双活集群的优先可用域和次级可用域分别选择其所属的数据中心。

      ![](https://cdn.smartx.com/internal-docs/assets/bd6c2df9/zbs_multiactive_user_guide_16.png)
   7. 单击**关联集群**，成功将 CloudTower 与 SMTX ZBS 双活集群完成关联。

      > **注意**：
      >
      > - 完成集群关联后，如果发现该集群有错误或遗漏的配置而需要重新安装部署，必须先取消其与 CloudTower 的关联。
      > - 若 CloudTower 时区与集群时区不一致，建议手动调整时区使两者一致。

## 为主节点配置 Meta 服务的优先级

# 为主节点配置 Meta 服务的优先级

在将双活集群的主机分配至优先可用域和次级可用域后，还需要对每个主节点配置 Meta 服务的优先级。

1. 登录主节点，执行 `$cat /etc/zbs/zbs.conf` 命令查看配置文件。通过其 `members` 字段查找各节点的 IP 地址列表。

   ```
   [root@m-31-176 17:48:06 ~]$cat /etc/zbs/zbs.conf
   [network]
   data_ip = 10.10.31.176
   heartbeat_ip = 10.10.31.176
   vm_ip = 192.168.31.176
   web_ip = 192.168.31.176

   [cluster]
   role = master
   members = 10.10.25.47,10.10.31.176,10.10.29.172,10.10.29.232
   zookeeper = 10.10.25.47:2181,10.10.31.176:2181,10.10.29.172:2181,10.10.29.232:2181,10.10.28.17:2181
   mongo = 10.10.25.47:27017,10.10.31.176:27017,10.10.29.172:27017,10.10.29.232:27017,10.10.28.17:27017
   cluster_mgt_ips = 192.168.23.131,192.168.25.47,192.168.29.172,192.168.29.232,192.168.31.176,192.168.31.204
   cluster_storage_ips = 10.10.23.131,10.10.25.47,10.10.29.172,10.10.29.232,10.10.31.176,10.10.31.204

   [zone]
   zone_ip_map = [{"zone_id": "924da84b-3afc-4b0d-bceb-eb761eec2caf", "member_ips": ["10.10.29.172", "10.10.29.232", "10.10.31.204"]}, {"zone_id": "default",   "member_ips": ["10.10.23.131", "10.10.25.47", "10.10.31.176"]}]
   ```
2. 配置主节点的 Meta 服务优先级。

   根据可用域拓扑配置，`members` 中位于优先可用域的节点，其优先级为 2；位于次级可用域的节点，其优先级为 1。若准备将 `10.10.25.47` 和 `10.10.31.176` 配置在优先可用域 `10.10.29.172` 和 `10.10.29.232` 配置在次级可用域，则可以通过执行以下命令实现：

   `$zbs-tool service set_priority meta 10.10.25.47:10100:2,10.10.31.176:10100:2,10.10.29.172:10100:1,10.10.29.232:10100:1`

   查看主节点的优先级配置结果。

   ```
   [root@m-31-176 17:51:34 ~]$zbs-tool service show meta
   -----------------  -----------------------------------------------------------------------------------------
   Name               meta
   Members            ['10.10.25.47:10100', '10.10.29.172:10100', '10.10.29.232:10100', '10.10.31.176:10100']
   Leader             10.10.25.47:10100
   Specified members  ['[10.10.25.47:10100', '10.10.31.176:10100', '10.10.29.172:10100', '10.10.29.232:10100]']
   Priority           10.10.25.47:10100:2,10.10.31.176:10100:2,10.10.29.172:10100:1,10.10.29.232:10100:1
   Current priority
   -----------------  -----------------------------------------------------------------------------------------
   ```

   配置完成后，Meta Leader 将始终在活跃节点里优先级较高的节点中产生。

## 部署后配置

# 部署后配置

完成双活集群部署后，必须通过配置硬件拓扑来确保数据副本跨可用域分布，这是实现故障切换能力的核心基础；同时，还需对 VMware vSphere 集群进行规划，并在 ESXi 主机上完成系列配置，从而保障在单一数据中心发生故障时，虚拟机与存储服务能由另一站点无缝接管，最终使双活集群的高可用性得以充分发挥。

## 部署后配置 > 配置双活集群的硬件拓扑

# 配置双活集群的硬件拓扑

构建双活集群必须启用 SMTX ZBS 的机架拓扑感知功能，不仅在规划时需要考虑尽量将服务器平均分布在不同的机架中，在集群部署结束后，还需要使用 CloudTower 的机架配置功能，如实呈现安装部署现场的主机与机架的位置和拓扑结构，为每个可用域添加机架和机箱，并将主机分配至可用域中。配置完成后，系统才能根据拓扑来调整数据的分配情况。

具体操作步骤请参考《SMTX ZBS 管理指南》的[配置机架拓扑](/smtxzbs/5.7.0/zbs_administration_guide/zbs_administration_guide_097)小节。

## 部署后配置 > 配置 ESXi 主机

# 配置 ESXi 主机

使用 SMTX ZBS 双活集群为 VMware vSphere 集群（客户端）提供块存储服务时，需要对 VMware vSphere 集群进行规划，并在 ESXi 主机上完成系列配置，利用 VMware vSphere 集群的 High Availability （vSphere HA）能力和指定虚拟机与主机间的规则，以保证在优先/次级可用域所在的数据中心故障后，另一个数据中心仍可以继续提供虚拟机和存储服务。

下面以两个集群的典型组网为例，描述在 SMTX ZBS 双活集群部署完成后，如何在 VMware vSphere 集群的 ESXi 主机上配置，以支持使用 SMTX ZBS 双活集群为 VMware vSphere 集群提供块存储服务。

![](https://cdn.smartx.com/internal-docs/assets/bd6c2df9/zbs_multiactive_user_guide_05.png)

## 部署后配置 > 配置 ESXi 主机 > 规划和调整 ESXi 主机的位置

# 规划和调整 ESXi 主机的位置

SMTX ZBS 双活集群的优先可用域和次级可用域隶属于两个不同的数据中心。为了保证在双活集群的其中一个可用域发生故障后，故障区域中 ESXi 主机上的虚拟机能 HA 到另一个存活的可用域的 ESXi 主机上，需要对 VMware vSphere 集群的所有 ESXi 主机进行规划并调整其所属的数据中心。

1. 将 VMware vSphere 集群的所有 ESXi 主机按照 SMTX ZBS 双活集群的优先可用域和次级可用域进行划分，每个可用域的 ESXi 主机为一组，其中运行着虚拟机的 ESXi 主机全部放进优先可用域所对应的组，记录每个 ESXi 主机所属的可用域。
2. 将属于优先/次级可用域所在数据中心的 ESXi 主机移动至对应的数据中心的机架中。

## 部署后配置 > 配置 ESXi 主机 > 创建 ESXi 主机组和虚拟机组

# 创建 ESXi 主机组和虚拟机组

将 VMware vSphere 集群的所有 ESXi 主机划分为两组，分别对应双活集群的优先可用域和次级可用域，需要在 VMware vCenter 中创建一个主机组，将优先可用域所在的数据中心的 ESXi 主机全部添加进主机组中；同时将集群中所有虚拟机添加进一个虚拟机组中。

1. 登录 vSphere Web Client，浏览并选中 VMware vSphere 集群。
2. 单击**配置**选项卡，选择**配置** > **虚拟机/主机组**。
3. 创建一个主机组 `Host-Group1`，对应双活集群的优先可用域。

   1. 单击**添加**按钮，填写主机组的名称，例如 `Host-Group1`，并在**类型**下拉菜单中选择**主机组**。
   2. 单击**添加**按钮，在弹出的**添加组成员**对话框中，勾选优先可用域对应的组内的所有 ESXi 主机。
   3. 单击**确定**，完成主机的添加。
   4. 单击**确定**，完成主机组的创建。
4. 创建一个虚拟机组 `VM-Group1`，对应 VMware vSphere 集群的所有虚拟机。

   1. 单击**添加**按钮，填写虚拟机组的名称，例如 `VM-Group1`，并在**类型**下拉菜单中选择**虚拟机组**。
   2. 单击**添加**按钮，在弹出的**添加组成员**对话框中，勾选集群内所有 ESXi 主机上的虚拟机。
   3. 单击**确定**，完成虚拟机的添加。
   4. 单击**确定**，完成虚拟机组的创建。

## 部署后配置 > 配置 ESXi 主机 > 启用 vSphere HA

# 启用 vSphere HA

启用 vSphere HA 后，可以结合后续设置的虚拟机与主机间的规则，控制 VMware vSphere 集群内 ESXi 主机上的虚拟机的放置位置，从而使得虚拟机在触发 HA 时，遵照规则调度到满足要求的主机上开机，以保持业务的连续性。

1. 使用 VMware vSphere Web Client 连接到 vCenter，浏览到 ESXi 主机。
2. 选择**配置** > **系统** > **高级系统设置**，并单击**编辑**按钮。在弹出的**编辑高级系统设置**参数表中，将 **Misc.APDHandlingEnable** 参数的值修改为 `1`。
3. 参考上述步骤，将集群中每个 ESXi 主机的 **Misc.APDHandlingEnable** 参数的值修改为 `1`。
4. 在 vCenter 中选中集群，选择**配置** > **服务** > **vSphere 可用性**，在 vSphere 可用性设置界面右侧选择**编辑**，然后在弹出的**编辑集群设置**对话框中开启 **vSphere HA** 选项。
5. 在**故障和响应**页面，进行以下设置：

   1. 开启**启用主机监控**选项。
   2. 将**主机故障响应**选项设置为**重新启动虚拟机**。
   3. 将**针对主机隔离的响应**选项设置为**关闭虚拟机电源再重新启动虚拟机**。
   4. 将**处于 PDL 状态的数据存储**选项设置为**禁用**。
   5. 选中并展开**处于 APD 状态的数据存储**，将**全部路径异常 (APD) 故障响应**选项设置为**关闭虚拟机电源并重新启动虚拟机 - 保守的重新启动策略**，将**响应恢复**选项设置为**禁用**，再将**响应延迟**的值设置为 `3` 分钟。
6. 单击**确定**。

## 部署后配置 > 配置 ESXi 主机 > 为虚拟机创建虚拟机/主机规则

# 为虚拟机创建虚拟机/主机规则

为了控制 VMware vSphere 集群中虚拟机的放置位置，需在 VMware vCenter 中为虚拟机设置虚拟机到主机的规则。

**虚拟机/主机规则**

VMware vSphere 集群提供了以下 4 组虚拟机与主机间关联性规则，可设置选定的虚拟机组的成员允许或者不允许在特定的主机组的成员上运行。

- `必须在组中的主机上运行`

  该策略为**必须**策略，虚拟机组中的虚拟机必须在主机组中的主机上运行。若主机组内的所有主机均不可用，vSphere HA 功能受限于此条规则中的**必须**性的强制要求，将无法将虚拟机自动迁移到主机组外的其他主机上，从而导致业务中断。
- `应在组中的主机上运行`

  该策略为**应当**策略，虚拟机组中的虚拟机应当（优先但不是必须）在指定的主机组中的主机上运行。若指定的主机组的所有主机均不可用，虚拟机组中的虚拟机还可以 HA 至其他主机上运行。
- `不得在组中的主机上运行`

  该策略为**不得在**策略，虚拟机组中的虚拟机不允许在指定的主机组中的主机上运行。若除主机组外的其他主机均不可用，vSphere HA 功能受限于此条规则中的**不得在**的强制性要求，无法将虚拟机自动迁移到指定主机组的主机上，从而导致业务中断。
- `不应在组中的主机上运行`

  该策略为**不应在**策略，虚拟机组中的虚拟机不应当（但可以）在主机组的主机上运行。若虚拟机所在的主机不可用，虚拟机可以优先迁移到指定主机组以外的其他主机上。由于是**不应在**策略，而非强制性要求，因此即使指定主机组以外的其他主机均不可用，系统还可以将虚拟机自动迁移到指定主机组的主机上，使得业务的连续性得以保障。

**操作步骤**

1. 使用 VMware vSphere Web Client 连接到 vCenter，浏览到 VMware vSphere 集群。
2. 单击**配置**选项卡，选择**配置** > **虚拟机/主机规则**，弹出**创建虚拟机/主机规则**对话框。
3. 单击**添加**按钮，弹出**创建虚拟机/主机规则**对话框。
4. 填写虚拟机到主机规则的名称，例如 `VM_Host_Rules1`，在**类型**下拉菜单中选择**虚拟机到主机**，在**虚拟机组**和**主机组**下拉菜单中选择对应的组 `VM-Group1` 和 `Host-Group1`。

   若无特殊的业务需求限制，建议选择**应当**策略，这样既可以满足配置要求，也能在故障场景发生时完成故障切换需要，保持业务的连续。如果您配置为**必须**策略，在站点故障时，由于规则的限制，可能会无法切换至合适的主机，从而导致业务被中断。
5. 单击**确定**。

## 部署后配置 > 配置 ESXi 主机 > 开启 ESXi 主机的 vMotion 服务

# 开启 ESXi 主机的 vMotion 服务

设置 VMware vSphere 集群内虚拟机与主机间的放置规则后，执行这些规则可能需要对虚拟机进行迁移，因此必须为集群中的 ESXi 主机开启 vMotion 服务。

1. 使用 VMware vSphere Web Client 连接到 vCenter，浏览 VMware vSphere 集群并选中一台 ESXi 主机。
2. 单击**配置**选项卡，选择**网络** > **VMkernel 适配器**。
3. 在 **VMkernel 适配器**列表中选中数据网络的适配器，单击**编辑...** 按钮。
4. 在**端口属性**页签的**可用服务**选项中，勾选 **vMotion** 前的复选框。
5. 单击 **OK**。
6. 参考步骤 1 ~ 5，对 VMware vSphere 集群中的所有 ESXi 主机启用 vMotion 服务。

## SMTX ZBS 节点的本地账户与默认安全设置说明 > SMTX ZBS 节点的本地账户说明

# SMTX ZBS 节点的本地账户说明

| root | 默认账户，默认禁止 SSH 远程登录。您可以使用该账户管理节点。 | - 请勿删除该账户。 - 强烈建议不要开启 root 账户通过 SSH 远程登录的权限。 - 以命令行方式升级集群时必须拥有 root 权限。 |
| smartx | 默认账户，默认支持 SSH 远程登录。您可以使用该账户通过 SSH 协议远程连接到节点。 | - 请勿删除该账户。 - 请勿更改 `/etc/sudoers` 里针对该账户的配置。 - 以该账户登录节点后，必须切换为 root 账户再执行运维操作。执行 `sudo su` 命令可免密码切换。 |
| 其他本地账户 | 由 root 账户创建的其他账户，账户权限由 root 账户决定。您可自行创建新的本地账户用于 SSH 远程登录。 | - |

# SMTX ZBS 节点的默认安全设置说明

SMTX ZBS 块存储系统安装完成后，会默认对节点进行如下安全设置。

| **设置项** | **默认设置说明** |
| --- | --- |
| 连接超时设置 | 登录节点后，如果在 30 分钟内未进行操作，与节点的连接将自动断开。 |
| 密码强度设置 | 除 root 和初始 smartx 外的账户设置密码时：  - 密码长度必须至少为 8 个字符。 - 密码必须至少包含 1 个数字、1 个大写字母、1 个小写字母和 1 个特殊字符。 - 重设的密码不能与前 5 次的密码重复。 |
| 登录失败后锁定设置 | 连续 5 次登录失败后，账户将被锁定 10 分钟。 |
| SSH 设置 | - 禁止 root 账户直接 SSH 远程登录节点。 - SSH 服务的 MACs 和 Ciphers 配置支持 CTR 和 SHA2 算法，不支持 CBC、MD5 和 SHA1 算法。 |
