## 关于本文档

# 关于本文档

SMTX 备份与容灾是由北京志凌海纳科技股份有限公司（以下简称 “SmartX”）自主研发的数据保护产品，可为运行在 SMTX OS（ELF）集群的虚拟机和 SMTX ZBS 块存储集群的存储卷提供全面的数据保护和灾难恢复解决方案。

同步复制与恢复主要提供以下功能：

- **同步复制**

  将运行在源集群中的保护对象同步复制到目标集群中，在目标集群创建对应的副本对象，并在网络无故障的情况下保证副本对象不丢失任何增量数据，确保数据的安全性和可用性。
- **故障转移**

  当保护对象发生故障时，可使用副本对象接替保护对象继续提供服务。
- **故障转移测试**

  在测试环境中模拟故障转移场景，通过创建测试虚拟机复现真实故障切换流程，在不影响实际保护对象的前提下，验证故障转移方案的可用性与可靠性。
- **故障恢复**

  保护对象在发生故障转移后，当其恢复正常时，将故障转移期间副本对象产生的数据增量传输回保护对象，并恢复至由保护对象提供服务。故障恢复可以确保故障转移期间产生的数据变化不会丢失，并确保保护对象在恢复后业务能够无缝切换回原始的生产环境，保障业务的连续性。
- **永久故障转移**

  在保护对象无法恢复的情况下，将保护对象永久切换到副本对象，并使用副本对象持续提供服务。

## 概述 > 应用场景

# 应用场景

同步复制与恢复可应用于以下场景：

- **灾难恢复**

  防止因自然或人为灾害、勒索软件攻击、恐怖事件等导致的大规模数据损坏和丢失。一旦灾难发生，可快速将业务切换至灾备集群，并迅速恢复业务运行。
- **灾备演练**

  同步复制与恢复可用于真实灾备演练和模拟灾备演练：

  - 真实灾备演练：通过故障转移与故障恢复功能进行，用于在接近真实灾难的条件下验证灾备接管能力。演练过程中存在业务停机时间，业务将由副本虚拟机接管；演练结束后，可将演练期间产生的数据同步回源端虚拟机，恢复正常生产运行。
  - 模拟灾备演练：通过故障转移测试功能进行，用于在不影响生产业务的情况下验证灾备方案。演练过程中，生产虚拟机保持正常运行，演练数据不会同步回源端虚拟机，不会对生产环境造成影响。
- **数据复用**

  利用集群原生的快照重建功能实现数据复用。通过从副本虚拟机的快照重建虚拟机进行开发、测试、故障排除等活动，避免对生产环境造成任何影响。数据复用功能能够在不影响主要业务的前提下，为开发人员和测试团队提供高效且隔离的环境，加快开发和测试过程，同时确保生产环境的稳定性和安全性。

## 相关概念

# 相关概念

本章解释了手册中使用的专业术语。在阅读过程中，如果遇到不熟悉的概念，可在此快速查找其定义和详细解释。

**复制服务**

复制服务负责管理同步复制计划，并执行虚拟机的同步复制、故障转移、故障转移测试、故障恢复和永久故障转移等任务调度。复制服务负责保障复制对象的数据一致性。

**站点**

站点是指由一个 CloudTower 管理的一个故障域。一个站点可以包含一个或多个集群和数据中心。每个站点在地理位置、硬件设备、管理平台和软件服务等方面都完全隔离，且不共享任何资源。

站点可以分为以下两种类型：

- **源站点**：源站点通常是实际运行业务的站点。复制服务将源站点的虚拟机数据复制到目标站点的副本对象，实现数据的同步。
- **目标站点**：目标站点通常是用于容灾的站点，用于存储从源站点复制过来的副本对象。

**站内复制**

源集群和目标集群位于同一站点。当原虚拟机不可用时，可切换使用站点内其他集群中的副本虚拟机，实现集群级别的容灾。若管理站点的 CloudTower 发生故障，无法发起故障转移。

**跨站点复制**

源集群和目标集群位于不同站点。当源站点内的原虚拟机不可用时，无论管理源站点的 CloudTower 是否发生故障，均可切换至目标站点的副本虚拟机，实现站点级别的容灾。

跨站点复制包括以下两种类型：

- **入站复制**：目标站点为当前站点，源站点为其他关联站点。
- **出站复制**：源站点为当前站点，目标站点为其他关联站点。

**源集群**

通常为生产集群，承载实际运行的业务应用和数据，是原虚拟机所属的集群，位于源站点。

**目标集群**

通常为用于容灾的集群，用于存储从源集群复制过来的副本对象，位于目标站点。

**保护对象**

作为复制数据源的原始对象。进行虚拟机复制时，保护对象通常是运行的生产环境中的虚拟机，负责承载实际的业务应用和数据。

**副本对象**

通过同步复制计划创建的保护对象的完整副本，包含和保护对象相同的数据和配置。副本对象通常用于灾难恢复和容灾目的，在保护对象发生故障时，它可以快速接替保护对象，确保业务的连续性和可用性。副本对象通常存储在用于容灾的目标集群。

**同步复制计划**

用于规划和安排数据的复制操作。通过同步复制计划，用户可以定义复制对象、复制方向、目标集群、复制服务、同步策略、副本虚拟机和测试虚拟机设置等信息。

**复制对象**

一个保护对象和其对应的副本对象称为一组复制对象。进行虚拟机复制时，一个原虚拟机及其副本虚拟机称为一组复制对象。

**恢复点创建周期**

同步复制任务自动创建恢复点的周期，用于指定复制对象进入`已同步`状态后，创建恢复点的时间间隔。

**恢复点**

恢复点是在每次为复制对象执行异步复制、同步复制（已同步状态下周期性创建）、故障转移（立即生成恢复点模式）、故障转移测试（立即生成恢复点模式）或故障恢复时生成的数据状态，以快照形式保存在复制对象上。恢复点记录了复制对象的数据和配置，反映了复制对象在执行上述操作时的状态。

恢复点可以使复制对象在需要时快速回滚到特定的数据状态，例如在灾难恢复场景中将副本对象恢复到某个先前的时间点，以恢复副本对象的正常运行，保障业务的连续性和可用性。

**故障处理**

故障发生后可进行的恢复操作，包括故障转移、故障转移测试、故障恢复和永久故障转移。

- **故障转移**

  故障转移是一种应对保护对象不可用的操作。当保护对象不可用时，可以将副本对象回滚到与指定恢复点一致的状态，并切换至使用副本对象。通过故障转移，系统可以在短时间内将服务快速恢复到故障发生前的状态，保障业务的连续性和稳定性。
- **故障转移测试**

  故障转移测试是一种模拟容灾演练的操作，在不中断源端生产业务的前提下，通过生成测试虚拟机，在测试虚拟机验证恢复机制的有效性。演练过程中的增量数据不会同步回被保护的虚拟机。
- **故障恢复**

  故障恢复是指当发生故障的保护对象恢复正常使用后，副本对象将故障转移期间变化的数据回传，然后切换回使用保护对象。故障恢复旨在将复制对象之间的数据进行同步，确保保护对象可以在故障解决后继续提供服务。
- **永久故障转移**

  永久故障转移是当保护对象无法从故障中恢复时采取的措施。在保护对象发生永久性故障时，副本对象将正式接替保护对象的服务角色，以确保业务的持续运行和可用性。

**RPO** (Recovery Point Objective)

数据恢复点目标，主要指在故障转移场景中允许的数据损失的时间范围。RPO 以时间为度量单位，标示了从最近执行的复制时间点到发生故障时的时间跨度。较小的 RPO 值表示系统要求在发生故障时允许的数据损失较少。较大的 RPO 值意味着在故障发生时系统可以容忍较长时间范围内的数据丢失。

**RTO** (Recovery Time Objective)

数据恢复时间目标。主要指业务系统从崩溃到恢复运行所能容忍的时间。较短的 RTO 值表示业务系统要求在发生故障时能够迅速恢复，较长的 RTO 值则意味着业务系统可以容忍更长的恢复时间。

**VPC**（Virtual Private Cloud）

虚拟专有云，是一套为云资源构建的逻辑隔离的网络环境，用户可以在 VPC 中自主设置 VPC 子网、IP 地址、路由表、浮动 IP、网关服务和安全服务等，以实现网络隔离和安全控制。副本虚拟机和原虚拟机可以部署在相同或不同的 VPC 中，并支持灵活的网络映射。测试虚拟机需部署在和原虚拟机不同的 VPC 中。

**VPC 子网**

VPC 子网是组成 VPC 的基础网络模块，用于划分 VPC 的 IP 地址空间。位于同一 VPC 子网的虚拟机默认可以直接通信，跨 VPC 子网虚拟机在在路由可达的情况下也可以互相通信。

**浮动 IP**

浮动 IP 是一种网络地址转换（NAT）机制，通过一对一映射的方式，将外部子网中的 IP 地址动态绑定到 VPC 内部的虚拟机上。这使得外部网络可以主动访问 VPC 内的虚拟机。浮动 IP 为 VPC 内的虚拟机提供了灵活的外部访问能力，同时也保证了网络安全性。

**网络映射**

旨在解决原虚拟机和副本虚拟机/测试虚拟机之间所处的网络环境差异问题的一项配置规则。

由于副本虚拟机/测试虚拟机和原虚拟机往往处于不同的网络环境，若副本虚拟机/测试虚拟机完全复制原虚拟机的网络配置，可能无法在目标集群环境正常配置或连通网络。通过配置网络映射规则，可将原虚拟机所属的虚拟机网络映射到目标集群中对应的虚拟机网络，从而保证副本虚拟机/测试虚拟机能够正常与网络相连。

**IP 映射**

旨在解决原虚拟机和副本虚拟机/测试虚拟机在不同网络环境下的 IP 地址不匹配的问题的一项配置规则。

由于副本虚拟机/测试虚拟机和原虚拟机往往处于不同的网络环境，若副本虚拟机/测试虚拟机完全复制原虚拟机的 IP 地址，可能无法在目标集群正常联通网络。通过配置 IP 映射规则，在故障转移和故障转移测试时，复制服务可以自动为副本虚拟机/测试虚拟机设置正确的 IP 地址，确保副本虚拟机/测试虚拟机可以按照预期的网络配置提供服务，无需手动去更新虚拟机 IP。

**ELF**

ELF 平台即指 SMTX OS 原生虚拟化计算平台，是当 SMTX OS 以超融合形态直接部署在 x86\_64/AArch64 架构的服务器上时，该超融合软件中基于 KVM 的虚拟化引擎组件。ELF 除提供虚拟机生命周期管理、电源操作、高可用、冷热迁移等基本功能外，通过与存储组件 ZBS 相结合，还能提供秒级快照、模板和克隆等高级的虚拟机服务。

ELF 简单易用，对 SMTX OS 超融合集群的所有操作均可通过 CloudTower 完成。

**CloudTower**

SmartX 多集群资源集中管理平台，旨在管理站点内数据中心及跨数据中心的多个 SmartX 集群和系统服务，并提供完整和标准的 RESTful API 和多种语言的 SDK。

## 设计方案 > 方案概要

# 方案概要

SMTX 备份与容灾的设计方案采用了 Kubernetes Operator 模式，遵循声明式编程（declarative programming）范式，只需对复制相关资源设定期望的状态，而无需描述其实现流程，系统即能自行确定需要执行的操作以达到用户期望的状态。

Kubernetes Operator 模式包含了资源（Resource）和控制器（Controller）两个基本概念：

- **资源**

  Kubernetes 将被管理的对象抽象为“资源”。资源包含两种状态：期望状态和实际状态。期望状态通常由用户进行定义，实际状态通常由控制器负责更新和维护。

  Kubernetes 系统通过控制器监视资源对象，并执行业务逻辑，让资源的实际状态最终达到期望状态，从而达成用户的预期结果。

  Kubernetes 提供了多种默认的资源类型，也支持用户创建自定义资源（Custom Resource，简称 CR）。

  复制与恢复模块包含以下自定义资源：

  | 英文名称 | 中文名称 | 含义 |
  | --- | --- | --- |
  | ReplicationGroup | 复制组 | 一个保护对象对应一个 ReplicationGroup，一个 ReplicationGroup 包含一个 ReplicationSlice。 |
  | ReplicationSlice | 复制恢复点链 | 代表一个保护对象的复制恢复点链。一个复制恢复点链由一个全量恢复点和多个增量恢复点组成。 |
  | ReplicationRestorePoint | 复制恢复点 | 一个可以用于故障转移的基本对象，记录了保护对象的数据和配置，反映了保护对象在复制时的状态。 |
  | MultiStorageConsistentSnapshot | 数据一致快照组 | 由多个在存储上数据一致的快照组成的组。 |
  | SyncReplicationPlan | 同步复制计划 | 包含保护对象、恢复点创建周期、保留策略和副本对象设置。 |
  | SyncReplicationJob | 同步复制任务 | 代表一个保护对象的同步复制任务。 |
  | ReplicationJob | 异步复制任务 | 代表一个保护对象的异步复制任务。 |
  | ReplicaGroup | 复制对象副本 | 代表一个故障转移或故障转移测试中的副本对象。 |
  | FailoverExecution | 故障转移或故障转移测试执行 | 手动触发的一次故障转移或故障转移测试执行操作，包含多个 SyncReplicationFailoverJob、BootJob 和 ReIPJob。 |
  | SyncReplicationFailoverJob | 复制组同步复制故障转移或故障转移测试任务 | 代表一个保护对象的同步复制故障转移或故障转移测试任务。 |
  | BootJob | 开机任务 | 代表一个或多个虚拟机的开机任务。 |
  | ReIPJob | IP 映射任务 | 代表一个或多个虚拟机的 IP 映射任务。 |
  | FailbackExecution | 故障恢复执行 | 手动触发的一次故障恢复执行操作，包含多个 FailbackJob，保护对象为虚拟机时还包含 BootJob 和 ReIPJob。 |
  | FailbackJob | 故障恢复任务 | 代表一个副本对象的故障恢复任务。 |
  | FailbackReplicationJob | 恢复回传任务 | 代表一个副本对象的恢复回传任务，是故障恢复任务的一部分。 |
  | PermanentFailoverJob | 永久故障转移任务 | 代表一个副本对象的永久故障转移任务。 |
- **控制器**

  针对每一种资源，复制服务均有与之对应的控制器。控制器会监听资源的创建、编辑、删除等事件，并且根据资源的期望状态和实际状态，触发对应的执行逻辑，直至资源达到期望状态。如果对应资源的相关操作失败，且在有限的重试后无法达到期望状态，则对应的资源对象最终会处于失败或异常状态，且控制器不会再重试操作。

  例如，用户通过界面设置同步复制计划的保护对象后，系统会更新 SyncReplicationPlan 资源的期望状态（即需要保护的对象列表）。负责 SyncReplicationPlan 资源的控制器，将根据 SyncReplicationPlan 资源的期望状态，自动创建 ReplicationGroup 和 SyncReplicationJob 资源，从而使保护对象被同步复制所保护。

## 设计方案 > 方案特点

# 方案特点

同步复制与恢复的设计方案具有如下特点：

- **无代理**

  无需在保护对象中安装代理，减少管理和运维负担。
- **持续同步（RPO = 0 ）**

  副本对象数据持续与保护对象保持同步。在网络抖动等异常情况出现并恢复后，能继续同步。
- **高可用**

  依赖于 SMTX OS 的高可用功能，当复制服务所在节点发生故障时，可自动在其他节点继续运行并提供服务。

## 同步复制工作机制

# 同步复制工作机制

同步复制通过在目标端创建与源端数据实时一致的副本，实现 RPO = 0 的数据保护目标。该过程由 Controller-manager 统一调度，通过异步复制任务与 ZBS CDP Job 的高效协同，完成从初始对齐到实时同步的状态演进。

## 初始化与副本创建

同步复制首先通过全量异步复制任务建立数据基准。系统获取源虚拟机的配置信息及 ZBS 卷快照（记录当前时刻的全量状态），并通过存储网络在目标集群生成副本虚拟机。此时，副本虚拟机的数据状态与源端创建快照时的状态保持一致。

## 数据追赶

全量初始化期间，原虚拟机仍然会持续存在业务 I/O。为消除初始化阶段产生的增量数据差异，系统进入数据追赶阶段，通过增量异步复制任务与 ZBS CDP 机制实现动态对齐：

1. Track（增量追踪）：在增量异步复制开始前，系统便会开始精准记录源端发生的所有数据变更。
2. Delta Copy（增量对齐）：在增量异步复制任务完成后，系统依据 Track 阶段记录的数据变更，将增量快照后产生的增量数据同步至副本端。该步骤旨在业务运行过程中快速补齐数据缺口，使副本虚拟机与原虚拟机的状态首次达成一致。

## 实时同步

当历史增量数据同步完成后，系统进入稳定的同步复制状态，此时通过 Mirror（双写）机制实现实时数据同步。原虚拟机产生的写 I/O 将同时写入源端和目标端，只有当两端均完成写入后，该 I/O 才会返回成功。通过双写机制，系统确保后续任意写入在源端与目标端之间始终保持一致，从而实现 RPO = 0 的实时复制能力。

## 数据路径 > 同步数据路径

# 同步数据路径

将保护对象同步到副本对象时，涉及数据异步复制、数据同步和配置同步等操作，此过程一般涉及以下步骤：

1. **构建数据链路**

   复制服务需要能够通过存储网络同时联通源集群和目标集群，用于保护对象向副本对象异步复制数据。源集群和目标集群之间也需要能够通过存储网络进行联通，用于保护对象向副本对象同步数据。

   ![](https://cdn.smartx.com/internal-docs/assets/963b915a/sync_replication_whitepaper_04.png)

   复制服务会向 CloudTower 注册，由 CloudTower 维护服务的连接地址。用户可以在 CloudTower UI 界面上将复制服务与集群进行关联。进行同步复制前，源集群和目标集群需要关联同一个复制服务。
2. **通过全量异步复制初始化副本对象**

   复制服务在初始化阶段会执行全量异步复制，以获取保护对象的当前状态，并将其配置及全量数据同步至副本对象，从而完成副本对象的初始构建。
3. **将副本对象与保护对象同步**

   初始化完成后，复制服务会进行数据追赶，确保保护对象与副本对象数据达成一致。在此过程中，系统将先创建增量异步复制任务，并通过 ZBS Track 以记录增量复制期间的写入变化。在增量异步复制完成后，使用 ZBS Delta Copy 功能来拷贝增量复制期间的写入，直至 ZBS Delta Copy 完成。
4. **在已同步状态下 ZBS 执行双写**

   保护对象与副本对象数据达成一致后，ZBS 将会从 Delta Copy 阶段退出并自动进入 Mirror。在开启 ZBS Mirror 后， ZBS 会在保护对象和副本对象上执行双写。此时复制对象进入已同步状态，保护对象上所有的写入都会立即同步给副本对象，确保 RPO=0。
5. **在副本对象创建恢复点**

   在已同步期间，复制服务将会定期对副本对象创建快照作为同步复制的恢复点。后续当保护对象发生故障后，可执行故障转移任务，将副本对象从指定恢复点状态启动以接替保护对象继续提供服务。

## 数据路径 > 故障恢复数据路径

# 故障恢复数据路径

当保护对象从故障中恢复后，故障恢复功能可将业务运行从副本对象切换回保护对象，在此过程中，副本对象需将保护对象故障期间发生变更的配置信息和数据回传给保护对象，完成数据的同步。

1. **构建数据链路**

   复制服务需要通过存储网络同时联通源集群和目标集群，用于副本对象向保护对象回传故障期间产生的配置和数据变更。该过程为异步复制。

   ![](https://cdn.smartx.com/internal-docs/assets/963b915a/sync_replication_whitepaper_05.png)
2. **获取副本对象的当前状态**

   复制服务向 CloudTower 发送请求，对副本对象进行快照，并获得保护对象快照的元数据。元数据包含了副本对象的配置信息以及 ZBS 卷快照，记录了副本对象在当前时刻的状态。
3. **将副本对象的配置同步给保护对象**

   由于在故障转移期间，副本对象在接替保护对象提供服务的过程中可能涉及到配置信息的更新，因此，为确保在故障恢复后保护对象能够正常提供服务，复制服务会向 CloudTower 发出请求，将副本对象的配置同步给保护对象。
4. **将副本对象的数据复制给保护对象**

   完成保护对象配置信息的同步后，副本对象的快照数据将传输给保护对象。复制服务将通过指纹对比的方式，识别出副本对象相对于保护对象增量数据，并将数据传输给保护对象。
5. **在副本对象创建恢复点**

   在将副本对象的数据传输给保护对象后，复制服务将会对副本对象创建快照作为复制的恢复点，并作为后续保护对象数据的复制基准。若保护对象再次发生故障，依然可执行故障转移任务，将副本对象从指定恢复点状态启动以接替保护对象继续提供服务。

## 工作执行流程 > 基本执行逻辑

# 基本执行逻辑

复制服务的 controller-manager 模块中包含了一组控制器，用于控制每种资源的状态。当用户对某项资源执行操作后，可视为调整了该资源的期望状态。控制器在观察到用户的操作后，会根据资源当前的实际状态和期望状态，不断执行调谐（Reconcile）逻辑，使得资源的实际状态最终达到期望状态。

复制服务的基本执行逻辑如下：

1. 用户在 CloudTower UI 上进行某项操作后，CloudTower UI 将操作请求发送至 CloudTower Server；
2. CloudTower Server 将操作请求发送至 kube-apiserver，由 kube-apiserver 对相应的资源对象进行创建、编辑和删除操作；
3. controller-manager 中相应的控制器观察到 kube-apiserver 中增加、修改、删除事件的发生，尝试查询最新的资源对象，进行相应的逻辑处理，并更新资源对象的实际状态；
4. CloudTower Server 观察最新的资源对象状态，并在 CloudTower UI 界面上展示。

![](https://cdn.smartx.com/internal-docs/assets/963b915a/sync_replication_whitepaper_06.png)

下面以同步复制计划的创建过程为例说明复制服务基本的执行逻辑，编辑和删除操作的逻辑也基本相同。

1. 用户在 CloudTower UI 执行创建同步复制计划操作后，CloudTower UI 将操作请求发送至 CloudTower Server；
2. CloudTower Server 将该操作请求发送至 Replication Service 的 kube-apiserver 模块，kube-apiserver 将创建一个 SyncReplicationPlan 资源对象；
3. controller-manager 模块的 SyncReplicationPlan 控制器观察到 kube-apiserver 创建了新的 SyncReplicationPlan 后，将根据用户的设置，为每个待复制的保护对象创建一个 ReplicationGroup 资源对象，同时根据初始化时间，并在对应时间点创建一个 SyncReplicationJob 资源对象，触发 SyncReplicationJob 控制器执行相关同步复制操作；
4. CloudTower Server 观察到 ReplicationGroup 的创建后，会在同步复制计划详情页面展示复制对象，并且根据 SyncReplicationJob 的状态，展示同步复制任务的执行状态。

## 工作执行流程 > 同步复制任务的执行流程

# 同步复制任务的执行流程

创建同步复制计划后，系统会根据同步复制计划设定的初始化时间自动创建同步复制任务。执行任务时，如果复制服务需要操作保护对象或副本对象，会向对象所属的站点 CloudTower 发送请求。

![](https://cdn.smartx.com/internal-docs/assets/963b915a/sync_replication_whitepaper_07.png)

如下步骤以正常创建和执行一个同步复制任务的流程为例：

1. 用户在 CloudTower UI 界面上创建一个同步复制计划，并指定初始化时间为「立即开始」。前端将向 CloudTower Server 发送创建 SyncReplicationPlan 资源对象的请求；
2. CloudTower Server 向复制服务下发创建 SyncReplicationPlan 的请求；
3. controller-manager 中的 SyncReplicationPlan 控制器在观察到 SyncReplicationPlan 的创建后，读取到 SyncReplicationPlan 中配置了「立即开始」，将会创建一个或多个 SyncReplicationJob 资源对象。
4. SyncReplicationJob 控制器观察到 SyncReplicationJob 创建后，会创建一个全量异步复制任务 ReplicationJob 资源对象进行副本对象的初始化；
5. SyncReplicationJob 控制器观察到全量异步复制任务成功后，将通过源端 ZBS 的 API 接口为保护对象所有的卷开启 Track；
6. SyncReplicationJob 控制器创建一个增量异步复制任务 ReplicationJob 资源对象开始数据追赶；
7. SyncReplicationJob 控制器观察到增量异步复制任务成功后，将通过源端 ZBS 的 API 接口为保护对象所有的卷开启 Delta Copy；
8. SyncReplicationJob 控制器观察到 ZBS Delta Copy 完成后，更新 SyncReplicationJob 进入已同步状态。此时 ZBS 会在保护对象和副本对象上执行双写，保护对象上所有的写入都会立即同步给副本对象；
9. SyncReplicationJob 控制器将根据配置中的恢复点创建周期，在已同步状态下定期通过 CloudTower 的 API 接口，为副本虚拟机创建虚拟机快照，该虚拟机快照将会作为已同步生成的恢复点。

## 工作执行流程 > 故障转移的执行流程

# 故障转移的执行流程

用户可以通过选择已有恢复点或者立即生成恢复点对同步复制对象进行故障转移，两种模式的内部处理流程基本相同。

在跨站点容灾场景中，故障转移任务只能在目标站点发起。执行任务时，如果复制服务需要操作保护对象或副本对象，会向对象所属的站点 CloudTower 发送请求。

![](https://cdn.smartx.com/internal-docs/assets/963b915a/sync_replication_whitepaper_08.png)

如下为虚拟机故障转移的流程：

1. 用户在 CloudTower UI 界面创建故障转移任务后，前端将向 CloudTower Server 发送创建 FailoverExecution 资源对象的请求；
2. CloudTower Server 向复制服务下发创建 FailoverExecution 的请求；
3. controller-manager 中的 FailoverExecution 控制器在观察到 FailoverExecution 的创建后，将会创建一个或多个 SyncReplicationFailoverJob 资源对象。SyncReplicationFailoverJob 控制器观察到 SyncReplicationFailoverJob 创建后，如果以“立即生成恢复点”模式进行故障转移，将会根据副本对象当前的数据状态创建一个恢复点；

   如果以“选择已有恢复点”模式进行故障转移，SyncReplicationFailoverJob 控制器会通过 CloudTower 的 API 接口，将副本虚拟机回滚到指定恢复点状态；
4. SyncReplicationFailoverJob 将根据当前复制计划中网络/IP 映射规则对副本虚拟机进行网络映射和 VPC 网卡的 IP 映射，并根据用户选项进行 VPC 浮动 IP 设置；
5. FailoverExecution 控制器观察到 SyncReplicationFailoverJob 完成后，将会创建一个或多个 BootJob 资源对象。BootJob 控制器观察到 BootJob 创建后，在等待指定的启动延迟后，会通过 CloudTower 的 API 接口，将副本虚拟机开机；
6. 若用户在创建故障转移任务时，在复制计划中设置了 IP 映射规则，且指定恢复点中虚拟机工具为运行中状态，对应的副本虚拟机才会创建 ReIPJob。ReIPJob 控制器观察到 ReIPJob 创建后，将会监控副本虚拟机的开机。在开机成功后，会通过 CloudTower 的 API 接口，将根据 IP 映射规则，将副本虚拟机 VLAN 网卡的 IP 地址更改为映射后的 IP。

## 工作执行流程 > 故障恢复的执行流程

# 故障恢复的执行流程

用户可以对故障转移中的复制对象进行故障恢复。

在跨站点容灾场景中，故障转移任务只能在目标站点发起。执行任务时，如果复制服务需要操作保护对象或副本对象，会向对象所属的站点 CloudTower 发送请求。

![](https://cdn.smartx.com/internal-docs/assets/963b915a/sync_replication_whitepaper_10.png)

故障转移虚拟机的流程如下，类似于复制与故障转移虚拟机流程的结合：

1. 用户在 CloudTower UI 界面上创建故障恢复任务后，前端将向 CloudTower Server 发送创建 FailbackExecution 资源对象的请求；
2. CloudTower Server 向复制服务下发创建 FailbackExecution 的请求；
3. controller-manager 中的 FailbackExecution 控制器在观察到 FailbackExecution 的创建后，将会创建一个或多个 FailbackJob 资源对象。FailbackJob 控制器观察到 FailbackJob 创建后，会创建 FailbackReplicationJob 进行数据回传。FailbackReplicationJob 控制器观察到 FailbackReplicationJob 创建后，通过 CloudTower 的 API 接口，为故障恢复的副本虚拟机创建虚拟机快照；
4. FailbackReplicationJob 控制器读取虚拟机快照中的虚拟机配置信息，通过 CloudTower 的 API 接口，同步原虚拟机使其配置与虚拟机快照相同；
5. FailbackReplicationJob 控制器解析虚拟机快照，获取该虚拟机每块虚拟盘对应的 ZBS Snapshot 位置，并和对应 ZBS 建立连接，开始读取目标端 ZBS Snapshot 的数据；
6. 复制服务不断将读取的 ZBS Snapshot 流式写入到源端 ZBS 中，直到完成所有虚拟盘的数据回传；
7. FailbackJob 控制器将根据恢复点中的记录，将原虚拟机网卡配置恢复，并根据用户选项进行 VPC 浮动 IP 设置；
8. FailbackExecution 控制器观察到 FailbackJob 完成后，将会创建一个或多个 BootJob 资源对象。BootJob 控制器观察到 BootJob 创建后，在等待指定的启动延迟后，会通过 CloudTower 的 API 接口，将原虚拟机开机；
9. 若用户在创建故障转移任务时，指定恢复点中虚拟机工具为运行中状态，对应的原虚拟机才会创建 ReIPJob。ReIPJob 控制器观察到 ReIPJob 创建后，将会监控原虚拟机的开机。在开机成功后，会通过 CloudTower 的 API 接口，对原虚拟机进行 VLAN IP 配置。

## 高级特性 > 容错机制

# 容错机制

复制服务具备容错能力，确保同步复制在各种异常情况下仍能保持数据同步的连续性和可靠性。。

- **网络抖动容忍**

  同步复制能够适应网络环境的不稳定性。当集群间发生网络抖动或短暂中断等情况时，受保护的对象仍然能正常运行业务系统持续监控网络状态，检测到网络恢复后，将通过数据追赶机制补齐积压数据，自动恢复增量数据同步。
- **异常场景处理**

  同步复制内置了异常检测与恢复机制，能够应对多种故障场景。在故障发生时，复制服务会自动创建恢复点以避免数据丢失并通过健康检查、自动重连和数据追赶等技术手段，确保在异常恢复后能够继续保持增量数据同步。

## 高级特性 > 高可用

# 高可用

复制服务以虚拟机的形式运行在 SMTX OS（ELF）集群中，因此，复制服务具有 SMTX OS（ELF）虚拟机的高可用特性。当复制服务虚拟机所在的主机宕机时，系统会在集群内另一台健康主机上重建一台复制服务虚拟机，以保证复制服务可以继续运行并提供服务。

复制服务触发 HA 时，如果存在正在执行的同步复制、故障转移、故障恢复和永久故障转移任务，将在虚拟机 HA 后继续执行。
