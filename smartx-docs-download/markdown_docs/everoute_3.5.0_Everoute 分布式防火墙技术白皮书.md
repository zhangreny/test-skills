## 概述

# 概述

随着云计算的繁荣和普及，企业上云、应用现代化改造逐渐成为企业 IT 的核心需求，而 IT 现代化对基础设施的三大核心组件：计算、存储、网络的要求却越来越高。虚拟化技术的应用极大的提升了基础设施的效率，并带来了便利性，同时也将应用划分为更细的粒度，这给应用管理和控制增加了复杂度。国内法律法规也对云和数据中心安全提出了更高的要求。

伴随着虚拟化网络和 SDN 逐渐成为数据中心互联的基础，如何在虚拟化网络环境下保障应用安全成为客户最重要的考虑项之一，这不仅是相关法规的要求，也是网络安全威胁不断上升和升级进化的时代背景下必然需要。

## 概述 > 面临的挑战

# 面临的挑战

## 以边界防御为主的传统数据中心并不安全

传统数据中心网络安全手段以加强边界安全为主。边界安全模型认为内部网络是可信任的，安全威胁主要来自边界之外，只要在边界做好防护就能高枕无忧。但在组织业务的云化、虚拟化和移动化的演变趋势下，传统的边界已被拉伸并逐渐变得模糊。一旦数据中心从内部被攻破，整个业务系统和全部核心数据都将暴露在攻击范围之中。数据中心内网已不再是可信的安全区域。

同时，数据中心正朝着服务器虚拟化、容器化、微服务和融合基础设施方向进行演变。当今的应用已经从单体架构转型为微服务架构，各个微服务组件之间呈现出高度分布式的结构特点。这些变化扩大了数据中心内部的受攻击面，内部网络将更容易受到攻击。

## 传统网络分段的安全手段无法应对新兴的安全威胁

传统网络一般只能按照 VLAN（Virtual LAN，虚拟局域网）或者 IP 子网方式，在网络广播域之间进行分段隔离，如下图所示。

![](https://cdn.smartx.com/internal-docs/assets/e274bbdc/whitepaper-1.png)

传统网络分段隔离的粒度粗，影响范围大。随着业务发展，应用架构的复杂度增大，传统的网络分段方式将很难满足日益增多的虚拟服务对网络安全的要求。同时，边界防火墙安全规则以网络地址为基础，网络管理员或者安全管理员需要编写和管理大量的以 IP 地址（段）为基本单位的安全策略来实现复杂的流量控制。“精细化安全”的要求，意味着安全策略必须越来越细致，直至所有的安全策略中必须以“每个 IP 地址和端口”为匹配条件。如果数据中心网络中任何 IP 地址发生变化，或 IP 的用途发生变化（例如，某 IP 上原本仅开通 HTTP/HTTPS 服务，需要新增 FTP 服务），管理员必须逐一修改这些基于 IP 地址的安全规则。这不仅带来了巨大的管理开销，还不可避免地将发生各种配置错误，最终导致安全漏洞和网络连通性等问题频发。

## 传统网络和安全模式的成本居高不下

由于硬件设备无法实现按需扩容，在所有基于硬件的网络和安全项目中，首次采购和部署就必须满足最大规格要求，而容量预留和高可用架构的成本高昂。为满足业务的不断扩展，一旦现有硬件的使用量超出安全预留阈值，或者需要引入全新的安全功能，就必须对硬件设备进行升级、扩容、测试，以及割接，这些通常需要数周乃至数月的时间。上述现象均给企业增加了各种有形和无形的成本，而这些也无法在以硬件为核心的基础架构内进行优化和精简，这些居高不下的成本逐渐成为企业面临的难题之一。

综上所述，传统网络和安全与业务之间缺乏联动，传统安全设备和机制无法随时记录和更新 IP 地址与业务的对应关系，不能及时响应各种层出不穷的新型网络威胁；传统网络与安全架构的扩展节奏已经无法满足业务发展的敏捷弹性需求，成为制约企业数字化转型和发展的短板。走向云时代的企业应用基础架构必须启动现代化网络和安全机制改造，以适应新的应用开发与部署模式，而不是迫使应用和业务向旧有网络技术和安全方法妥协。

## 概述 > 解决方案

# 解决方案

随着虚拟化的普及以及虚拟化网络改造的深入，数据中心内部的业务种类越来越多，应用体系也变得越来越复杂，内外部的安全威胁愈演愈烈。因此，基于零信任理念的网络架构和更细粒度的网络隔离功能——“微分段”——应运而生。

微分段（也称做“微隔离”）是一种面向虚拟化世界的网络隔离技术，将数据中心内部网络分成若干个可以单独保护的逻辑区域，支持跨数据中心物理边界和不同云环境创建逻辑区域，重点用于阻止进入企业数据中心网络内部后的横向平移（也即“东西向移动”）攻击，是软件定义网络和安全的一种具体实践。

基于微分段的网络能够快速实现数据中心内部互联、安全合规、弹性扩展和自动修复，通过自动化方法快速添加、更新和删除网络配置（包括转发策略、安全策略等）。IT 管理人员可借助微分段技术，在工作负载、应用程序、操作系统或虚拟机级别对数据中心内部进行按需分组，灵活部署分组间安全策略，对不同业务节点提供细粒度且有针对性的安全隔离和管控。微分段天生的自动化、精细化和零信任的特点，使它可以及时应对种种不可预测的安全威胁，减少 IT 团队经常面对的繁杂易错的手工配置。

Everoute 分布式防火墙的微分段功能，基于 SMTX OS 的虚拟化场景（SMTX 虚拟机服务，即 “ELF 平台”）进行设计，通过使用 Kubernetes 云原生框架和技术构建的管控平台，结合虚拟分布式交换机，支持虚拟机隔离、自定义安全策略、默认拒绝通信三种形态的分布式防火墙策略，主要解决数据中心虚拟网络安全面临的挑战，为业务提供灵活精细的网络安全保障，符合“零信任架构”对网络安全的要求。

![](https://cdn.smartx.com/internal-docs/assets/e274bbdc/whitepaper-2.png)

SMTX OS 虚拟分布式交换机（Virtual Distributed Switch，VDS）是一组物理连通的物理网口（可结合网口绑定）定义的虚拟化层面的分布式交换机，可以为存储网络、管理网络和虚拟机网络提供基础。

![](https://cdn.smartx.com/internal-docs/assets/e274bbdc/whitepaper-3.png)

Everoute 分布式防火墙基于零信任的网络架构，支持以应用为中心设置安全策略。在应用微分段功能后，数据中心的网络安全模型如下图所示，数据中心外部通过边界防火墙进行隔离，内部使用微分段功能对每个虚拟机网络提供安全规则保护。管理员将不必过分关注网络拓扑和配置，极大减少了管理员的负担，同时也提高了数据中心网络安全的可靠性和灵活性。

![](https://cdn.smartx.com/internal-docs/assets/e274bbdc/whitepaper-4.png)

## 技术特点

相较于传统的网络分段方案，Everoute 微分段功能主要有以下技术特点：

1. 为数据中心提供不同粒度的分组

   Everoute 微分段管控粒度更精细，可以针对每一台虚拟机、每一个 K8s Pod 进行分组并部署安全策略，形成的逻辑安全域更为精准，符合“零信任”网络安全模型对最小权限访问的要求。
2. 基于业务属性下发安全策略

   传统网络一般以网络 IP 地址为基础来配置和管理安全策略，由于数据中心每台虚拟机的每个虚拟网卡均设置 IP 地址，在业务发展或发生变化时，都会引起大量的变更；Pod 的 IP 更是会随着 Pod 的重建和迁移变更，无法与 Pod 承载的业务绑定。SMTX OS 微分段依据业务属性，通过标签或安全组定义目标虚拟机组及 Pod 组，并面向目标虚拟机组及 Pod 组设定安全策略，而不是以 IP 地址作为安全策略的基本条件。这就为管理员屏蔽了网络 IP 地址这个难于管理的元素。
3. 灵活适配数据中心网络拓扑变化

   数据中心内部节点数量多，变化频率高，虚拟机可能按照不同的业务要求进行灵活迁移。Everoute 微分段自动适配虚拟机的变化，自动关联虚拟机与对应的安全策略，减少上述过程中的人工参与和管理安全策略的开销。

## 方案优势

Everoute 分布式防火墙有如下优势：

- **易使用**

  - 无需安装任何插件，基于业务的虚拟机组与虚拟机 IP 地址自动关联，自动适配组安全规则，简化安全配置。
  - 采用声明式 API，通过代码的方式实现安全即代码（SaaC: Security as a Code），用户只需关注期望结果，无需关心中间过程。
  - 提供友好的图形化界面，实现灵活易用的配置管理。
  - 无需调整物理网络上的交换、路由、安全等配置。
- **可扩展**

  - 采用分布式虚拟防火墙架构，具备横向扩展性。
  - 支持统一管理多个 SMTX OS（ELF）集群或 SMTX ELF 集群上的安全策略。
  - 支持跨数据中心的统一安全策略管理。
  - 支持跨虚拟机和 Pod 的统一安全策略管理。
- **高可用**

  - 多控制器实例组成高可用集群，无单点故障。
  - 控制与转发分离，控制平面故障不影响网络数据转发。
- **广泛兼容**

  - 支持任意底层网络架构，无任何物理网络品牌、型号和功能依赖。
  - 纯软件实现方案，无任何硬件设备（网卡、服务器）依赖。
  - 支持多种架构平台（Intel x86\_64、Hygon x86\_64、鲲鹏 AArch64 和飞腾 AArch64 架构）混合部署和使用。

## 概述 > 典型应用 > 数据中心服务间的“零信任安全”

# 数据中心服务间的“零信任安全”

随着部署到数据中心虚拟化服务平台的应用越来越多，服务间东西向流量在整个网络流量的占比也越来越重，而传统安全模型却常常无法控制数据中心东西向的网络流量安全。

微分段基于零信任模型，必须通过白名单才能允许东西向流量的通信。通过与业务平台的联动，Everoute 分布式防火墙自动为业务资源提供精细化的策略，有效地消除了数据中心横向移动引发的内部威胁，大大减少了总攻击面，给数据中心东西向业务流量提供了坚实的安全保护。

## 概述 > 典型应用 > 异常虚拟机隔离

# 异常虚拟机隔离

网络攻击手段种类多样，潜伏期也越来越长，例如高级可持续威胁（APT）入侵与攻击，可以绕过基于代码的传统安全方案（如防病毒软件、防火墙和 IPS 等），并长时间潜伏于系统中。Everoute 的微分段功能可以针对异常虚拟机提供隔离策略。

在应用“完全隔离策略”后，该虚拟机与系统网络完全隔离，不允许任何出入流量。即使该虚拟机被攻击甚至被攻破，此策略也可以阻止该虚拟机对系统其他业务产生影响。

当系统管理员需要针对某个虚拟机进行诊断或测试时，还可以设置“诊断隔离策略”，仅允许管理员配置的对象与该虚拟机进行通信。

![](https://cdn.smartx.com/internal-docs/assets/e274bbdc/whitepaper-11.png)

## 概述 > 典型应用 > 快速拦截风险流量

# 快速拦截风险流量

用户在刚部署 Everoute 分布式防火墙后，可能暂时未完全理清业务虚拟机之间全部的访问需求。此时若有明确的风险端口（如 SSH 端口）和环境隔离需求，可先为关键业务虚拟机配置相关的拒绝策略来拦截风险流量。此外，若在业务运行期间发现了新的安全威胁，也可立即设置拒绝策略以防止潜在的网络攻击。

## 概述 > 典型应用 > 业务部门分组安全

# 业务部门分组安全

企业数据中心承载着不同业务部门的 IT 需求，而每个业务部门的权限与安全策略都不同，业务部门之间的访问策略与规则也各有区别。利用 Everoute 分布式防火墙，可以对不同部门管理和维护的应用虚拟机和 Pod 进行灵活分组，制定并执行不同部门组内和组间的访问安全策略。

---

## 概述 > 典型应用 > 灵活定义内外部服务访问策略

# 灵活定义内外部服务访问策略

在某些场景下，数据中心内部的某些应用可能需要访问外部的服务，但不应该访问数据中心内部某些服务。这些不应该被访问到的内部服务可能与外部服务使用相同的 IP。此时可以通过设置自定义安全策略（放行策略）的 IP 地址白名单来允许内部应用访问数据中心外的服务，并通过取消勾选**配置对称策略**防止应用访问内部服务。

例如，运行内部服务的 VM3 和外部服务使用了相同的 IP 地址 `172.20.20.1/26`，内部应用 VM1 和 VM2 预期可访问外部服务但不能访问 VM3。此时，可为 VM1 和 VM2 创建放行策略，将 `172.20.20.1/26` 添加到出流量白名单，并取消勾选**配置对称策略**来满足要求。

![](https://cdn.smartx.com/internal-docs/assets/e274bbdc/whitepaper-14.png)

## 概述 > 典型应用 > 虚拟机与容器之间的访问控制

# 虚拟机与容器之间的访问控制

在虚拟化平台上运行容器时，需要对容器与虚拟机之间的网络流量进行精细的访问控制，以保障平台的安全性。容器与虚拟机之间可能存在紧密的业务关联，需要根据具体的业务场景配置面向虚拟机和容器的统一访问策略，允许必要的通信，同时阻止未经授权的访问。

Everoute 不仅支持在虚拟机之间、Pod 之间设置安全策略，还支持在 Pod 与虚拟机之间设置安全策略。

## 产品架构

# 产品架构

Everoute 分布式防火墙使用软件定义方式为数据中心虚拟化网络提供网络和安全功能，与硬件完全解耦，支持网络和安全的灵活管理和动态控制，实现网络数据的转发与控制分离。 其主要由管理平面、控制平面和数据平面组成，三个平面组件之间相互独立，并通过 API Server 无缝配合。此外，Everoute 还支持为 Kubernetes 集群提供容器网络 CNI 和网络安全能力。

## 产品架构 > 管理平面

# 管理平面

管理平面主要由 Everoute operator 、Web UI，以及 API 组成，用户可以通过 Web UI 或 API 的方式访问 Everoute operator。

管理平面主要提供以下功能：

- 许可管理：查看或更新 Everoute 软件许可。
- 安装包管理：管理用于部署、升级 Everoute 服务的文件。
- Everoute 服务运维管理：多 Everoute 服务管理，部署、卸载、一键热升级 Everoute 服务，关联 SMTX OS（ELF）集群或 SMTX ELF 集群，Everoute Controller 虚拟机生命周期管理。
- Everoute 服务监控告警。
- 分布式防火墙管理：

  - 管理网络与安全策略，包括默认拒绝通信、全局白名单、自定义安全策略（包括放行策略和拒绝策略）和虚拟机隔离。
  - 查看生效的安全策略：展示安全策略命中次数并支持邮件通知，提供下载策略日志的接口。

其中，Everoute 服务的运维管理、微分段安全策略的配置等由 Everoute operator 负责。

## 产品架构 > 控制平面

# 控制平面

控制平面采用集中控制和分布式代理的架构，通过一组由 3 个或 5 个控制器节点所组成的 Everoute 控制器集群，对 ELF 集群进行集中式管理，并将控制代理 Everoute Agent 分布在每个节点上部署，以负责学习虚拟机 IP 地址和生成安全规则。

![](https://cdn.smartx.com/internal-docs/assets/e274bbdc/whitepaper-6.png)

- **Everoute 控制器集群**

  由 3 个或 5 个控制器节点组成，每个控制器（Controller）占用少量资源，无单点故障。

  控制器集群利用 etcd 提供主节点选举和实时监控。当有节点发生变化或者掉线后，集群可以第一时间检测到节点的健康状态，并实现主从切换，以满足业务连续性。同时 etcd 还为控制平面提供数据的持久化。

  控制器集群还提供可扩展的南北向接口：北向接口即声明式 API，南向接口即与 Everoute Agents 通信的接口。
- **控制代理**

  部署在每一个节点上，负责学习虚拟机的 IP 地址及 IP 地址的超时管理，并通过与控制器的南向接口通信，订阅 Controller 生成的安全策略对应的安全组信息及其变化，将安全策略转换成安全规则，然后通过 Datapath 接口为节点上的 VDS 数据转发平面编辑安全规则的转发表。
- **CloudTower Plugin**

  Controller 虚拟机通过 CloudTower Plugin 订阅控制台中发生的所有与防火墙策略相关的事件，例如安全策略、安全组、虚拟机标签、虚拟机的编辑操作等，然后对安全规则执行对应的增删改操作。
- **K8sMgmt Plugin**

  Controller 虚拟机通过 K8sMgmt Plugin 订阅容器管理平台发生的工作负载集群相关事件，例如工作负载集群的创建与删除、证书更新等。
- **EIC Plugin**

  Controller 虚拟机通过 EIC Plugin 连接各个工作负载集群，订阅工作负载集群的 Pod 变更事件，对相关安全规则执行对应的删改操作。
- **声明式 API**

  与传统命令式 API 相比，声明式 API 使用起来更加简洁，并提供了更好的抽象性。使用声明式 API，用户只需提交一个定义好的 API 对象来声明所期望的最终状态，而不需要关心执行的过程。
- **Controller 南向接口**

  负责与 Everoute Agents 进行通信。

控制平面主要根据安全策略中的虚拟机标签信息、安全组信息、策略对象和协议信息进行计算，将以虚拟机标签、安全组和协议为要素的安全策略展开为对应的虚拟机分组、Pod 分组和 IP 地址信息，供 Agent 订阅生成对应的安全规则。每次对虚拟机标签或安全组进行更新后，Controller 将自动计算并更新安全规则。

虚拟机关联的标签及安全组不会随着其热迁移或业务调整发生变化，而安全规则随着虚拟机对应主机节点的变化进行动态更新，因此可以做到业务无感知，同时也不会影响业务连通性。

## 产品架构 > 数据平面

# 数据平面

数据平面是网络数据转发和策略执行的主体，通过 OpenFlow 实现基于 Open vSwitch（OVS）的数据转发和策略控制，这种模式支持 L3 和 L4 层安全规则的灵活编排，以及对 TCP、UDP、ICMP 协议做流量过滤。

![](https://cdn.smartx.com/internal-docs/assets/e274bbdc/whitepaper-7.png)

数据平面负责执行由控制面生成的转发规则和安全规则，提供通过 ARP（IPv4）、NDP（IPv6）地址学习的 IP 地址发现机制，实时动态更新策略对象与 IP 地址间的映射关系。如果虚拟机安装了 VMTools，也会同时从虚拟机内部获取实际配置的 IP 地址。

数据平面主要承担以下职责：

- 虚拟网络流量转发
- Endpoint IP 地址发现
- 虚拟机网络隔离
- 执行有状态安全规则

  分布式防火墙分为有状态转发和无状态转发两种模式。Everoute 提供有状态分布式防火墙功能，根据网络连接状态过滤网络流量，与基于 ACL 无状态包过滤的网络安全策略相比，Everoute 的有状态分布式防火墙功能将更安全。
- 策略日志

  采集虚拟机流量匹配的安全策略信息，生成策略日志，并维护安全策略命中计数。

## 分布式防火墙策略

# 分布式防火墙策略

Everoute 支持**全局安全策略**、**自定义安全策略**、和**虚拟机隔离**的三种分布式防火墙策略，其中**自定义安全策略**包含**拒绝策略**和**放行策略**，**全局安全策略**包含**全局白名单**和**默认拒绝通信**。为保证应用这些策略，虚拟机所在的集群必须与 Everoute 服务关联，以及虚拟机网络所属 VDS 与 Everoute 服务也需关联。自定义安全策略可通过 Pod 安全组应用到 Pod 上，Pod 需使用 EIC CNI，且其节点的 EIC 专用网卡所使用的虚拟机网络所属 VDS 需要与 Everoute 服务关联。

若 Everoute 分布式防火墙关联了 1.3.0 或更高版本的可观测性服务，可通过安全策略命中计数及安全策略日志查看安全策略生效情况。

以上策略应用于虚拟机和 Pod 的优先级，按照从高到低的顺序依次为：**虚拟机隔离策略** > **自定义安全策略（拒绝策略）** > **自定义安全策略（放行策略）** = **全局白名单** > **默认拒绝通信**。策略生效优先级示意图如下。

## 分布式防火墙策略 > 全局安全策略

# 全局安全策略

全局安全策略包括默认拒绝通信和全局白名单。

## 默认拒绝通信

默认拒绝通信的应用对象可为所有虚拟机网络所属的虚拟分布式交换机都关联了 Everoute 服务的虚拟机，且这些虚拟机未作为自定义安全策略的策略对象且未被隔离。当虚拟机的虚拟机网络所属的虚拟分布式交换机，没有全部关联该策略所属的 Everoute 服务时，该策略可对虚拟机中已关联该 Everoute 服务的虚拟机网络生效。可以对每个 Everoute 服务设置是否开启此策略。

## 全局白名单

为确保用户在数据中心之外所部署的服务（例如堡垒机）与数据中心内虚拟机正常通信，Everoute 分布式防火墙支持设置全局白名单。全局白名单对已设置隔离策略的虚拟机无法生效。

- 若对虚拟机已设置完全隔离或诊断隔离策略，则全局白名单失效，虚拟机只能应用完全隔离或诊断隔离策略。
- 若对虚拟机仅设置了自定义安全策略，则虚拟机将首先应用这些策略，然后再应用全局白名单，确保白名单内的 IP 地址可以与虚拟机进行通信。

## 分布式防火墙策略 > 自定义安全策略

# 自定义安全策略

自定义安全策略的应用对象可为携带标签的虚拟机、包含在安全组中的虚拟机、IP 或 Pod（使用 EIC CNI），且这些虚拟机的虚拟机网络所属的虚拟分布式交换机需要关联 Everoute 服务，Pod 所属虚拟机节点 EIC 专用网卡的虚拟机网络所属的虚拟分布式交换机需要关联 Everoute 服务。当虚拟机的虚拟机网络所属的虚拟分布式交换机，全部未关联该策略所属的 Everoute 服务时，该虚拟机无法作为策略对象。

## 策略类型

自定义安全策略有以下两种安全策略类型，可在创建自定义安全策略时配置。

- **放行策略**

  放行策略是一种只允许信任流量通过防火墙的策略。为策略对象配置了放行策略后，除了策略中明确指定的流量，其他流量都不允许通过。这种严格的控制机制确保了只有经过授权流量才被允许进入系统，从而极大地提高了系统的安全性。
- **拒绝策略**

  拒绝策略是一种阻止特定流量的策略。与放行策略不同的是，拒绝策略只阻止在策略中明确指定的流量，其余流量能否通行则取决于系统中配置的其他安全策略。拒绝策略提供了一种简单、灵活的管理机制，可以高效控制和响应已知的安全威胁，在保护关键资源的同时保持网络的开放性。

若同时为虚拟机配置了完全隔离或诊断隔离策略，则自定义安全策略失效，虚拟机只能应用完全隔离或诊断隔离策略。

## 策略对象

自定义安全策略的策略对象，可通过安全组或虚拟机标签与虚拟机的映射关系、安全组与 Pod 的映射关系进行动态更新。

- 虚拟机标签可通过 CloudTower 管理平台进行设置。虚拟机与特定的标签完成关联后，Everoute 分布式防火墙通过标签对虚拟机进行动态分组。当标签关联的虚拟机一旦发生变化，例如新业务上线，此时只需要对新上线的虚拟机设置对应的标签，系统会自动生成安全策略和安全规则。
- 安全组可通过 Everoute 服务进行设置。安全组根据成员类型分为虚拟机安全组、IP 安全组和 Pod 安全组。可在虚拟机安全组中添加标签组合作为动态成员，也可以添加指定虚拟机作为静态成员。如需对特定业务的虚拟机应用相同的安全策略，可以将其添加至同一个安全组中，并在各类所需策略中使用安全组作为策略对象；可在 IP 安全组中添加 IP 地址作为成员；可在 Pod 安全组中配置 Pod 所属工作负载集群、Pod 所属 K8s 名字空间及 Pod 标签作为动态成员。

> **说明**：
>
> - 通过虚拟机标签添加策略对象时，可选择一个或多个标签。当选择多个标签时，虚拟机的筛选逻辑为**逻辑与**，即筛选出的虚拟机为同时关联了所有标签的虚拟机。
> - 支持同时通过虚拟机标签和安全组的方式添加多组策略对象。例如，一个策略对象通过安全组添加，另一个策略对象通过虚拟机标签添加。多组策略对象之间为**逻辑或**的关系，即包含在任一组策略对象中的虚拟机都作为此自定义安全策略的策略对象，并应用相关的安全规则。
> - 使用 IP 安全组作为策略对象时，该 IP 相关的策略会下发至所有关联集群的所有节点上。

## 安全规则

安全规则支持多种对象：虚拟机、Pod、IP 地址、CIDR 地址段、IP 地址范围，分为入流量规则和出流量规则。可以通过安全规则对象灵活指定策略对象的**出方向**和**入方向**的规则。

安全规则支持 TCP、UDP、ICMP、IPIP 协议、FTP、TFTP 动态协议，其中 TCP 协议和 UDP 协议支持指定端口和端口范围。

根据所选择的策略类型制定不同的安全规则：

- **白名单规则**

  如果策略类型为放行策略，则安全规则将按照白名单方式进行制定，只允许白名单内的网络流量流入或流出，白名单之外的网络流量默认不允许通过
- **黑名单规则**

  如果策略类型为拒绝策略，则安全规则将按照黑名单方式进行制定，不允许黑名单内的网络流量流入或流出，黑名单之外的网络流量不受影响，是否允许通过由其他安全策略决定。

## 对称策略

对称策略的核心机制是为白名单对象配置对称规则，以确保白名单和策略对象的双向通信。例如，为实现虚拟机 A 访问虚拟机 B，需要在 A 上设置允许向 B 发送流量的出规则，同时在 B 上设置允许从 A 接收流量的入规则，它们之间才能实现正常通信。这种情况下，虚拟机 A 的出流量规则和虚拟机 B 的入流量规则被称为一组对称策略。

在自定义安全策略（放行策略）中，若为策略对象设置了出、入流量白名单，Everoute 分布式防火墙会为策略对象针对入流量白名单设置入流量规则，允许接受所有白名单中虚拟机和 Pod 的流量；同时针对出流量白名单设置出流量规则，允许向出流量白名单发送流量。同时，为保证出、入流量白名单生效，还需为白名单中虚拟机和 Pod 设置对称策略。根据不同的白名单设置方法，设置对称策略的方式也有所不同，具体说明如下：

- **使用虚拟机标签、虚拟机安全组和 Pod 安全组添加白名单**

  该方式下，由于虚拟机标签或安全组关联的全部为 Everoute 服务管理范围内的虚拟机和 Pod，Everoute 服务自动为白名单中虚拟机和 Pod 设置对称策略，确保白名单可生效。
- **使用 IP 安全组或 IP 地址添加白名单**

  使用 IP 安全组或 IP 地址添加白名单时，若 IP 地址在 Everoute 服务管理范围内，则用户可选择是否为白名单配置对称策略。若不配置，则仅对策略对象配置相关规则，实际流量的处理方式将取决于白名单中虚拟机和 Pod 所属的其他安全策略。

  若 IP 地址在 Everoute 服务管理范围外，则 Everoute 无法为此类 IP 地址设置对称策略。

| **App-Service-Policy** | 放行策略 | App | **Object**: Web  **Protocol**: TCP  **Ports**: 6000 | **Object**: DB  **Protocol**: TCP  **Ports**: 8000 |

## 分布式防火墙策略 > 虚拟机隔离

# 虚拟机隔离

虚拟机隔离策略的应用对象为单个虚拟机。当虚拟机受到攻击或者用户想对虚拟机进行隔离调试时，可以对虚拟机设置隔离策略。隔离策略分为两种：完全隔离和诊断隔离。

- **完全隔离**：应用此策略后，所有出入虚拟机的流量（包括所有二层流量和三层流量）将全部禁止。虚拟机与外界完全隔离，并禁止连接网络。用户只能通过管理控制器登录虚拟机。
- **诊断隔离**：指用户指定访问虚拟机（入流量）的白名单，以及虚拟机访问对象（出流量）的白名单，以帮助排查和分析此虚拟机的故障。
