## 双活集群简介

# 双活集群简介

双活是 SMTX OS 的一种数据保护和容灾机制，提供 RPO（数据恢复点目标）为 0，RTO（恢复时间目标）为分钟级的站点失效容错能力。

SMTX OS 双活集群以拉伸形态部署，由两个可用域和一个仲裁节点组成。两个可用域与仲裁节点间通过网络连接通信。当一个可用域失效后，另一个可用域仍可以继续提供服务，从而获得可用域级别的容灾能力。

**可用域**

两个可用域一般位于同一个城市有一定距离的物理节点所在区域，分为优先可用域和次级可用域。每个可用域至少包含 2 个节点，并且均可以完全独立地提供计算和存储服务，集群正常工作时业务主要运行在优先可用域上。

当两个可用域工作正常时，集群会将每个数据块在两个可用域内分别保存副本，以保证当一个可用域失效时，从另外一个存活的可用域中可以获得完整的数据用于恢复业务。

**仲裁节点**

设置仲裁节点主要是为了保证集群中选主类服务正常运行，并使得业务在优先或次级可用域中自动切换。仲裁节点仅仅参与选主算法，保存部分元数据，而不存放任何用户数据。

仲裁节点可以是物理机或虚拟机，需要部署在与优先可用域和次级可用域不同的物理故障区。一般情况下，仲裁节点与优先可用域、次级可用域均位于不同的数据中心。如果仲裁节点与可用域位于同一故障区，例如同一个 IDC 机房内，在遭遇 IDC 整体下线的场景时，双活集群将无法自动恢复业务。

## 双活集群简介 > 相关概念

# 相关概念

在阅读本文档前，您需要先了解与 SMTX OS 双活集群相关的一些概念，以帮助您更好地理解双活集群的安装部署和运维操作。

**超融合**

超融合是一种 IT 基础架构，通过将计算虚拟化、分布式存储、网络及各类运维管理等服务融合部署在通用的 x86\_64/AArch64 架构的服务器上，大幅简化基础设施类资源的交付、上线和运维。

**SMTX OS 节点**

SMTX OS 软件可运行在物理服务器上或者虚拟机中，而节点特指运行了 SMTX OS 软件的单个物理服务器或者虚拟机。节点是组成 SMTX OS 集群的基本单位。

**SMTX OS 集群**

SMTX OS 集群属于逻辑概念。在实际生产环境中，一个 SMTX OS 集群至少由 3 个节点通过网络互连组成。

**虚拟分布式交换机（VDS）**

SMTX OS 利用虚拟化技术将一组物理连通的物理网口（可结合网口绑定实现网络高可用）转化为虚拟化层的分布式交换机（Virtual Distributed Switch, VDS），为主机之间提供网络连接，并为处理各类型系统流量及业务流量提供基础。

SMTX OS 通过虚拟分布式交换机管理物理网口绑定和 VLAN ID 等，从而实现网络高可用和网络隔离。

**ELF**

ELF 平台即指 SMTX OS 原生虚拟化计算平台，是当 SMTX OS 以超融合形态直接部署在 x86\_64 或 AArch64 架构的服务器上时，该超融合软件中基于 KVM 的虚拟化引擎组件。ELF 除提供虚拟机生命周期管理、电源操作、高可用、冷热迁移等基本功能外，通过与存储组件 ZBS 相结合，还能提供秒级快照、模板和克隆等高级的虚拟机服务。

ELF 简单易用，对 SMTX OS 集群的所有操作均可通过 CloudTower 完成。

**ZBS**

SMTX OS 内置的分布式块存储软件，它负责接管服务器的所有存储资源（包括所有的 SSD 和 HDD），将多个服务器的存储资源池化，构建统一的存储池，为虚拟机提供存储服务。

**VMware ESXi 平台**

ESXi 是 VMware 公司推出的一套服务器虚拟化解决方案（vSphere）中的一个关键组件，可独立安装和运行在物理服务器上，是 VMware 的裸机虚拟机管理程序。VMware ESXi 平台特指运行着 ESXi 程序的虚拟化平台。

**SCVM**

当 SMTX OS 结合 VMware ESXi 平台以超融合的形态进行部署时，SMTX OS 软件需安装在一种特殊的虚拟机中，该虚拟机即为 SCVM（SmartX Controller Virtual Machine）。

SCVM 能直接管理主机上的硬盘设备（SSD/HDD），使得这些硬盘统一由 ZBS 负责管理，并通过网络将 SCVM 组成分布式存储集群，为 VMware ESXi 主机中的虚拟机提供存储服务。

**主节点**

SMTX OS 集群中额外运行元数据服务的节点。

**存储节点**

SMTX OS 集群中未运行元数据服务的普通节点。

**副本数**

使用多副本技术提供数据冗余，一份数据会在不同主机上保存多个副本，以提高数据的可靠性和安全性。

**物理盘池**

由 SMTX OS 集群中某个节点的一块或多块物理盘组成，不支持跨节点构建。SMTX OS 支持在单个节点上配置多个物理盘池，以提升节点的性能上限并充分利用物理盘的性能。每增加一个物理盘池都会增加一定的 CPU 核心和内存消耗。根据节点容量规格不同，可配置 1、2 或 4 个物理盘池。

**存储分层模式**

把集群内的存储设备分为缓存层和容量层，其中容量层是所有节点数据分区构成的空间，用于存放冷数据，冷数据根据设定的数据冗余策略，以副本或纠删码的形态存储。不同存储配置下缓存层的构成不同。

对于 SMTX OS（ELF）集群，当节点为混闪配置、多种类型 SSD 全闪配置或单一类型多种属性 SSD 全闪配置时，缓存盘与数据盘将独立部署，高速介质为缓存，低速介质为容量。数据会以副本形态先写入缓存，待数据变冷后下沉到容量层。

- 混闪配置和多种类型 SSD 全闪配置：缓存层分为写缓存和读缓存；写缓存用于存放新写入的数据，读缓存用于存放被频繁访问的已下沉的数据，写缓存与读缓存的容量比为 8:2。
- 单一类型多种属性 SSD 全闪配置：缓存层均为写缓存，用于存放新写入的数据。

由于双活集群仅支持副本策略，因此单一类型单一属性 SSD 全闪配置部署为不分层模式即可。

**存储不分层模式**

不设置缓存层。除了含有系统分区的物理盘，剩余的所有物理盘都作为数据盘使用。使用存储不分层模式时只能使用全闪配置。

**容量规格**

SMTX OS 按照不同的物理盘容量上限区分了三种容量规格，其中小容量规格单个主机最大支持 50 TB，标准容量规格单个主机最大支持 128 TB，大容量规格单个主机最大支持 256 TB。小容量规格仅适用于从 SMTX OS 6.1.0 以下版本升级上来的集群，具体可参考《SMTX OS 升级指南》的**确认数据分区容量和节点容量规格**小节。

**常驻缓存模式**

SMTX OS 支持常驻缓存模式，将虚拟卷（ELF 平台）或 NFS 文件（VMware ESXi 平台）中的数据保留在缓存层，从而获得更稳定的高性能。

**CloudTower**

SmartX 多集群资源集中管理平台。CloudTower 运行在虚拟机内，通过使用少量的计算和存储资源便可管理上千台虚拟机。除虚拟机生命周期管理、存储管理、网络管理、硬件管理、监控和报警等基本服务外，CloudTower 还提供全局搜索、虚拟机资源优化、事件审计、虚拟机回收站和报表等功能，同时它也提供完整和标准的 RESTful API 和多种语言的 SDK，帮助运维管理员轻松管理和使用多个 SMTX OS 和 SMTX ZBS 集群。

**LACP 动态链路聚合**

LACP（Link Aggregation Control Protocol，链路聚合控制协议）是一种基于 IEEE802.3ad 标准的实现链路动态聚合与解聚合的协议，属于在链路聚合中常用的协议之一。链路聚合组中启用了 LACP 协议的成员端口通过发送 LACP PDU 报文进行交互，双方对能够发送和接收报文的端口明确并达成一致，确定承担业务流量的链路。此外，当聚合条件发生变化时，如某个链路发生故障，LACP 模式会自动调整聚合组中的链路，组内其他可用成员链路接替故障链路维持负载平衡。因此在不进行硬件升级时，使用 LACP 协议也可以增加设备之间的逻辑带宽，提高网络的可靠性。

动态链路聚合模式需要本端和对端设备同时启用 LACP 协议，所选择的活动接口必须保持一致，然后才能建立链路聚合组 LAG（Link Aggregation Group）。LAG 是指将若干条以太链路捆绑在一起形成一条逻辑链路，也称为 Eth-Trunk 链路。

SMTX OS 支持 LACP 动态链路聚合功能。在虚拟分布式交换机中，需选择与对端交换机配置 LACP 端口相连的两个物理网口，并且网口绑定模式选择为 OVS bonding 中的 `balance-tcp` 或 Linux Bonding 中的 `802.3ad` 模式。设置完成后即可启用 LACP 动态链路聚合功能。

**部署控制节点**

在部署双活集群时，需要通过浏览器访问集群中优先可用域或次级可用域中某个节点的管理 IP，进入集群部署界面。此节点即为集群部署时的控制节点，简称部署控制节点。

## 双活集群简介 > 部署形态

# 部署形态

SMTX OS 以超融合架构部署为双活集群时，支持以下两种部署形态：

- 采用 SmartX 原生虚拟化 ELF 部署

  SMTX OS 直接运行在优先/次级可用域的物理服务器上，通过 iSCSI 接入协议为优先/次级可用域节点上的虚拟机提供虚拟磁盘进行存储。仲裁节点可以是物理服务器或者虚拟机，SMTX OS 运行在物理服务器或者虚拟机中。
- 搭配 VMware ESXi 虚拟化部署

  SMTX OS 与 VMware ESXi 平台相结合，运行在优先/次级可用域中 ESXi 主机的 SCVM 虚拟机内，通过 NFS 协议为 ESXi 主机提供存储服务，而虚拟机服务则由 VMware ESXi 提供。仲裁节点可以是物理服务器或者虚拟机，SMTX OS 运行在物理服务器或者虚拟机中。

本文档仅描述采用 SmartX 原生虚拟化 ELF 部署的部署流程，搭配 VMware ESXi 虚拟化部署的相关内容请参考[《SMTX OS 双活集群安装部署指南（VMware ESXi 平台）》](/smtxos/6.3.0/vmware_multiactive_installation/vmware_multiactive_installation_preface)。

## 双活集群简介 > 部署场景

# 部署场景

在施工现场，由于采用的虚拟化平台不同，安装部署 SMTX OS 双活集群所面临的场景可能存在以下两种：

- 在 SMTX Halo 超融合一体机上安装部署 SMTX OS

  该场景只适用于采用 SmartX 原生虚拟化 ELF 平台的双活集群。

  SMTX Halo 超融合一体机在出厂前已经过严格的测试，产品整机能力也已充分优化与验证，因此在现场的一体机上安装部署 SMTX OS 双活集群时，您可以无需校验集群的硬件兼容性，从确认集群的软件版本开始，实施集群的安装部署。
- 在已经准备好的服务器上安装部署 SMTX OS

  该场景适用于两种虚拟化平台：ELF 平台和 VMware ESXi 平台。

SMTX OS 必须运行在硬件兼容的服务器上才能充分发挥其高可靠性和高性能。为了确保在安装部署集群前做好相应的准备，我们建议您认真了解[构建双活集群的要求](/smtxos/6.3.0/elf_multiactive_installation/elf_multiactive_installation_05)，现场逐项核实相关要求，特别是硬件、网络的要求，并做好集群和网络规划。

## 双活集群简介 > 安装方式

# 安装方式

SMTX OS 支持使用独立物理盘安装，在安装 SMTX OS 映像文件时，您可根据系统提示选择将 SMTX OS 安装在两块物理盘中配置软 RAID 1 或使用独立的物理盘安装 SMTX OS。

- **将 SMTX OS 安装在独立的硬 RAID 1 中**

  如果希望系统盘故障时不影响集群系统运行并且能够通过硬件 RAID 重建故障盘，或者希望将系统分区与缓存和数据物理分隔、降低系统盘的 IO 频率以降低故障率，可选择此安装方式。该方式将使用两块 SSD 与存储控制器组成的硬 RAID 1 作为 **SMTX 系统盘**，用于引导和安装 SMTX OS，无需单独的**启动盘**。安装完成后，节点内其他物理盘作为**缓存盘**或**数据盘**用于存放缓存或数据。

  使用此方式安装 SMTX OS 前，请先阅读[将 SMTX OS 安装在硬 RAID 1 中的配置推荐](/smtxos/6.3.0/elf_installation_guide/elf_installation_guide_73)了解推荐配置，以更好地保证系统稳定性和数据安全。
- **将 SMTX OS 安装在软 RAID 1 中**

  如果不需要将系统分区与缓存和数据做物理隔离，或者因硬件限制无法配置单独的硬 RAID 1，可选择此安装方式。该方式需要准备一块单独的物理盘作为**启动盘**用于引导加载程序。在安装 SMTX OS 时，系统将使用两块 SSD 安装 SMTX OS 并配置软 RAID 1，这两块 SSD 称为**含元数据分区的缓存盘**或**含元数据分区的数据盘**。安装完成后，**含元数据分区的缓存盘**和**含元数据分区的数据盘**的剩余存储空间将用于存放缓存或数据，节点内其他物理盘作为**缓存盘**或**数据盘**。

## 构建双活集群的要求 > 集群要求

# 集群要求

规划双活集群时，其规模、优先/次级可用域、仲裁节点、副本数和拓扑需遵照以下要求进行设计。

- **集群规模**

  单个双活集群最少应包含 5 个节点，其中优先可用域 2 个节点，次级可用域 2 个节点，1 个仲裁节点。推荐优先可用域和次级可用域分别至少 3 个节点。为了发挥集群的最高性能，我们推荐单个集群最多不超过 64 个节点。
- **物理盘池数**

  SMTX OS 支持为节点配置物理盘池，每个物理盘池对应一个存储服务，单个集群支持配置最多 255 个物理盘池。不同容量规格的节点支持的物理盘池数量如下：

  - 标准容量规格：支持配置 1 或 2 个物理盘池。
  - 大容量规格：支持配置 1、2 或 4 个物理盘池。

  为实现资源均衡并提升集群稳定性，建议为集群内所有节点配置相同数量的物理盘池。
- **优先/次级可用域**

  优先可用域和次级可用域需最少包含 2 个节点，且均为主节点，并且两个可用域的所有主节点应尽量分布在不同的机架和机箱上，以确保更高的可靠性。
- **仲裁节点**

  仲裁节点可以部署在虚拟机或者物理服务器中。
- **副本数**

  双活集群必须设置 3 副本的存储策略。

请在规划集群时同时记录优先/次级可用域内的每个节点和仲裁节点的服务器序列号，同时标记出两个可用域内每个主节点的服务器序列号。

## 构建双活集群的要求 > 网络要求 > 网络类型、网络部署模式与网络拓扑

# 网络类型、网络部署模式与网络拓扑

在部署双活集群时需要创建虚拟分布式交换机并关联网络。

## 网络类型

虚拟分布式交换机至少需要创建三种类型的虚拟网络：管理网络、存储网络和虚拟机网络。每个虚拟网络可使用其所属的虚拟分布式交换机所关联的物理网口（网卡）进行通信。

- **管理网络**：用于管理集群。
- **存储网络**：用于集群内各节点间数据交换的网络，该网络只能在集群内部使用。
- **虚拟机网络**：用于虚拟机业务通信和对外提供服务。

在双活集群部署结束后，还可以单独为双活集群配置以下两个网络：

- **接入网络**：仅需在集群作为存储集群且需要为集群外部的计算端提供存储服务时创建。
- **迁移网络**：为虚拟机迁移而单独创建的网络。虚拟机在集群内或集群间热迁移时，默认使用存储网络进行迁移。单独为集群创建专属的迁移网络后，跨集群热迁移的数据将通过该网络进行传输，从而避免占用存储网络的带宽。

SMTX OS 将管理网络、存储网络、接入网络和迁移网络统称为系统网络。

SMTX OS 网络的相关概念、网络结构和分层，以及网络数据流的更多介绍，请参考文档[《SMTX OS 网络技术白皮书》](/smtxos/6.3.0/network_whitepaper/network_whitepaper_preface_generic)。

## 网络部署模式

在规划双活集群的网络时，可以根据您的部署环境和实际业务需求，选择为双活集群的系统网络分别设计一个独享的虚拟分布式交换机，以及每个虚拟分布式交换机分别关联一个或者多个物理网口（网络分离部署模式），也可以设计其中几个系统网络共享同一个虚拟分布式交换机和一个或者多个物理网口的带宽。

当双活集群的两个或以上的系统网络（管理网络、存储网络和迁移网络）共享一个虚拟分布式交换机时，即为网络融合部署；当每个系统网络分别独享一个虚拟分布式交换机时，即为网络分离部署。

## 构建双活集群的要求 > 网络要求 > 网络配置要求

# 网络配置要求

双活集群里两个可用域的数据是实时同步的。在正常情况下，所有的数据写入操作都会确保在两个可用域写入存储磁盘后，才会返回成功。而读取操作会优先选择从数据访问点所在可用域内的数据副本上获取数据。因此双活集群的网络配置对集群的写性能影响较大。

请参考《SMTX OS 网络技术白皮书》中针对 ELF 平台的网络拓扑描述，以及双活集群的网络连接要求，确保集群的网络配置满足以下要求。

- **网络带宽**

  双活集群的管理网络带宽不低于 1 Gbps，虚拟机网络带宽不低于 1 Gbps，若虚拟机网络和管理网络共用相同的网口，则可复用管理网络带宽。

  可用域之间的存储网络带宽不低于 10 Gbps，仲裁节点与优先/次级可用域的存储网络之间的带宽应不低于 100 Mbps。如果管理网络和存储网络使用网络融合部署模式，即两个网络共用同一个网口，则网络带宽必须不低于 10 Gbps。

  双活集群的写入带宽上限为两个可用域间的最大存储网络带宽。如果期望更高的吞吐量，则需要对可用域间带宽进行扩容。
- **网络连接**

  集群的优先可用域与次级可用域的主机必须连接到 L2 层管理网络和存储网络。

  仲裁节点与优先/次级可用域的节点之间可通过 L2 层或 L3 层存储网络进行连接。

  > **说明：**
  >
  > 仲裁节点与优先/次级可用域节点通过存储网络连接时不支持 NAT 地址转换。
- **网络延时**

  集群内优先可用域与次级可用域之间的存储网络 ping 延时在 5 ms 以下。

  仲裁节点与优先可用域或次级可用域之间的存储网络 ping 延时在 100 ms 以下。

  数据写入将受到可用域间的链路往返延时影响，尽量降低可用域间存储网络 ping 延时将改善性能影响。当可用域间存储网络 ping 延时为 5 ms 时：

  - 对于队列深度为 1 的小块 I/O（I/O 大小 <= 4k），其影响最为明显，性能将下降至正常情况下的 10% 左右。
  - 对于高队列深度（64）或者大块 I/O（I/O 大小 >= 64k），其性能下降幅度较小，约为 10%。

# 双活集群节点 IP

仲裁节点需要与优先/次级可用域节点的 L2 或 L3 层存储网络连通，因此需为其规划一个 IP 用于存储网络，以及对应的子网掩码和网关等信息。如果希望双活集群支持通过 SSH 方式登录仲裁节点，建议为其再配置一个 IP 用于管理网络。

规划完以上 IP 后，需要将集群每个节点的 IP 与服务器序列号（即服务标签）一一对应。

# 双活集群管理虚拟 IP

设置集群的管理虚拟 IP 是为了保证集群管理界面的高可用。访问此 IP 时，可以自动通过集群内任一可用的主机管理 IP 访问集群。

  - 169.254.0.0/16
  - 240.0.0.0/4
  - 224.0.0.0/4
  - 198.18.0.0/15
  - 127.0.0.0/8
  - 0.0.0.0/8

## 构建双活集群的要求 > 启用其他高级特性/功能要求 > 启用机架拓扑感知

# 启用机架拓扑感知

为了确保在机架断电或损坏等极端状况下，系统还能在其他机架的主机上找到存活的副本，继续提供数据访问能力和增加数据可用性，SMTX OS 提供了机架感知功能。

启用机架拓扑感知功能要求在规划集群时，尽量将服务器平均分布在不同的机架中，从而尽量使数据的多个副本均匀地分配到不同机架和机箱的主机上。

此外，还需在 CloudTower 管理平台正确配置双活集群的物理拓扑。即在双活集群部署结束后，使用 CloudTower 的机架配置功能，如实呈现安装部署现场的主机与机架的位置和拓扑结构，然后系统将根据拓扑来调整数据的分配情况。

## 在节点上安装 SMTX OS

# 在节点上安装 SMTX OS

通过浏览器访问优先/次级可用域节点和仲裁节点的 IPMI 管理台 IP，检查 BIOS 设置，挂载 SMTX OS 映像文件，并远程安装 SMTX OS 软件。

**前提条件**

- 已获得双活集群完整的 IP 地址分配表，并且对每个可用域节点和仲裁节点已配置完 IPMI 管理台 IP。
- 已获取 VMware ESXi 安装文件。

**注意事项**

在安装 SMTX OS 前，建议将所有服务器的引导模式都设置为 `UEFI`。如果服务器使用的 RAID 卡为 `MegaRAID 9560-8i`，请注意将引导模式设置为 `UEFI`。

> **说明：**
>
> - 由于在虚拟机和物理服务器中安装 SMTX OS 软件的步骤相似，下面以仲裁节点部署在物理服务器中为例，描述其安装过程。
> - SMTX OS 不支持在双活集群中启用 SR-IOV 特性，因此在[检查 BIOS 设置](/smtxos/6.3.0/elf_installation_guide/elf_installation_guide_23)时，可忽略该特性的设置要求。

## 在节点上安装 SMTX OS > 检查 BIOS 设置 > 戴尔 Intel x86 服务器

# 戴尔 Intel x86 服务器

请对每个节点执行以下操作。

**操作步骤**

1. 打开浏览器，在浏览器的地址栏输入节点的 IPMI 管理台 IP，登录该节点的 IPMI 管理台。
2. 登录成功后，找到**虚拟控制台**，单击**启动虚拟控制台**，进入虚拟控制台界面。
3. 在界面顶端单击**启动**，在弹出的**开机控制**窗口中单击 **BIOS 设置**，然后在弹出的**确认启动操作**对话框中单击**是**。
4. 在界面顶端单击**功率**，在弹出的**电源控制**窗口中单击**复位系统（热启动）**，然后在弹出的**确认电源操作**对话框中单击**是**，完成后等待服务器重启。
5. 当服务器进入 **System Setup** 界面时，单击 **System BIOS**，进入 **System BIOS Settings** 界面。

   ![](https://cdn.smartx.com/internal-docs/assets/6e80c98e/installationguide_239.png)
6. 在界面中单击 **Boot Settings**，将 **Boot Mode** 选项设置为 **UEFI**，以提升部署效率。

   ![](https://cdn.smartx.com/internal-docs/assets/6e80c98e/installationguide_234.png)
7. 单击 **Back**，返回 **System BIOS Settings** 界面后，单击 **Processor Settings**，检查并确认 **Virtualization Technology** 选项为 **Enabled**，以确保启用 CPU 虚拟化（VT-x）和 I/O 虚拟化（VT-d）特性。

   ![](https://cdn.smartx.com/internal-docs/assets/6e80c98e/installationguide_235.png)

   > **说明**：
   >
   > IOMMU 的实现因 CPU 和服务器厂商不同而有所差异，请以实际服务器 BIOS 设置界面中的选项为准。本示例中，启用 IOMMU 需将 **Virtualization Technology** 设置为 **Enabled**。
8. 单击 **Back**，返回 **System BIOS Settings** 界面后，单击 **System Profile Settings**，检查并确认 **System Profile** 选项为 **Performance**，以确保关闭电源的节能模式。

   ![](https://cdn.smartx.com/internal-docs/assets/6e80c98e/installationguide_236.png)
9. 检查并确认 **Workload Profile** 选项为 **Virtualization Optimized Performance Profile**，以确保系统针对虚拟化场景优化性能。

   ![](https://cdn.smartx.com/internal-docs/assets/6e80c98e/installationguide_237.png)
10. （可选）若计划后续开启 SR-IOV 特性，单击 **Back**，返回 **System BIOS Settings** 界面后，单击 **Integrated Devices**，检查并确认 **SR-IOV Global Enable** 选项为 **Enabled**。

SMTX OS 使用的是向导式的部署界面，您只需按照界面指引，依次完成每个步骤的设置即可成功部署双活集群。但部署时可能还需要为启用某些特性或功能进行特殊设置和操作，建议您在部署集群前，认真阅读[部署须知](/smtxos/6.3.0/elf_multiactive_installation/elf_multiactive_installation_32)，了解每个特性或功能的注意事项。

## 部署 SMTX OS 双活集群 > 部署须知

# 部署须知

对于即将部署的双活集群，必须确保即将关联同一虚拟分布式交换机的所有物理网口互相连通。

当接入双活集群管理网络或存储网络的物理交换机启用 LACP 动态链路聚合，或者为双活集群的管理网络或存储网络配置了 VLAN，或者集群启用了常驻缓存模式时，请参照如下的步骤和要求进行设置和操作。部分配置需在部署结束后才能进行，请同时做好相应的记录。

> **注意**：
>
> SMTX OS 不支持在双活集群中启用 RDMA 功能，因此在[启用 LACP](/smtxos/6.3.0/elf_installation_guide/elf_installation_guide_31)时，请忽略对 RDMA 功能的设置要求。

## 部署 SMTX OS 双活集群 > 部署须知 > 启用 LACP

# 启用 LACP

当接入 SMTX OS 管理网络或存储网络的物理交换机启用了 LACP 动态链路聚合时，需要在部署集群的**第 5 步：配置网络**阶段，为管理网络或存储网络配置虚拟分布式交换机时，在**物理网口**区域，为绑定的网口设置绑定模式。

- 若已启用 RDMA 功能，选择 **802.3ad(LACP)**。
- 若未启用 RDMA 功能，选择 **balance-tcp**。

如果接入**管理网络**的物理交换机启用了 LACP 动态链路聚合，还需要在部署集群前执行以下操作，以绑定管理网口，否则请忽略。

> **注意**：
>
> 当物理交换机为静态 LACP 模式时，请勿执行以下配置，以免影响网络稳定性。

**操作步骤**

1. 使用默认账户 `root` 登录需要设置网口绑定的节点的 SMTX OS 系统。
2. 执行 `cd /etc/sysconfig/network-scripts/` 命令，进入网口配置文件的访问路径。
3. 使用 `ip a` 命令查看所有网卡的接口信息，再使用 `ethtool <port>` 查看每个网卡接口的速率，以判断出用于管理网络的网口，其中 `<port>` 表示实际网口的名称。

   - 当管理网络和存储网络采用分离部署模式时，**Speed** 显示 1000 Mb/s 及以上的网口可以作为用于管理网络的网口。
   - 当管理网络和存储网络采用融合部署模式时，**Speed** 显示 10000 Mb/s 及以上的网口可以作为用于管理网络和存储网络共用的网口。
4. 在网口配置文件路径下执行以下命令，将用于管理网络的网口绑定为一个网口（名称为 `pre_bond`）。

   ```
   network-preconfig create-lacp --nics <nics>
   ```

   其中，`<nics>` 表示需要绑定的网口名称，网口名称之间需用空格隔开，且名称外侧需加引号（"）。例如，当需要绑定网口 eth0 和 eth1 时，则执行 `network-preconfig create-lacp --nics "eth0 eth1"`。

   当输出以下信息时，表明 pre\_bond 网口创建完成，绑定管理网口成功。

   ```
   create lacp bond port success
   ```
5. 可选：如果物理交换机没有为管理网络配置 Trunk VLAN，需要执行以下命令，为 pre\_bond 网口配置 IP 地址、子网掩码和网关。如果物理交换机已为管理网络配置 Trunk VLAN，则无需执行该步骤，请参考[配置 VLAN](/smtxos/6.3.0/elf_installation_guide/elf_installation_guide_50) 小节进行操作。

   ```
   network-preconfig config-bond-address --ipaddr <ipaddr> --netmask <netmask> --gateway <gateway>
   ```

   其中，带有`<>`符号的字段表示管理网络的参数。

   | 参数 | 说明 |
   | --- | --- |
   | `ipaddr` | 表示实际规划的 IP 地址。 |
   | `netmask` | 表示实际规划的子网掩码。 |
   | `gateway` | 表示实际规划的网关。 |

   当系统输出以下信息时，表明 pre\_bond 网口的 IP 地址、子网掩码和网关配置成功。

   ```
   config ip address for pre_bond interface success
   ```
6. 可选：如果需要重新绑定网口，可先执行以下命令清除所有配置，然后根据输出信息执行下一步操作。

   ```
   network-preconfig clear-config
   ```

   当输出以下信息时，表明 pre\_bond 网口的配置清理成功，请参考步骤 3 ～ 4 重新绑定管理网口。

   ```
   delete pre_bond configuration
   ```

   如果为管理网口配置过 VLAN 子接口，还会输出以下信息，表明 VLAN 子接口的配置也被清理。若需要重新创建 VLAN 子接口，请参考**配置 VLAN** 进行部署集群前的配置。

   ```
   delete pre_vlan configuration
   ```

## 部署 SMTX OS 双活集群 > 部署须知 > 配置 VLAN

# 配置 VLAN

当物理交换机为 SMTX OS 管理网络或存储网络配置了 VLAN 时，需要在部署集群的**第 5 步：配置网络**阶段，在为管理网络或存储网络配置虚拟分布式交换机时，填写 VLAN ID。

如果物理交换机为管理网络配置了 Trunk VLAN，还需要在部署集群前执行以下操作，以为管理网口配置子接口，否则请忽略。

**操作步骤**

1. 用默认账户 `root` 登录每个节点的 SMTX OS 系统。
2. 使用 `cd /etc/sysconfig/network-scripts/` 命令，进入网口配置文件的访问路径。
3. 使用 `ip a` 命令查看所有网卡的接口信息，再使用 `ethtool <port>` 查看每个网卡接口的速率，以判断并确定用于管理网络的网口，其中 `<port>` 表示实际网口的名称。

   - 当管理网络和存储网络采用分离部署模式时，**Speed** 显示 1000 Mb/s 及以上的网口可以作为用于管理网络的网口。
   - 当管理网络和存储网络采用融合部署模式时，**Speed** 显示 10000 Mb/s 及以上的网口可以作为用于管理网络和存储网络共用的网口。
4. 在网口配置文件路径下执行以下命令，为网口创建 VLAN 子接口，并配置 IP、网关和子网掩码。

   ```
   network-preconfig create-vlanif --iface <iface> --ipaddr <ipaddr> --netmask <netmask> --gateway <gateway> --vlanid <vlanid>
   ```

   其中，带有`<>`符号的字段表示管理网络的参数。

   | 参数 | 说明 |
   | --- | --- |
   | `iface` | 表示网口名称。 - 当计划接入 SMTX OS 管理网络的物理交换机启用了 LACP 动态链路聚合时，若当前节点用于管理网络的网口为 pre\_bond 网口，请替换为 `pre_bond`。 - 若当前节点用于管理网络的网口不为 pre\_bond 网口，请替换为实际的网口名称，如 `eth0`。 |
   | `ipaddr` | 表示实际规划的 IP 地址。 |
   | `netmask` | 表示实际规划的子网掩码。 |
   | `gateway` | 表示实际规划的网关。 |
   | `vlanid` | 表示实际规划的 VLAN ID。 |

   当输出以下信息时，表明 VLAN 子接口创建完成，管理网口已成功配置子接口。

   ```
   create vlan interface success
   ```
5. 可选：如果需要重新为管理网口配置 VLAN 子接口，可先执行以下命令清除所有配置，然后根据输出信息执行下一步操作。

   ```
   network-preconfig clear-config
   ```

   当输出以下信息时，表明 VLAN 子接口的配置清理成功，请参考步骤 3 ～ 4 重新为管理网口配置 VLAN 子接口。

   ```
   delete pre_vlan configuration
   ```

   如果绑定过管理网口，还会输出以下信息，表明 pre\_bond 网口的配置也已被清理。若需要重新绑定管理网口，请参考**启用 LACP** 进行部署集群前的配置。

   ```
   delete pre_bond configuration
   ```

## 部署 SMTX OS 双活集群 > 配置 Port Channel 哈希策略

# 配置 Port Channel 哈希策略

当接入 SMTX OS 集群存储网络的物理交换机启用了 LACP 动态链路聚合时，建议在物理交换机上配置 Port Channel 哈希策略，在默认参数的基础上增加根据源端口和目的端口作为哈希字段，以便充分利用多网口的带宽。下文分别以 Mellanox 交换机和华为交换机为例，介绍配置 Port Channel 哈希策略的操作。

## Mellanox 交换机

1. 执行以下命令，以管理员身份登录交换机，进入配置模式。

   ```
   switch > enable
   switch # configure terminal
   ```
2. 执行 `show interfaces port-channel load-balance` 命令，检查当前 Port Channel 哈希字段是否包含 `source-destination-port`。

   - 若输出结果中已包含 `source-destination-port`，则无需执行后续配置操作。
   - 若输出结果中不包含 `source-destination-port`，请继续参考步骤 3～5 进行配置。
3. 执行以下命令，在现有字段的基础上增加 `source-destination-port`。

   ```
   switch (config) # port-channel load-balance ethernet source-destination-mac source-destination-port
   ```
4. 执行 `show interfaces port-channel load-balance` 命令，检查 Port Channel 哈希字段。

   若输出结果中包含 `source-destination-port` ，说明配置成功。

   ```
   switch (config) # show interfaces port-channel load-balance
   source-destination-mac, source-destination-port
   ```
5. 执行 `configuration write` 命令，保存配置。

## 华为交换机

1. 执行以下命令，以管理员身份登录交换机，进入配置模式。

   ```
   <6865-2>system-view
   Enter system view, return user view with return command.
   ```
2. 执行 `display load-balance profile` 命令，查看负载分担模板的详细信息，确认输出结果中 IP 一行的哈希字段是否包含 `l4-src-port` 和 `l4-dst-port`。

   - 若输出结果中已包含 `l4-src-port` 和 `l4-dst-port`且不包含 `dst-qp`，则无需修改。输出示例如下：

     ```
     [~6865-2]display load-balance profile
     Load-balance Profile: default
     Packet    HashField
     ---------------------------------------------------
     IP             src-ip          dst-ip
                 l4-src-port     l4-dst-port
     IPv6           src-ip          dst-ip
                 l4-src-port     l4-dst-port
     L2             src-mac         dst-mac
     MPLS           top-label       2nd-label
     Eth-Trunk      universal-id(1) hash-mode(1)
     Stack          universal-id(1) hash-mode(1)
     ```
   - 若输出结果中不包含 `l4-src-port` 和 `l4-dst-port` 或包含 `dst-qp`，则需要手动修改，否则会影响 RDMA 的连接。请执行如下命令更新默认 profile，添加 `l4-src-port` 和 `l4-dst-port` 并删除 `dst-qp`。

     ```
     [~6865-2]load-balance profile default
     [~6865-2-load-balance-profile-default]ip src-ip dst-ip l4-src-port l4-dst-port
     [*6865-2-load-balance-profile-default]commit
     [~6865-2-load-balance-profile-default]quit
     ```

     更新完成后，再次执行 `display load-balance profile` 命令，确认配置结果符合预期。
3. 执行 `display interface Eth-Trunk 1 | include "Hash"` 命令查看 Eth-Trunk 接口的哈希策略。

   - 若输出结果中 Eth-Trunk 已经默认采用系统的 profile 作为哈希策略，则无需修改。输出示例如下：

     ```
     [~6865-2]display interface Eth-Trunk 1 | include "Hash"
     Switch Port, PVID :    1, TPID : 8100(Hex), Hash Arithmetic : profile default, The Maximum Frame Length is 9216
     ```
   - 若输出结果中 Eth-Trunk 未默认采用系统的 profile 作为哈希策略，请执行如下命令配置 Eth-Trunk。

     ```
     [~6865-2]interface Eth-Trunk 1
     [~6865-2-Eth-Trunk1]load-balance enhanced profile default
     [~6865-2-Eth-Trunk1]commit
     [~6865-2-Eth-Trunk1]quit
     ```

     配置完成后，再次执行 `display interface Eth-Trunk 1 | include "Hash"` 命令，确认配置结果符合预期。

## 部署 SMTX OS 双活集群 > 检查并设置双活集群部署的环境 > 为优先/次级可用域节点设置部署的环境

# 为优先/次级可用域节点设置部署的环境

**注意事项**

当物理交换机为集群的管理网络配置了 VLAN 时，请使用 `root` 账户登录每个优先/次级可用域节点的 SMTX OS 系统，然后从步骤 3 开始执行。

**操作步骤**

1. 使用默认账户 `root` 登录每个优先/次级可用域节点的 SMTX OS 系统，确认用于管理网络和存储网络的网口。

   1. 使用 `cd /etc/sysconfig/network-scripts/` 命令，进入网口配置文件的访问路径。
   2. 使用 `ip a` 命令查看所有网卡的接口信息，再使用 `ethtool <port>` 查看每个网卡接口的速率，以判断并确定用于管理网络和存储网络的网口，其中 `<port>` 表示实际网口的名称。

      - 当管理网络和存储网络采用分离部署模式时，**Speed** 显示 1000 Mb/s 及以上的网口可以作为用于管理网络的网口；**Speed** 显示 10000 Mb/s 及以上的网口可以作为用于存储网络的网口。
      - 当管理网络和存储网络采用融合部署模式时，**Speed** 显示 10000 Mb/s 及以上的网口可以作为用于管理网络和存储网络共用的网口。
2. 确认当前配置路径下是否存在管理网络网口的对应配置文件，如果网口名称是 `ens4f0np0` ，那么预期会存在 `ifcfg-ens4f0np0` 配置文件。如果当前配置路径下不存在管理网络网口的对应配置文件，可以从当前配置路径下已有的其他网口配置文件拷贝后进行修改。如网口名称是 `ens4f0np0` 且配置文件不存在，而当前路径存在 `ifcfg-ens4f0` 配置文件，那么可以执行如下操作来生成网口对应配置文件：

   1. 使用 `cp ifcfg-ens4f0 ifcfg-ens4f0np0` 命令拷贝配置文件。
   2. 使用 `vi ifcfg-ens4f0np0` 命令打开 `ifcfg-ens4f0np0` 配置文件。
   3. 将 `DEVICE=ens4f0` 修改为 `DEVICE=ens4f0np0`。
3. 配置用于管理网络的网口。

   使用 `vi ifcfg-<port>` 命令，打开用于管理网络的网口的配置文件，并进行以下修改，其中 `<port>` 表示该网口的名称。

   1. 修改 **BOOTPROTO=dhcp** 为 **BOOTPROTO=static**
   2. 修改 **ONBOOT=no** 为 **ONBOOT=yes**
   3. 在配置文件末尾添加以下内容：

      ```
      IPADDR=manageIP_address
      NETMASK=manageIP_netmask
      GATEWAY=manageIP_gateway
      ```

      其中 **`manageIP_address`**、**`manageIP_netmask`** 和 **`manageIP_gateway`** 为实际规划的优先/次级可用域节点的 SMTX OS 管理 IP 地址、子网掩码和网关。
4. 为双活集群的部署控制节点配置存储网络的网口。

   使用 `vi ifcfg-<port>` 命令，打开用于存储网络的网口的配置文件，并进行以下修改，其中 `<port>` 表示该网口的名称。

   1. 修改 **BOOTPROTO=dhcp** 为 **BOOTPROTO=static**
   2. 修改 **ONBOOT=no** 为 **ONBOOT=yes**
   3. 在配置文件末尾添加以下内容：

      ```
      IPADDR=control_storageIP_address
      NETMASK=control_storageIP_netmask
      ```

      其中 `control_storageIP_address` 和 `control_storageIP_netmask` 为实际规划的部署控制节点的 SMTX OS 存储 IP 和子网掩码。
5. （可选）在每个节点的 SMTX OS 系统中，执行以下命令，修改主机名称，其中 `<name>` 表示重命名后的主机名称。建议使用辨识度较高的主机名称，从而在部署集群的配置主机时，可以更方便地依据主机名称辨认和勾选集群节点。

   ```
   hostname <name>
   su
   ```
6. 在每个优先/次级可用域节点中执行 `systemctl restart network` 命令，重启服务。
7. 两个可用域中所有节点的网口都配置完成后，测试节点网络，确保节点间的管理网络能够互相连通。
8. 在每个优先/次级可用域节点中执行 `systemctl status nginx` 命令，确认 nginx 服务已启动。
9. 在每个优先/次级可用域节点中执行 `systemctl status zbs-deploy-server` 命令，确认 zbs-deploy-server 服务已启动。
10. 使用 date 命令将节点的时间调整为当前时间，例如 `date -s "2022-11-08 15:21:23"`。在每个优先/次级可用域节点中执行命令后，系统将同步所有节点的时间，避免因集群内节点的时间设置不同步，导致集群无法正常工作。

## 部署 SMTX OS 双活集群 > 检查并设置双活集群部署的环境 > 为仲裁节点设置部署的环境

# 为仲裁节点设置部署的环境

1. 登录仲裁节点的 SMTX OS 系统，确认并设置存储网络和管理网络的网卡、IP 地址和子网掩码等信息。

   1. 使用 **root** 账户登录 SMTX OS 系统，使用 `cd /etc/sysconfig/network-scripts/` 命令，进入配置文件访问路径。
   2. 使用 `ip a` 命令查看所有网卡的接口信息，再使用 `ethtool <port>` 查看每个网卡接口的速率，以判断并确定用于存储网络和管理网络的网口，其中 `<port>` 表示实际网口的名称。

      **Speed** 显示 1000 Mb/s 及以上，表示该网口可设置为存储网络或管理网络的网口。
   3. 配置用于存储网络的网口。

      使用 `vi ifcfg-<port>` 命令，打开用于存储网络的网口的配置文件，并进行以下修改，其中 `<port>` 表示该网口的名称。

      1. 修改 **BOOTPROTO=dhcp** 为 **BOOTPROTO=static**
      2. 修改 **ONBOOT=no** 为 **ONBOOT=yes**
      3. 在配置文件末尾添加以下内容：

         ```
         IPADDR=storageIP_address
         NETMASK=storageIP_netmask
         MACADDR=storageIP_mac
         ```

         其中 **`storageIP_address`**、**`storageIP_netmask`** 和 **`storageIP_mac`** 为实际规划的仲裁节点的 SMTX OS 存储 IP 地址、子网掩码和 MAC 地址。
   4. （可选）若为仲裁节点规划了管理 IP 时，参考下面步骤为其配置用于管理网络的网口。

      使用 `vi ifcfg-<port>` 命令，打开用于管理网络的网口配置文件，并进行以下修改，其中 `<port>` 表示该网口的名称。

      1. 修改 **BOOTPROTO=dhcp** 为 **BOOTPROTO=static**
      2. 修改 **ONBOOT=no** 为 **ONBOOT=yes**
      3. 在配置文件末尾添加以下内容：

         ```
         IPADDR=manageIP_address
         MACADDR=manageIP_mac
         NETMASK=manageIP_netmask       
         GATEWAY=manageIP_gateway
         ```

         其中 **`manageIP_address`**、**`manageIP_mac`**、**`manageIP_netmask`** 和 **`manageIP_gateway`** 为实际规划的仲裁节点的 SMTX OS 管理 IP、管理网卡 MAC 地址、子网掩码和网关。
   5. 执行 `systemctl restart network` 命令，重启服务。
2. 输入 `ping control_storageIP_address` 命令，确认仲裁节点与双活集群两个可用域节点 L2 层存储网络连通。其中 **`control_storageIP_address`** 为双活集群部署控制节点的存储 IP。

   > **注意**：
   >
   > 若仲裁节点与集群两个可用域里的主机 L2 层存储网络无法连通, 请参考[为双活集群配置静态路由](/smtxos/6.3.0/elf_multiactive_installation/elf_multiactive_installation_38)的步骤，手动配置可用域的部署控制节点与仲裁节点间的静态路由，以确保仲裁节点与集群可用域里的部署控制节点之间的存储网络可连通。
3. （可选）输入以下命令，修改主机名称，使主机名称更有辨识度。在部署集群的配置主机阶段，可以根据主机名称辨认集群节点。

   ```
   hostname <name>
   su
   ```
4. 执行命令 `systemctl status nginx`，确认 nginx 服务启动。
5. 执行命令 `systemctl status zbs-deploy-server`，确认 zbs-deploy-server 服务启动。
6. 使用 date 命令将仲裁节点的时间调整为当前时间，例如 `date -s "2022-11-08 15:21:23"`，并确认与可用域节点时间保持一致。
7. （可选）如果仲裁节点安装在虚拟机中，请按照此虚拟机的 Hypervisor 类型自行配置高可用 HA 方案。

## 部署 SMTX OS 双活集群 > 检查并设置双活集群部署的环境 > 为双活集群配置静态路由（可选）

# 为双活集群配置静态路由（可选）

**前提条件**

在[为仲裁节点设置部署的环境](/smtxos/6.3.0/elf_multiactive_installation/elf_multiactive_installation_37)时，若仲裁节点与集群两个可用域里的主机之间 L2 层存储网络无法连通，此时需要手动配置仲裁节点与部署控制节点（在部署集群时准备通过其管理 IP 访问集群进行部署的节点）间的静态路由。

**操作步骤**

1. 在仲裁节点上添加到部署控制节点的静态路由。

   1. 使用 **root** 账户登录 SMTX OS 系统, 执行下述命令，添加仲裁节点到部署控制节点的静态路由。

      ```
      ip route add 10.10.1.100/32 via 10.10.2.2 dev eth0
      ```

      其中 `10.10.1.100` 为部署控制节点的存储 IP，`10.10.2.2` 为规划的仲裁节点到部署控制节点所在可用域的网关地址，请用户根据实际情况替换。
   2. 执行命令 `ip route show`，查看仲裁节点的路由表，确认上述添加的路由条目已刷新至路由表。
2. 在部署控制节点上添加到仲裁节点的静态路由。

   1. 使用 **root** 账户登录 SMTX OS 系统, 执行下述命令，添加部署控制节点到仲裁节点的静态路由。

      ```
      ip route add 10.10.2.1/32 via 10.10.1.253 dev eth1
      ```

      其中 `10.10.2.1` 为仲裁节点的存储 IP，`10.10.1.253` 为规划的部署控制节点所在可用域到仲裁节点的网关地址。
   2. 执行命令 `ip route show`，查看部署控制节点的路由表，确认上述添加的路由条目已刷新至路由表。
3. 使用 **root** 账户登录仲裁节点或部署控制节点的 SMTX OS 系统，确认仲裁节点与部署控制节点之间的网络可连通。其中 `storageIP_address` 为部署控制节点或者仲裁节点的存储 IP。

   ```
   ping storageIP_address
   ```

**配置实例**

以下图中的同城双活集群为例，若集群两个可用域 IDC A、IDC B 和 IDC C 内仲裁节点的网络配置如下：

![](https://cdn.smartx.com/internal-docs/assets/0ecdf211/multiactive_installation_002.png)

- IDC A 和 IDC B 位于同一子网内，各节点的存储网络均位于 `10.0.255.0/24` 网段； IDC C 位于另一子网，仲裁节点的存储网络位于 `40.0.0.0/24` 网段。
- 将 IDC A 的前两个节点设置为主节点，其存储网络分别配置为 `10.0.255.11/24`、`10.0.255.12/24`，第三个节点的存储网络配置为 `10.0.255.13/24`，三个节点到仲裁节点的路由网关为 `10.0.255.253`。
- 将 IDC B 的前两个节点设置为主节点，其存储网络分别配置为 `10.0.255.14/24`、`10.0.255.15/24`，第三个节点的存储网络配置为 `10.0.255.16/24`，三个节点到仲裁节点的路由网关为 `10.0.255.254`。
- IDC C 内仲裁节点的存储网络配置为 `40.0.0.11/24`。IDC C 到 IDC A 可用域网关配置为 `40.0.0.253`，IDC C 到 IDC B 可用域网关配置为 `40.0.0.254`。

可见，IDC C 内的仲裁节点与 IDC A、IDC B 的节点之间 L2 层存储网络不连通，需配置为 L3 层存储网络连通。

则 IDC A 内各节点至仲裁节点之间的静态路由应参考如下进行配置：

| 节点存储网络 | 静态路由配置 |
| --- | --- |
| 10.0.255.11/24 | 40.0.0.11/32 via 10.0.255.253 |
| 10.0.255.12/24 | 40.0.0.11/32 via 10.0.255.253 |
| 10.0.255.13/24 | 40.0.0.11/32 via 10.0.255.253 |

IDC B 内三个节点至仲裁节点之间的静态路由应参考如下进行配置：

| 节点存储网络 | 静态路由配置 |
| --- | --- |
| 10.0.255.14/24 | 40.0.0.11/32 via 10.0.255.254 |
| 10.0.255.15/24 | 40.0.0.11/32 via 10.0.255.254 |
| 10.0.255.16/24 | 40.0.0.11/32 via 10.0.255.254 |

仲裁节点 C 至两个可用域内每个节点之间的静态路由应参考如下进行配置：

| 序号 | 静态路由配置 |
| --- | --- |
| 1 | 10.0.255.11/32 via 40.0.0.253 |
| 2 | 10.0.255.12/32 via 40.0.0.253 |
| 3 | 10.0.255.13/32 via 40.0.0.253 |
| 4 | 10.0.255.14/32 via 40.0.0.254 |
| 5 | 10.0.255.15/32 via 40.0.0.254 |
| 6 | 10.0.255.16/32 via 40.0.0.254 |

## 部署 SMTX OS 双活集群 > 第 4 步：配置存储

# 第 4 步：配置存储

1. 根据集群要采用的存储分层或不分层模式，选择**分层**或**不分层**。

   系统会根据所选的模式和物理盘类型自动选择物理盘用途。

   > **注意**：
   >
   > 只有选择存储**分层**模式，才能支持启用常驻缓存模式。
2. 当采用存储分层模式部署时，若集群内所有节点均为单一类型多种属性 SSD 全闪配置，且 SMTX OS 安装在软 RAID 1 中，请选择系统盘用途为**含元数据分区的缓存盘**，否则请忽略。
3. 当集群节点满足多物理盘池的配置要求时，系统会根据节点的存储配置自动选择推荐的物理盘池数量，并分配物理盘及其用途。您也可以手动修改物理盘池数量，建议各节点的物理盘池数量保持一致。

   > **注意**：
   >
   > - 单集群最多支持配置 255 个物理盘池。
   > - 单物理池的数据盘总容量上限为 256 TB，缓存盘总容量上限为 51 TB。
   > - 存储分层模式下：混闪配置、多种类型 SSD 全闪配置，或单一类型多种属性 SSD 全闪配置，单物理盘池的物理盘数量上限为 64。
   > - 存储不分层模式下：单物理盘池的物理盘数量上限为 32。
   > - 集群部署完成后，不支持修改物理盘池数量。
4. 参考在确认[优先/次级可用域节点的配置要求](/smtxos/6.3.0/elf_multiactive_installation/elf_multiactive_installation_18)时已记录的每块物理盘的用途，确认系统自动选择的物理盘用途。

   - **SMTX 系统盘**：即[将 SMTX OS 安装在独立的硬 RAID 1 中](/smtxos/6.3.0/aftersales_guide/elf_installation_guide/elf_installation_guide_71)的步骤 4 所选择的物理盘，不属于任何物理盘池，不可修改用途。
   - **含元数据分区的缓存盘**和**含元数据分区的数据盘**：即[将 SMTX OS 安装在软 RAID 1 中](/smtxos/6.3.0/aftersales_guide/elf_installation_guide/elf_installation_guide_25)的步骤 5 所选择的物理盘，不可修改用途，可移至其他物理盘池。
   - **其余物理盘**：

     存储分层模式部署时：

     - 若节点为混闪配置，一般只有当系统判断的物理盘类型有误时，才需要手动修改用途，可将物理盘移至其他物理盘池或选择**不挂载**。
     - 若节点为多种类型 SSD 全闪配置，当计划预留常驻缓存比例高于 75% 时，需要手动将所有**数据盘**用途修改为**缓存盘**；其余情况下，只有当系统判断的物理盘类型有误时，才需要手动修改物理盘用途。可将物理盘移至其他物理盘池或选择**不挂载**。
     - 若节点为单一类型且多种属性的 SSD 全闪配置，物理盘默认为**数据盘**，需手动将高性能 SSD 的用途修改为**缓存盘**；当计划预留常驻缓存比例高于 75% 时，需要手动将所有**数据盘**的用途修改为**缓存盘**。可将物理盘移至其他物理盘池或选择**不挂载**。

     存储不分层模式部署时：所有物理盘默认为数据盘，不可修改用途，可移至其他物理盘池或选择不挂载。
5. 单击**配置网络**，进入下一步。

## 部署 SMTX OS 双活集群 > 第 5 步：配置网络

# 第 5 步：配置网络

## 配置虚拟分布式交换机

选择管理网络和存储网络所要采用的网络部署模式，系统会根据所选模式自动创建虚拟分布式交换机并关联网络，默认情况下，SMTX OS 的网络部署模式为分离式部署。

### 选择分离式部署

选择分离式部署时，系统会创建两个虚拟分布式交换机并分别关联管理网络和存储网络，需要您分别对两个虚拟分布式交换机进行配置。

1. 配置管理网络所关联的虚拟分布式交换机。

   1. 填写虚拟分布式交换机的名称。
   2. 可选：当接入 SMTX OS 管理网络的物理交换机为管理网络配置了 VLAN 时，请在 **VLAN ID** 字段下填写实际规划的 VLAN ID。
   3. （可选）若要保证管理网络的流量使用需求，可以启用 **QoS 策略**并进行以下设置。

      | 字段 | 说明 | 取值/示例 |
      | --- | --- | --- |
      | 优先级及权重 | 该网络在虚拟分布式交换机中的带宽分配优先级。 - 高（100） - 中（60） - 低（30） - 自定义权重：可设置 0 ~ 100 之间的权重 | “100” |
      | 预留带宽 | 该网络在虚拟分布式交换机中保证预留的带宽。  虚拟分布式交换机中所有虚拟网络的预留带宽之和不允许超过总带宽的 75%。 | “1000 Mbps” |
      | 最大带宽 | 启用**限制带宽**后，将限制该网络可达到的最大带宽。 | “3000 Mbps” |
      | 突发通信量上限 | 启用**限制带宽**后，设置该网络以超出最大带宽的速率进行通信时，最多可完成通信的数据量。 | “300 Mb” |
   4. 在**物理网口**区域的**关联网口**字段下，给每个主机选择用于管理网络的物理网口。

      - 可以根据显示的网口速率、连接状态和 MAC 地址等判断合适的网口，所选的网口速率需为 1 GbE 及以上。
      - 若各个主机的关联网口名称相同，可以在设置完一个主机的关联网口后，单击右侧的 ... 图标，选择**选中其他主机上的同名网口**。
      - 当接入 SMTX OS 管理网络的物理交换机启用了 LACP 动态链路聚合时，请为需要绑定网口的主机设置**网口绑定模式**为 `balance-tcp`。如下图。

        ![](https://cdn.smartx.com/internal-docs/assets/0ecdf211/multiactive_installation_015.png)

   下图为已配置 VLAN，已启用 Qos 策略，且未设置网口绑定模式的配置界面。

   ![](https://cdn.smartx.com/internal-docs/assets/0ecdf211/multiactive_installation_016.png)
2. 配置存储网络所关联的虚拟分布式交换机。

   1. 填写虚拟分布式交换机的名称。
   2. （可选）当接入 SMTX OS 存储网络的物理交换机为存储网络配置了 VLAN 时，请在 **VLAN ID** 字段下填写实际规划的 VLAN ID。若要启用 RDMA 功能，则此处填写的 VLAN ID 必须为 0。
   3. （可选）若要保证存储网络的流量使用需求，可以启用 **QoS 策略**，设置方式与管理网络相同。
   4. 在**物理网口**区域的**关联网口**字段下，给每个主机选择用于存储网络的物理网口。

      - 可以根据显示的网口速率、连接状态和 MAC 地址等判断合适的网口，所选的网口速率需为 10 GbE 及以上。
      - 若各个主机的关联网口名称相同，可以在设置完一个主机的关联网口后，单击右侧的 ... 图标，选择**选中其他主机上的同名网口**。
      - 当接入 SMTX OS 存储网络的物理交换机启用了 LACP 动态链路聚合时，请为需要绑定网口的主机设置**网口绑定模式** 为 `balance-tcp`。

### 选择融合部署

选择融合部署时，系统会创建一个虚拟分布式交换机并同时关联管理网络和存储网络，需要您对该虚拟交换机进行配置。

1. 填写虚拟分布式交换机的名称。
2. （可选）当接入 SMTX OS 管理网络和存储网络的物理交换机为管理网络或存储网络配置了 VLAN 时，请在对应网络的 **VLAN ID** 字段下填写实际规划的 VLAN ID。
3. （可选）若要保证管理网络或存储网络的流量使用需求，可以为对应网络启用 **QoS 策略**并进行以下设置。

   | 字段 | 说明 | 取值/示例 |
   | --- | --- | --- |
   | 优先级及权重 | 该网络在虚拟分布式交换机中的带宽分配优先级。 - 高（100） - 中（60） - 低（30） - 自定义权重：可设置 0 ~ 100 之间的权重 | “100” |
   | 预留带宽 | 该网络在虚拟分布式交换机中保证预留的带宽。  虚拟分布式交换机 中所有虚拟网络的预留带宽之和不允许超过总带宽的 75%。 | “1000 Mbps” |
   | 最大带宽 | 启用**限制带宽**后，将限制该网络可达到的最大带宽。 | “3000 Mbps” |
   | 突发通信量上限 | 启用**限制带宽**后，设置该网络以超出最大带宽的速率进行 通信时，最多可完成通信的数据量。 | “300 Mb” |
4. 在**物理网口**区域的**关联网口**字段下，给每个主机选择管理网络和存储网络共用的物理网口。

   - 可以根据显示的网口速率、连接状态和 MAC 地址等判断合适的网口，所选的网口速率需为 10 GbE 及以上。
   - 当接入 SMTX OS 管理网络和存储网络的物理交换机启用了 LACP 动态链路聚合时，请为需要绑定网口的主机设置**网口绑定模式**为 **balance-tcp**。

## 为主节点配置 Meta 服务的优先级

# 为主节点配置 Meta 服务的优先级

在将集群的主机分配至优先可用域和次级可用域后，还需要对每个主节点配置 Meta 服务的优先级。

1. 登录主节点，执行 `$cat /etc/zbs/zbs.conf` 命令查看配置文件。通过其 `members` 字段查找各节点的 IP 地址列表。

   ```
   [root@m-31-176 17:48:06 ~]$cat /etc/zbs/zbs.conf
   [network]
   data_ip = 10.10.31.176
   heartbeat_ip = 10.10.31.176
   vm_ip = 192.168.31.176
   web_ip = 192.168.31.176

   [cluster]
   role = master
   members = 10.10.25.47,10.10.31.176,10.10.29.172,10.10.29.232
   zookeeper =  10.10.25.47:2181,10.10.31.176:2181,10.10.29.172:2181,10.10.29.232:2181,10.10.28.17: 2181
   mongo =  10.10.25.47:27017,10.10.31.176:27017,10.10.29.172:27017,10.10.29.232:27017,10.10.28 .17:27017
   cluster_mgt_ips =  192.168.23.131,192.168.25.47,192.168.29.172,192.168.29.232,192.168.31.176,192.168.3 1.204
   cluster_storage_ips =  10.10.23.131,10.10.25.47,10.10.29.172,10.10.29.232,10.10.31.176,10.10.31.204

   [zone]
   ```

2. 配置主节点的 Meta 服务优先级。

   根据可用域拓扑配置，`members` 中位于优先可用域的节点，其优先级为 2；位于次级可用域的节点，其优先级为 1。若准备将 `10.10.25.47` 和 `10.10.31.176` 配置在优先可用域，`10.10.29.172` 和 `10.10.29.232` 配置在次级可用域，则可以通过执行以下命令实现：

## 部署后配置

# 部署后配置

完成双活集群部署后，必须通过配置硬件拓扑来确保数据副本跨可用域分布，这是实现故障切换能力的核心基础；同时，还需设置为双活集群设置虚拟机高可用性，从而保障在单一数据中心发生故障时，虚拟机与存储服务能由另一站点无缝接管，最终使双活集群的高可用性得以充分发挥。此外，如果因业务需要，要求虚拟机在适当的主机上运行，还需为虚拟机设置虚拟机放置组规则。该规则将与虚拟机高可用功能协同工作，自动将虚拟机调度至符合要求的主机运行，从而保障业务连续性。

## 部署后配置 > 为双活集群设置虚拟机高可用性

# 为双活集群设置虚拟机高可用性

在双活集群部署成功后，必须为集群中的所有虚拟机设置高可用（HA）。设置后，双活集群默认遵循优先可用域优先的模式进行 HA 调度，即当优先可用域内节点资源满足虚拟机启动的前提下，虚拟机触发 HA 时将始终选择优先可用域内的节点作为 HA 目的地。当双活集群的优先可用域发生故障时，可用域内的虚拟机将自动在次级可用域的健康主机上重建。从而减少了业务中断时间，保证了业务的连续性。具体操作请参考《管理指南》的[为集群中的所有虚拟机设置高可用](/smtxos/6.3.0/os_administration_guide/os_administration_guide_049)小节。

## 部署后配置 > 设置虚拟机放置组规则（可选）

# 设置虚拟机放置组规则（可选）

如果因业务需求限制，要求虚拟机在适当的节点上运行，可以通过设置虚拟机放置组来实现。双活集群支持选择配置**可用域放置组策略**或**主机放置组策略**，您可根据实际需要选择其中一种进行配置。配置要求如下：

- **可用域放置组策略**

  对于运行在可用域中的系统服务虚拟机，或对延时敏感的业务虚拟机，建议配置此策略，以避免虚拟机所在主机故障后，自动 HA 到另一个可用域导致系统服务失效或影响业务稳定性。

  - **必须放置在优先/次级可用域上**

    勾选此策略后，需要指定双活集群中的优先可用域或次级可用域。当发生故障触发 HA 时，虚拟机仅会在所选可用域中的主机上重建，即使所选可用域内无可用主机，虚拟机也不会在另一可用域中的主机上重建。
  - **优先放置在优先/次级可用域上**

    勾选此策略后，需要指定双活集群中的优先可用域或次级可用域。当发生故障触发 HA 时，虚拟机会优先在所选可用域中的主机上重建，如果所选可用域中的主机均不可用，系统会自动在另一可用域中的主机上重建虚拟机。
- **主机放置组策略**

  - **必须放置/不放置在选定的主机上**

    若勾选了此策略，并设置为：

    - **必须放置在选定的主机上**，推荐分别在两个可用域内至少指定一台主机，否则有可能无法保证业务的连续性。例如，在同时用了此条策略和开启 HA 的情况下，若此策略中所选的主机都在优先可用域 A，当整个优先可用域 A 的主机都不可用时，由于该策略限制虚拟机 HA 的目的地必须在选定的主机上，因此无法将虚拟机自动 HA 到次级可用域 B 的主机上，从而导致业务中断。
    - **必须不放置在选定的主机上**，推荐不要指定某个可用域内的所有主机，否则有可能无法保证业务的连续性。例如，同时启用此条策略和开启 HA 的情况下，若此策略中所选的主机都在次级可用域 B，当整个优先可用域 A 都不可用时，由于该策略限制虚机 HA 的目的地必须不在选定的主机上，因此无法将虚拟机自动 HA 到次级可用域 B 的主机上，从而导致业务中断。
  - **优先放置/不放置在选定的主机上**

    若勾选了此策略，并设置为**优先放置在选定的主机上**，不推荐指定次级可用域 B 的主机，从而可以保障在触发 HA 后，虚拟优先 HA 到优先可用域 A 的主机上。即使整个优先可用域 A 不可用，由于此策略为**优先**策略，而非**必须**策略，系统最仍会自动将虚拟机 HA 到次级可用域 B 内的主机上，使得业务的连续性得以保障。
  - **优先/必须放置在相同/不同主机上**

    无特殊要求。

    勾选此策略后，若选择的为**优先**策略，则放置组成员的虚拟机在触发 HA 后，将优先在相同或不同的主机上重启，若可用主机不满足**优先**策略，系统仍可将虚拟机调度至可用主机上重启；若选择的为**必须**策略，则放置组成员的虚拟机在触发 HA 后，将限定只能在相同或不同的主机上重启，若可用主机不满足**必须**策略，而虚拟机无法在可用主机上重启，则业务将出现中断。

设置虚拟机放置组的具体操作步骤，请参考《管理指南》的[创建虚拟机放置组](/smtxos/6.3.0/os_administration_guide/os_administration_guide_038)小节。

## SMTX OS 节点的本地账户与默认安全设置说明 > SMTX OS 节点的本地账户说明

# SMTX OS 节点的本地账户说明

SMTX OS 节点支持以下三种本地账户：

| 本地账户 | 说明 | 注意事项 |
| --- | --- | --- |
| root | 默认账户，默认禁止 SSH 远程登录。您可以使用该账户管理 SMTX OS 节点。 | - 请勿删除该账户。 - 强烈建议不要开启 root 账户通过 SSH 远程登录的权限。 - 以命令行方式升级集群时必须拥有 root 权限。 |
| smartx | 默认账户，默认支持 SSH 远程登录。您可以使用该账户通过 SSH 协议远程连接到 SMTX OS 节点。 | - 请勿删除该账户。 - 请勿更改 `/etc/sudoers` 里针对该账户的配置。 - 以该账户登录节点后，必须切换为 root 账户再执行运维操作。执行 `sudo su` 命令可免密码切换。 |
| 其他本地账户 | 由 root 账户创建的其他账户，账户权限由 root 账户决定。您可自行创建新的本地账户用于 SSH 远程登录。 | - |

# SMTX OS 节点的默认安全设置说明

SMTX OS 系统安装完成后，会默认对 SMTX OS 节点进行如下安全设置。

| 设置项 | 默认设置说明 |
| --- | --- |
