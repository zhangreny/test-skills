# Figma Doc Uicase Generate

## Overview

按以下顺序执行：

**Step 1：盘点输入 -> Step 2：先分析需求文档并确认 -> Step 3：用受控预算读取 Figma 并确认 UI 场景 -> Step 4：合并文档与设计稿形成测试设计 -> Step 5：生成并 review 测试用例**

目标不是简单复述页面，而是先把 `场景`、`数值`、`展示`、`报错`、`状态` 拆清楚，再输出有层次的 UI 测试用例。

---

## Step 1：盘点输入

围绕用户给出的设计稿链接、节点、需求文档路径做一次轻量盘点：

- 标出会参与分析的 `.docx`、`.pdf`、`.md` 文件。
- 标出 Figma URL、`fileKey`、`node-id`。
- 记录用户额外约束：
  - 是否限制 Figma MCP 读取次数。
  - 是否要求先按 UI 场景拆解。
  - 是否要求 TestRail 风格输出。
- 明确哪些是“原始需求文档”，哪些只是补充说明。

这一阶段只盘点，不急着深入读 Figma。

---

## Step 2：先分析需求文档并确认

### 2.1 读取文档

- `.md` 直接读取。
- `.docx` / `.pdf` 使用当前环境可行的本地读取方式。
- 如果是 SmartX 需求文档，先读取并遵循 `../smartx-doc-2-product-and-keywords/SKILL.md`，优先提炼产品、功能摘要和关键词。

### 2.2 从文档中抽取测试设计前置信息

在进入 Figma 前，先完成这一轮结构化分析：

1. 功能目标
2. 涉及页面或入口
3. 关键对象
4. 关键数值口径
5. 关键展示字段
6. 关键校验或限制
7. 已知边界和异常
8. 当前不确定点

特别注意从文档中优先提炼这些信息：

- 哪些 UI 入口会受影响。
- 哪些数字需要显示、计算、比较、扣减、归还或聚合。
- 哪些文案属于提示、说明、错误、状态标签、tooltip。
- 哪些状态会切换，如正常、额度不足、已过期、禁用、隐藏、半透明。

### 2.3 需要澄清时的规则

如果用户提供了多个文档，且它们看起来不是同一 feature，暂停推进并明确指出冲突来源。

如果从文档里无法唯一确定“测试主体页面/弹窗/卡片”，先列出 2-4 个候选入口让用户确认，再继续读 Figma。

### 2.4 向用户展示并确认文档分析结果

在继续之前，显式展示这一轮确认包：

1. 功能摘要
2. 主要入口
3. 关键对象
4. 关键数值口径
5. 关键展示/报错类型
6. 当前不确定点

这一轮必须等待用户确认或修正。后续 Figma 精读和 testcase 拆点都要复用这组结论。

---

## Step 3：用受控预算读取 Figma 并确认 UI 场景

### 3.1 默认读取策略

优先少读、精读，不要一上来把整页设计稿全量展开。

默认顺序：

1. 先用一次 `get_metadata` 看页面结构、section 和关键 frame。
2. 只对高价值节点调用 `get_design_context`：
   - 业务卡片
   - 关键弹窗
   - 关键表单区域
   - 错误提示模板
   - 文案变化明显的节点
3. 只有在确实需要视觉确认时，再用截图类能力补充。

### 3.2 Figma MCP 调用预算

如果用户明确限制读取次数，必须把预算当作硬约束。

推荐预算分配：

1. `1 次`：根节点或目标 section 的 `get_metadata`
2. `3-5 次`：关键场景区块的 `get_design_context`
3. `1-3 次`：错误模板、tooltip、卡片等细节点的 `get_design_context`
4. 预留 `1 次`：补读最关键但前面遗漏的节点

如果预算不足以覆盖所有场景：

- 优先读“会改变测试结论”的节点，不优先读装饰层。
- 优先读错误提示、状态态、数字展示区，不优先读通用布局壳子。
- 明确告诉用户哪些节点是根据元数据推断，而不是精读确认。

### 3.3 Figma 分析目标

围绕文档结论，对设计稿完成以下分析：

1. UI 场景清单
2. 每个场景的入口
3. 关键控件或信息块
4. 数值展示位置
5. 数值比较规则
6. 正常态展示
7. 异常态展示
8. 状态态展示
9. 报错位置与提示方式
10. 与通用组件相比，哪些内容变了，哪些必须保持一致

优先显式记录这些 UI 维度：

- 卡片/弹窗/表单/表格/tooltip/footnote/footer error
- 提交前校验 vs 提交后反馈
- 单对象报错 vs 多对象联合报错
- 正常展示 vs 置灰/透明/隐藏/禁用/过期
- 数值标签、单位、上限、已用、可用、需要值

### 3.4 向用户展示并确认 UI 分析结果

在继续之前，显式展示这一轮 Figma 分析包：

1. 设计稿中确认的 UI 场景
2. 每个场景的关键对象
3. 每个场景的关键数值
4. 每个场景的关键展示
5. 每个场景的关键报错/状态
6. 哪些结论来自精读节点
7. 哪些结论只是基于元数据或文档推断
8. 剩余未澄清点

这一轮必须等待用户确认或修正。

---

## Step 4：合并文档与设计稿形成测试设计

### 4.1 构建统一测试对象表

在正式生成 testcase 前，先把“需要被验证的独立对象”列出来。

推荐格式：

| 测试对象 | 来源 | 关键变量 | 关键结果 |
|----------|------|----------|----------|
| 对象名 | 文档 / Figma | 数值 / 状态 / 入口 | 展示 / 报错 / 生效 |

至少覆盖：

- 文档里定义的业务对象
- Figma 中实际出现的卡片、弹窗、区域、报错组件
- 会影响用例拆分的状态对象

### 4.2 构建 UI 测试矩阵

把测试设计按主题分组，而不是按文件分组。

建议至少形成这些矩阵：

1. `entry_matrix`
   - 不同入口进入同一能力时，展示和校验是否一致
2. `state_matrix`
   - 正常、不足、过期、禁用、隐藏、半透明等状态
3. `number_matrix`
   - X>Y、X=Y、X<Y、0、已用满、多对象聚合
4. `copy_matrix`
   - 说明文案、tooltip、错误文案、状态标签
5. `display_matrix`
   - 卡片、tooltip、footer、inline error、table row、footnote

### 4.3 必须区分“场景”“数值”“展示”

生成 testcase 前，先完成这三层拆解：

1. 场景层
   - 用户从哪里进入
   - 做了什么动作
   - 命中了哪一类规则
2. 数值层
   - 输入值、需求值、可用值、最大值、已用值
   - 单对象值还是聚合值
   - 比较逻辑是否有边界值
3. 展示层
   - 数字显示在哪
   - 文案显示在哪
   - 错误以什么形式出现
   - 状态视觉是否变化

如果三层没有拆开，不要直接开始写 testcase。

### 4.4 如需补齐粒度，读取 testcase pattern

如果用户要求 TestRail 风格、分层清晰、避免大而粗的 case，先读取并遵循：

- `../testcase-pattern-learning/SKILL.md`
- 按需读取 `../testcase-pattern-learning/tutorial-all-groups/all-groups-common.md`
- 如果 feature 属于网络、安全、LB、VPC、DFW 等，再按需读取 `../testcase-pattern-learning/tutorial-all-groups/network-group.md`
- 如果需要输出 TestRail 层级，再按需参考 `../testcase-generate/references/testrail_default.md`

pattern 只用于补粒度、命名和矩阵，不用于照抄旧业务内容。

### 4.5 向用户展示并确认测试设计

在进入 testcase 生成前，必须同时展示：

1. 测试对象表
2. 建议测试主题分组
3. UI 场景矩阵
4. 数值矩阵
5. 展示矩阵
6. 必须显式展开的边界点
7. 可以抽样覆盖的维度
8. 当前仍存在的疑点

必须等待用户确认后再生成最终用例。

---

## Step 5：生成并 review 测试用例

### 5.1 输出目标

默认输出“以 UI 为核心的测试用例”，而不是泛泛的功能点列表。

每条 case 应尽量只验证一个核心结果，并且显式体现：

- 入口或前提
- 数值条件
- 展示或报错结果

### 5.2 推荐分组

优先按主题组织，不按文件组织。常见分组：

1. 基础配置与展示
2. 数值口径与统计展示
3. 正常提交流程
4. 边界值与额度比较
5. 错误提示与文案展示
6. 状态变化与禁用/过期展示
7. 多入口一致性
8. 回归与通用组件一致性

### 5.3 输出格式要求

如无额外要求，复用 TestRail 风格层级：

- `## 模块`
- `### 子模块`
- `#### 用例`
- `描述：`
- `Step1`
- `Step2`
- `Step3`

不要输出模板说明文字，不要把分析过程混进最终用例正文。

### 5.4 reviewer 复核终稿

输出前必须把最终 testcase 当作 review 对象，重新对照以下输入做一轮复核：

- 原始需求文档
- 已确认的文档分析结果
- 已确认的 Figma UI 分析结果
- 已确认的测试对象表和矩阵

复核重点：

1. 是否漏掉文档里明确存在的入口、对象、条件、状态
2. 是否漏掉设计稿里明确变化的数值、文案、展示位置、错误方式
3. 是否把“数值规则”写成了泛泛展示检查
4. 是否把多个核心结果混在同一条 case
5. 是否漏掉 `X>Y`、`X=Y`、`X<Y` 这类关键边界
6. 是否漏掉单对象与多对象联合报错
7. 是否漏掉正常态与异常态、过期态、禁用态的一致性回归
8. 是否把通用组件未变化部分误写成大量无效 testcase

只有在“关键需求本身存在歧义，且不澄清会明显影响 testcase 正确性”时，才回头问用户。

### 5.5 最终对用户的输出要求

- 以 `【测试用例处理完成】` 开头
- 直接展示 review 后的最终测试用例正文
- 除非用户额外要求，否则不附加长篇过程解释

---

## 质量检查

完成前确认：

- 已先读文档，再读 Figma，而不是只看设计稿截图。
- 已控制 Figma MCP 读取次数，并说明预算使用情况。
- 已在 testcase 生成前显式输出过“文档分析确认”“UI 分析确认”“测试设计确认”。
- 已把场景、数值、展示三层拆开。
- 最终 case 重点覆盖页面场景、数值规则、展示结果、报错方式、状态变化和一致性回归。
- 最终 case 粒度清晰，单条 case 不臃肿。
